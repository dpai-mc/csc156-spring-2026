🎯Learning Goals for Week 9

By the end of this week you should be able to:

  1. 1 Explain the structure and forensic significance of Linux file systems, including directory hierarchies, inode metadata, and the /proc and /sys virtual file systems.
  2. 2 Identify and extract user and group account artifacts from /etc/passwd, /etc/shadow, and /etc/group to establish account timelines and privilege assignments.
  3. 3 Recover and interpret shell history files, recently accessed files, and user-level configuration artifacts to reconstruct user activity.
  4. 4 Analyze Linux system logs — including syslog, auth.log, and the systemd journal — to correlate events, identify authentication anomalies, and build an incident timeline.
  5. 5 Examine scheduled task artifacts (cron jobs, systemd timers, at jobs) for evidence of persistence mechanisms and unauthorized automation.
  6. 6 Compare Linux and Windows forensic methodologies, articulating key differences in artifact locations, tool sets, and investigative strategies.

🔍Key Concepts and Themes

Keep these ideas at the forefront as you work through the modules and lab:

🗂️
Inode Metadata
Inodes store access, modification, and change timestamps (atime, mtime, ctime) that are central to timeline reconstruction on Linux systems.
👤
User Account Artifacts
The passwd, shadow, and group files — combined with login records — reveal account creation, privilege escalation, and unauthorized access patterns.
💻
Shell History as Evidence
Shell history files provide a near-chronological record of commands executed by a user, including potential attacker commands if history was not cleared.
📄
Log Correlation
Combining syslog, auth.log, and the systemd journal entries across time allows investigators to reconstruct event sequences and identify anomalies.
⏰
Persistence via Scheduling
Cron jobs and systemd timers are common attacker persistence mechanisms. Knowing where to look — and what normal looks like — is critical for detection.
⚖️
Live vs. Post-Mortem
Choosing between live triage (volatile data capture on a running system) and post-mortem imaging (offline disk analysis) affects what evidence is available and how it is collected.
🪟
Linux vs. Windows Differences
Artifact locations, log formats, file system structures, and tool sets differ substantially between platforms. A skilled investigator adapts methodology to the operating system.
🔗
Chain of Custody
All Linux acquisition actions — just like Windows — must be documented with hash verification to maintain evidence integrity and legal admissibility.
🚀 Looking Ahead — Week 11: Memory Forensics

With the host-based arc complete, Week 11 shifts the investigation from persistent disk artifacts to volatile memory. You will be introduced to the principles of RAM acquisition and analysis, and you will begin working with the Volatility framework — the industry-standard tool for memory forensics. The Linux knowledge you built this week pays off immediately: Volatility profiles are OS-specific, and understanding Linux process structures, user-space memory layout, and kernel objects will help you interpret memory dumps from Linux targets. Come to Week 10 with a solid grasp of Linux processes and the /proc file system — it is the conceptual bridge between disk forensics and memory forensics.
