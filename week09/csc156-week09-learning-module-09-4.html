Linux logging has evolved through three generations of technology that often coexist on the same system. Understanding the architecture tells you where logs live, what format they use, and critically — what gaps an attacker could exploit or create.

syslog vs. rsyslog vs. journald — What's the Difference?

syslog (legacy) Plain text — /var/log/syslog Older SysV-init systems Human-readable; easily grepped; easily deleted or truncated by attacker

💡 Why Both Exist on Ubuntu 22.04

On web-prod-03 (Ubuntu 22.04), both journald and rsyslog are active. journald collects everything first; rsyslog subscribes to journald's socket and writes selected messages to /var/log/syslog, /var/log/auth.log, etc. This means you have two independent copies of many events — a powerful cross-validation tool. If the plain-text logs are tampered with, the binary journal may still be intact.

journalctl Query Syntax — Forensic Essentials

The journalctl command is your primary interface to the binary journal. On a live system, run as root; on a forensic image, point it at the journal directory with -D.

# Read journal from a mounted forensic image (dead-box analysis)
journalctl -D /mnt/evidence/var/log/journal

# Filter by time window — critical for incident scoping
journalctl --since "2026-03-01 20:00:00" --until "2026-03-01 22:00:00"

# Show only SSH authentication events (unit filter)
journalctl -u ssh.service

# Show kernel messages only (equivalent to dmesg)
journalctl -k

# Show messages from a specific process ID
journalctl _PID=1337

# Show messages from a specific executable path
journalctl _EXE=/usr/bin/sudo

# Output in JSON for scripted analysis
journalctl -o json-pretty --since "2026-03-01"

# Show boot sessions — each -b N is a past boot
journalctl --list-boots
journalctl -b -1 # previous boot session

Log Rotation and Its Forensic Implications

Linux systems automatically rotate log files on a schedule managed by logrotate (configured in /etc/logrotate.conf and /etc/logrotate.d/). Understanding rotation is essential for reconstructing complete timelines.

File Name Age / Status Forensic Note
/var/log/auth.log Current (active writes) Most recent events; may be incomplete if rotation just occurred
/var/log/auth.log.1 Previous rotation (plain text) Immediately prior period; check mtime for rotation timestamp
/var/log/auth.log.2.gz Two rotations ago (gzip compressed) Use zcat or zgrep to read without decompressing
/var/log/auth.log.N.gz N rotations ago Retention period depends on logrotate config (often 4–8 weeks)

⚠️ Forensic Implication of Log Rotation

Always examine all rotated log files, not just the current one. An attack that began weeks ago may only be visible in compressed rotated logs. Conversely, if rotated logs are missing for expected time periods, or if the compressed archives have been deleted, that absence is itself evidence of anti-forensic activity. Check logrotate configuration to understand the expected retention schedule, then verify whether that many rotation files actually exist.

# Search across ALL rotated auth logs (current + compressed)
zgrep "Accepted publickey" /var/log/auth.log*

# List all rotated logs with sizes and timestamps
ls -lh /var/log/auth.log*

# Decompress and concatenate for timeline tools
zcat /var/log/auth.log.2.gz | grep "Mar 1"

3 · Web Server Log Analysis

Web server access logs are often the first place evidence of an attack appears on a compromised server. Every HTTP request — including exploit attempts — is recorded. Learning to read these logs fluently is a core forensic skill.

Apache Combined Log Format — Annotated

Apache's default "combined" log format records one line per request. The example below is a real-looking malicious request — each field is color-coded and explained.

Apache Access Log — Annotated Line

203.0.113.47 - - [01/Mar/2026:21:14:33 +0000] "GET /admin/../../../etc/passwd HTTP/1.1" 400 512 "-" "sqlmap/1.7.8#stable (https://sqlmap.org)"
Client IP (203.0.113.47): The source IP of the request. Cross-reference with auth.log SSH logins. Multiple attack types from one IP = coordinated campaign.
Ident / Auth fields (- -): Almost always dashes in practice. Ignore.
Timestamp [01/Mar/2026:21:14:33 +0000]: Request time in server local time (UTC here). Note the timezone offset — normalize all timestamps to UTC for timeline analysis.
Request "GET /admin/../../../etc/passwd HTTP/1.1": Method + URI + protocol. This URI contains directory traversal sequences (../) attempting to escape the web root and read /etc/passwd. A clear attack indicator.
Status Code (400): HTTP response code. 400 = Bad Request (server rejected the traversal). A 200 response to a traversal attempt would indicate successful exploitation. Watch for 200s on suspicious URIs.
Bytes Transferred (512): Response body size in bytes. A large byte count on a suspicious request may indicate data was successfully returned to the attacker.
Referer ("-"): The page that linked to this request. A dash means no referrer (direct request or automated tool). Legitimate users usually have referrers; scanners often do not.
User-Agent ("sqlmap/1.7.8..."): The client software string. sqlmap is an automated SQL injection tool — its presence in a User-Agent is an unambiguous attack indicator. Legitimate browsers never identify as sqlmap.

Web Attack Signatures in Access Logs

The following attack patterns leave characteristic signatures in web server logs. Recognizing them is a core skill tested on the CHFI and GCFE certifications.

Goal: Escape the web root to read sensitive files (/etc/passwd, /etc/shadow, application config files with credentials).

Log signatures to look for:

  • ../ or %2e%2e%2f (URL-encoded) in the URI
  • ..%2F, ..%5C, %252e%252e%252f (double-encoded)
  • References to /etc/passwd, /etc/shadow, /proc/self/environ in URIs
grep -E '(\.\./|%2e%2e|%252e)' /var/log/apache2/access.log

⚠️ Status Code Matters

A 400 or 403 response means the server blocked the attempt. A 200 response to a traversal URI is critical evidence of successful exploitation — the server returned the requested file contents.

Goal: Inject SQL code into database queries via URL parameters or POST data to extract data, bypass authentication, or execute commands.

Log signatures to look for:

  • SQL keywords in URI parameters: UNION, SELECT, FROM, WHERE, OR 1=1, DROP TABLE
  • URL-encoded SQL: %27 (single quote), %20OR%201%3D1
  • Automated tool User-Agents: sqlmap, Havij, jSQL
  • High volume of requests to the same endpoint with varying parameters
grep -iE '(union.*select|or.*1=1|%27|%20or%20|sqlmap)' /var/log/apache2/access.log

Goal: Map the application's attack surface — discover directories, files, technologies, and vulnerabilities — before targeted exploitation.

Recognizable scanner User-Agents:

  • Nikto — web vulnerability scanner
  • sqlmap — SQL injection automation
  • Nmap Scripting Engine — port/service scanner
  • dirbuster, gobuster, feroxbuster — directory brute-forcers
  • python-requests, curl/ — often used by custom exploit scripts
  • Masscan, ZGrab — mass internet scanners

Behavioral signatures:

  • Hundreds of 404 responses in rapid succession (directory brute-force)
  • Requests for common CMS paths (/wp-admin, /phpmyadmin, /.git/) on a non-CMS server
  • Sequential IP addresses or identical timing patterns (automated)
grep -iE '(nikto|sqlmap|dirbuster|gobuster|masscan|python-requests)' /var/log/apache2/access.log

Goal: After uploading a web shell (a malicious script that executes commands), the attacker accesses it repeatedly to run commands on the server.

Log signatures:

  • POST requests to unusual file names in upload directories (e.g., POST /uploads/image.php)
  • Requests to .php files in directories that should only contain static content (/images/, /uploads/)
  • Large POST bodies to unexpected endpoints (command input)
  • 200 responses to requests for recently created PHP files
  • User-Agent strings like Mozilla/5.0 but with unusual parameter patterns (manual attacker interaction)

🔗 Cross-Artifact Correlation

If you identify a suspicious PHP file being accessed in the web logs, cross-reference with the filesystem: check the file's mtime (when it was written), compare to the first access log entry for that file, and look for the upload event in earlier log entries. This reconstructs the complete web shell deployment timeline.

4 · Log Tampering Detection

🔑 Core Forensic Principle

The absence of logs is evidence. When an attacker deletes or truncates log files, they leave a different kind of evidence: the gap itself. A sophisticated forensic examiner treats missing, incomplete, or inconsistent logs as a positive finding — not a dead end. Document every gap with the same rigor as a positive artifact.

Indicator 1 — Timestamp Gaps

A healthy, continuously running server produces a nearly unbroken stream of log entries. Gaps where no entries appear for extended periods — especially during the suspected intrusion window — are a primary tampering indicator.

# Extract timestamps from auth.log to find gaps
awk '{print $1, $2, $3}' /var/log/auth.log | head -5
# Output example:
# Mar 1 20:55:01
# Mar 1 20:55:03
# Mar 1 21:32:17 ← last entry before gap
# Mar 2 08:11:44 ← next entry — ~10.5 hour gap on an active server!

# Cross-reference: did the system reboot? Check wtmp
last -F reboot

Indicator 2 — File mtime Inconsistency

Log files are written sequentially — their modification time (mtime) should reflect the time of the last written entry. If the mtime of a log file is earlier than the last entry it contains, or if the mtime is suspiciously round (e.g., exactly midnight), the file has likely been touched or replaced.

# Check modification times of all log files
ls -la --time-style=full-iso /var/log/

# Compare mtime of auth.log to the timestamp of its last entry
stat /var/log/auth.log
tail -1 /var/log/auth.log

# If mtime is BEFORE the last log entry timestamp → file was backdated
# If mtime is much LATER than last entry → file was modified after logging stopped

Indicator 3 — Truncated Files

The simplest anti-forensic technique is truncating a log file to zero bytes (cat /dev/null > /var/log/auth.log or truncate -s 0 /var/log/auth.log). This preserves the file's existence (so a basic check passes) but destroys all content.

# Find zero-byte or suspiciously small log files
find /var/log/ -type f -name "*.log" -size 0

# Find log files smaller than 1KB (suspicious for an active server)
find /var/log/ -type f -name "*.log" -size -1k

# Check inode change time (ctime) — ctime updates even if mtime is faked
stat /var/log/auth.log # look at "Change:" field

Indicator 4 — Cross-Source Correlation

The most powerful tampering detection technique is cross-source correlation — comparing the same events across multiple independent log sources. An attacker who deletes /var/log/auth.log may forget that the same SSH login event is also recorded in the binary journal.

📌 Cross-Source Pairs to Correlate

  • auth.log ↔ journald (SSH events)
  • auth.log ↔ wtmp (login sessions)
  • Web access log ↔ web error log (same request, two views)
  • Cron execution in syslog ↔ crontab file contents
  • Package install in dpkg.log ↔ binary presence on disk

⚠️ Remote Syslog as Anti-Tamper

Organizations that forward logs to a remote syslog server or SIEM make it much harder for attackers to destroy evidence. On web-prod-03, check /etc/rsyslog.conf for remote forwarding rules — if logs were forwarded, the SOC may have intact copies even if local logs were wiped.

Cron is the standard Linux job scheduler. It executes commands at defined time intervals — making it a natural persistence mechanism. An attacker-planted cron job can re-download malware, beacon to a C2 server, or re-add a deleted SSH key on a schedule.

Cron File Locations — Examine All of These

LocationScopeHow to Read
/etc/crontab System-wide; includes username field cat /etc/crontab
/etc/cron.d/ System-wide; package-installed cron files ls -la /etc/cron.d/ then cat each file
/etc/cron.hourly/, daily/, weekly/, monthly/ Scripts run at those intervals by /etc/crontab ls -la /etc/cron.*/
/var/spool/cron/crontabs/username Per-user crontabs (created with crontab -e) cat /var/spool/cron/crontabs/root — always check root's crontab
/var/spool/cron/username Per-user (RHEL path variant) Same approach as above

Cron Entry Syntax

# ┌─ minute (0-59)
# │ ┌─ hour (0-23)
# │ │ ┌─ day of month (1-31)
# │ │ │ ┌─ month (1-12)
# │ │ │ │ ┌─ day of week (0-7, 0=Sun)
# │ │ │ │ │
* * * * * command_to_execute

# Example: run every minute (attacker beacon frequency)
* * * * * /tmp/.update >/dev/null 2>&1

Red Flag Cron Patterns

  • Commands referencing /tmp/, /dev/shm/, or other world-writable directories
  • Commands that pipe to bash or sh (e.g., curl http://... | bash)
  • Output redirected to /dev/null on both stdout and stderr (>/dev/null 2>&1) — hiding execution evidence
  • Executables with names that mimic system tools (e.g., .update, systemd-check, cron.d)
  • Very high frequency (* * * * * = every minute) — typical of C2 beacons
  • Base64-encoded commands: echo "base64string" | base64 -d | bash
  • Cron files owned by a non-root user in /etc/cron.d/
  • Cron entries with wget or curl downloading from external IPs or unusual domains

systemd unit files define services, timers, and other system units. An attacker can install a malicious .service file that runs their payload on every boot — and it will look identical to a legitimate service to a casual observer.

Key Locations to Examine

  • /etc/systemd/system/ — Administrator-installed units (highest priority); attacker-installed units most commonly appear here
  • /lib/systemd/system/ — Package-installed units (lower priority)
  • /run/systemd/system/ — Runtime units (volatile; lost on reboot)
  • ~/.config/systemd/user/ — Per-user units (run without root)

Anatomy of a Malicious Unit File

# /etc/systemd/system/systemd-network-helper.service
# (name chosen to blend in with legitimate systemd services)

[Unit]
Description=Network Helper Service
After=network.target

[Service]
ExecStart=/usr/local/bin/.netd ← hidden binary (dot-prefixed)
Restart=always ← restarts if killed
RestartSec=30
User=root ← runs as root

[Install]
WantedBy=multi-user.target ← starts on every boot

Detection Commands

# List all enabled services — look for unfamiliar names
systemctl list-unit-files --type=service --state=enabled

# Find recently modified unit files (within last 30 days)
find /etc/systemd/system/ -mtime -30 -name "*.service"

# Examine a specific unit file
cat /etc/systemd/system/suspicious.service

# Check service execution history in journal
journalctl -u suspicious.service --since "2026-03-01"

SUID (Set User ID) is a file permission bit that causes a program to execute with the privileges of its owner, not the user who runs it. A SUID binary owned by root runs as root regardless of who executes it. This is a legitimate mechanism (e.g., /usr/bin/passwd needs root to write /etc/shadow), but attackers abuse it by creating or modifying SUID binaries to provide persistent root access.

Legitimate SUID Examples

  • /usr/bin/passwd — changes passwords
  • /usr/bin/sudo — privilege escalation
  • /usr/bin/ping — raw socket access
  • /usr/bin/su — switch user

Attacker SUID Abuse

  • Copy /bin/bash to /tmp/bash and set SUID
  • Set SUID on find, vim, python3 (GTFOBins)
  • Create custom SUID binary that spawns root shell
  • Modify existing SUID binary to add backdoor

Finding SUID/SGID Binaries

# Find ALL SUID binaries on the system
find / -perm -4000 -type f -ls 2>/dev/null

# Find ALL SGID binaries
find / -perm -2000 -type f -ls 2>/dev/null

# Find SUID/SGID in unusual locations (not /usr/bin, /bin, /sbin)
find /tmp /var /opt /home /root -perm /6000 -type f -ls 2>/dev/null

# Compare against a known-good baseline (if available)
# Any SUID binary NOT in the baseline is a red flag

⚠️ GTFOBins Reference

The GTFOBins project (referenced by name — do not follow external links) documents how legitimate Unix binaries can be exploited when given SUID permissions. If you find an unexpected SUID binary like find, vim, python3, perl, or nmap, treat it as a critical finding — these can all spawn root shells.

Adding an attacker-controlled public key to a user's authorized_keys file is one of the most common and reliable persistence techniques. It requires no password, survives reboots and password changes, and is invisible to login monitoring that only watches for password authentication.

Locations to Examine

  • /root/.ssh/authorized_keys — Examine first and always
  • /home/username/.ssh/authorized_keys — For every user account
  • Non-standard paths defined in /etc/ssh/sshd_config via AuthorizedKeysFile directive

Red Flags in authorized_keys

  • Keys added after the last known-good deployment date (check file mtime and compare key comments)
  • Keys with no comment field, or generic comments like user@kali, root@attacker
  • More keys present than expected for the account
  • Key options prepended to the key (e.g., command="/bin/bash", no-pty) — used to restrict or control what the key can do
  • authorized_keys file owned by a different user than the home directory owner
# Check authorized_keys for all users
for home in /root /home/*; do
  echo "=== $home/.ssh/authorized_keys ==="
  cat "$home/.ssh/authorized_keys" 2>/dev/null
done

# Check sshd_config for non-standard AuthorizedKeysFile path
grep -i "AuthorizedKeysFile" /etc/ssh/sshd_config

The Linux dynamic linker allows a shared library to be loaded before all others via the LD_PRELOAD environment variable or the /etc/ld.so.preload file. An attacker can use this to inject malicious code into every process on the system — overriding standard library functions like read(), write(), or open() to hide files, processes, or network connections.

How Attackers Use It

  • Place a malicious shared library (.so file) on disk, often in /tmp/ or a hidden directory
  • Add its path to /etc/ld.so.preload — this causes it to load into every process system-wide
  • The malicious library overrides functions to hide the attacker's files, processes, and network connections from standard tools like ls, ps, and netstat

Detection

# Check /etc/ld.so.preload — should be empty or non-existent on most systems
cat /etc/ld.so.preload

# Find all .so files in suspicious locations
find /tmp /var /dev/shm /home /root -name "*.so" -o -name "*.so.*" 2>/dev/null

# Check environment of running processes for LD_PRELOAD
cat /proc/*/environ 2>/dev/null | tr '\0' '\n' | grep LD_PRELOAD

🚨 Critical Implication

If /etc/ld.so.preload contains a malicious library, you cannot trust the output of any standard tool on the live system. The library may be hiding files, processes, and connections from you. This is a key reason why forensic analysis from a dead-box image (where the malicious library cannot execute) is more reliable than live analysis when rootkits are suspected.

A Loadable Kernel Module (LKM) rootkit is the most sophisticated Linux persistence and concealment technique. It operates at the kernel level — giving it the ability to intercept and modify any system call, hide any file or process, and make itself essentially invisible to user-space tools.

How LKM Rootkits Work (Conceptual)

  • A kernel module is compiled and loaded with insmod or modprobe
  • Once loaded, the rootkit modifies the system call table — the kernel's table of function pointers for every OS operation
  • It replaces legitimate syscall handlers (e.g., getdents for directory listing) with its own versions that filter out attacker-controlled files and processes
  • The result: ls, ps, netstat, and other tools return filtered results — the attacker's artifacts are invisible

Detection Approaches

  • Dead-box analysis is essential — on a live system with an active LKM rootkit, user-space tools cannot be trusted
  • Check /var/log/kern.log or journalctl -k for unexpected module load events (insmod)
  • Compare lsmod output against a known-good baseline (may be filtered by rootkit on live system)
  • Look for unsigned or recently compiled kernel modules in /lib/modules/$(uname -r)/
  • Memory forensics with Volatility (covered in Module 10) can detect LKM rootkits by analyzing kernel memory directly
# Check kernel log for module load events
journalctl -k | grep -i "module\|insmod\|modprobe"

# List currently loaded modules (may be filtered on live compromised system)
lsmod

# Find recently modified kernel module files
find /lib/modules/ -name "*.ko" -mtime -30 -ls

🔗 Connection to Week 10

LKM rootkit detection through memory analysis is covered in depth in Module 10 — Memory (RAM) Forensics using the Volatility Framework. The linux_check_syscall and linux_lsmod Volatility plugins can detect rootkit-modified syscall tables and hidden modules directly from a memory dump, bypassing the rootkit's user-space filtering entirely.

  • Log rotation forensic implications — all rotated files must be examined; missing rotated logs are evidence of tampering.
  • Key log files — auth.log, web server access/error logs, wtmp/btmp, dpkg.log, and audit logs; their owners, formats, and forensic value.
  • Apache Combined Log Format — all seven fields and their forensic significance; HTTP status codes as attack success indicators.
  • Web attack signatures — directory traversal, SQL injection, scanner User-Agents, and web shell access patterns.
  • Log tampering detection — timestamp gaps, mtime inconsistency, truncated files, and cross-source correlation as the primary detection strategy.
  • Linux persistence mechanisms — cron (all file locations, red flag patterns), systemd unit files, SUID/SGID binaries (find commands), SSH authorized_keys, LD_PRELOAD hijacking via /etc/ld.so.preload, and LKM rootkits at the conceptual level.
