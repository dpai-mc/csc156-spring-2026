Investigation Scenario — Week 9 Context

📋 Scenario — Used Across All Five Week 9 Modules

You've just been handed a forensic image of a compromised Ubuntu 22.04 web server. Where do you start?

It is 9:47 PM. Your organization's Security Operations Center has flagged anomalous outbound traffic from web-prod-03, a public-facing Ubuntu 22.04 LTS server hosting a customer portal. The server was taken offline by the network team at 21:32 and a forensic image was captured using dd. The image file — web-prod-03.img — is sitting on your forensic workstation.

You have no access to the original hardware. The system owner reports the server has been running for 217 days without a reboot. The last known-good deployment was 14 weeks ago. You have been asked to determine: Was the server compromised? If so, how, when, and what data may have been accessed?

Every concept, directory, artifact, and technique you study this week connects back to answering that question. Keep this scenario in mind as you work through Modules 9.1–9.5.

Why Linux Matters in Forensics

Linux is not just an alternative desktop operating system — it is the dominant platform for servers, cloud infrastructure, embedded systems, and network appliances. Understanding Linux forensics is no longer a specialty skill; it is a core competency for any digital forensic examiner.

~96%
of the top 1 million web servers run Linux (Netcraft, 2024)
>90%
of public cloud workloads run on Linux-based VMs or containers
100%
of the top 500 supercomputers in the world run Linux
~75%
of IoT and embedded devices run a Linux kernel

Where You Will Encounter Linux in Investigations

  • Web and application servers — Apache, NGINX, Node.js deployments on Ubuntu, RHEL, Debian, CentOS
  • Database servers — MySQL, PostgreSQL, MongoDB typically hosted on Linux
  • Cloud instances — AWS EC2, Azure VMs, GCP Compute Engine default images are Linux
  • Containers and orchestration — Docker images and Kubernetes nodes are Linux-based
  • Network infrastructure — Firewalls, routers, and NAS devices often run embedded Linux
  • Developer workstations — Many software engineers use Ubuntu, Fedora, or Arch Linux
  • Attacker infrastructure — Threat actors frequently stage operations on compromised Linux VPS instances

Common Distributions You Will Encounter

Distribution Common Use Forensic Notes
Ubuntu (LTS) Web servers, developer workstations, cloud images Snap packages add artifact locations under /var/lib/snapd; journald + syslog both present
Debian Stable servers, embedded systems Very similar artifact locations to Ubuntu; minimal pre-installed software reduces noise
RHEL / CentOS / Rocky Enterprise servers, regulated environments SELinux audit logs at /var/log/audit/audit.log are extremely valuable; dnf history in /var/log/dnf.log
Kali Linux Penetration testing, attacker toolkits Presence on a non-pentester system is a significant indicator; tool installation artifacts visible in dpkg logs
Alpine Linux Docker base images, minimal containers Very sparse logs; busybox replaces standard utilities; limited shell history by default
Amazon Linux 2 / AL2023 AWS EC2 default instances Cloud-init logs at /var/log/cloud-init.log record instance provisioning history

Note: Artifact locations are largely standardized across distributions by the Filesystem Hierarchy Standard (FHS), but log formats, init systems (systemd vs. SysV), and package managers differ.

⚠️ Examiner Awareness

Never assume a Linux system is "just like" another Linux system. Always determine the distribution, version, and init system (systemd vs. SysV init) early in your examination — these affect where logs are stored, how services are managed, and what persistence mechanisms are available to an attacker.

Linux vs. Windows — Key Forensic Dimensions

If you are coming from a Windows forensics background (as most examiners do), Linux will challenge some deeply ingrained assumptions. The most important conceptual shift is understanding the "everything is a file" philosophy versus Windows' registry-centric design.

🧠 Core Concept — "Everything Is a File"

In Linux, almost every system resource is represented as a file in the filesystem: hardware devices (/dev/sda), running processes (/proc/1234/), network sockets, and system configuration. This is fundamentally different from Windows, where configuration is stored in a binary database (the Registry) separate from the filesystem.

Forensic implication: On Linux, you can often read configuration and state information using standard text-processing tools (cat, grep, awk) directly from files. On Windows, you need specialized Registry parsing tools. Linux artifacts are often more human-readable — but they are also easier for an attacker to modify or delete.

Side-by-Side Comparison

Dimension Linux 🐧 Windows 🪟
Configuration Storage Plain-text files in /etc/ — human-readable, version-controllable Binary Registry hive files (NTUSER.DAT, SOFTWARE, SYSTEM, etc.) — requires specialized parsing
Log Format Mix of plain-text (/var/log/syslog) and binary journal (journald) — journalctl needed for binary logs Binary EVTX format in Windows Event Logs — requires Event Viewer or parser (e.g., evtx_dump)
User Account Data /etc/passwd (user info), /etc/shadow (hashed passwords), /etc/group — all plain text SAM Registry hive (local accounts) or Active Directory (domain accounts) — binary, encrypted
Shell History ~/.bash_history, ~/.zsh_history — plain text, per-user, easily deleted or manipulated PowerShell history at %APPDATA%\...\PSReadLine\ConsoleHost_history.txt — similar plain-text risks
Persistence Mechanisms Cron jobs (/etc/cron*, crontab -l), systemd unit files (/etc/systemd/system/), init scripts, ~/.bashrc, ~/.profile, SUID binaries Run keys (Registry), Scheduled Tasks, Services, Startup folder, DLL hijacking
Installed Software Package manager databases: /var/lib/dpkg/ (Debian/Ubuntu), /var/lib/rpm/ (RHEL); install logs in /var/log/dpkg.log Registry keys under HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall; Windows Installer logs
Network Configuration /etc/network/interfaces, /etc/netplan/, /etc/hosts, /etc/resolv.conf — plain text Registry keys + ipconfig output; C:\Windows\System32\drivers\etc\hosts
Process / Memory Artifacts /proc/ virtual filesystem — live process data, memory maps, open file descriptors (live system only) Process memory via WinPMem or similar; Prefetch files, Superfetch/ReFS artifacts
Timestamps (MAC) ext4: mtime, atime, ctime (change time, not creation); crtime available on some filesystems via debugfs NTFS: Created, Modified, Accessed, MFT-Modified — $MFT and $STANDARD_INFORMATION can differ (timestomping)
Forensic Tooling Autopsy, The Sleuth Kit, Volatility (memory), grep/awk/find for artifact hunting Autopsy, FTK, Volatility, RegRipper, Eric Zimmerman tools

📌 Key Takeaway for Examiners

The absence of a centralized Registry means Linux forensics is more distributed — evidence is scattered across many individual files. This requires a systematic, directory-aware approach rather than a single-hive extraction workflow. The upside: many Linux artifacts are plain text, making them accessible with basic command-line tools.

📁

Linux Directory Hierarchy for Forensic Examiners

Linux organizes all files under a single root directory (/), defined by the Filesystem Hierarchy Standard (FHS). As a forensic examiner, you do not need to memorize every directory — you need to know which directories hold the most investigative value and why. Expand each entry below to learn its forensic significance.

🔑 Priority Legend

HIGH Examine in almost every investigation   MEDIUM Examine based on case type   REFERENCE Useful for context

What it contains: System-wide configuration files for almost every service and application. All files are plain text.

Key forensic artifacts:

  • /etc/passwd — User accounts (username, UID, GID, home directory, shell). Readable by all users. Check for unauthorized accounts or UID 0 (root) entries.
  • /etc/shadow — Hashed passwords and password aging data. Root-readable only. Cracked hashes may reveal weak credentials used in an attack.
  • /etc/group — Group memberships. Look for unauthorized additions to sudo or wheel groups.
  • /etc/sudoers and /etc/sudoers.d/ — Privilege escalation configuration. Attackers frequently modify this to grant persistent root access.
  • /etc/ssh/sshd_config — SSH server configuration. Check for PermitRootLogin yes, PasswordAuthentication yes, or custom AuthorizedKeysFile paths.
  • /etc/cron* and /etc/crontab — System-wide scheduled tasks. Common persistence location.
  • /etc/hosts — Static hostname-to-IP mappings. Attackers may redirect domains here to intercept traffic.
  • /etc/rc.local, /etc/init.d/ — Legacy startup scripts (SysV init systems).

🔍 Scenario Connection

On web-prod-03, you would examine /etc/passwd and /etc/sudoers early to determine if any new accounts were created or if privilege escalation paths were modified after the last known-good deployment.

What it contains: Log files from the kernel, init system, services, and applications. This is the primary timeline reconstruction source on Linux.

Key forensic artifacts:

  • /var/log/syslog or /var/log/messages — General system messages. Contains kernel messages, daemon activity, and service events.
  • /var/log/auth.log (Debian/Ubuntu) or /var/log/secure (RHEL) — Authentication events: SSH logins, sudo usage, su attempts, PAM events. Critical for intrusion analysis.
  • /var/log/apache2/ or /var/log/nginx/ — Web server access and error logs. Web-based attack vectors (SQLi, RCE, LFI) leave traces here.
  • /var/log/dpkg.log / /var/log/dnf.log — Package installation history. Look for attacker-installed tools.
  • /var/log/wtmp — Binary log of all logins/logouts. Parse with last command.
  • /var/log/btmp — Binary log of failed login attempts. Parse with lastb.
  • /var/log/lastlog — Last login time per user. Binary format; parse with lastlog.
  • /var/log/audit/audit.log — (RHEL/CentOS with auditd) Detailed audit trail of system calls, file access, and privilege use.

⚠️ Log Rotation and Deletion

Linux systems rotate logs on a schedule (typically managed by logrotate). Rotated logs are compressed as syslog.1, syslog.2.gz, etc. Attackers may delete or truncate log files — the absence of expected log entries is itself evidence. Always check /var/log/ for unexpectedly empty files or unusual timestamps.

What it contains: Personal files, configuration, and history for each non-root user. Each user has a subdirectory: /home/username/.

Key forensic artifacts:

  • ~/.bash_history — Bash command history. Attackers may clear this (history -c, unset HISTFILE), but timestamps may survive in the filesystem metadata. Check HISTTIMEFORMAT setting.
  • ~/.ssh/ — SSH keys and configuration:
    • authorized_keys — Public keys that can authenticate as this user. Attacker-added keys are a primary persistence mechanism.
    • known_hosts — Systems this user has connected to (lateral movement evidence).
    • id_rsa, id_ed25519 — Private keys (potential exfiltration risk).
  • ~/.bashrc, ~/.bash_profile, ~/.profile — Shell initialization scripts. Attackers inject malicious commands here for persistence.
  • ~/.local/share/recently-used.xbel — Recently accessed files (desktop environments).
  • ~/.wget-hsts, ~/.curl_history — Download history artifacts.
  • Browser profiles (Firefox: ~/.mozilla/, Chrome: ~/.config/google-chrome/) — Browsing history, saved credentials.

What it contains: The home directory for the root (superuser) account. Separate from /home — root lives at /root/, not /home/root/.

Why it is critical: Any attacker who achieved privilege escalation will have left artifacts here. This directory is readable only by root, but on a forensic image you have full access.

Key forensic artifacts:

  • /root/.bash_history — Root's command history. Every command run as root after escalation may appear here.
  • /root/.ssh/authorized_keys — Attacker-added SSH keys for persistent root access. Check this immediately.
  • /root/.ssh/known_hosts — Systems root has SSH'd to (lateral movement, C2 connections).
  • /root/.bashrc, /root/.profile — Root shell initialization; malicious code injected here runs with full privileges.
  • Any scripts, tools, or downloaded files dropped by an attacker after escalation.

🔍 Scenario Connection

On web-prod-03, /root/.ssh/authorized_keys and /root/.bash_history are among the first artifacts to examine after confirming the image is properly mounted. An attacker who gained root access almost certainly interacted with these files.

What it contains: Temporary files created by the system and users. /tmp is cleared on reboot; /var/tmp persists across reboots.

Why attackers love it: /tmp is world-writable (any user can write here), making it a favorite staging area for malware, exploit scripts, and downloaded tools. It requires no special permissions to use.

Key forensic artifacts:

  • Malware binaries or scripts staged for execution
  • Exploit code downloaded during an attack
  • Output files from data exfiltration staging
  • Compiler output from in-memory build-and-execute attacks
  • Deleted file remnants recoverable through unallocated space analysis

⚠️ Volatility Note

Because /tmp is cleared on reboot, files staged there are lost when the system restarts. This is one reason live acquisition (capturing a running system) is sometimes necessary — you preserve /tmp contents that would otherwise be gone after a shutdown.

  • /proc/ — Virtual filesystem exposing live kernel and process data. Only exists on a running system — not present in a dead-box image. Contains per-process directories (/proc/PID/) with memory maps, open files, and command-line arguments.
  • /usr/bin/, /usr/sbin/, /usr/local/bin/ — Standard binary locations. Look for unusual executables or modified system binaries (rootkit indicators).
  • /opt/ — Optional/third-party software. Web applications, commercial tools, and attacker-installed software often live here.
  • /srv/ — Service data (web root files sometimes here alongside /var/www/). Web shells and malicious files may be planted in web-accessible directories.
  • /var/www/ — Default Apache/NGINX web root. Check for web shells (e.g., .php files with eval(base64_decode(...)) patterns).
  • /var/spool/cron/crontabs/ — Per-user cron job files. Persistence mechanism frequently used by attackers.
  • /lib/systemd/system/ and /etc/systemd/system/ — Systemd service unit files. Malicious services installed as systemd units persist across reboots.
  • /dev/shm/ — Shared memory filesystem (RAM-backed). Used by some malware families to stage payloads in memory without touching disk.

Forensic Acquisition Approach

Before examining any artifact, you must acquire the evidence in a forensically sound manner. For Linux systems, the first decision is whether to perform a dead-box (offline) acquisition or a live acquisition of a running system. This decision has significant consequences for what evidence you can recover.

The Dead-Box vs. Live Acquisition Decision

🔒 Dead-Box Acquisition

System is powered off before imaging

  • Full disk image captured with dd, dcfldd, or FTK Imager
  • No risk of evidence modification during acquisition
  • Preserves deleted file remnants in unallocated space
  • Hash verification is stable (no writes occurring)
  • Loses: RAM contents, running processes, network connections, /proc/ data, /tmp contents (if tmpfs)

Best when: System can be taken offline without business impact; volatile data is not critical; legal/chain-of-custody requirements favor offline imaging.

⚡ Live Acquisition

System remains running during acquisition

  • RAM captured with LiME (Linux Memory Extractor) kernel module
  • Running processes, network connections, and /proc/ data preserved
  • /tmp and /dev/shm/ contents captured
  • Encryption keys may be recoverable from memory
  • Risk: Every action on the live system modifies evidence; requires careful documentation

Best when: Malware is suspected to be memory-resident only; encrypted volumes are mounted; volatile network connections are critical; system cannot be shut down.

⚠️ Order of Volatility

When performing live acquisition, always collect evidence in order of volatility — most volatile first. The general order for Linux: CPU registers and cache → RAM → Network connections and routing tables → Running processes → /tmp → Disk image → Remote logs. This principle ensures you capture the most transient evidence before it disappears.

Read-Only Mount Procedure

Whether you acquired a dead-box image or are working from a live capture, you must mount the filesystem read-only to prevent any writes to the evidence. On Linux, this is accomplished with the mount command and specific flags.

# Step 1: Create a mount point directory
sudo mkdir -p /mnt/evidence

# Step 2: Mount the image read-only with noexec and nosuid for safety
sudo mount -o ro,noexec,nosuid,noatime -t ext4 /path/to/web-prod-03.img /mnt/evidence

# Step 3: Verify the mount is read-only
mount | grep /mnt/evidence

# Expected output should show: (ro,noexec,nosuid,noatime,...)

Mount Flag Explanations

Flag Meaning Forensic Purpose
ro Read-only Prevents any writes to the evidence image — the most critical flag for evidence integrity
noexec No execution of binaries Prevents accidentally running malware or attacker tools present on the image
nosuid Ignore SUID/SGID bits Prevents SUID binaries on the evidence image from running with elevated privileges on your forensic workstation
noatime Do not update access times Prevents read operations from updating file atime timestamps — preserves original timestamp evidence
-t ext4 Filesystem type Explicitly specifies the filesystem type; use blkid or file to determine the correct type before mounting

🔗 Connection to Week 6 — Hash Verification

Before mounting any evidence image, always verify its integrity using a cryptographic hash. As you learned in Week 6, compute the SHA-256 hash of the image and compare it to the hash recorded at acquisition time. If the hashes match, the image has not been altered. If they do not match, the image's chain of custody is broken and its admissibility is in question.

# Verify image integrity before mounting
sha256sum /path/to/web-prod-03.img
# Compare output to the hash recorded in your acquisition documentation

💡 Hardware Write Blockers on Linux

When working with physical media (not image files), always use a hardware write blocker between the evidence drive and your forensic workstation. On Linux, if a hardware write blocker is unavailable, you can use hdparm or configure the kernel to treat the device as read-only — but a hardware write blocker is always preferred for court-admissible evidence.
