📋 Investigative Scenario

A Linux server known internally as victoria-v8 was flagged by network monitoring as potentially compromised. The system was a publicly accessible server. After the alert was raised, the machine was taken offline and a forensic image of its primary partition (de>victoria-v8.sda1.img) was captured using de>dd for offline analysis.

You have been assigned as the forensic analyst on this case. Your task is to mount the disk image read-only, verify its integrity, and conduct a structured investigation to determine: how the attacker gained access, what they did once inside, what tools or files they introduced, and what evidence of persistence or anti-forensics exists. All findings must be supported by specific artifacts — file paths, log excerpts, and timestamps.

⚠ Scope Note: Two memory dump files also exist for this case (de>victoria-v8.kcore.img and de>victoria-v8.memdump.img). Memory forensics is covered in a later unit. This lab focuses exclusively on the disk partition image de>victoria-v8.sda1.img.

Phase 1 Image Acquisition, Integrity Verification & Mount

📋 Objective

Before any analysis begins, you must establish the integrity of the forensic evidence. This phase covers downloading the image, verifying its cryptographic hashes, examining the partition structure, and mounting the filesystem read-only so that no accidental writes can occur.

🚫 Evidence Integrity Rule: Never mount a forensic image read-write. Always use the de>ro (read-only) flag. If you accidentally mount read-write, the image is considered potentially contaminated and must be re-downloaded.

🔢 Step 1 — Verify the ZIP Hash

Before extracting, confirm the ZIP file matches the published hash:

de># On Linux / WSL
md5sum    victoria-v8.sda1.img.zip
sha1sum   victoria-v8.sda1.img.zip

# On Windows PowerShell
Get-FileHash victoria-v8.sda1.img.zip -Algorithm MD5
Get-FileHash victoria-v8.sda1.img.zip -Algorithm SHA1

Expected values:

AlgorithmExpected Hash
MD5cba614f59020ce8910346cc43056692f
SHA1cddc70ca67db4f3cfca4d48c755c43bb286738c3
⚠ If hashes do not match: Do not proceed. Re-download the file. A hash mismatch means the file is corrupt or has been altered.

💾 Step 2 — Extract and Hash the Raw Image

After confirming the ZIP hash, extract the image and record the hash of the raw de>.img file:

de>unzip victoria-v8.sda1.img.zip
md5sum  victoria-v8.sda1.img
sha1sum victoria-v8.sda1.img

Record both hash values in your lab report. These become the evidence hashes you will reference throughout the investigation to prove the image was not modified during analysis.

🔍 Step 3 — Examine Partition Structure

Use Sleuth Kit's de>mmls to view the partition layout:

de>mmls victoria-v8.sda1.img

Note the partition start offset (in sectors), partition type, and size. You will need the start offset to mount the correct filesystem. Also run de>fsstat to identify the filesystem type: 

de>fsstat victoria-v8.sda1.img
📋 Document & Answer

What is the partition layout? What filesystem type is present? What OS and distribution does de>fsstat or the image metadata suggest?

🔒 Step 4 — Mount the Image Read-Only

Create a mount point and mount the image read-only with no execution permissions:

de>sudo mkdir /mnt/victoria
sudo mount -o ro,loop,nodev,noexec victoria-v8.sda1.img /mnt/victoria

Confirm the mount is read-only:

de>mount | grep victoria
# Should show: ro (read-only)
✓ Alternatively in Autopsy: Add the image as a new Data Source in Autopsy. Autopsy always accesses images read-only. Use Autopsy for browsing the directory tree and extracting files, and the CLI mount for running commands like de>last, de>grep, and log parsing.
📋 Document & Answer

Provide a screenshot or terminal output confirming the image is mounted read-only. Record the mount path you will use for all subsequent phases.

✅ Phase 1 Checklist

    >1ZIP file MD5 and SHA1 verified and match published values. >2Raw de>.img file MD5 and SHA1 recorded as evidence hashes. >3Partition structure documented using de>mmls. >4Filesystem type and OS identified using de>fsstat. >5Image mounted read-only (or loaded in Autopsy) and confirmed.
Phase 2 User Accounts & Authentication Files

📋 Objective

Examine all user account-related files on the compromised server. Identify which accounts exist, which are suspicious, which service or account triggered the initial alert, and whether any accounts show signs of attacker creation or modification. Document every non-system account with supporting evidence.

👤 Step 1 — Enumerate /etc/passwd

View all user accounts defined on the system:

de>cat /mnt/victoria/etc/passwd

Each entry follows this format:

de>username:x:UID:GID:comment:/home/dir:/bin/shell

Flag any account where:

    >UID is 0 (root-level privilege) but the username is not de>root >Shell is de>/bin/bash or de>/bin/sh for a non-human account >Home directory is unusual (e.g., de>/tmp, de>/var/tmp) >Username looks generic, random, or out of place for the server role
📋 Document & Answer

List all non-system accounts (UID ≥ 1000 or any with de>/bin/bash shell). Flag any that appear suspicious and explain why.

🔒 Step 2 — Examine /etc/shadow

Check password hashes and account status:

de>cat /mnt/victoria/etc/shadow

Look for:

    >Accounts with a valid password hash (not de>* or de>!) that should be service accounts >Recently changed passwords — the third field is days since Jan 1, 1970 >Accounts with no password expiry configured
📋 Document & Answer

Are there accounts in de>/etc/shadow with active password hashes that should not have them? Note password change dates for all flagged accounts.

👥 Step 3 — Check /etc/group and /etc/sudoers

Identify group memberships and sudo privileges:

de>cat /mnt/victoria/etc/group
cat /mnt/victoria/etc/sudoers
ls -la /mnt/victoria/etc/sudoers.d/

Look for unexpected accounts added to de>wheel, de>sudo, or de>admin groups.

📋 Document & Answer

Which accounts have sudo or elevated group membership? Were any unexpected accounts added to privileged groups?

🔐 Step 4 — Review SSH Authorized Keys

Check all user home directories for planted SSH keys:

de>find /mnt/victoria/home /mnt/victoria/root \
  -name "authorized_keys" 2>/dev/null | \
  xargs -I{} sh -c 'echo "=== {} ===" && cat {}'
📋 Document & Answer

Were any SSH authorized keys found? Record the key type, comment field, and which user account they belong to. Could these be attacker-planted backdoor keys?

✅ Phase 2 Checklist

    >1All accounts in de>/etc/passwd enumerated; suspicious ones flagged with reasoning. >2de>/etc/shadow reviewed for active hashes on unexpected accounts. >3Group memberships and sudo privileges reviewed and documented. >4All de>authorized_keys files located and contents recorded.
Phase 3 Shell History & User Activity Artifacts

📋 Objective

Shell history files are among the most valuable artifacts in a Linux compromise investigation. They record commands typed by users and attackers, revealing tools downloaded, reconnaissance performed, files modified, and anti-forensic activity. This phase examines bash history and related shell startup files for all relevant accounts.

⚠ Anti-Forensics Awareness: Sophisticated attackers sometimes clear history with de>history -c, set de>HISTSIZE=0, or redirect history to de>/dev/null. The absence of history is itself a finding worth documenting.

📄 Step 1 — Locate All History Files

Find all shell history files on the system:

de>find /mnt/victoria -name ".*history" -type f 2>/dev/null
cat  /mnt/victoria/root/.bash_history
ls -la /mnt/victoria/root/
ls -la /mnt/victoria/home/
cat  /mnt/victoria/home/[username]/.bash_history
📋 Document & Answer

List every history file found, its path, owner, size, and last-modified timestamp. If a history file is missing, empty, or symlinked to de>/dev/null, document that as a potential anti-forensic indicator.

🔎 Step 2 — Analyze Command History

Carefully review the contents of each history file. Look for evidence of:

    >Reconnaissance: de>uname -a, de>id, de>whoami, de>cat /etc/passwd, network scans >Privilege escalation: de>sudo su, de>su -, exploit execution >File downloads: de>wget, de>curl, de>ftp — note full URLs >Tool installation: de>gcc compile commands, de>chmod +x, package installs >Anti-forensics: de>rm -rf on log files, de>history -c, log tampering >Persistence setup: crontab edits, service installs, SSH key additions
📋 Document & Answer

List every significant command found. Group by category: reconnaissance, escalation, downloads, persistence, anti-forensics. What tools did the attacker use? Were any files downloaded — if so, record the full URLs or filenames.

📄 Step 3 — Review Shell Startup Files

Check shell startup and profile files for injected commands:

de>cat /mnt/victoria/root/.bashrc
cat /mnt/victoria/root/.bash_profile
cat /mnt/victoria/root/.profile
cat /mnt/victoria/home/[username]/.bashrc

Look for unusual de>PATH modifications, aliases that shadow system commands, or commands that execute automatically on login.

📋 Document & Answer

Were any shell startup files modified? Document any unusual aliases, PATH changes, or auto-executed commands found.

🔍 Step 4 — Check for Dropped Files

Based on history findings, look for files the attacker may have left behind:

de>ls -la /mnt/victoria/tmp/
ls -la /mnt/victoria/var/tmp/
ls -la /mnt/victoria/dev/shm/ 2>/dev/null
# Recently modified files
find /mnt/victoria -newer /mnt/victoria/etc/passwd \
  -not -path "*/proc/*" -type f 2>/dev/null | head -40
📋 Document & Answer

Were any attacker-dropped files found? For each: record full path, size, permissions, timestamps. Use de>file or de>strings to identify what each file is. What do these files suggest about the attacker's goals or capabilities?

✅ Phase 3 Checklist

    >1All shell history files located; paths, sizes, and timestamps recorded. >2History contents analyzed and commands categorized by type. >3Shell startup files reviewed for injected commands or PATH manipulation. >4Attacker-dropped files in staging directories identified and analyzed. >5Anti-forensic indicators (missing/cleared history, deleted logs) documented.
Phase 4 Log Analysis & Attack Timeline Reconstruction

📋 Objective

System logs are the primary record of what happened on the server and when. This phase examines authentication logs, service logs, and login records to reconstruct the attack timeline — identifying brute-force attempts, successful logins, privilege escalation, and any evidence of log tampering or deletion.

💡 Log Location Note: Debian/Ubuntu systems store authentication events in de>/var/log/auth.log. Red Hat/CentOS systems use de>/var/log/secure. Check which file exists on this image and adapt your commands accordingly.

📊 Step 1 — Survey Available Logs

First inventory what logs exist, their sizes, and modification timestamps:

de>ls -lah /mnt/victoria/var/log/
# Check for web server logs
ls -lah /mnt/victoria/var/log/apache2/ 2>/dev/null
ls -lah /mnt/victoria/var/log/httpd/    2>/dev/null

Pay close attention to:

    >Log files with a size of 0 bytes — may indicate deliberate truncation >Log files with unexpectedly recent modification times >Missing log files that should normally be present (e.g., no de>auth.log at all) >Rotated logs (de>auth.log.1, de>auth.log.2.gz) that may predate tampering
📋 Document & Answer

List all log files found in de>/var/log/ with their sizes and timestamps. Are any logs missing, empty, or suspiciously small? Document this as a potential anti-forensic finding.

🔐 Step 2 — Authentication Log Analysis

Examine the primary authentication log for failed and successful login events:

de># View the full auth log
cat /mnt/victoria/var/log/auth.log

# Filter for failed SSH attempts (brute force)
grep "Failed password" /mnt/victoria/var/log/auth.log | head -30

# Filter for successful logins
grep "Accepted" /mnt/victoria/var/log/auth.log

# Check for invalid user attempts
grep "Invalid user" /mnt/victoria/var/log/auth.log | head -20

# Count failed attempts per source IP
grep "Failed password" /mnt/victoria/var/log/auth.log | \
  awk '{print $11}' | sort | uniq -c | sort -rn
📋 Document & Answer

What IP addresses attempted logins? How many failed attempts occurred and from which IPs? Which accounts were targeted? Were any login attempts successful — if so, record the username, source IP, and timestamp of the first successful login.

👤 Step 3 — Login Record Analysis (wtmp / btmp / lastlog)

Use binary login record files to reconstruct who logged in and when:

de># Successful logins (wtmp)
last -f /mnt/victoria/var/log/wtmp

# Failed login attempts (btmp)
lastb -f /mnt/victoria/var/log/btmp 2>/dev/null | head -30

# Last login per user (lastlog)
lastlog --root /mnt/victoria
⚠ Compatibility Note: The de>last and de>lastb commands may need to be run on a Linux system with a compatible de>utmp format. If you get garbled output, try running on a similar-vintage Linux VM or use Autopsy to browse the raw file.
📋 Document & Answer

List all successful logins from de>wtmp: username, source IP, login time, and session duration. Cross-reference with de>auth.log findings. Do the records agree, or are there discrepancies that suggest log tampering?

🌐 Step 4 — Service and Web Server Logs

If a web server (Apache, nginx, or other) is installed, examine its access and error logs:

de># Apache access log
cat /mnt/victoria/var/log/apache2/access.log 2>/dev/null
cat /mnt/victoria/var/log/httpd/access_log   2>/dev/null

# Look for scanning, exploit attempts, or webshell activity
grep -iE "(cmd=|exec|passthru|shell|wget|curl|base64)" \
  /mnt/victoria/var/log/apache2/access.log 2>/dev/null

# Check installed services
ls /mnt/victoria/etc/init.d/
ls /mnt/victoria/etc/xinetd.d/ 2>/dev/null
📋 Document & Answer

What services were running on the server? Are there web server logs present? If so, do they show scanning, exploit attempts, or webshell requests? What service was ultimately attacked?

📅 Step 5 — Build the Attack Timeline

Using all log sources examined in this phase, construct a chronological timeline of the attack. Your timeline should answer:

    >When did the first suspicious activity appear in logs? >When did the first successful unauthorized login occur? >What did the attacker do immediately after gaining access? >Were there multiple phases of the attack (initial access, escalation, persistence)? >When was the last recorded attacker activity?
📋 Document & Answer

Present your attack timeline as a table or numbered list with: timestamp, log source, event description, and significance. Include at least 5 distinct timestamped events that together tell the story of the compromise from first probe to last known activity.

✅ Phase 4 Checklist

    >1All log files in de>/var/log/ inventoried with sizes and timestamps. >2Auth log analyzed for failed attempts; source IPs and targeted accounts documented. >3Successful logins identified with username, source IP, and timestamp. >4de>wtmp and de>btmp records analyzed and cross-referenced with auth log. >5Service and web server logs examined for exploit attempts. >6Chronological attack timeline constructed with at least 5 timestamped events.
Phase 5 Persistence Mechanisms & Attacker Footprint

📋 Objective

After gaining access, attackers typically install persistence mechanisms to ensure they can return even if the initial vulnerability is patched or the compromised account's password is changed. This phase identifies what the attacker installed, modified, or exploited to maintain long-term access, and assesses the overall attacker footprint on the system.

📅 Step 1 — Cron Jobs

Check all cron directories and per-user crontabs for attacker-added scheduled tasks:

de># System-wide cron
cat  /mnt/victoria/etc/crontab
ls -la /mnt/victoria/etc/cron.d/
ls -la /mnt/victoria/etc/cron.daily/
ls -la /mnt/victoria/etc/cron.hourly/
ls -la /mnt/victoria/etc/cron.weekly/
ls -la /mnt/victoria/etc/cron.monthly/

# Per-user crontabs (stored in /var/spool/cron)
ls -la /mnt/victoria/var/spool/cron/
ls -la /mnt/victoria/var/spool/cron/crontabs/ 2>/dev/null
📋 Document & Answer

Were any cron jobs found that were not part of the base OS installation? For each suspicious entry: record the schedule, the command, and which user it runs as. What would the cron job accomplish?

⚙ Step 2 — Startup Services and Init Scripts

Check for attacker-added services or modified init scripts:

de># SysV init scripts
ls -la /mnt/victoria/etc/init.d/
ls -la /mnt/victoria/etc/rc.local 2>/dev/null
cat  /mnt/victoria/etc/rc.local 2>/dev/null

# Check rc directories for enabled services
ls -la /mnt/victoria/etc/rc2.d/
ls -la /mnt/victoria/etc/rc3.d/

# Systemd (if present)
ls -la /mnt/victoria/etc/systemd/system/ 2>/dev/null
ls -la /mnt/victoria/lib/systemd/system/ 2>/dev/null
📋 Document & Answer

Are there any startup scripts or services that appear attacker-added or modified? Document the script name, content summary, and what it would do at boot.

🔐 Step 3 — SUID/SGID Binaries and Modified System Files

Look for SUID binaries that could provide persistent privilege escalation:

de># Find all SUID binaries
find /mnt/victoria -perm -4000 -type f 2>/dev/null

# Find all SGID binaries
find /mnt/victoria -perm -2000 -type f 2>/dev/null

# Look for world-writable files in sensitive locations
find /mnt/victoria/etc /mnt/victoria/bin /mnt/victoria/sbin \
  -perm -o+w -type f 2>/dev/null
📋 Document & Answer

List any SUID/SGID binaries that appear unusual or were not part of the standard OS install. Were any system binaries replaced or modified? How would an attacker use these for persistence or privilege escalation?

🔌 Step 4 — Rootkit and Trojanized Binary Indicators

Check for signs that core system binaries may have been replaced:

de># Check file sizes of common binaries against expected values
ls -la /mnt/victoria/bin/ls
ls -la /mnt/victoria/bin/ps
ls -la /mnt/victoria/bin/netstat 2>/dev/null
ls -la /mnt/victoria/usr/bin/find
ls -la /mnt/victoria/sbin/ifconfig 2>/dev/null

# Look for hidden directories (names starting with . or spaces)
find /mnt/victoria -maxdepth 4 -name ".*" -type d 2>/dev/null | \
  grep -v "^/mnt/victoria/proc"

# Check /dev for unusual files (rootkits sometimes hide here)
find /mnt/victoria/dev -type f 2>/dev/null
⚠ Rootkit Warning: If system binaries like de>ps, de>ls, or de>netstat have been replaced by a rootkit, running them on the live system would hide attacker processes. This is why offline forensic analysis from a clean system is critical.
📋 Document & Answer

Were any signs of rootkit installation found — unusual files in de>/dev, hidden directories, or system binaries with unexpected sizes or timestamps? Document all indicators found.

💡 Step 5 — Overall Attacker Assessment

Based on all evidence gathered across Phases 1–5, answer the following:

📋 Document & Answer

a) What vulnerabilities or weaknesses did the attacker exploit to gain initial access?

b) Were the attacks successful? Which succeeded and which (if any) failed?

c) What did the attacker obtain or accomplish on this system?

d) Based on the tools used, commands run, and methods employed — does the attack appear automated (e.g., scripted brute-force, bot) or manual? Explain your reasoning.

e) What could have prevented this compromise? List at least three specific security controls.

✅ Phase 5 Checklist

    >1All cron locations checked; any attacker-added jobs documented. >2Init scripts and startup services reviewed for unauthorized entries. >3SUID/SGID binaries enumerated; unusual ones flagged and explained. >4Core system binaries checked for signs of replacement or tampering. >5Hidden directories and unusual files in de>/dev investigated. >6Attacker assessment questions (a–e) answered with supporting evidence.
Findings Incident Narrative & Final Report

📝 Overview

Your final deliverable for this lab is a written incident narrative that synthesizes all findings from Phases 1–5 into a coherent, evidence-backed account of what happened on the victoria-v8 server. This is the kind of report a forensic analyst would deliver to management, legal counsel, or a security operations team.

💡 Key Principle: Every conclusion in your report must be tied to a specific artifact — a file path, a log entry, a timestamp, a command from history. Unsupported assertions are not findings; they are speculation.

📄 Report Structure

Your report should include the following sections. There is no strict length requirement — focus on accuracy, clarity, and evidence. A thorough report will typically run 3–6 paragraphs plus supporting tables or artifact excerpts.

1. Case Summary

One short paragraph: what system, what image, when it was acquired, and what tools you used.

2. System Identification

OS and distribution, kernel version, hostname, installed services. Cite the artifacts (e.g., de>/etc/os-release, de>/etc/hostname, de>fsstat output).

3. Account Analysis Summary

List all flagged accounts, their UIDs, and why they are suspicious. Include any planted SSH keys or unauthorized sudo access found in Phase 2.

4. Attack Narrative

A chronological account of the attack from first probe to last known activity. Use your Phase 4 timeline as the backbone. Incorporate relevant findings from Phases 3 and 5 to fill in what the attacker did during the session.

5. Persistence and Impact

What did the attacker leave behind? What access would they retain after a reboot? What data or capabilities did they gain? What is the overall impact to the system and its users?

6. Conclusions and Recommendations

Was the attack automated or manual? What vulnerabilities were exploited? What three or more specific controls would have prevented or detected this attack?
