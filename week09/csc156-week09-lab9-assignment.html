Investigative Scenario

Case Number: CSC156-2026-LAB09  |  Examiner: (Your Name)  |  Date: (Today's Date)

This lab is based on a real-world incident investigated by the Honeynet Project as part of their Forensic Challenge 7 (published March 2011). A Linux server — referred to in the challenge as victoria-v8 — was possibly compromised and submitted to the Honeynet Project community for forensic analysis. The disk image you will examine (victoria-v8.sda1.img) is a sector-level copy of that server's primary hard drive, captured at the time of the incident.

Your task is to conduct a structured dead-box examination of the disk image to determine:

  • What service was running on the server and which account triggered the initial alert
  • What kind of system was targeted (OS, installed services, configuration)
  • When and how the attacker gained access, and from which source IP address
  • What attacks were launched and whether they succeeded
  • What files or tools the attacker downloaded to the server
  • Whether the attacker established any persistence mechanisms
Your Investigation Goals

Across five phases you will: mount and orient yourself to the disk image, analyze user accounts and authentication logs, recover shell history artifacts, examine key log files in /var/log, and identify attacker persistence mechanisms. You will consolidate your findings into a structured summary at the end. Note: this lab focuses exclusively on disk-based evidence — memory analysis is not covered here.

Lab Objectives

By completing this lab, you will demonstrate the ability to:

  1. Mount a Linux forensic image read-only and document the mounting process with hash verification.
  2. Extract and interpret user account data from /etc/passwd, /etc/shadow, and /etc/sudoers to identify suspicious accounts.
  3. Parse authentication logs (auth.log or secure) and binary login history files (wtmp, btmp) to reconstruct login events.
  4. Recover bash history artifacts and analyze command-line evidence, including detecting anti-forensic techniques.
  5. Analyze key /var/log files to reconstruct attacker activity and identify log tampering indicators.
  6. Identify at least one attacker persistence mechanism and document its location, type, and forensic significance.
  7. Produce a structured findings summary suitable for inclusion in a forensic report.

Phase 1 — Image Mounting and Initial Orientation

Before examining any artifacts, you must verify the forensic image's integrity and mount it in a forensically sound manner. This mirrors the acquisition principles from Week 6 — applied here to a Linux image.

Steps

  1. Verify the zip file hash before extracting. Use md5sum or sha1sum on the downloaded victoria-v8.sda1.img.zip and compare against the values in the callout above. Record your results in the response field below. Then extract the zip to obtain victoria-v8.sda1.img.
  2. Mount the image read-only using a write blocker or the following command pattern: 
    sudo mount -o ro,noexec,noatime -t auto \
    /path/to/victoria-v8.sda1.img /mnt/evidence
    If using Autopsy, add the extracted image file as a new data source. Document your mount method.
  3. Identify the filesystem type and partition layout. Run mmls (Sleuth Kit) or use Autopsy's partition view to note the partition table type (MBR or GPT), partition offsets, and filesystem type (ext4, XFS, etc.).
  4. Survey the top-level directory structure. Navigate to the image root and list the contents of /etc, /var/log, /home, and /root. Note which forensically significant directories are present.
  5. Document file system timestamps for the root directory using stat /mnt/evidence or Autopsy's metadata view.
  • You are hashing the zip file (victoria-v8.sda1.img.zip) against the provided values — this confirms the download is intact before you extract. The extracted .img file will have a different hash.
  • If the image contains multiple partitions, you may need to use an offset: mount -o ro,loop,offset=<bytes> image.img /mnt/evidence. Use mmls output to calculate the offset (sectors × 512).
  • The noatime flag is critical — without it, mounting the image updates file access timestamps and can contaminate your evidence.
  • If hash verification fails, stop immediately and notify your instructor — do not continue the investigation on a potentially corrupted image. Re-download from the OneDrive folder and try again.
  • In Autopsy, verify the hash in the Data Source Integrity section after ingestion completes.

Record the zip filename you downloaded, the hash algorithm you used, the provided hash value, the hash value you computed, and whether they match. Then record the extracted image filename.

Record the exact command or tool used to mount the image. List the partition table type, number of partitions, filesystem type(s), and partition sizes.

List the directories and notable files you found in /etc, /var/log, /home, and /root. Note anything that looks unusual or is missing.

Phase 2 — User Account and Authentication Analysis

Examine the user account files and authentication records to determine who had access to the system, whether any accounts are suspicious, and when authentication events occurred.

2A — User Account Files

  1. Examine /etc/passwd. Open the file and review each entry. Look for: duplicate UID 0 entries, accounts with /bin/bash or other interactive shells that should not have them, home directories in unusual locations, and recently created accounts.
  2. Examine /etc/shadow. Review password hash formats ($id$salt$hash) and note any accounts with recently changed passwords (field 3), locked accounts (! or * prefix), or accounts with no expiry set (field 5 is empty).
  3. Examine /etc/group and /etc/sudoers. Identify accounts with elevated privileges — sudo group membership, wheel group, or explicit NOPASSWD entries in sudoers.
  • A second UID 0 entry in /etc/passwd is a critical finding — it means a hidden root-level account exists.
  • Service accounts (e.g., www-data, daemon) should have /usr/sbin/nologin or /bin/false as their shell. An interactive shell on a service account is a red flag.
  • /etc/shadow requires root to read. If Autopsy does not display it directly, use icat to extract it by inode number.

List all non-system user accounts (typically UID ≥ 1000). For each, record the username, UID, GID, home directory, and shell. Flag any that appear suspicious and explain why.

Note the password hash algorithms in use, any accounts with recently changed passwords, and all accounts with sudo or elevated privileges. Include any NOPASSWD entries found.

2B — Authentication Log Analysis

  1. Locate and open the authentication log. On Debian-based systems (including older Ubuntu) this is /var/log/auth.log; on RHEL/CentOS it is /var/log/secure. This image predates systemd's journald, so look for plain-text log files rather than binary journal files.
  2. Search for SSH login events. Filter for Accepted (successful) and Failed (unsuccessful) SSH events. Note source IP addresses, usernames, and timestamps. 
    grep -E "Accepted|Failed" auth.log | head -50
  3. Look for brute-force patterns: many failed attempts from a single IP address shortly before a successful login.
  4. Search for sudo/privilege escalation events: 
    grep "sudo\|su:" auth.log
  5. Parse binary login history files using the appropriate commands: 
    last -f /var/log/wtmp       # successful logins
lastb -f /var/log/btmp      # failed login attempts
lastlog                      # last login per user
  • The wtmp and btmp files are binary — do not attempt to open them with a text editor or cat. Use last and lastb.
  • If btmp is empty or missing, that itself may be a finding — an attacker may have cleared it. Record the file's mtime using stat and compare it to the timeline.
  • Correlate the source IP from auth.log with web server logs (Phase 4) — they may be the same attacker host.
  • In Autopsy, you can view text log files directly in the file browser. For binary log files, export them and parse with last/lastb in a Linux terminal.

List the significant authentication events you found: failed logins, successful logins, sudo usage, and privilege escalation. For each, include the timestamp, username, source IP (if applicable), and event type. Identify any brute-force patterns.

Summarize the output from last and lastb. Note the first and last successful login for any suspicious account. Record whether btmp was present, empty, or missing, and the file's mtime.

Phase 3 — Bash History Recovery and Command-Line Artifact Analysis

Shell history files can reveal exactly what an attacker did after gaining access. Examine history files for all user accounts, including root, and look for evidence of anti-forensic activity.

Steps

  1. Locate bash history files for all user accounts. Check:
    • /root/.bash_history — root user history (highest forensic value)
    • /home/<username>/.bash_history — for each user account found in Phase 2
    • Also check for .zsh_history or .fish_history if non-bash shells are installed
  2. Record timestamps for each history file using stat. The mtime can help confirm when the session occurred.
  3. Examine shell startup scripts for anti-forensic configuration: 
    cat /root/.bashrc
cat /root/.bash_profile
cat /etc/profile
    Look for: unset HISTFILE, HISTSIZE=0, HISTFILESIZE=0, or HISTCONTROL=ignorespace.
  4. Check for deleted history files using The Sleuth Kit: 
    fls -r -d evidence.img | grep bash_history
    A deleted but recoverable inode may allow partial history recovery.
  5. Analyze history file contents for: privilege escalation commands (sudo su, sudo -i), file download commands (wget, curl), network tools (nc, ncat), file permission changes (chmod, chown), and cron or service modifications.
  • An empty .bash_history file is itself a finding — it may indicate the attacker ran history -c or set HISTSIZE=0. Check the file's inode metadata with stat to see if it was recently truncated.
  • If HISTTIMEFORMAT was set, each history entry will be preceded by a Unix timestamp line starting with # — these are very useful for building a timeline.
  • Even if the history file is empty or missing, commands may appear in logs (Phase 4) or cron entries (Phase 5), providing corroboration.
  • Look at the bash history of the web server service account (e.g., www-data) if it has a home directory — attackers sometimes escalate from web shell to interactive session.

List each .bash_history file found (or expected but missing). For each, record the file path, size in bytes, mtime (last modified), and whether timestamps were enabled in the file.

List all commands from the history file(s) that are potentially related to the incident. Include the full command and, if timestamps are available, the time. Briefly explain why each command is significant.

Did you find evidence that the attacker attempted to hide their command-line activity? List any indicators found (e.g., HISTFILE/HISTSIZE manipulation in startup scripts, truncated or deleted history file, history -c usage documented in logs).

Phase 4 — Log File Analysis

The /var/log directory contains multiple log sources that provide a broader picture of system activity. Analyze the logs relevant to this incident and look for signs of log tampering.

4A — System and Web Server Logs

  1. Examine /var/log/syslog or /var/log/messages for the incident time window. First note the date range of entries in the file, then filter for the relevant period. Look for service starts/stops, unexpected kernel messages, and cron execution events: 
    # Check the date range of log entries first
head -1 syslog; tail -1 syslog
# Then filter for a specific date (replace with the date found in the log)
grep "Feb 18\|Feb 19" syslog
  2. Identify what network services were running. Look at /etc/inetd.conf, /etc/xinetd.d/, and any service directories under /etc/init.d/ to understand what services were exposed. Note any services you would not expect on a typical server — these may be the attack vector.
  3. Examine web server access logs in /var/log/apache2/access.log or /var/log/apache/access.log (this is an older Debian-era system; paths may vary). Search for:
    • Directory traversal patterns: ../
    • Common scanner user-agents: sqlmap, nikto, nmap
    • High volumes of 4xx errors from a single IP (potential brute force or scanning)
    • Unusual POST requests to admin paths or CGI scripts
    awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -20
  4. Examine the web server error log (error.log) for application errors, CGI execution failures, or file-not-found patterns that suggest scanning activity.
  5. Examine the package manager log (/var/log/dpkg.log) for any software installed or removed around the incident time. Attackers often install tools like netcat, nmap, or wget.
  • Check for rotated logs (syslog.1, syslog.2.gz). If the incident occurred before the last rotation, you may need to decompress archived logs with zcat or zgrep.
  • Correlate the attacker IP from auth.log (Phase 2) with web server access logs — many attacks begin with web exploitation before pivoting to SSH.
  • In Autopsy, the keyword search module can help locate log entries containing specific IP addresses across all log files simultaneously.

4B — Log Tampering Detection

  1. Check for timestamp gaps in auth.log and syslog that align with the suspected intrusion window — missing log entries during active attack periods suggest tampering.
  2. Compare file system timestamps (mtime) on log files with the most recent entry in those files. If mtime is earlier than the last log entry timestamp, the file may have been modified.
  3. Look for truncated log files: 
    ls -lh /var/log/auth.log
wc -l /var/log/auth.log
    An unusually small file may indicate truncation.
  4. Check for immutable file attributes: 
    lsattr /var/log/auth.log
    The i flag indicates an immutable file (attacker may have set this to prevent log growth after tampering).
  • The absence of log entries is itself evidence — document gaps explicitly. Note the timestamp of the last entry before the gap and the first entry after it.
  • This image predates systemd journald, so there is no binary journal to fall back on — if auth.log is wiped, deleted-inode recovery with fls/icat (Sleuth Kit) may be your only option for partial log recovery.
  • Some attackers only delete specific lines rather than the whole file — look for entries that reference nearby events (e.g., the session opened) without the corresponding "session closed" entry.

List notable events from syslog or messages related to the incident time window. Include the timestamp, event type, and significance of each entry.

Identify the network services you found running on this system. Then summarize suspicious activity found in the web server or service logs: include the attacker IP(s), types of requests or attacks attempted, any successful exploit indicators, and the time window of activity.

List any packages installed or removed around the incident time (from dpkg.log). Document any log tampering evidence found: timestamp gaps, mtime inconsistencies, truncated files, or immutable flags.

Phase 5 — Persistence Mechanism Identification

Attackers who establish initial access often install one or more persistence mechanisms to ensure continued access even after a reboot or password change. Examine the four most common persistence categories on Linux systems.

Required Finding

You must identify and document at least one persistence mechanism for full credit on this phase. Document its exact file path, content, and your assessment of what it does.

5A — Scheduled Tasks (Cron)

  1. Examine all cron locations: 
    cat /etc/crontab
ls -la /etc/cron.d/
ls -la /var/spool/cron/crontabs/
  2. Look for red flags: entries using wget, curl, or bash to download/execute remote content; base64-encoded payloads; entries running from /tmp or /var/tmp; entries running as root under an unusual username.

5B — systemd Services

  1. List unit files in user-writable locations: 
    ls -la /etc/systemd/system/
ls -la /lib/systemd/system/
  2. Examine any recently modified or unfamiliar service files. Look for ExecStart paths pointing to /tmp, /var/tmp, or hidden directories; Restart=always for persistence; services with WantedBy=multi-user.target that lack a recognizable package origin.

5C — SUID/SGID Binaries

  1. Find all SUID binaries on the image: 
    find /mnt/evidence -perm -4000 -type f 2>/dev/null
  2. Compare the list against expected system binaries. Unexpected SUID bits on binaries like python3, vim, find, nmap, or nc indicate an attacker-created privilege escalation path.

5D — SSH Authorized Keys

  1. Check authorized_keys files for all users and root: 
    cat /root/.ssh/authorized_keys
cat /home/<username>/.ssh/authorized_keys
  2. Review SSH daemon configuration for tampering: 
    cat /etc/ssh/sshd_config
    Flag: PermitRootLogin yes, PasswordAuthentication yes (if the policy was to use keys only), non-standard Port settings.
  • When examining cron entries, pay attention to the timing — a cron job scheduled for @reboot or every few minutes is more suspicious than one scheduled weekly.
  • For SUID binary analysis: document the binary path, owner, and permissions. Compare the binary's mtime against standard system installation dates — a recently modified system binary is a major red flag.
  • An empty ~/.ssh/ directory may indicate the attacker created and then removed the authorized_keys file. Check for a deleted inode with fls.
  • Multiple persistence mechanisms are common — finding one does not mean you are done. Check all four categories before writing your summary.

List all cron entries found. For each suspicious entry, record the file location, the full cron entry text, the schedule, the command being run, and your assessment of its purpose.

List any suspicious systemd unit files found (include the file path, ExecStart value, and enabled status). List any unauthorized SUID binaries found (include the binary path, owner, and permissions).

List any authorized_keys files found and their contents. Note any suspicious configuration in sshd_config. If no keys were found, state that explicitly.

Identify the single most significant persistence mechanism you found. Describe what it is, where it is located, what it does, and what forensic evidence supports your conclusion. This is a required finding for full credit.
