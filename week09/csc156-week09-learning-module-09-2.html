ext2/3/4 File System Architecture

The extended file system family is the standard for Linux. Understanding its internal structure is essential because forensic tools like The Sleuth Kit operate directly on these structures — and knowing what each component contains tells you where evidence lives and why it survives deletion.

ext2 — Second Extended File System (1993)

ext2 established the core architecture that ext3 and ext4 still use today. It introduced the inode-based metadata model and block-group layout that remain the foundation of Linux file systems.

  • No journaling — changes are written directly to disk; an unclean shutdown can leave the file system in an inconsistent state requiring fsck
  • Fixed inode count — set at format time; running out of inodes (even with free space) prevents new file creation
  • Maximum volume: 32 TiB; maximum file size: 2 TiB
  • Timestamps: mtime, atime, ctime stored at 1-second resolution; no creation time (crtime)
  • Forensic note: Still encountered on older embedded Linux systems, legacy servers, and some USB drives; the lack of journaling means fewer metadata traces but also fewer recovery aids

⚠️ No Journal = Fewer Traces

On ext2, there is no journal to examine for recently written or partially committed data. Deleted file recovery relies entirely on inode and block bitmap analysis. If bitmaps have been overwritten, recovery becomes significantly harder.

ext3 — Third Extended File System (2001)

ext3 added journaling to the ext2 architecture — the single most significant change for both system reliability and forensic analysis. It is backward-compatible with ext2; an ext3 volume can be mounted as ext2 (disabling journaling).

  • Three journaling modes:
    • journal — both metadata and data blocks are written to the journal before the main filesystem; maximum protection and maximum forensic value, but slowest
    • ordered — only metadata is journaled; data blocks are flushed to disk before the metadata journal entry is committed; the default mode on most Linux distributions
    • writeback — only metadata is journaled; data blocks may be written at any time relative to metadata; fastest, least safe
  • Journal location: stored as a hidden file (inode 8) at .journal in the root of the volume; accessible with debugfs
  • Timestamps: same as ext2 — 1-second resolution; no crtime
  • Forensic note: The journal is a circular buffer. In journal mode, recently deleted or overwritten data may still exist in the journal for a short time after deletion — a valuable recovery source

💡 Journal as Evidence

The ext3/ext4 journal records pending metadata operations. Even after a file is deleted and its inode is marked free, the journal may still contain the inode's previous state, including block pointers to the file's data. Examining the journal with debugfs can reveal recently deleted file metadata before it is overwritten.

ext4 — Fourth Extended File System (2008–present)

ext4 is the default file system for most modern Linux distributions (Ubuntu, Debian, RHEL, CentOS, Rocky Linux). It is the file system you will most frequently encounter in Linux forensic investigations.

  • Extents: replaces the old indirect-block chain with extent trees — a file maps to one or more contiguous block ranges; reduces fragmentation and speeds up large-file access
  • Nanosecond timestamps: mtime, atime, ctime, and crtime are all stored at nanosecond resolution — a significant improvement over ext3's 1-second granularity
  • crtime (creation time): ext4 is the first member of the ext family to store a true file creation time in the inode; accessible via debugfs or istat (TSK)
  • Larger volumes: maximum volume size 1 EiB; maximum file size 16 TiB
  • Inline data: very small files (< ~60 bytes) may be stored directly inside the inode, leaving no separate data blocks
  • Journaling: same three modes as ext3; ordered remains the default
  • Delayed allocation: blocks are not allocated until data is actually flushed; can complicate recovery of partially written files

✅ Why ext4 Matters Most for This Course

The Ubuntu 22.04 server image (web-prod-03.img) uses ext4. All The Sleuth Kit commands and timestamp analysis techniques in this module apply directly to that image. ext4's nanosecond crtime is particularly valuable: it can definitively establish when a file was created, even if an attacker modified its mtime.

Block Group Layout

ext4 divides the volume into block groups of equal size. Each block group is a self-contained unit with its own metadata structures. This design means metadata is distributed across the disk rather than concentrated at the beginning — reducing seek times and providing redundancy through superblock backups.

The superblock is the master record of the file system. It is located at byte offset 1024 from the start of the volume (block group 0) and is backed up at the beginning of selected block groups (typically groups 1, 3, 5, 7, and powers of 3 and 5).

Key forensic fields in the superblock:

  • s_magic — magic number 0xEF53 identifies this as an ext2/3/4 volume
  • s_blocks_count — total block count; used to calculate volume size
  • s_inodes_count — total inode count (fixed at format time)
  • s_free_blocks_count / s_free_inodes_count — current free counts; a sudden drop may indicate bulk file creation (e.g., malware unpacking)
  • s_mtime — last mount time; s_wtime — last write time; s_lastcheck — last fsck time
  • s_uuid — volume UUID; useful for correlating images to system logs that reference the UUID
  • s_last_mounted — path where the volume was last mounted (e.g., /)
  • s_feature_compat, s_feature_incompat — feature flags indicating which ext4 features are enabled (extents, large files, etc.)

🔍 Forensic Significance

The superblock's s_mtime and s_wtime fields record the last time the volume was mounted and written. On a dead-box image, these timestamps reflect the state at acquisition — they can help confirm or challenge the system owner's account of when the server was last active.

An inode (index node) is a fixed-size data structure that stores all metadata about a file or directory — except its name. Filenames are stored in directory entries, which point to inode numbers. This separation is what makes deleted file recovery possible: when a file is deleted, the directory entry is removed but the inode (and its block pointers) may remain readable until overwritten.

Key fields stored in an ext4 inode:

i_mode
File type (regular, directory, symlink, device) and permission bits (rwxrwxrwx)
i_uid / i_gid
Owner user ID and group ID — maps to /etc/passwd and /etc/group
i_size
File size in bytes; for deleted files this field may still hold the original size
i_links_count
Number of hard links pointing to this inode; 0 means the file has been unlinked (deleted)
i_block / extent tree
Block pointers (ext2/3) or extent tree root (ext4) — maps inode to data blocks on disk
i_flags
Flags: extents flag (0x80000), immutable (0x10), append-only (0x20) — immutable/append-only are attacker-used anti-forensic techniques
i_atime
Last access time — when the file was last read
i_mtime
Last modification time — when file content was last changed
i_ctime
Inode change time — when inode metadata last changed (NOT creation time)
i_crtime (ext4 only)
True creation time — when the file was first created; stored in extra inode space

⚠️ Critical Distinction: ctime ≠ Creation Time

This is one of the most common mistakes in Linux forensics. ctime is inode change time — it updates whenever the inode metadata changes (permissions, ownership, link count, timestamps). It is not the file creation time. On ext2 and ext3, there is no creation time at all. On ext4, the true creation time is crtime, accessible only through debugfs or istat (The Sleuth Kit). Do not report ctime as "file created" in your forensic report.

The journal is a circular write-ahead log stored as a special file within the ext3/ext4 volume. Before any metadata (or data, in journal mode) is written to its final location, a transaction is written to the journal. After the write completes successfully, the transaction is marked committed and the journal space is eventually reused.

Forensic value of the journal:

  • Deleted file metadata recovery: After a file is deleted, its inode state (including block pointers) may persist in uncommitted or recently committed journal transactions. Examining the journal can reveal block pointers that are no longer accessible through the inode table.
  • Timeline reconstruction: Journal transactions carry timestamps, providing an independent timeline of metadata operations that can corroborate or contradict other timestamp evidence.
  • Anti-forensic detection: If an attacker used shred or wipe on a file, the journal may contain evidence of the multiple overwrite passes as successive metadata transactions.
  • Partial write evidence: In ordered mode, if a system crashed mid-write, the journal may contain the partial transaction — evidence of activity at the time of the crash or shutdown.
# Open image in debugfs (read-only mode)
sudo debugfs -R "logdump -a" /path/to/web-prod-03.img

# Dump journal to a file for offline analysis
sudo debugfs -R "dump <8> /tmp/journal.bin" /path/to/web-prod-03.img

💡 Journal Size and Retention

The journal is a circular buffer — once it fills, old transactions are overwritten. On a busy server, the journal may only retain the last few seconds to minutes of transactions. On a server that was shut down cleanly (as web-prod-03 was), the journal is likely in a consistent state and may retain more history than a system that crashed.

XFS Basics for Forensic Examiners

While ext4 dominates Ubuntu and Debian systems, XFS is the default file system on Red Hat Enterprise Linux (RHEL) 7+, CentOS, Rocky Linux, and Fedora. As a forensic examiner you will encounter XFS on enterprise servers, database hosts, and high-performance storage systems. This section provides the orientation you need to work with XFS images without mistaking it for ext4.

📌 Scope Note

This module provides a working overview of XFS for forensic examiners. A full XFS deep-dive is beyond the scope of CSC 156. The goal is that you can identify XFS, understand its key differences from ext4, and apply The Sleuth Kit tools correctly when you encounter an XFS image.

What Is XFS?

XFS is a high-performance 64-bit journaling file system originally developed by Silicon Graphics (SGI) in 1993 and contributed to the Linux kernel in 2001. It is designed for large files and high-throughput I/O, making it popular for database servers and NAS appliances.

Feature ext4 XFS
Default on Ubuntu, Debian, most consumer Linux RHEL 7+, CentOS, Rocky Linux, Fedora

⚠️ XFS Has No crtime — Critical for Forensic Reports

Unlike ext4, XFS does not store a file creation time (crtime). On an XFS volume, you have only atime, mtime, and ctime. This means you cannot definitively establish when a file was created from the file system metadata alone on an XFS system. You must rely on other evidence sources — log entries, journal transactions, or application-level timestamps — to establish file creation timelines. Always note this limitation explicitly in your forensic report when examining XFS volumes.

Identifying XFS in the Field

Before mounting or running TSK tools, confirm the file system type. Using the wrong type specification will produce errors or incorrect results.

# Identify file system type from an image file
sudo blkid /path/to/image.img
# Output example: /path/to/image.img: UUID="..." TYPE="xfs"

# Alternative: use the file command on the raw image
file /path/to/image.img
# Output: SGI XFS filesystem data (blksz 4096, ...)

# Mount XFS image read-only for examination
sudo mount -o ro,noexec,nosuid,noatime -t xfs /path/to/image.img /mnt/evidence

🔗 TSK Works on Both ext4 and XFS

The Sleuth Kit commands (fls, icat, ils, istat) work on XFS images. Specify the file system type explicitly with the -f xfs flag. The output format and field meanings are consistent across file system types, making it straightforward to apply the same workflow you learn for ext4 to XFS volumes.

Linux Timestamps Deep Dive

Timestamps are among the most powerful — and most misunderstood — pieces of evidence in Linux forensics. Every file on an ext4 volume carries up to four timestamps in its inode. Understanding precisely what each one records, what updates it, and how it can be manipulated is essential before you draw any timeline conclusions.

atime
Access Time
Records the last time the file's content was read. Updated when a process opens and reads the file.
Forensic use: Indicates when a file was last accessed — but reliability is severely compromised by noatime mount option and relatime kernel default.
mtime
Modification Time
Records the last time the file's content was modified. Updated when data is written to the file.
Forensic use: Most reliable indicator of when a file was last written. Attackers may manipulate with touch -m (timestomping).
ctime
Change Time
Records the last time the inode metadata changed: permissions, ownership, link count, or any timestamp update.
Forensic use: Cannot be set by a normal user with touch — only the kernel updates ctime. Harder to fake than mtime; often reveals when an attacker modified file attributes.
crtime
Creation Time
ext4 only. Records when the file was first created. Stored in extra inode space; not exposed by standard stat.
Forensic use: Definitive creation evidence. Access via debugfs or istat (TSK). Not available on ext2, ext3, or XFS.

💡 "Not Allocated" is Good News for Recovery

When istat reports Not Allocated and num of links: 0, the file has been deleted — but the inode still contains block pointers. As long as those blocks have not been overwritten by new data, you can recover the file's content with icat. This is the foundation of metadata-based deleted file recovery.

The noatime Mount Option — Impact on Forensic Evidence

By default, Linux updates a file's atime every time it is read. On a busy server, this creates enormous write overhead (a read becomes a write to update the inode). For this reason, most production Linux servers are mounted with noatime or relatime.

Mount Option Behavior Forensic Impact
atime (default) atime updated on every file read atime is reliable evidence of last access — but rarely used in production due to performance cost
noatime atime is never updated atime is frozen at the time of last write — completely unreliable as access evidence; common on servers and SSDs
relatime atime updated only if it is older than mtime or ctime, or more than 24 hours old atime provides approximate access evidence (within ~24 hours); the Linux kernel default since 2.6.30
strictatime atime updated on every read (explicit override) atime is fully reliable — but almost never used in production

⚠️ Always Check Mount Options Before Interpreting atime

Before drawing any conclusions from atime evidence, verify the mount options used on the target system. Check /etc/fstab on the image and look for noatime or relatime in the options column. On web-prod-03, you should check /etc/fstab early in your examination. If noatime is present, discard atime as access evidence entirely and focus on mtime, ctime, and crtime for your timeline.

🧠 Timestamp Scenario — Think It Through

You find a file /tmp/.x11-unix/proc on the web-prod-03 image with these timestamps:

  • crtime: 2026-03-08 03:19:02 UTC
  • mtime: 2026-02-14 09:00:00 UTC
  • ctime: 2026-03-08 03:19:02 UTC
  • atime: 2026-02-14 09:00:00 UTC

The mtime and atime match a date before the crtime. The ctime matches the crtime. What does this pattern suggest?

Analysis: The file was created on 2026-03-08 (crtime), but its mtime was set to 2026-02-14 — a date in the past. This is a classic timestomping pattern: an attacker used touch -m -t 202602140900 /tmp/.x11-unix/proc to make the file appear older than it is. The ctime, however, cannot be set by a normal user — it reflects the true moment the inode was last changed (2026-03-08), which matches crtime and exposes the manipulation. This file warrants immediate investigation.

Deleted File Recovery

When an attacker deletes a file on Linux, they typically call rm, which invokes the unlink() system call. Understanding exactly what unlink() does — and does not do — is the foundation of deleted file recovery on ext4.

What unlink() Actually Does

1
Directory entry is removed
The filename-to-inode mapping in the parent directory is deleted. The filename is gone. The inode number it pointed to is no longer reachable through normal directory traversal.
2
Inode link count is decremented
The i_links_count field in the inode is reduced by 1. If it reaches 0 (no remaining hard links), the inode is marked as free in the inode bitmap — but the inode structure itself is not immediately zeroed.
3
Data blocks are marked free in the block bitmap
The blocks that held the file's content are marked as available for reuse in the block group's block bitmap. The actual data in those blocks is not overwritten — it remains on disk until another file is written to those blocks.
4
Inode block pointers may be cleared (ext4 behavior)
On ext4, the kernel may zero the block pointers in the inode at deletion time (unlike ext3, which left them intact). This means metadata-based recovery is harder on ext4 — you may need to fall back to file carving if block pointers have been cleared.

⚠️ ext4 Block Pointer Clearing

This is a critical difference from ext3. On ext3, deleted inodes typically retained their block pointers, making metadata-based recovery straightforward. On ext4, the kernel often zeroes the extent tree or block pointers upon deletion. If istat shows an unallocated inode with no block pointers, the inode's pointers have been cleared and you must use file carving to recover the content from unallocated space. Always attempt istat first — if block pointers remain, use icat; if not, use Scalpel or Foremost.

The Sleuth Kit Workflow: fls → istat → icat

The Sleuth Kit (TSK) provides a command-line toolkit for analyzing forensic images at the file system layer. The three commands you will use most for deleted file recovery are fls, istat, and icat. They are designed to work together in a pipeline.

fls (file listing) lists directory contents from a forensic image, including files that have been deleted but whose directory entries or inode records remain. Deleted entries are prefixed with an asterisk (*).

# List all files (including deleted) in /var/www/html (inode 786433)
fls -f ext4 -r -d /path/to/web-prod-03.img 786433

r/r 2883591: index.php
r/r 2883592: config.php
* r/r 2883600: .cache.php
* r/r 2883601: update.sh

# Lines starting with * are deleted files
# The number after r/r is the inode number — use with istat and icat

# Common flags:
# -r recursive (traverse subdirectories)
# -d show only deleted files
# -l long format (includes timestamps)
# -p display full path

In this example, .cache.php and update.sh were deleted from the web root. Their inode numbers (2883600 and 2883601) are still visible. You would next use istat to examine their metadata and icat to recover their content.

ils (inode listing) lists inodes that are currently unallocated (freed) in the file system. Unlike fls, which traverses directory trees, ils works directly from the inode bitmap — it can find deleted files even when the parent directory entry has been overwritten.

# List all unallocated inodes in the image
ils -f ext4 /path/to/web-prod-03.img

💡 ils vs fls — When to Use Each

Use fls when you want to browse a specific directory and find deleted entries within it. Use ils when you want a comprehensive list of all unallocated inodes across the entire volume — useful for large-scale deleted file surveys or when directory entries have been destroyed. Both outputs can be fed into TSK's mactime tool for timeline creation.

icat (inode cat) extracts the data content of a file identified by its inode number, writing it to stdout. It works on both allocated and unallocated inodes — as long as the inode still contains valid block pointers.

# Recover deleted file .cache.php (inode 2883600) to a file
icat -f ext4 /path/to/web-prod-03.img 2883600 > /evidence/recovered/.cache.php

# Verify the recovered file is not empty
file /evidence/recovered/.cache.php
recovered/.cache.php: PHP script, ASCII text

# Compute hash of recovered file for chain of custody
sha256sum /evidence/recovered/.cache.php

# If icat returns empty output, block pointers were cleared
# → fall back to file carving with Scalpel or Foremost

✅ Always Hash Recovered Files

Every file you recover with icat should be immediately hashed with sha256sum and the hash recorded in your case notes. This documents the integrity of the recovered file at the moment of extraction and is essential for chain of custody if the evidence is used in legal proceedings.

File Carving — When Metadata Is Gone

When inode block pointers have been cleared (common on ext4) or when directory entries have been overwritten, metadata-based recovery fails. File carving is the fallback technique: it scans raw disk data looking for known file-type signatures (headers and footers) and extracts matching byte sequences from unallocated space — without relying on any file system metadata.

📌 File Carving Reviewed from Module 5.5

You covered the principles of file carving in Module 5.5 (Week 5): file signatures (magic bytes), header/footer carving, and the challenges posed by fragmented files. This section focuses on applying those principles specifically to Linux images using Scalpel and Foremost.

Scalpel is a fast, open-source file carver based on the original Foremost design. It reads a configuration file (scalpel.conf) that defines file types by their header and footer byte sequences, then carves matching sequences from the raw image.

Key configuration concepts:

  • Each line in scalpel.conf defines: extension case-sensitivity max-size header footer
  • Uncomment the file types you want to carve; comment out those you do not need (reduces noise and processing time)
  • Set max-size conservatively — too large wastes time and disk space on false positives
# Example scalpel.conf entries (uncommented = active)
php y 500000 \x3c\x3f\x70\x68\x70 \x3f\x3e
sh y 100000 \x23\x21\x2f\x62\x69\x6e \x0a
jpg y 10000000 \xff\xd8\xff\xe0 \xff\xd9

# Run Scalpel against the image
scalpel -c /etc/scalpel/scalpel.conf -o /evidence/scalpel-output/ /path/to/web-prod-03.img

# Output directory will contain subdirectories per file type
# e.g., /evidence/scalpel-output/php-0-y-500000/ contains carved PHP files
# Review audit.txt in output directory for carving statistics

Foremost was developed by the U.S. Air Force Office of Special Investigations and is the predecessor to Scalpel. It uses the same header/footer carving approach and is available in most Linux forensic distributions. Foremost and Scalpel are functionally similar; the choice between them is largely one of preference and configuration flexibility.

# Carve common file types from the image using built-in signatures
foremost -t php,jpg,pdf,zip -i /path/to/web-prod-03.img -o /evidence/foremost-output/

# Carve all supported types (slower, more output)
foremost -t all -i /path/to/web-prod-03.img -o /evidence/foremost-output-all/

# Review output/audit.txt for carving results and statistics

⚠️ File Carving Limitations

File carving recovers content but no metadata — recovered files will have no original filename, timestamp, permissions, or ownership. Fragmented files (split across non-contiguous blocks) may be partially recovered or corrupted. Always verify carved files: check that they open correctly, compute their hash, and document them as "carved from unallocated space" in your report — not as files with verified provenance.

✅ Complete Deleted File Recovery Decision Workflow

  1. Run fls -d on the target directory to find deleted file entries and their inode numbers.
  2. Run istat on each deleted inode to check: Is it unallocated? Does it have block pointers? What are its timestamps?
  3. If block pointers exist → use icat to extract the file content. Hash the output immediately.
  4. If block pointers are cleared → use ils to survey all unallocated inodes for any with surviving pointers, then fall back to Scalpel or Foremost for carving from unallocated space.
  5. Run Scalpel or Foremost against the full image (or extracted unallocated space) to recover files by signature.
  6. Document everything: inode numbers, recovery method, output hashes, and any limitations noted (e.g., partial recovery, no metadata).
