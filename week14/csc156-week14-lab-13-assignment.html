1 · Scenario: The Meridian Financial Data Breach

📈 Scenario — Meridian Financial Services

Meridian Financial Services, a mid-size investment advisory firm headquartered in Sacramento, CA, detected anomalous outbound data transfers on March 15, 2026. The Security Operations Center (SOC) escalated the alert and engaged your forensic team on March 16, 2026.

Over the following three weeks your team executed a textbook investigation: write-blocked imaging of three workstations and one file server, volatile memory captures, network packet captures from the perimeter firewall, and malware triage of suspicious executables. Technical analysis is now complete.

The Chief Legal Officer has requested a formal forensic report suitable for potential use in civil litigation and regulatory proceedings. Your task in this lab is to produce that report using the documentation standards (Module 14.1), report structure (Module 14.2), and QA checklist (Module 14.4) covered this week.

📚 Background Reading

Before starting this lab, you should have completed Module 14.1 (Documentation Standards), Module 14.2 (Anatomy of a Professional Forensic Report), and Module 14.4 (Courtroom Testimony, QA, and Report Finalization). These modules provide the standards and templates you will apply here.

Upon successful completion of this lab, you will be able to:

  1. Review the evidence tables in Section 3 above. Confirm you understand each finding and its relationship to the overall breach timeline.

  4. Check language objectivity. Review each finding description. Flag any language that expresses opinion, speculation, or subjective judgment. Rewrite if necessary using third-person, factual language.
💡 Hint: Building a Chronological Timeline

Start with the earliest event and work forward. For the Meridian case, the timeline begins with the malware compile date (F-06, 2026-03-09) and ends with the SOC alert on 2026-03-15. Key milestones include: malware compilation → domain registration → persistence established (Registry) → data staging (file creation) → file server access → exfiltration → file deletion → SOC detection. Refer to Module 14.1, Section 3 for timestamp formatting standards.

List all key events in UTC chronological order using the format specified above. Include exhibit references for every entry.

5 · Phase 2 – Draft the Report

✍️ Phase 2 Goal

Complete each of the 10 standard report sections defined in Module 14.2. Draft your content directly in the text areas below. Write in objective, third-person language appropriate for a legal audience. Reference exhibits by number and include hash values where applicable.

⚠️ Important Reminders

  • Use objective, factual language throughout. Avoid first-person pronouns (e.g., write "The examiner" not "I").
  • Clearly distinguish facts (what the evidence shows) from opinions/conclusions (what the examiner interprets).
  • Reference all evidence by exhibit number and hash value.
  • All timestamps must be in UTC format.
  • The Executive Summary should be written last, even though it appears as Section 3 of the report.
💡 Hint: General Report Writing Tips

Refer to Module 14.2, Section 3 for writing standards. Key principles: (1) Write for a non-technical reader—define technical terms on first use or include a glossary in the appendices. (2) Present negative findings (what was NOT found) when relevant. (3) Each finding should answer: What was found? Where? When? How was it found? Why does it matter? (4) Use tables, timelines, and structured formatting to improve readability.

Report Section 1: Title Page

Include: report title, case number (MFS-2026-0042), examiner name, examiner affiliation, date of report, classification/confidentiality marking, and version number.

Report Section 2: Table of Contents

List all report sections and subsections with page number placeholders. This will be finalized when the report is formatted.

Report Section 3: Executive Summary

1–2 page overview of key findings and conclusions written for a non-technical audience (management, legal counsel, jury). Summarize: what was examined, what was found, and what it means. Write this section LAST after completing all other sections.

💡 Hint: Executive Summary Best Practices

The executive summary should answer three questions: (1) What were we asked to do? (2) What did we find? (3) What does it mean? Keep it to 1–2 paragraphs. Avoid technical details—save those for the Findings section. Write this section last so it accurately reflects the complete report. See Module 14.2, Section 2 for the "write last, place first" principle.

Report Section 4: Scope and Objectives

Describe: what was requested by the client/counsel, what evidence items were examined, what investigative questions were addressed, and any limitations or constraints on the examination.

Report Section 5: Evidence Description

For each evidence item: physical description, serial number/identifier, hash values (acquisition and verification), chain of custody reference, and condition upon receipt. Reference the evidence table in Section 3 of this lab.

💡 Hint: Evidence Description Standards

For each item, include: (1) Exhibit number, (2) Physical description (make, model, capacity), (3) Serial number or unique identifier, (4) Condition upon receipt, (5) How and when it was acquired (tool, write-blocker, date/time UTC), (6) Hash values (both acquisition and verification), (7) Chain of custody log reference. Refer to Module 14.1, Section 2 for documentation taxonomy and Week 6 acquisition standards.

Report Section 6: Tools and Methodology

List all forensic tools (name, version, purpose), hardware used, and describe the forensic methodology/process followed. Reference accepted standards (e.g., NIST SP 800-86, SWGDE). Explain the examination workflow.

Report Section 7: Findings

Present each finding organized by investigative question or evidence item. For each finding include: what was found, where (exhibit #, file path, offset), when (UTC timestamp), how it was found (tool/method), and supporting hash values. Use objective, factual language. Include negative findings where relevant. Reference findings F-01 through F-06 from Section 3.

💡 Hint: Writing Effective Findings

Each finding should follow a consistent structure: (1) Finding number and title, (2) Evidence source (exhibit number), (3) Tool used for discovery, (4) Factual description of what was observed, (5) Supporting data (timestamps, hashes, file paths), (6) Forensic significance. Remember to document negative findings—what you did NOT find can be as important as what you did find. Avoid conclusions in this section; save interpretation for Section 8. See Module 14.2, Section 3.

Report Section 8: Analysis and Interpretation

Explain what the findings mean in the context of the investigation. Connect individual findings into a coherent narrative. Clearly label any examiner opinions or interpretations and distinguish them from established facts. Address each investigative question from the Scope section.

💡 Hint: Analysis vs. Findings

The Findings section states what the evidence shows. The Analysis section explains what it means. For example, Finding: "238 MB of outbound TLS traffic was transmitted to IP 185.220.101.34." Analysis: "The volume of outbound traffic (238 MB) is consistent with the size of the staged archive (245 MB, pre-compression), suggesting the client PII data was successfully exfiltrated." Always use qualifying language for opinions: "is consistent with," "suggests," "supports the conclusion that." Avoid definitive statements unless the evidence is unambiguous.

Report Section 9: Conclusions

Summarize your opinions based on the findings and analysis. Address each investigative question directly. Clearly distinguish conclusions (supported opinions) from facts. Note any limitations or areas requiring further investigation.

Report Section 10: Appendices

Include supporting materials: hash verification logs, chain of custody forms (reference by form number), tool output excerpts, timeline exports, annotated screenshots, and a glossary of technical terms. List each appendix with a letter designation (Appendix A, B, C, etc.).

💡 Hint: What Goes in the Appendices

Appendices contain supporting evidence that is too detailed for the main report body but essential for verification and legal defensibility. At minimum, include: (1) Hash logs proving image integrity, (2) Chain of custody documentation, (3) Complete timeline, (4) Representative tool output (not every log line—selected excerpts that support your findings), (5) A glossary defining every technical term used in the report. For this lab, you may describe what each appendix would contain rather than producing full tool output. See Module 14.2, Section 2 for appendix standards.
