1. The Integration Challenge

Siloed Analysis vs. Integrated Investigation

Most forensic training (including this course) teaches skills one domain at a time: Week 7 covered Windows Registry, Week 11 covered memory, Week 12 covered network captures. This is pedagogically necessary—but it creates a dangerous habit:

The Silo Trap

Siloed analysis means examining each evidence source independently and reporting findings as disconnected lists. The result is a report that says "the disk shows X" and "the network shows Y" without ever explaining how X and Y relate to the same attacker action at the same moment in time.

Integrated investigation treats all evidence sources as fragments of a single story. The examiner's job is reconstruction—assembling those fragments into a narrative that answers: Who did what, when, how, and what was the impact?

Why Corroboration Matters

In legal and organizational proceedings, findings supported by multiple independent evidence sources carry dramatically more weight than single-source findings. Consider:

  • Single source: "A file named exfil.zip was created on the suspect's workstation at 14:32 UTC." — Useful, but an opposing expert could argue timestamp manipulation.
  • Corroborated: "A file named exfil.zip was created at 14:32 UTC (NTFS $MFT), the same file's hash appears in an outbound FTP transfer at 14:33 UTC (PCAP), and the firewall log shows an ALLOW decision for that FTP session at 14:33 UTC." — Three independent sources confirm the same event. Timestamp manipulation across all three simultaneously is implausible.

Integration transforms individual artifacts into evidence chains that are far more defensible in court, more convincing to executives, and more resistant to alternative explanations.

The Cost of Failing to Integrate — Real-World Consequences
  • Missed connections: A malware indicator in memory may only make sense when correlated with a phishing email artifact on disk and a DNS query in the network log. Examined in isolation, each artifact might seem benign.
  • Contradictory findings: Without integration, your report might inadvertently contain findings from one source that contradict findings from another—destroying credibility.
  • Incomplete timelines: No single evidence source captures every action. Disk images miss network-only activity; PCAPs miss local file operations. Only integration reveals the complete sequence.
  • Vulnerability to cross-examination: An opposing expert will ask "Did you verify this finding against any other evidence source?" If the answer is "no," the finding's weight diminishes.

2. Building a Unified Timeline

The unified timeline is the primary deliverable of evidence integration. It arranges events from all evidence sources into a single chronological sequence, showing how actions on one system or network correspond to actions elsewhere.

Four-Step Methodology

Step 1: Extract Domain-Specific Timelines

Before you can integrate, you need clean per-source timelines. For each evidence source in your investigation:

  1. Identify all timestamped artifacts (file MAC times, log entries, packet timestamps, call records, etc.).
  2. Export them into a structured format (spreadsheet, CSV, or tool-native timeline).
  3. Record the time source for each artifact: which clock generated this timestamp?
  4. Note the timestamp resolution: seconds? milliseconds? minutes only?

Tools that help: Plaso/log2timeline (disk images), Wireshark export (PCAPs), Volatility timeliner plugin (memory), manual extraction from mobile reports.

Example — Extracting from a Disk Image

# Using log2timeline/plaso to generate a super-timeline from a disk image
log2timeline.py --storage-file case42.plaso /evidence/disk.E01

# Filter to the relevant time window
psort.py --output-time-zone UTC \
  -w case42_timeline.csv \
  case42.plaso "date > '2026-03-15 02:00:00' AND date < '2026-03-15 03:00:00'"
Step 2: Normalize Timestamps

Different evidence sources often use different time zones, different clock sources, and different levels of precision. Before merging, you must normalize:

  1. Convert all timestamps to a single reference timezone — UTC is the forensic standard.
  2. Document each source's original timezone in your case notes.
  3. Assess clock accuracy — Was the system clock synchronized via NTP? Was the mobile device on carrier time? Was the PCAP captured by a sensor with a known-good clock?
  4. Calculate and document clock skew if present (see Section 3 below).
Key Principle

Never assume two clocks agree. A 3-minute skew between a workstation clock and a network sensor clock can make events appear out of order—or worse, make corroboration appear to fail when it should succeed.

Step 3: Merge into a Single Chronological Sequence

With normalized timestamps, merge all domain timelines into one master timeline. For each entry, record:

FieldDescription
Timestamp (UTC)Normalized event time
SourceEvidence source (e.g., "Disk Image — workstation01.E01")
DomainForensic domain (Disk, Network, Memory, Mobile, Log)
ArtifactSpecific artifact type (e.g., "$MFT entry", "PCAP frame #4821")
DescriptionWhat happened — factual, objective language
Corroboration StatusCorroborated / Single-source / Contradicted / Inferred (see Section 4)
Confidence NoteAny caveats (clock skew applied, low-resolution timestamp, etc.)

Practical tip: Use a spreadsheet for your master timeline. Sort by timestamp. Color-code rows by domain for visual scanning. This becomes an appendix in your final report.

Step 4: Identify Patterns, Gaps, and Conflicts

Once merged, review the unified timeline for:

  • Corroborated clusters: Multiple sources confirming the same event within a narrow time window. These are your strongest findings.
  • Temporal gaps: Periods where no evidence exists. Document these—they may indicate anti-forensics, evidence destruction, or simply limitations of your sources.
  • Conflicts: Two sources that appear to contradict each other (e.g., a file creation timestamp that precedes the network session that delivered the file). These require the resolution techniques in Section 3.
  • Causal sequences: Events that logically depend on prior events (e.g., malware execution must follow malware delivery). Verify that your timeline preserves causal ordering.

Worked Example — Unified Timeline (2-Minute Window)

The following sanitized example shows how five evidence sources corroborate a data exfiltration event within a 2-minute window. All timestamps are normalized to UTC.

2026-03-15 02:31:04 UTC Firewall Log

2026-03-15 02:31:05 UTC PCAP (Sensor)

2026-03-15 02:31:12 UTC Disk Image ($MFT)

2026-03-15 02:31:14 UTC Memory Dump

Integration Insight

No single source tells the full story. The firewall log confirms the connection was allowed. The PCAP reveals the credentials and data volume. The disk image shows which file was exfiltrated. Memory confirms the responsible process. The mobile SMS suggests coordination with an accomplice. Together, they form an evidence chain that is extremely difficult to refute.

3. Timestamp Conflict Resolution

When merging timelines from heterogeneous sources, timestamp conflicts are expected, not exceptional. A conflict occurs when two or more sources present information that appears temporally inconsistent. Your job is to determine the cause and document your resolution.

Common Causes of Timestamp Conflicts

CauseDescriptionTypical MagnitudeDetection Method
Clock SkewSystem clock drifted from true time (no NTP sync, dead CMOS battery)Seconds to minutesCompare system clock to NTP server logs or known-accurate source
Timezone MisconfigurationSystem configured for wrong timezone; timestamps recorded in local time without offset indicatorHours (exact multiples)Offset is exactly 1–12 hours; correlate with known events
TimestompingDeliberate modification of file timestamps by attacker (anti-forensics)Variable — often yearsCompare $STANDARD_INFORMATION vs. $FILE_NAME in MFT; check $UsnJrnl
Resolution MismatchOne source records seconds, another only minutes; apparent 0–59 second discrepancyUp to 59 secondsCheck artifact documentation for timestamp granularity
Evidence Source LatencyLog aggregation delay; SIEM ingestion lag; mobile carrier timestamp vs. device timestampSeconds to minutesCompare direct-source timestamps to aggregated timestamps

Decision Framework for Resolving Conflicts

Step-by-Step Resolution Process
  1. Identify the conflict: Two or more sources show the same event at different times, or show a causal sequence in the wrong order.
  2. Determine each source's clock authority:
    • Was the system NTP-synchronized? (Check registry keys on Windows: HKLM\SYSTEM\CurrentControlSet\Services\W32Time; check /var/log/syslog for ntpd on Linux)
    • Was the PCAP sensor using a GPS-synchronized clock?
    • Was the mobile device on automatic carrier time?
  3. Calculate the skew: If you can identify a common event visible in both sources (e.g., the same TCP SYN), measure the time difference. This is your skew offset.
  4. Apply the correction: Adjust the less-authoritative source by the calculated skew. Document the adjustment in your timeline notes.
  5. If no correction is possible: Document the conflict, state which source you consider more authoritative and why, and note the uncertainty range in your findings.

Worked Example — Resolving a 47-Second Skew

A Windows workstation shows a file creation at 02:31:12 UTC. The network sensor (GPS-synchronized) shows the FTP transfer of that file beginning at 02:30:25 UTC. The file cannot have been transferred before it was created—so either the disk timestamp is wrong, or the network timestamp is wrong.

Resolution: The network sensor uses a GPS clock (authoritative). The workstation's last NTP sync was 6 days ago. Checking a known shared event (a DNS query visible in both the workstation's DNS client log and the PCAP), we find the workstation clock is 47 seconds ahead of true time. Applying a −47s correction to the disk timestamp gives 02:30:25 UTC — the file creation and FTP transfer now align correctly.

Documentation: "Workstation clock skew of +47 seconds determined by comparing DNS query for update.microsoft.com at PCAP timestamp 02:28:03 UTC with Windows DNS Client Event ID 3008 at system time 02:28:50. All workstation timestamps adjusted by −47s in this report."

Detecting Timestomping

Timestomping is the deliberate modification of file timestamps to mislead investigators. On NTFS:

  • The $STANDARD_INFORMATION attribute (easily modified by user-level tools like Timestomp) stores the timestamps visible in Windows Explorer.
  • The $FILE_NAME attribute (requires kernel-level access to modify) stores a separate set of timestamps.
  • Detection: If $SI created/modified dates are earlier than $FN created dates, timestomping is indicated — because the $FN created date reflects when the MFT entry was allocated, which cannot legitimately precede the file's creation.
  • Additional check: The $UsnJrnl (Update Sequence Number Journal) records file operations with its own timestamps. A file showing a 2020 $SI timestamp but a 2026 $UsnJrnl entry for its creation is clearly timestomped.

When timestomping is detected, use the $FN and $UsnJrnl timestamps as the authoritative source, and document the discrepancy as evidence of anti-forensic activity.

Report Writing Tip

Always document timestamp conflicts and their resolutions in your report's methodology section. Never silently adjust timestamps. Transparency about uncertainty strengthens—not weakens—your credibility.

4. Corroboration Classification System

Every finding in your integrated timeline should be classified by its level of evidentiary support. This classification drives how you present findings in your report and how much weight they should carry in conclusions.

ClassificationDefinitionMinimum CriteriaReport Language Guidance
Corroborated Finding confirmed by two or more independent evidence sources within a consistent time window ≥2 sources, same event, timestamps consistent after normalization "The evidence establishes that…" / "Multiple sources confirm…"
Single-Source Finding supported by only one evidence source; no contradicting evidence exists 1 source only; no conflicting data from other sources "Analysis of [source] indicates that…" / "Based on [artifact]…"
Contradicted Finding from one source is contradicted by another source; conflict unresolved 2+ sources with irreconcilable information after resolution attempts "[Source A] indicates X; however, [Source B] indicates Y. This discrepancy could not be resolved because…"
Inferred Finding not directly observed in any source but logically deduced from other established facts Logical necessity based on corroborated findings + documented reasoning "Based on the established sequence of events, it can be inferred that…" / "The evidence is consistent with…"

Applying the Classification — Guidance

When to Use Each Classification

Corroborated — Your Strongest Findings

  • Use for key findings that support your conclusions.
  • In the worked timeline above, the FTP exfiltration is corroborated: firewall log + PCAP + disk artifact + memory artifact all confirm the same event.
  • These findings can support definitive language in your conclusions.

Single-Source — Valid but Weaker

  • Common for artifacts where only one evidence type captures the activity (e.g., a registry key change visible only in the disk image).
  • Still reportable and valuable—but acknowledge the limitation.
  • Ask yourself: "Is there another source that should show this event? If so, why doesn't it?" Document the answer.

Contradicted — Requires Transparency

  • Do not hide contradictions. They damage credibility if discovered by opposing counsel.
  • Present both sides, explain your resolution attempt, and state which source you find more reliable and why.
  • If you cannot resolve the contradiction, say so explicitly.

Inferred — Use Sparingly and Label Clearly

  • Inference is necessary when evidence is incomplete (it almost always is).
  • Always state the logical basis: "Because A is established and B is established, C necessarily follows."
  • Never present inferences as directly observed facts.
🎯 Practice: Classify the Finding

For each scenario below, select the appropriate corroboration classification.

Scenario A: Your disk image shows a file backdoor.exe created at 14:05 UTC. Your PCAP shows an HTTP download of a file with the same SHA-256 hash at 14:04 UTC from a known-malicious IP. Your IDS log shows an alert for "known malware download" at 14:04 UTC from the same IP.
Scenario B: A Windows Prefetch file shows that wce.exe (a credential-dumping tool) was executed at 15:22 UTC. No other evidence source (PCAP, memory, logs) contains any reference to this tool or activity at that time.
Scenario C: The suspect's mobile phone shows no outbound communication between 02:00–03:00 UTC. However, the corporate proxy log shows web browsing from the suspect's authenticated account during that same window. The suspect claims they were asleep and their phone was off.
Scenario D: You have established (corroborated) that the attacker gained RDP access at 01:15 UTC and that sensitive files were exfiltrated at 02:31 UTC. Between those times, a credential-dumping tool was executed (single-source: Prefetch only). You conclude the attacker used the stolen credentials to access the file server where the exfiltrated files resided — but no direct evidence of this lateral movement exists.
