üîé Hardware Identification in Practice

Accurate hardware identification is the first step in any forensic examination. It informs your entire strategy, from tool selection to expected artifacts.

Identifying Key Components

Here are practical methods and tools to identify the major hardware components you learned about in Module 4.1:

üñ•Ô∏è CPU (Central Processing Unit)
Identify model, architecture, core count, and virtualization support.
Practical Identification:
  • Windows: Task Manager (Performance tab), System Information (`msinfo32`), Device Manager. Tools like `CPU-Z` provide comprehensive details.
  • Linux: `lscpu`, `cat /proc/cpuinfo`.
  • Physical: CPU model number is often etched on the CPU heat spreader (requires removing cooler).
# Linux example
lscpu

# Windows (run in cmd/powershell)
wmic cpu get Name,NumberOfCores,NumberOfLogicalProcessors,VirtualizationFirmwareEnabled
Forensic Relevance Reminder:

CPU architecture (x86/x64/ARM) affects memory forensics tools and malware analysis. CPU cache contains highly volatile data (Module 4.1).

üíæ RAM (Random Access Memory)
Determine type (DDR3/4/5), size, speed, and configuration.
Practical Identification:
  • Windows: Task Manager (Performance > Memory), System Information (`msinfo32`). Tools like `HWiNFO` provide slot-by-slot details.
  • Linux: `sudo dmidecode --type memory`, `free -h`.
  • Physical: RAM sticks have labels indicating type (DDR4), speed (3200MHz), and size (8GB).
# Linux example
sudo dmidecode --type memory | grep -E "Size:|Type:|Speed:"

# Windows (run in cmd/powershell)
wmic ComputerSystem get TotalPhysicalMemory
wmic MemoryChip get BankLabel,Capacity,MemoryType,Speed,Tag
Forensic Relevance Reminder:

RAM holds critical volatile evidence (encryption keys, running processes) that is lost on power loss (Module 4.1).

üîå Motherboard & Chipset
Identify manufacturer, model, chipset, and BIOS/UEFI version.
Practical Identification:
  • Windows: System Information (`msinfo32`), `wmic baseboard get product,manufacturer`. Tools like `CPU-Z` or `HWiNFO` are excellent.
  • Linux: `sudo dmidecode -t baseboard`, `sudo dmidecode -t system`.
  • Physical: Manufacturer and model are usually printed prominently on the motherboard itself.
# Linux example
sudo dmidecode -t baseboard | grep -E "Manufacturer|Product"

# Windows (run in cmd/powershell)
wmic baseboard get Manufacturer,Product,SerialNumber
Forensic Relevance Reminder:

Motherboard firmware (BIOS/UEFI) stores boot order, system time, and can be home to rootkits (Module 4.1, 4.4).

üåê NIC (Network Interface Card)
Locate MAC address, type (wired/wireless), and identify manufacturer.
Practical Identification:
  • Windows: `ipconfig /all` in Command Prompt, Network Connections details.
  • Linux: `ip a` or `ifconfig -a` in terminal.
  • Physical: MAC address is often on a sticker on the NIC itself or the Ethernet port.
# Linux example
ip a | grep "link/ether"

# Windows (run in cmd/powershell)
ipconfig /all | findstr "Physical Address"
Forensic Relevance Reminder:

MAC addresses uniquely identify devices in network logs (Module 4.1). USB Wi-Fi adapters leave connection artifacts (Module 4.3).

üíø Storage Devices
Identify type (HDD/SSD), interface (SATA/NVMe/USB), capacity, model, serial.
Practical Identification:
  • Windows: Disk Management (`diskmgmt.msc`), Device Manager, `wmic diskdrive get model,size,serialnumber`. Tools like `CrystalDiskInfo` provide detailed SMART data.
  • Linux: `lsblk`, `fdisk -l`, `smartctl -a /dev/sda`.
  • Physical: Drives have labels with make, model, serial number, and capacity. Interface type is visible (SATA connectors, M.2 slot).
# Linux example
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT,FSTYPE

# Windows (run in cmd/powershell)
wmic diskdrive get Model,SerialNumber,Size
Forensic Relevance Reminder:

HDD vs. SSD impacts deleted file recovery (TRIM, garbage collection). RAID requires special documentation (Modules 4.2, 4.3).

Safety First: Physical Inspection

When physically inspecting hardware, always ensure the system is **powered off and unplugged**. Discharge static electricity before touching components. üîÑ Boot Process Analysis in Practice

Analyzing how a system boots provides critical insights into its configuration, potential tampering, and installed operating systems.

Accessing & Documenting BIOS/UEFI Settings

The firmware settings are a treasure trove of information. Accessing them requires precise timing during boot.

‚öôÔ∏è Step-by-Step: Accessing BIOS/UEFI ‚ñº
  1. Power On: Start the computer.
  2. Press Hotkey: Immediately and repeatedly press the designated hotkey (e.g., Del, F2, F10, F12, Esc). The exact key varies by manufacturer.
  3. Navigate: Once in the BIOS/UEFI setup utility, use arrow keys (BIOS) or mouse (UEFI) to navigate.
  4. Document: Photograph every screen or carefully transcribe all relevant settings. Focus on:
    • Firmware type (BIOS/UEFI)
    • Firmware version and date
    • System date/time (compare to actual time)
    • Boot order/priority
    • Secure Boot status (for UEFI)
    • Any passwords or security settings
    • Fast Boot/Quick Boot settings
  5. Exit Safely: Always exit without saving changes unless explicitly instructed to do so in a controlled lab environment.
‚ö†Ô∏è
WARNING: Do NOT Change Settings on Suspect Systems!

On a live forensic case, you **MUST NOT** change any BIOS/UEFI settings. Only view and document. Changing settings modifies evidence and can lead to inadmissibility.

Analyzing Partition Structure (MBR vs. GPT)

The partition structure dictates how the disk is logically organized and where file systems reside. Identifying MBR or GPT is fundamental.

ü™ü Windows: `diskpart`
Command-line utility to manage disk partitions.
Usage:

Run `diskpart` in an elevated Command Prompt or PowerShell.

C:\> diskpart
DISKPART> list disk

# Look for 'Gpt' column. An asterisk (*) means GPT.
# If no asterisk, it's MBR.

DISKPART> select disk [disk number]
DISKPART> list partition

# Document partition types (Primary, Extended, Logical, System, Recovery)
# Identify EFI System Partition (ESP) if present (indicates UEFI/GPT)
Forensic Relevance:

Identifies partitioning scheme (MBR/GPT), lists all visible partitions, and helps locate the ESP.

üêß Linux: `fdisk` / `gdisk`
`fdisk` for MBR, `gdisk` for GPT partitioning.
Usage:

Run these commands in a terminal, typically with `sudo`.

# To list all disks and partition types (MBR/GPT)
sudo fdisk -l

# If it reports "Disklabel type: gpt", use gdisk
sudo gdisk -l /dev/sda

# Otherwise, use fdisk for MBR
sudo fdisk -l /dev/sda

# Document partition types, sizes, start/end sectors.
# Look for EFI System Partition (type EF00 in gdisk).
Forensic Relevance:

Essential for understanding disk layout, identifying system partitions, and detecting hidden or deleted partitions (Module 4.4).

Identifying the Bootloader

The bootloader is the program that loads the operating system. Its location and type are key forensic indicators.

  • MBR Systems: The bootloader is typically found in the bootstrap code of the MBR (e.g., GRUB for Linux, Windows Boot Manager for Windows 7/XP).
  • UEFI/GPT Systems: Bootloaders are files located in the EFI System Partition (ESP), typically in directories like \EFI\Microsoft\Boot\bootmgfw.efi (Windows) or \EFI\ubuntu\grubx64.efi (Ubuntu Linux).
üîç
Example: Multi-Boot System Artifacts

If a system has both Windows and Linux installed (a dual-boot setup), a UEFI system would have bootloader entries for both operating systems within its ESP. An MBR system would have a bootloader (like GRUB) that can then chain-load to the Windows bootloader. Documenting these entries helps understand the system's history and user's intent.

üíæ Storage Technology Application

Your understanding of HDD and SSD technologies (Module 4.2) and storage interfaces (Module 4.3) directly impacts your evidence collection strategy.

HDD vs. SSD: Acquisition Decisions

The type of drive fundamentally changes your expectations for deleted data recovery and the urgency of your actions.

  • HDD (Hard Disk Drive):
    • Expectation: High likelihood of deleted file recovery (data persists until overwritten).
    • Action: Use a hardware write blocker, create a bit-for-bit image. Time is less critical for deleted data, but prompt collection is always best.
  • SSD (Solid State Drive):
    • Expectation: Low likelihood of deleted file recovery (TRIM and garbage collection erase data quickly).
    • Action: **IMMEDIATE ACQUISITION IS CRITICAL.** Use a hardware write blocker (if available for NVMe) or a validated software write blocker. Focus on existing files and volatile data.

RAID Systems: The Documentation Imperative

RAID (Redundant Array of Independent Disks) configurations require extreme caution. **Documentation is NOT optional‚Äîit is critical.**

  • Before Disassembly:
    • Photograph the physical disk arrangement, cabling, and labels.
    • Access the RAID controller's BIOS/UEFI to record the RAID level (0, 1, 5, 6, 10), stripe size, and disk order.
    • Label each disk clearly (e.g., "RAID 5 Disk 1 of 4").
  • Acquisition: Image *each individual disk* in the array using a hardware write blocker. Never image the logical volume directly without first imaging the physical disks.
  • Reconstruction: Use forensic tools (like X-Ways, R-Studio) to reconstruct the array from the individual disk images using the documented configuration.
‚ö†Ô∏è
RAID Mistake = Permanent Data Loss!

Failure to properly document and acquire RAID disks can lead to irreversible data loss. This is a common and devastating mistake for inexperienced examiners.

Interface-Specific Write Blocking

The storage interface dictates your choice of write blocker (Module 4.3).

  • SATA: Widely supported by all major hardware write blockers (e.g., Tableau T8, WiebeTech UltraBlock). This is the easiest interface for write blocking.
  • NVMe (M.2, U.2, PCIe): Limited hardware write blocker support. You may need specialized NVMe write blockers (e.g., Tableau T9) or rely on validated software write blockers (e.g., FTK Imager's read-only mode in a forensic workstation).
  • USB: Dedicated USB hardware write blockers are common and affordable. Always use one for external USB drives.
üí°
Validation is Key for Software Write Blockers

If you must use a software write blocker (e.g., for NVMe where hardware isn't available), you **must** validate its functionality before casework. Test it on a known good drive, attempt to write, and verify that no changes occurred (e.g., by hashing before and after).

üß™ Lab Activity 4: Preparation Guide

Lab Activity 4 is your opportunity to apply the knowledge from Week 4. Read through this guide carefully to ensure you're well-prepared.

Lab Activity 4 Overview

Objective: Develop practical skills in identifying hardware components, analyzing boot configurations, and understanding their forensic significance.

Total Points: 20 points

Deliverable: A single PDF document (`LastName_FirstName_Lab4.pdf`) containing all three parts with screenshots and forensic observations.

Part 1: Hardware Component Identification (8 points)

Task: Examine a provided computer system (physical or virtual) and document all major hardware components.

  • CPU: Model, specifications (cores, architecture), virtualization features.
    Tools: Task Manager, msinfo32, CPU-Z (Windows); lscpu, /proc/cpuinfo (Linux).
  • RAM: Type (DDR), size (GB), speed (MHz), configuration (number of sticks).
    Tools: Task Manager, msinfo32, HWiNFO (Windows); dmidecode (Linux).
  • Storage Devices: For EACH drive: Type (HDD/SSD), interface (SATA/NVMe/USB), capacity, manufacturer, model, serial number.
    Tools: Disk Management, Device Manager, CrystalDiskInfo (Windows); lsblk, fdisk -l, smartctl (Linux).
  • Motherboard: Manufacturer, model, chipset, BIOS/UEFI version.
    Tools: msinfo32, wmic, HWiNFO (Windows); dmidecode (Linux).
  • NIC: MAC address for each network adapter (wired/wireless).
    Tools: ipconfig /all (Windows); ip a, ifconfig (Linux).
  • **Any additional hardware:** USB devices, expansion cards (e.g., discrete GPU).
üí°
Documentation Tip: For each component, include a screenshot from a software tool showing its details AND a photograph of any physical labels if you have physical access. Explain the **forensic significance** of each piece of information.

Part 2: Boot Process Analysis (8 points)

Task: Access and document BIOS/UEFI settings, and analyze the partition structure and boot sequence.

  • BIOS/UEFI Settings:
    • Access method (e.g., "Pressed F2 repeatedly after power on").
    • Firmware type (BIOS/UEFI), version, date.
    • Secure Boot status (for UEFI).
    • Boot order/priority.
    • System date/time vs. actual time (note any discrepancies).
    • Any passwords or Fast Boot settings.

    Tools: Physical access to system, camera for screenshots.
  • Partition Structure:
    • Identify partitioning scheme (MBR or GPT).
    • Document ALL partitions (type, size, file system).
    • Identify EFI System Partition (ESP) if GPT.

    Tools: `diskpart` (Windows); `fdisk -l`, `gdisk -l` (Linux).
  • Boot Process Analysis:
    • Document the boot sequence from power-on to OS login.
    • Identify the bootloader location and type (e.g., "Windows Boot Manager in MBR" or "GRUB on ESP").
    • Note any unusual boot behavior or multi-boot options.

    Tools: Observation, research.
‚ö†Ô∏è
Virtual Machine Safety: If using a VM, you can safely experiment with changing boot order or partition schemes (e.g., converting MBR to GPT) to observe effects, but always revert to a snapshot or create a new VM for the actual lab submission.

Part 3: Storage Technology Comparison (4 points)

Task: Compare HDD and SSD characteristics from a forensic perspective.

  • Use a comparison table (provided in lab instructions or create your own).
  • For each storage type (HDD & SSD), document:
    • Data persistence after deletion.
    • TRIM support and behavior.
    • Wear leveling impact.
    • Expected deleted file recovery success rate.
    • Key acquisition considerations (e.g., time sensitivity, write blocker type).
  • Conclude by explaining which storage type presents greater forensic challenges and why, drawing on your Week 4 knowledge.

üì∞ Case Study Application: Hardware in Real Investigations

Understanding hardware and boot processes isn't just academic; it's the bedrock of real-world investigations. Here are some examples of how this knowledge is critical:

üîç Case: The USB Data Exfiltration ‚ñº
Background:

A former employee is suspected of stealing company secrets. They deny connecting any external storage devices to their workstation.

Hardware Knowledge Applied:
  • The investigator knew that USB controllers (Module 4.1) and the operating system (Module 4.3) log information about connected USB devices.
  • Analysis of the system's registry (Windows) or log files (Linux) revealed the Vendor ID, Product ID, and serial number of a specific USB flash drive, along with timestamps of its connection.
Impact:

This evidence directly contradicted the suspect's statement, proving the device was connected. The serial number allowed law enforcement to later identify and seize the specific USB drive from the suspect's home, which contained the stolen data.

üí°
Lesson: USB connection artifacts are powerful. Knowing where to find them (via USB controllers) is key.
üîç Case: The Anti-Forensic Boot Attempt ‚ñº
Background:

A suspect's computer was found powered off. Investigators suspected an attempt to boot from a live operating system to wipe the drive or access encrypted data.

Hardware Knowledge Applied:
  • The investigator accessed the system's BIOS settings (Module 4.4) and found that the boot order had been changed to prioritize USB devices and CD-ROM drives over the internal hard drive. This was an unusual configuration for a personal computer.
  • They also noted that "Fast Boot" was disabled, which is sometimes done to allow more time to access boot menus.
Impact:

While no evidence of a successful wipe was found, the altered boot order and disabled Fast Boot were strong indicators of an attempt to boot from external media. This information supported the prosecution's narrative of the suspect's intent to conceal evidence.

üí°
Lesson: BIOS/UEFI settings are crucial for understanding system state and user intent.
üîç Case: The UEFI Rootkit (LoJax) ‚ñº
Background:

A high-profile organization was repeatedly re-infected with sophisticated malware, even after wiping and reinstalling operating systems on compromised machines.

Hardware Knowledge Applied:
  • Traditional disk forensics and OS-level antivirus were ineffective. Forensic experts suspected a low-level infection.
  • They understood that UEFI firmware (Module 4.4) is a persistent storage location that loads *before* the operating system.
  • Specialized firmware analysis tools were used to extract and examine the UEFI/BIOS contents of the motherboards.
Impact:

This analysis revealed the presence of a UEFI rootkit (LoJax) embedded directly in the motherboard's firmware. This malware allowed attackers to maintain persistence regardless of OS reinstallation. Detecting it required deep knowledge of boot processes and firmware architecture.

üí°
Lesson: Don't overlook firmware. Malware can hide in the deepest layers of a system.

üéÆ Interactive Scenarios

Apply your knowledge to practical decision-making in these forensic scenarios.

üéØ Scenario 1: The Encrypted Laptop

You're dispatched to seize a suspect's laptop. You arrive to find the laptop powered on and at the Windows login screen, indicating full-disk encryption (BitLocker) is active. The internal drive is a 1TB NVMe SSD. You have a hardware write blocker for SATA drives, but no NVMe-compatible hardware write blocker. You also have forensic software (e.g., FTK Imager) with software write-blocking capabilities on your forensic workstation.

What is your BEST course of action?

A) Power off the laptop immediately, remove the NVMe SSD, and connect it to your SATA hardware write blocker for imaging.
B) Perform a live acquisition of the entire NVMe SSD while the system is still running and unlocked, using your forensic software's software write blocker.
C) Wait for an NVMe hardware write blocker to become available, then return to the scene to acquire the powered-off SSD.
D) Reboot the laptop to access the UEFI settings and disable BitLocker, then proceed with standard acquisition.

üéØ Scenario 2: The Legacy Desktop

You are examining an older desktop computer from a 2010 fraud case. The system contains a 500GB Western Digital SATA II HDD, was powered off when seized, and shows no signs of encryption. You need to recover deleted financial documents.

What is your BEST approach for acquiring and analyzing this drive?

A) Connect the drive directly to your forensic workstation, copy the visible files, and then search for deleted files using a logical file recovery tool.
B) Connect the drive via a SATA hardware write blocker, create a bit-for-bit forensic image, and then use file carving tools on the image to recover deleted documents.
C) Boot the system to Windows 7, enable "show hidden files," and copy all potentially relevant files to an external drive.
D) Assume deleted files are unrecoverable due to the age of the system and focus only on existing documents.

üéØ Scenario 3: The Corrupted Server

You respond to a server crash where the primary 4TB storage volume is inaccessible. Investigation reveals it was part of a 4-disk RAID 5 array. The server is now off. The IT manager tells you "just pull the drives, they're numbered 1-4."

What is your BEST course of action?

A) Trust the IT manager's numbering, pull the drives, and attempt to reconstruct the RAID 5 array in your lab.
B) Before removing any drives, carefully document the physical layout, cable connections, and any labels. Then, access the RAID controller BIOS (if possible) to confirm the RAID level, disk order, and stripe size. Finally, image each disk individually with a write blocker.
C) Since the server is off, the data is stable. You can simply take images of the drives and worry about the RAID configuration later during analysis.
D) As it's a RAID 5, you only need to image 3 of the 4 drives, as one can fail without data loss.
