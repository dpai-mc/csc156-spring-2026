üîé Part 1: Hardware Component Identification (8 points)

Examine a provided computer system (physical or virtual) and document all major hardware components. For each component, you will identify its specifications and explain its forensic significance.

Instructions:

  1. Obtain access to a computer system (e.g., your personal computer, a lab computer, or a virtual machine).
  2. For each component listed below, use appropriate tools (software and/or physical inspection) to identify its specifications.
  3. Capture **screenshots** of software tool output and/or **photographs** of physical labels.
  4. Document your findings in your lab report, including the forensic significance of each identified detail.

Components to Identify & Document:

  • Example Tools: Task Manager (Windows), `msinfo32` (Windows), `CPU-Z` (Windows), `lscpu` (Linux), `cat /proc/cpuinfo` (Linux).

    **Forensic Significance:** CPU architecture impacts malware analysis and memory forensics tool selection. CPU cache contains volatile data.

  • Example Tools: Task Manager (Windows), `msinfo32` (Windows), `HWiNFO` (Windows), `sudo dmidecode --type memory` (Linux), `free -h` (Linux).

    **Forensic Significance:** RAM holds critical volatile evidence (encryption keys, running processes) that is lost on power loss.

  • Example Tools: Disk Management (Windows), Device Manager (Windows), `CrystalDiskInfo` (Windows), `lsblk` (Linux), `fdisk -l` (Linux), `smartctl` (Linux).

    **Forensic Significance:** Drive type (HDD vs. SSD) impacts deleted file recovery. Interface (SATA/NVMe) determines write blocker needs and acquisition speed.

  • Example Tools: `msinfo32` (Windows), `wmic baseboard get product,manufacturer` (Windows), `CPU-Z` (Windows), `HWiNFO` (Windows), `sudo dmidecode -t baseboard` (Linux).

    **Forensic Significance:** Firmware (BIOS/UEFI) stores boot order, system time, and can be a target for persistent malware (rootkits).

  • Example Tools: `ipconfig /all` (Windows), `ip a` (Linux), `ifconfig -a` (Linux).

    **Forensic Significance:** MAC addresses uniquely identify devices in network logs, helping to reconstruct network activity.

  • **Forensic Significance:** USB devices leave connection artifacts. Specialized cards might indicate specific user activities.

Safety Note: When performing physical inspection, ensure the system is **powered off and unplugged**. Discharge static electricity. For your lab, prioritize software tools and screenshots/photos of labels.

üîÑ Part 2: Boot Process Analysis (8 points)

Access and document the system's BIOS/UEFI settings, analyze the partition structure, and identify the bootloader.

Instructions:

  1. Access the system's BIOS/UEFI setup utility.
  2. Document the relevant settings as specified below.
  3. Analyze the disk's partition structure.
  4. Identify the bootloader.

Elements to Analyze & Document:

  • **Forensic Significance:** Helps identify system age and potential for firmware-based malware.

  • **Forensic Significance:** Altered boot order can indicate attempts to boot from external media (e.g., live OS, malicious USB).

  • **Forensic Significance:** Disabled Secure Boot might suggest attempts to install unauthorized OS or bootloaders.

  • **Forensic Significance:** Time discrepancies can impact timeline reconstruction; may indicate dead CMOS battery or deliberate alteration.

  • Example Tools: `diskpart` (`list disk` command) (Windows), `fdisk -l` or `gdisk -l` (Linux).

    **Forensic Significance:** Dictates how partitions are structured and where bootloaders are located. Affects tool choice for parsing.

  • Example Tools: `diskpart` (`list partition` command) (Windows), `fdisk -l` or `gdisk -l` (Linux).

    **Forensic Significance:** Reveals OS installations, recovery partitions, and potential hidden areas.

  • **Forensic Significance:** Bootloaders can be targeted by malware (bootkits). Identifies the primary boot mechanism.
