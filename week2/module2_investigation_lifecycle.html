1.2 Multiple Framework Acknowledgment

Why Multiple Frameworks Exist

Different organizations have developed investigation frameworks, but they all share the same core principles:

  • NIST (National Institute of Standards and Technology): Four-phase model focused on federal government
  • IOCE (International Organization on Computer Evidence): Six-phase international standard
  • ACPO (Association of Chief Police Officers, UK): Four principles-based approach
  • FBI/DOJ: Law enforcement-specific methodology

Common Thread: All frameworks emphasize:

  • Systematic approach (don't skip steps)
  • Evidence integrity (don't alter original)
  • Documentation (record everything)
  • Legal admissibility (follow accepted procedures)

This Course: We use the six-phase model because it provides the most detailed breakdown of investigation activities, making it easier to learn and apply.

Identification is the process of recognizing that an incident has occurred and determining whether forensic investigation is warranted. This phase answers four critical questions:

What Happened?
  • Incident type (breach, theft, malware, etc.)
  • Initial indicators of compromise
  • When incident was discovered
What Systems Are Affected?
  • Endpoints (workstations, laptops, mobile)
  • Servers and databases
  • Network infrastructure
  • Cloud services
What's the Potential Impact?
  • Data exfiltration
  • Financial loss
  • Regulatory compliance
  • Reputation damage
What Are the Legal Implications?
  • Criminal investigation potential
  • Civil litigation likelihood
  • Regulatory reporting requirements
  • Evidence preservation obligations

Section 3: Phase 2 - Preservation

"The golden rule of digital forensics: Don't change what you're investigating. Every action you take can potentially alter evidence."

3.1 The Five Core Preservation Principles

Click each principle to explore its importance:

1. Minimize Changes to Original Evidence
‚ñº

Why It Matters: Any modification to evidence can be used to challenge its admissibility in court. Even seemingly harmless actions like opening a file can change metadata.

Best Practices:

  • Use write blockers for all storage device access
  • Never boot from suspect hard drives
  • Use forensically sound tools that don't modify evidence
  • Document any unavoidable changes

Example: Opening a Microsoft Word document updates its "last accessed" timestamp, potentially destroying evidence of when the file was actually last used.

2. Document Everything
‚ñº

Why It Matters: Documentation proves you followed proper procedures. In court, if it's not documented, it didn't happen.

What to Document:

  • Date, time, and location of each action
  • Who performed each action
  • Tools and versions used
  • System state and observations
  • Any anomalies or unexpected findings

Documentation Tools: Chain of custody forms, photographs, video recordings, detailed notes, tool output logs.

3. Follow Forensically Sound Methods
‚ñº

Why It Matters: Using validated, accepted methods ensures your evidence will be admissible and your findings defensible.

Forensically Sound Requirements:

  • Use validated forensic tools (commercial or open-source)
  • Follow established standards (NIST, SWGDE)
  • Use repeatable, documented procedures
  • Maintain proper chain of custody

Tool Validation: Before using any tool in casework, test it on known data sets to verify it produces accurate, expected results.

4. Maintain Security and Control
‚ñº

Why It Matters: Evidence must be protected from tampering, unauthorized access, and environmental damage.

Security Measures:

  • Restrict physical access to evidence (locked room, evidence locker)
  • Use evidence bags and tamper-evident seals
  • Log all access to evidence
  • Encrypt forensic images during storage and transfer
  • Backup all evidence images to separate locations

Environmental Protection: Store evidence in climate-controlled environments, protected from static, magnetic fields, and physical damage.

5. Consider Order of Volatility
‚ñº

Why It Matters: Some evidence disappears quickly. RAM contents are lost when a system powers off. Network connections terminate. You must collect volatile data first.

Order of Volatility (collect in this order):

  1. CPU registers, cache (nanoseconds)
  2. RAM contents (seconds to minutes)
  3. Network connections, running processes (seconds to hours)
  4. Hard drive / SSD (persistent until overwritten)
  5. Backup media, printed documents (very stable)

Critical Decision: For a running system, capture RAM and live data BEFORE powering off for disk imaging.

Contains:

  • Routing tables and ARP cache
  • Network shares and mapped drives
  • Open ports and listening services

Collection Commands: netstat, ipconfig, arp -a, route print (Windows); ifconfig, netstat, ss (Linux)

3. Running Processes ‚è±Ô∏è Lost in hours

Contains:

  • List of running programs and services
  • Process IDs and parent-child relationships
  • User accounts and logged-in sessions
  • Scheduled tasks and startup programs

Collection Tools: Task Manager, Process Explorer, pslist, tasklist (Windows); ps, top (Linux)

4. File System / Hard Drive üìÖ Persistent until overwritten

Contains:

  • Active files and directories
  • Deleted files (in unallocated space)
  • File metadata (creation, modification times)
  • System logs and event files
  • Registry (Windows), configuration files

Collection Method: Forensic imaging with write blocker. Data persists until deliberately overwritten or drive fails.

5. Backups / Archives (Least Volatile) üì¶ Very stable

Contains:

  • Backup tapes, external drives
  • Cloud backups and snapshots
  • Printed documents
  • Archived email and documents

Section 4: Phase 3 - Collection/Acquisition

"A forensic image is not just a copy‚Äîit's an exact, bit-for-bit replica that includes deleted files, unallocated space, and hidden data that normal copying misses."

4.1 Forensic Imaging vs. Logical Copying

Understanding the critical difference between forensic imaging and logical copying is essential.

Forensic Imaging (Bit-for-Bit Copy)

What It Is: A complete, sector-by-sector copy of an entire storage device, capturing every bit of data including active files, deleted files, unallocated space, and file slack.

‚úì What's Captured:

  • All active files and directories
  • Deleted files (recoverable from unallocated space)
  • File slack (partial data at end of files)
  • Unallocated disk space
  • System areas (boot sector, partition tables)
  • Hidden data and alternate data streams

Tools: FTK Imager, EnCase, dd, dc3dd, Guymager

Output Formats: E01 (Expert Witness), AFF (Advanced Forensic Format), raw/dd

Court Admissibility: ‚úÖ Accepted as forensically sound evidence

Time Required: Slower - several hours for large drives (500GB = ~4-6 hours)

Logical Copying (File-Level Copy)

What It Is: A file-by-file copy of only active, visible files‚Äîsimilar to drag-and-drop or copy/paste. Does NOT capture deleted files or hidden data.

‚úì What's Captured:

  • Active files currently visible in file system
  • Directory structure
  • Basic file metadata (may be altered during copy)

‚ùå What's MISSED:

  • Deleted files
  • Unallocated space
  • File slack
  • Hidden system areas
  • Some metadata (timestamps may change)

Tools: Windows File Explorer, macOS Finder, robocopy, rsync

Court Admissibility: ‚ùå Generally NOT accepted as forensically sound

Time Required: Faster - minutes to hours depending on file count

‚ö†Ô∏è Critical Rule

For forensic investigations requiring court-admissible evidence, ALWAYS use forensic imaging, never logical copying. Logical copies miss critical evidence and metadata changes during copying can be challenged in court.

4.2 Write Blockers: The Essential Tool

Write blockers are hardware or software devices that allow read-only access to storage devices, preventing any modifications to the original evidence.

Why Write Blockers Are Essential

The Problem: Simply connecting a drive to a computer can cause the operating system to automatically:

  • Write timestamps when files are accessed
  • Create system files (Thumbs.db, .DS_Store)
  • Update the file system journal
  • Run antivirus scans that modify metadata

The Solution: Write blockers intercept write commands and block them while allowing read operations to proceed normally.

Interactive Simulation: Write Blocker in Action

investigator@forensics:~$ sudo fdisk -l
Disk /dev/sdb: 500GB (suspect hard drive connected via write blocker)
investigator@forensics:~$ sudo blockdev --getro /dev/sdb
1 (read-only mode enabled ‚úì)
investigator@forensics:~$ sudo touch /mnt/evidence/testfile.txt
touch: cannot touch '/mnt/evidence/testfile.txt': Read-only file system
‚úì Write blocker successfully prevented modification!
investigator@forensics:~$ sudo dcfldd if=/dev/sdb of=evidence.dd hash=sha256
500000000000 bytes (500 GB) copied
SHA-256 hash: a3c5f8b9d7e4a1c2b8f3e9d4a7c1b5e8...
‚úì Forensic image created successfully with read-only access

Types of Write Blockers:

  • Hardware Write Blockers: Physical devices (Tableau, WiebeTech) - most reliable, trusted by courts
  • Software Write Blockers: Programs that mount drives read-only (less reliable, can be bypassed by OS)

4.3 Hash Verification: Proving Evidence Integrity

Cryptographic hash functions create a unique "digital fingerprint" of data. If even one bit changes, the hash value changes completely. This proves evidence hasn't been altered.

Interactive Hash Calculator

MD5 Hash:
9e107d9d372bb6826bd81d3542a419d6
SHA-256 Hash:
a3c5f8b9d7e4a1c2b8f3e9d4a7c1b5e8f2a9d6c3b7e4f1a8c5d9e2b6f3a7c1b4

Try it: Change even one character in the input above and watch how the entire hash changes completely. This "avalanche effect" makes hashes perfect for detecting any modifications to evidence.

Hash Algorithm Recommendations

  • MD5: Fast but cryptographically broken (collisions possible). Still used for quick verification but NOT for court evidence.
  • SHA-1: Also broken. Avoid for new casework.
  • SHA-256: ‚úÖ Current standard. Cryptographically secure, widely accepted in court.
  • SHA-512: Even stronger than SHA-256, increasingly used for high-security cases.

Best Practice: Compute SHA-256 hash at time of imaging, then verify hash matches when you later mount or analyze the image.

‚ö†Ô∏è Critical Verification Step

If the source hash and image hash DO NOT MATCH, the image is unusable and cannot be used as evidence. Possible causes include:

  • Write blocker failure (original was modified during imaging)
  • Drive read errors or bad sectors
  • Imaging software bug or corruption

Action Required: Re-image from the original and investigate the cause of the mismatch. Never proceed with mismatched hashes.
