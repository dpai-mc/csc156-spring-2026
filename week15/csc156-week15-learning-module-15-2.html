Preparation Forensic Readiness Activities

The Preparation phase happens before an incident occurs. For a forensic examiner, preparation means ensuring you can acquire, preserve, and analyze evidence quickly when the call comes at 3:47 AM.

Key Forensic Activities

  • Evidence source inventory: Document all systems, their logging configurations, and where evidence resides (endpoints, servers, network devices, cloud services).
  • Tool validation and kit readiness: Maintain a forensic workstation with validated tools — write blockers tested, imaging software current, Volatility profiles built for organizational OS versions. Week 4 Week 6
  • Baseline documentation: Capture known-good states — registry hive baselines, normal network traffic patterns, authorized scheduled tasks, expected services. Week 7 Week 9
  • Hash libraries: Maintain known-good hash sets (NSRL) and known-bad hash sets (malware repositories) for rapid triage. Week 5
  • Chain of custody templates: Pre-built documentation forms ready for immediate use. Week 3
  • Legal authority clarification: Understand organizational authority to image systems, capture memory, and monitor network traffic — before an incident. Week 3
MedTech Application

If MedTech had maintained forensic readiness, they would already have: centralized logging enabled on all servers, a pre-built Volatility profile for their Windows Server 2019 environment, and documented legal authority for the forensic examiner to capture memory from production systems without additional approvals.

Detection & Identification How Forensic Engagement Begins

Detection is when the organization first recognizes that something is wrong. The forensic examiner's initial role is to help confirm whether a true incident has occurred and begin scoping its extent.

Network-Based Detection

  • PCAP analysis with Wireshark: Examine captured traffic for C2 beaconing patterns, unusual protocols, data exfiltration signatures. Week 12
  • IDS/Firewall log review: Correlate alerts with network flow data to identify scope of compromise. Week 12
  • DNS query log analysis: Detect domain generation algorithms (DGA) or known malicious domains. Week 12

Host-Based Detection

  • Windows Event Log analysis: Look for failed logon bursts (Event ID 4625), new service installations (7045), cleared logs (1102). Week 7 Week 8
  • Linux log review: Check /var/log/auth.log, /var/log/syslog, and application logs for unauthorized access. Week 9
  • File system anomalies: Recently modified system files, unexpected executables in temp directories. Week 7

Memory-Based Detection

  • Volatility process analysis: Identify suspicious processes, injected code, hidden processes via DKOM. Week 11
  • Network connection enumeration: Use netscan to find active C2 connections in memory. Week 11

Malware IOC Matching

  • Hash comparison: Compare suspicious files against known-malware hash databases. Week 5 Week 13
  • YARA rule scanning: Apply YARA rules to disk and memory to detect malware signatures. Week 13
  • Static analysis indicators: Strings, imports, packer signatures on suspicious binaries. Week 13
MedTech Application

The SOC detected outbound connections via firewall logs Week 12. As the forensic examiner, your first step is to confirm the compromise by checking memory on the database server for active C2 connections Week 11 and correlating with Windows Event Logs for unauthorized authentication events Week 7.

Containment Preserve Evidence Before Containment Actions

⚠️ Critical Principle: Evidence Before Containment

Containment actions (isolating systems, blocking IPs, disabling accounts) can destroy volatile evidence. The forensic examiner's most urgent responsibility is to capture volatile evidence before containment actions alter or destroy it. This is the single most important procedural principle in IR forensics.

Order of Volatility (RFC 3227)

Capture evidence in order from most volatile to least volatile:

  1. CPU registers and cache — lost in microseconds (generally not capturable in practice)
  2. Memory (RAM) — lost on power-off or reboot Week 11
  3. Network state — active connections, routing tables, ARP cache Week 12
  4. Running processes — process list, open files, loaded DLLs Week 11
  5. Disk contents — file system, swap/pagefile Week 6 Week 7
  6. Removable media — USB drives, external storage Week 6
  7. Backup/archival data — tape, off-site backups
🚫 Common Forensic Mistakes During Containment
  • Rebooting a compromised system before capturing memory: Destroys all volatile evidence — running malware, active C2 connections, injected code, decrypted data in RAM. This is the #1 forensic mistake in IR.
  • Pulling the network cable before capturing network state: Loses active connection information and may trigger malware self-destruct routines.
  • Running antivirus scans before imaging: Modifies file timestamps, quarantines evidence files, and alters the file system state.
  • Logging into the system with a domain admin account: Creates authentication artifacts that contaminate evidence and may alert the attacker.
  • Shutting down Linux systems with shutdown instead of capturing memory first: Loses process state, tmpfs contents, and active network connections.

Forensic Acquisition Sequence During Containment

  1. Capture volatile memory using validated tools (FTK Imager, DumpIt, LiME for Linux) Week 11
  2. Document network connections — netstat, active sessions, ARP table Week 12
  3. Capture running process state — process list, open handles, loaded modules Week 11
  4. Implement network isolation (now safe to disconnect)
  5. Create forensic disk image with write blocker Week 6
  6. Verify image integrity with cryptographic hash Week 5
  7. Document chain of custody for all acquired evidence Week 3
MedTech Application

The IR Manager wants to immediately isolate the database server by pulling its network cable. You must intervene: "Before we isolate, I need 10 minutes to capture a memory dump and document active connections. Once we pull that cable, we lose the C2 session data and any decrypted credentials the malware is holding in RAM." After memory capture, you give the go-ahead for network isolation, then proceed with forensic imaging.

Eradication Finding Every Attacker Foothold

Eradication is where the forensic examiner's deep technical analysis directly drives operational decisions. The goal: identify every persistence mechanism, backdoor, and compromised credential so the attacker cannot return after recovery.

The forensic examiner must check all common Windows persistence locations: Week 7 Week 8

  • Registry Run/RunOnce keys: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run and user-level equivalents
  • Scheduled Tasks: C:\Windows\System32\Tasks\ and schtasks entries
  • Windows Services: New or modified service entries (Event ID 7045)
  • Startup folders: %AppData%\Microsoft\Windows\Start Menu\Programs\Startup
  • WMI Event Subscriptions: Persistent WMI consumers that survive reboot
  • DLL search order hijacking: Malicious DLLs in application directories
  • COM object hijacking: Modified CLSID registry entries

Linux systems have their own set of persistence vectors: Week 9

  • Cron jobs: /etc/crontab, /etc/cron.d/, user crontabs (crontab -l)
  • Systemd units: Malicious .service or .timer files in /etc/systemd/system/
  • SSH authorized_keys: Attacker-added public keys in ~/.ssh/authorized_keys
  • SUID/SGID binaries: Legitimate binaries given SUID bit, or attacker binaries with SUID
  • Shell profile scripts: .bashrc, .profile, /etc/profile.d/ modifications
  • LD_PRELOAD / shared library injection: /etc/ld.so.preload or environment variable persistence
  • Modified PAM modules: Backdoored authentication modules
  • Kernel modules: Loadable kernel module rootkits (/etc/modules-load.d/)

Verification Techniques

  • Hash all suspicious files and compare against threat intelligence feeds Week 5 Week 13
  • YARA rule scanning across all affected systems to find additional malware variants Week 13
  • Memory analysis for injected code: Use Volatility malfind plugin to detect process injection Week 11
  • Timeline analysis: Correlate file creation timestamps with known attack timeline to find all dropped files Week 7 Week 8
  • Network IOC sweep: Search all logs for communication with identified C2 infrastructure Week 12

Root Cause Identification

The forensic examiner must determine how the attacker got in (initial access vector) to prevent recurrence:

  • Web server logs: Exploitation of web application vulnerabilities Week 12
  • Email artifacts: Phishing email delivery and user interaction
  • Authentication logs: Brute force, credential stuffing, or stolen credentials Week 7 Week 9
  • Vulnerability correlation: Match exploited software versions with known CVEs
MedTech Application

Your memory analysis Week 11 reveals a process injected into svchost.exe. You extract the injected code, hash it, and match it to a known RAT variant Week 13. Timeline analysis shows the malware was first written to disk 12 days ago via a scheduled task created through a compromised service account. You check all three affected servers for the same persistence mechanism — finding it on the domain controller but not the file server (which was compromised via lateral movement using harvested credentials).

Recovery & Post-Incident Verification and Lessons Learned

During recovery, systems are restored to operational status. The forensic examiner's role shifts to verification:

  • Validate restored systems: Compare restored system state against known-good baselines to confirm no attacker artifacts remain Week 5 Week 7
  • Monitor for re-compromise: Enhanced logging and network monitoring on recovered systems Week 12
  • Verify eradication completeness: Run IOC scans on restored systems; check that all identified persistence mechanisms are absent Week 13
  • Credential verification: Confirm all compromised credentials have been rotated
  • Forensic imaging of recovered state: Capture a new baseline image post-recovery for future comparison Week 6

The forensic report you produce becomes the authoritative record of what happened and drives organizational improvement: Week 14

  • Complete timeline reconstruction: Merge evidence from disk, memory, network, and logs into a unified attack narrative Week 7–12
  • Root cause documentation: Clearly identify the initial access vector and contributing factors
  • Impact assessment: Document what data was accessed, exfiltrated, or destroyed
  • Recommendations: Specific, actionable security improvements based on findings
  • IR process improvements: What worked well? What slowed the response? What evidence was unavailable due to insufficient logging?
  • Legal and regulatory deliverables: Breach notification content, law enforcement reports, regulatory filings Week 3 Week 14

Question 1: Consequences of Lost Evidence

During the MedTech incident, a well-meaning system administrator reboots the compromised database server before the forensic examiner arrives, believing it will "clear the malware." Which of the following evidence types is permanently lost as a result?

✓ Correct! Memory (RAM) is the most volatile evidence type. On reboot, all contents are lost: running processes, injected malware code, active network connections, decrypted data, and credentials held in memory. This is why the order of volatility (RFC 3227, Week 6) places memory capture as the highest priority, and why "capture memory before any containment action" is the cardinal rule of IR forensics. File system data (A) persists on disk. Event Logs (C) are stored on disk and survive reboot. Network PCAPs (D) are stored on the capture system, not the compromised host.
✗ Not quite. The correct answer is B. Memory (RAM) is the most volatile evidence type — it is completely lost on reboot. This includes running malware, active C2 sessions, injected code, and decrypted credentials. Disk-based evidence (file system, Event Logs) survives a reboot, and network PCAPs are stored on separate capture systems. This is precisely why the order of volatility (RFC 3227) prioritizes memory capture above all other evidence types.

Question 2: Phase Identification During Active Eradication

The MedTech forensic examiner discovers a malicious scheduled task on the domain controller that executes a PowerShell script every 6 hours, re-establishing a reverse shell to the attacker's C2 server. The examiner documents the task, captures the script, hashes both artifacts, and then coordinates with the system administrator to remove the task and block the C2 IP at the firewall. Which IR phase is this activity primarily part of?

✓ Correct! This is Eradication. The defining activity of the Eradication phase is identifying and removing all attacker footholds — persistence mechanisms, backdoors, and malware. While the discovery of the scheduled task involves "finding" something (which sounds like Detection), the context matters: the incident has already been detected and contained. The examiner is now systematically hunting for all persistence mechanisms to ensure the attacker cannot return. The firewall block is a supporting containment measure, but the primary activity — finding and removing the scheduled task — is eradication.
✗ Not quite. The correct answer is C — Eradication. The key distinction: during Eradication, the examiner systematically identifies and removes all attacker persistence mechanisms (scheduled tasks, registry keys, malicious services, SSH keys, etc.). While the activity involves "finding" artifacts, the incident has already been detected and contained — this is now the methodical removal of attacker footholds. The firewall block supports containment, but the primary action (removing the scheduled task and verifying no other persistence exists) defines Eradication.

Question 3: Technique Mapping Exercise

You are planning your forensic analysis for the MedTech incident. You need to determine whether the attacker exfiltrated patient data from the database server. Which combination of techniques and their source weeks would be MOST appropriate for this specific investigative question?

✓ Correct! To determine if data was exfiltrated, you need to examine: (1) Network traffic — Wireshark/PCAP analysis (Week 12) to identify large outbound transfers, unusual protocols, or connections to known-bad IPs; (2) Memory — Volatility network connection analysis (Week 11) to find active or recent connections the malware used for exfiltration; (3) Host logs — Windows Event Logs (Week 7–8) to identify what processes accessed the database and initiated network connections. This combination directly addresses the exfiltration question by examining both the network path and the host-level evidence of data access.
✗ Not quite. The correct answer is B. To investigate data exfiltration, you need techniques that reveal: (1) what left the network (PCAP analysis, Week 12), (2) what connections existed in memory (Volatility, Week 11), and (3) what processes accessed the data (Event Logs, Week 7–8). Option A focuses on persistence/malware identification rather than exfiltration. Option C applies Linux tools to a Windows server scenario. Option D lists supporting activities, not analytical techniques for answering the exfiltration question.
