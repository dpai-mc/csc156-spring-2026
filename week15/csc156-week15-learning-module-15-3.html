Module 15.3 – Working Under Pressure: Forensic Triage, Evidence Prioritization, and IR Plan Development

Estimated Time: 1.5 – 2.0 hours  |  Objectives Supported: Week 15 Objectives 5 & 6

This module addresses the practical reality that incidents impose time pressure. You will learn to apply the RFC 3227 order of volatility as a decision tool, prioritize evidence collection using a three-factor framework, and develop an incident response plan outline that integrates forensic acquisition, analysis, and reporting.

1. Scenario Continuation — The Clock is Ticking

⏱ Week 15 Scenario Update — 11:45 PM Tuesday

The situation at Meridian Healthcare has escalated. The initial compromise you identified in Module 15.2 — the confirmed database server (DB-PROD-01) — is no longer the only concern. Network monitoring has flagged suspicious outbound connections from two additional web application servers (WEB-02 and WEB-03) that sit in the same VLAN.

Your IR Commander has delivered the directive:

"I need a preliminary forensic assessment of all three systems and an incident response plan on my desk by 7:00 AM. We have a potential HIPAA breach notification clock starting at sunrise."

Here is the problem: you have one forensic examiner (you), one junior analyst, and seven hours. You cannot do everything. You must decide what to collect first, which system to examine first, and how to structure your response — all before the evidence disappears and the legal deadline arrives.

This module gives you the decision frameworks to make those calls confidently.

2. Order of Volatility as a Decision Tool (RFC 3227)

RFC 3227 (Guidelines for Evidence Collection and Archiving) establishes a principle you first encountered in Week 4 and applied in Week 11 (memory forensics): collect the most volatile evidence first, because it disappears the fastest.

Key Insight

The order of volatility is not just a list to memorize — it is a real-time decision tool. When you arrive on-scene with limited time, it tells you exactly what to reach for first.

PriorityEvidence LayerVolatilityLifespanCollection Tools
1CPU Registers / CacheMost volatileNanosecondsGenerally not collectible in field; captured indirectly via memory dump
2RAM (System Memory)Extremely volatileLost at power-offFTK Imager, WinPmem, LiME, DumpIt, Magnet RAM Capture
3Network StateVery volatileSeconds–minutesnetstat, ss, Wireshark (live capture), tcpdump
4Running ProcessesVolatileUntil terminatedtasklist, ps aux, Volatility (from RAM image)
5Disk (File Systems)Semi-persistentUntil overwrittenFTK Imager, dd, dc3dd, Guymager
6Removable MediaPersistentUntil degaussed/destroyedSame as disk imaging tools
7Remote Logs / BackupsLeast volatileRetention-policy dependentSIEM export, log aggregator queries, backup software

Applying Volatility to Our Scenario

At 11:45 PM, all three Meridian servers are still powered on. The moment IT pulls a network cable or reboots a system, you lose Layers 1–4 permanently. Your first action on each system must be a RAM capture, followed by network state documentation, before any containment step that might alter the running system.

Connection to Prior Weeks

Week 4: You learned the concept of volatility during evidence acquisition planning.
Week 6: You practiced forensic imaging with hash verification — the same integrity principles apply to RAM captures.
Week 11: You used Volatility to analyze memory dumps — those dumps exist because someone followed this exact collection priority.

Triage Tool Categories (Awareness Level)

Note

The following tools automate the triage collections listed above. You are not expected to use them hands-on in this course, but you should be aware of their purpose and capabilities.

KAPE (Kroll Artifact Parser and Extractor) — Windows

Purpose: Automated collection and parsing of Windows forensic artifacts. Uses modular "targets" (what to collect) and "modules" (how to parse).

  • Can collect a full triage package from a live Windows system in minutes
  • Outputs structured data ready for timeline analysis
  • Maintained by Eric Zimmerman (same developer as the EZ Tools suite from Week 8)
  • Free for law enforcement and educational use
CyLR (CyLR Live Response) — Cross-Platform

Purpose: Lightweight, cross-platform artifact collection tool. Runs as a single executable with no installation required.

  • Supports Windows, Linux, and macOS
  • Collects files based on a configurable artifact list
  • Outputs a ZIP archive for offline analysis
  • Minimal footprint on the target system
UAC (Unix-like Artifacts Collector) — Linux/macOS/Unix

Purpose: Shell-based triage collector for Unix-like systems. Gathers volatile data, logs, configuration files, and persistence artifacts.

  • No compilation or dependencies required — pure shell script
  • Collects live system state (network connections, processes, loaded modules)
  • Packages logs, cron, user data, and SSH artifacts
  • Ideal for Linux servers where installing software is undesirable

4. Prioritization Decision Framework

When multiple systems are affected, you cannot examine them all simultaneously. You need a structured way to decide: which system do I analyze first?

The Three-Factor Model: Severity × Scope × Volatility

FactorQuestion It AnswersScale (1–5)
SeverityHow critical is the data/system? What is the potential harm?1 = Low-value test system
5 = Production system with PII/PHI
ScopeHow many users, records, or systems are affected?1 = Single user/host
5 = Enterprise-wide / thousands of records
VolatilityHow quickly is evidence disappearing or being altered?1 = Powered-off, stable
5 = Live, actively under attack, evidence degrading

Priority Score = Severity + Scope + Volatility (range: 3–15). Higher score = examine first.

Recommended Order
  1. DB-PROD-01 (Score: 14) — Highest severity and scope. Capture RAM immediately, then begin live triage. This is your primary evidence source and your biggest legal exposure.
  2. WEB-02 (Score: 12) — Highest volatility factor. Assign your junior analyst to capture RAM and preserve web/auth logs before the hourly rotation destroys evidence.
  3. WEB-03 (Score: 8) — Lower priority. Preserve logs remotely if possible; schedule full triage after the first two systems are secured.

Key decision: Even though WEB-02 has the highest volatility score (5), DB-PROD-01's combined severity and scope make it the overall priority. However, the volatility of WEB-02 means you should not wait — delegate its RAM capture to run in parallel.

Real-World Note

This framework is a guide, not a rigid formula. Professional judgment matters. If WEB-02's logs rotate in 15 minutes and DB-PROD-01 is stable, you might grab WEB-02's logs first (a 2-minute task) before starting the longer DB-PROD-01 RAM capture. The framework helps you justify your decisions — critical when writing your report and testifying.

5. Business and Legal Constraints on Prioritization

Technical factors alone do not determine priority. Business operations and legal obligations impose hard constraints that can override pure forensic logic.

Breach Notification Deadlines

RegulationNotification WindowImplication for Forensics
HIPAA (healthcare)60 days from discoveryClock starts when breach is "discovered" — your triage findings may trigger the clock
GDPR (EU data subjects)72 hours from awarenessExtremely aggressive; preliminary findings needed within hours
State breach laws (CA, NY)Varies; often "expedient" or 30–60 daysMust determine scope of exposure quickly to assess notification requirements
PCI-DSS (payment cards)Immediate to acquirer; 72 hrs forensic engagementPFI (PCI Forensic Investigator) must be engaged within 72 hours
Scenario Impact

Meridian Healthcare is a HIPAA-covered entity. The IR Commander's 7:00 AM deadline is driven by the need to determine whether PHI was actually exfiltrated — because that determination starts the 60-day notification clock. Your forensic triage must answer: "Did data leave the building?"

System Availability Requirements

  • Revenue-generating systems: Taking a production e-commerce server offline for imaging may cost thousands per hour. You may need to perform live acquisition.
  • Life-safety systems: Hospital systems, SCADA/ICS — cannot be taken offline. Triage must be non-disruptive.
  • SLA obligations: Contractual uptime guarantees may limit your containment options.

Legal Hold Obligations

  • Once litigation is reasonably anticipated, a legal hold requires preservation of all potentially relevant evidence.
  • This may expand your collection scope — you must preserve systems you might otherwise deprioritize.
  • Coordinate with legal counsel before making any decision to skip or defer evidence collection.
  • Connection to Week 3: The legal authority and chain-of-custody principles you learned early in the course become critical under legal hold conditions.

How Constraints Modify Your Priority Matrix

Think of business/legal constraints as override rules:

  • If a notification deadline is imminent → prioritize determining scope of exposure over deep analysis
  • If a system cannot go offline → plan for live acquisition methods only
  • If legal hold is active → you cannot deprioritize any potentially relevant system, even if its technical score is low

6. Building an Incident Response Plan

An Incident Response Plan (IRP) is a pre-written document that tells your organization how to handle incidents before they happen. In our scenario, Meridian Healthcare should already have one. In Lab 14, you will develop an IR plan outline for a given scenario. Here are the key sections every plan needs:

IR Plan Template Walkthrough

Section 1: Purpose and Scope

Defines what the plan covers (which systems, which incident types, which personnel). Sets boundaries — e.g., "This plan covers information security incidents affecting Meridian Healthcare production systems. Physical security incidents are covered under a separate plan."

Section 2: Roles and Responsibilities

Lists every role on the IR team with specific duties. Who declares an incident? Who authorizes containment? Who communicates with media? Who performs forensic acquisition? (Connection to Module 15.1 — CSIRT structure.)

Section 3: Incident Classification and Severity Levels

Defines categories (malware, unauthorized access, data breach, DoS, insider threat) and severity tiers (e.g., SEV-1 through SEV-4). Each severity level triggers different response actions, escalation paths, and timelines.

Section 4: Detection and Reporting Procedures

How are incidents detected? (SIEM alerts, user reports, threat intelligence feeds.) How does someone report a suspected incident? What information must be included in the initial report?

Section 5: Forensic Acquisition Procedures

Step-by-step procedures for evidence collection at each severity level. References the order of volatility. Specifies approved tools, hash verification requirements, and chain-of-custody documentation. (Connection to Weeks 4 and 6.)

Section 6: Communication Plan

Internal communication: who is notified, through what channels, at what intervals. External communication: legal counsel, law enforcement, regulators, affected individuals, media. Specifies who is authorized to speak externally (usually legal + PR only).

Section 7: Containment Strategy

Short-term containment (stop the bleeding — isolate affected systems, block attacker IPs). Long-term containment (maintain operations while preparing for eradication). Documents forensic considerations: always capture volatile evidence before containment actions that alter system state.

Section 8: Recovery Criteria

Defines what "recovered" means: What conditions must be met before systems return to production? Who authorizes return to service? What monitoring is required post-recovery? How long is the heightened monitoring period?

In Lab 14, you will be asked to develop an IR plan outline for a provided scenario. You do not need to write a complete 50-page plan — you will populate each of the eight sections above with scenario-specific content in a structured outline format. This template gives you the skeleton.

7. IR Playbooks — Scenario-Specific Procedures

While an IR Plan provides the overall framework, playbooks provide step-by-step procedures for specific incident types. Think of the IR Plan as the constitution and playbooks as the specific laws — they operate within the plan's structure but provide tactical detail.

Worked Example: Ransomware Incident Playbook

Ransomware Response Playbook — Key Steps

  1. Detection & Identification
    • Confirm ransomware indicators: ransom note files, encrypted file extensions, mass file modification events
    • Identify affected systems (check for lateral spread)
    • Determine ransomware variant (ransom note text, file extension, YARA signature match)
  2. Immediate Containment (First 30 Minutes)
    • Isolate affected systems from network (disconnect, do NOT power off)
    • Block known C2 domains/IPs at firewall
    • Disable affected user accounts
    • Capture RAM from isolated systems before any other action
  3. Forensic Collection
    • RAM dump (may contain encryption keys in memory)
    • Disk image of affected systems
    • Network logs showing lateral movement and C2 traffic
    • Email logs (identify initial delivery vector)
  4. Analysis
    • Determine initial access vector (phishing email? RDP brute force? exploit?)
    • Map lateral movement path
    • Identify all affected systems and data
    • Check for data exfiltration before encryption (double extortion)
  5. Eradication
    • Remove ransomware binaries and persistence mechanisms
    • Patch exploited vulnerabilities
    • Reset all potentially compromised credentials
  6. Recovery
    • Restore from clean backups (verify backup integrity first)
    • Rebuild systems that cannot be confirmed clean
    • Monitor for re-infection indicators
  7. Post-Incident
    • Complete forensic report
    • Conduct lessons-learned meeting
    • Update detection rules based on observed TTPs
    • Assess whether to notify law enforcement (FBI IC3)
Playbook Overview: Data Exfiltration / Insider Threat

Trigger: DLP alert, unusual data volume to external destination, or HR-reported suspicious employee behavior.

Key differences from ransomware playbook:

  • Legal/HR coordination is immediate (potential employment action)
  • Covert investigation may be required — do not alert the subject
  • Focus on proving what data left and where it went (network forensics emphasis)
  • Email, cloud storage, USB device logs are primary evidence sources
  • Chain of custody is especially critical if termination or prosecution follows
Playbook Overview: Web Application Compromise

Trigger: WAF alert, web server log anomalies, defacement, or customer reports of malicious redirects.

Key differences from ransomware playbook:

  • Web server logs are the primary initial evidence source (connection to Week 12)
  • Determine attack type: SQL injection, file upload, RCE, XSS
  • Check for web shells (persistent backdoor files on the server)
  • Database integrity verification if SQL injection confirmed
  • May require application-level forensics (not just OS-level)
  • Customer data exposure assessment drives notification decisions

Directly relevant to our scenario: WEB-02 and WEB-03 at Meridian Healthcare may require this playbook.

Playbooks Reference Forensic Procedures

Notice how each playbook references specific forensic techniques you have already learned: RAM capture (Week 11), disk imaging (Week 6), network log analysis (Week 12), persistence mechanism identification (Weeks 7–9), malware analysis (Week 13). Playbooks are where your individual skills become an orchestrated response.

Question 1 — Time-Constrained Collection Ordering

You arrive at an incident scene with three affected systems. System A is a Windows file server (powered on, no active attacker connection detected). System B is a Linux web server (powered on, active C2 beacon confirmed, logs rotate every 30 minutes — next rotation in 12 minutes). System C is a Windows workstation (powered off, suspected patient zero).

In what order should you begin evidence collection, and what is your first action on each system? Justify your answer using the order of volatility and the prioritization framework.

Recommended Order:

  1. System B (Linux web server) — FIRST
    • First action: Preserve the rotating logs immediately (12-minute deadline is your hardest constraint), then capture RAM.
    • Justification: Highest volatility (active C2 + imminent log rotation). The 12-minute window creates an absolute deadline that overrides other factors. Logs lost are lost forever.
  2. System A (Windows file server) — SECOND
    • First action: Capture RAM (volatile, lost at power-off), then collect registry hives and event logs.
    • Justification: Powered on = volatile evidence exists. No immediate deadline pressure, but evidence degrades over time.
  3. System C (Windows workstation) — THIRD
    • First action: Create a forensic disk image (system is off, so no volatile data to capture).
    • Justification: Powered off = no volatile evidence remaining. Disk evidence is stable. This system can wait without evidence loss.

Key principle: The order of volatility tells you what to collect on each system; the prioritization framework (with the volatility factor weighted heavily here) tells you which system to approach first.

Question 3 — Mini-Playbook Drafting

Your organization's monitoring system detects that a database administrator's account is downloading the entire customer database (2.3 million records) to a personal cloud storage service at 2:00 AM. The DBA is not scheduled to work tonight.

Draft a 5-step mini-playbook for this incident. For each step, identify: (a) the action, (b) who is responsible, and (c) what forensic evidence you would collect or preserve.

Sample Mini-Playbook: Suspected Insider Data Exfiltration

  1. Confirm and Document the Alert
    • Action: Verify the DLP/monitoring alert is not a false positive. Screenshot the alert, confirm the account, confirm the destination.
    • Responsible: SOC Analyst / IR Coordinator
    • Evidence: DLP alert logs, SIEM event records, proxy logs showing cloud storage upload
  2. Covert Preservation (Do NOT Alert the Subject)
    • Action: Preserve network logs, proxy logs, database audit logs, and authentication logs. Begin RAM capture of the DBA's workstation if physically accessible without alerting them.
    • Responsible: Forensic Examiner + Network Engineer
    • Evidence: Proxy/firewall logs, DB audit trail, workstation RAM, authentication logs
  3. Engage Legal and HR
    • Action: Notify legal counsel and HR. Determine whether to allow continued activity (to build a stronger case) or immediately contain.
    • Responsible: IR Commander + Legal Counsel + HR
    • Evidence: Document the decision and authorization in writing
  4. Containment (When Authorized)
    • Action: Disable the DBA account, block the cloud storage destination at the proxy, isolate the workstation.
    • Responsible: System Administrator (directed by IR Commander)
    • Evidence: Capture workstation RAM and disk image after account disable; preserve the cloud storage upload records
  5. Scope Assessment and Reporting
    • Action: Determine exactly what data was exfiltrated, whether it reached the cloud destination, and whether any other accounts or systems were involved.
    • Responsible: Forensic Examiner
    • Evidence: Database query logs, network PCAP (if available), cloud storage API logs, forensic workstation image analysis

Key differences from a ransomware playbook: Covert investigation, HR/legal engagement before containment, focus on proving intent and scope rather than stopping destruction.
