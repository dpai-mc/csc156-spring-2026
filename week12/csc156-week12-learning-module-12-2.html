2 Wireshark Interface Orientation

Wireshark is the most widely used network protocol analyzer in the world, and it is a standard tool in digital forensics labs. Before you apply a single filter, you need to understand the interface layout and how Wireshark organizes packet data.

2.1 The Three-Pane Layout

When you open a PCAP file in Wireshark, the main window is divided into three panes. Each pane shows the same selected packet at a different level of detail:

PaneLocationWhat It ShowsForensic Use
Packet ListTopOne row per packet: number, time, source, destination, protocol, length, infoScan for patterns, sort/filter, identify time ranges
Packet DetailsMiddleProtocol tree for the selected packet — expandable layers from Frame → Ethernet → IP → TCP/UDP → ApplicationInspect specific header fields, flag values, payload content
Packet BytesBottomRaw hexadecimal and ASCII representation of the selected packetVerify exact byte values, spot magic bytes, find embedded strings
💡 Tip

Click on any field in the Packet Details pane and the corresponding bytes are highlighted in the Packet Bytes pane. This is invaluable when you need to verify exactly what data was on the wire.

2.4 Statistics Menu — Your Investigation Starting Point

Before diving into individual packets, experienced examiners use the Statistics menu to get a high-level overview of the capture:

  • Protocol Hierarchy (Statistics → Protocol Hierarchy) — Shows the percentage breakdown of all protocols in the capture. Use this first to understand what types of traffic are present.
  • Conversations (Statistics → Conversations) — Lists all communication pairs (by Ethernet, IPv4, TCP, or UDP). Sort by bytes transferred to quickly identify the heaviest talkers.
  • Endpoints (Statistics → Endpoints) — Lists every unique address in the capture with packet/byte counts. Spot hosts that sent or received unusually large amounts of data.
  • I/O Graphs (Statistics → I/O Graphs) — Plots traffic volume over time. Look for spikes (possible exfiltration) or regular patterns (possible beaconing).
  • DNS (Statistics → DNS) — Summarizes all DNS queries and responses. Useful for identifying domains contacted by a suspect host.
💡 Tip

A good forensic workflow: Protocol Hierarchy → Conversations → targeted display filters. Start broad, then narrow down.

3.3 Guided Filter Walkthrough

Imagine you have just opened the Acme Corp PCAP (tens of thousands of packets). Here is how an examiner would systematically narrow the data:

  1. Step 1 — Get an overview: Use Statistics → Protocol Hierarchy to see what protocols are present. You notice HTTP, DNS, TLS, and a surprising amount of FTP traffic.
  2. Step 2 — Isolate the suspect host: Apply ip.addr == 192.168.1.105 — the web server flagged by the SOC. The packet count drops from 48,000 to 12,400.
  3. Step 3 — Focus on HTTP: Refine to ip.addr == 192.168.1.105 && http — now you see 3,200 HTTP packets. Sort by time to see the request sequence.
  4. Step 4 — Look for POST requests: Further narrow with ip.addr == 192.168.1.105 && http.request.method == "POST" — only 47 POST requests. Several come from an external IP: 203.0.113.42.
  5. Step 5 — Isolate the attacker: Apply ip.src == 203.0.113.42 — all traffic from this IP. You now have a focused dataset to investigate.

In five steps you went from 48,000 packets to a manageable set of attacker-related traffic. This is the power of display filters.

3.4 Mini-Exercise: Write the Filter

For each scenario below, write the Wireshark display filter you would use. Try writing your answer before revealing the model answer.

Scenario A

You want to see only DNS queries made by the host 10.0.0.25.

dns.flags.response == 0 && ip.src == 10.0.0.25

Explanation: dns.flags.response == 0 selects DNS queries (not responses). Adding ip.src == 10.0.0.25 limits results to queries originating from that host. This is useful for building a list of every domain the suspect system attempted to resolve.

Scenario B

You suspect a SYN port scan was launched from the IP 203.0.113.42. Write a filter to isolate SYN-only packets from that source.

ip.src == 203.0.113.42 && tcp.flags.syn == 1 && tcp.flags.ack == 0

Explanation: A SYN scan sends TCP SYN packets without completing the handshake. tcp.flags.syn == 1 && tcp.flags.ack == 0 matches SYN-only packets (not SYN-ACKs). Combined with the source IP, this reveals every port the attacker probed.

Scenario C

You want to find any packet in the entire capture that contains the string "UNION SELECT" (a common SQL injection payload).

frame contains "UNION SELECT"

Explanation: The frame contains filter performs a case-sensitive string search across the entire packet (all layers). This catches SQL injection payloads in HTTP requests, regardless of which specific HTTP field they appear in. Note: for URL-encoded variants like UNION%20SELECT, you would need frame contains "UNION%20SELECT" or use http.request.uri contains "UNION".

4 TCP Stream Reconstruction

Individual packets are fragments of a conversation. To understand what actually happened — what was requested, what was sent, what credentials were exchanged — you need to reconstruct the full TCP stream.

🔬 Examiner Note

"Follow TCP Stream" is one of the most powerful techniques in Wireshark. It reassembles all packets in a TCP session into a single, readable conversation — exactly as the two endpoints experienced it. Master this technique and you can read HTTP sessions, recover FTP transfers, and capture cleartext credentials.

4.1 How to Follow a TCP Stream

  1. In the Packet List pane, select any packet that belongs to the TCP session you want to examine.
  2. Right-click the packet and select Follow → TCP Stream (or use the menu: Analyze → Follow → TCP Stream).
  3. A new window opens showing the reconstructed conversation:
    • Red text = data sent by the client (the side that initiated the connection)
    • Blue text = data sent by the server (the side that accepted the connection)
  4. Use the Stream dropdown at the bottom to navigate between different TCP streams in the capture.
  5. Use "Show and save data as" to switch between ASCII, Hex Dump, C Arrays, Raw, or YAML views.

4.2 Follow HTTP Stream

For HTTP traffic specifically, you can use Follow → HTTP Stream, which reconstructs the HTTP request/response pairs in a more readable format. This is particularly useful because:

  • HTTP headers and body are displayed together in context
  • You can see the full request URI, cookies, form data, and server response
  • Chunked transfer encoding is automatically reassembled

4.3 Forensic Applications of Stream Reconstruction

Cleartext Credential Capture

Protocols that transmit credentials in plaintext are goldmines for forensic examiners:

  • HTTP Basic Authentication: Credentials are Base64-encoded (not encrypted!) in the Authorization header. Following the stream reveals the encoded string, which you can decode instantly.
  • FTP: The USER and PASS commands appear in plaintext in the stream. Example:
USER admin
331 Password required for admin
PASS P@ssw0rd123
230 User admin logged in
  • Telnet: The entire session — including keystrokes — appears in the stream. Every command the user typed is visible.

HTTP Session Reconstruction

Following an HTTP stream lets you see exactly what an attacker requested and what the server returned. You can read:

  • SQL injection payloads in GET/POST parameters
  • Server responses revealing database contents
  • File upload requests (webshells, malware)
  • Cookie values and session tokens

Data Exfiltration Evidence

If an attacker used HTTP POST to exfiltrate data, following the stream reveals the exact data that was sent, the destination URL, and any encoding applied.

⚠️ Limitation: Encrypted Traffic

Following a TLS/HTTPS stream shows only the encrypted ciphertext — you cannot read the content without the server's private key or a pre-master secret log. In forensic analysis, you may still extract metadata (server name from the TLS Client Hello SNI extension, certificate information) but not the payload. Use the filter tls.handshake.extensions_server_name to see which hostnames were requested over HTTPS.

5 Extracting Artifacts from PCAPs

Beyond reading conversations, Wireshark can extract complete files, credentials, and metadata from packet captures. This is network-layer file carving — the same concept you learned in Week 5, applied to network traffic instead of disk images.

🔗 Cross-Module Connection — Week 5 File Carving

In Week 5 you learned to recover files from disk by identifying magic bytes (file signatures) and carving data from unallocated space. The same principle applies here: files transferred over HTTP are embedded in TCP streams, and Wireshark can extract them by reassembling the stream and identifying the file boundaries. The magic bytes you learned to recognize (e.g., %PDF for PDF, PK for ZIP, ‰PNG for PNG) will help you verify extracted files.

5.1 Exporting HTTP Objects

  1. Go to File → Export Objects → HTTP.
  2. A dialog lists every file transferred over HTTP in the capture: images, scripts, HTML pages, downloads, uploaded files.
  3. Key columns: Hostname, Content Type, Size, Filename.
  4. Select individual files and click Save, or click Save All to export everything.
  5. After extraction: Hash each file (SHA-256) and document it in your forensic notes. Compare hashes against known-malware databases or the original server files.
💡 Tip

You can also export objects from other protocols: File → Export Objects → SMB (for Windows file shares), DICOM (medical imaging), IMF (email messages), and TFTP.

5.2 Credential Extraction

Wireshark can help you identify credentials transmitted in cleartext. Common sources:

ProtocolWhere Credentials AppearHow to Find Them
HTTP Basic AuthAuthorization: Basic dXNlcjpwYXNz header (Base64)Filter: http.authorization
HTTP Form LoginPOST body: username=admin&password=secretFilter: http.request.method == "POST" then follow stream
FTPUSER and PASS commands in plaintextFilter: ftp.request.command == "PASS"
TelnetEntire session in cleartext (including typed passwords)Filter: telnet then follow stream
SMTPAUTH LOGIN with Base64-encoded credentialsFilter: smtp.req.command == "AUTH"
🔬 Examiner Note

Base64 encoding is not encryption. It is trivially reversible. If you see dXNlcjpwYXNz in an HTTP Authorization header, decoding it with any Base64 tool yields user:pass. Always check Base64-encoded values in HTTP headers and SMTP AUTH exchanges.

5.3 DNS Query Extraction

Every DNS query in a PCAP tells you what domain a host tried to reach. This is forensically valuable because:

  • It reveals every website and service the suspect system communicated with
  • DNS queries occur even when the actual traffic is encrypted (HTTPS)
  • Queries to known-malicious domains are strong indicators of compromise
  • Unusual DNS patterns can reveal tunneling or DGA (Domain Generation Algorithm) activity

To extract all DNS queries: Statistics → DNS provides a summary. For detailed analysis, apply the filter dns.flags.response == 0 to see all queries, then examine the dns.qry.name field in the Packet Details pane.

🔗 Cross-Module Connection — Week 11 Memory Forensics

In Week 11 you used Volatility's netscan plugin to identify active network connections in a memory dump. If netscan showed a connection to 198.51.100.5:443, you can now search the PCAP for ip.addr == 198.51.100.5 to find the corresponding network traffic — and check DNS queries to learn what domain that IP resolved to. This is the host-to-network correlation loop.

6 tcpdump for Command-Line Capture

Wireshark is powerful, but it requires a graphical interface. In many forensic and incident-response scenarios, you need to capture packets on a headless server, a remote system over SSH, or as part of an automated script. That is where tcpdump comes in.

6.1 What Is tcpdump?

tcpdump is a command-line packet capture and analysis tool available on virtually every Linux and macOS system. It uses the same underlying capture library (libpcap) as Wireshark, and its output files are fully compatible with Wireshark for later analysis.

6.2 When to Prefer tcpdump Over Wireshark

  • Headless servers: Production servers rarely have a GUI. Use tcpdump to capture, then transfer the PCAP to your analysis workstation for Wireshark analysis.
  • Remote capture over SSH: You can run tcpdump on a remote system via SSH and pipe the output to a local file.
  • Scripted/automated capture: Incident response playbooks often include tcpdump commands for immediate evidence preservation.
  • Resource-constrained environments: tcpdump uses far less memory and CPU than Wireshark.

6.3 Essential tcpdump Syntax

7 Identifying Suspicious Traffic Patterns

Now that you can navigate Wireshark, apply filters, follow streams, and extract artifacts, the critical forensic skill is knowing what to look for. This section teaches you to distinguish normal network behavior from indicators of malicious activity.

7.1 Baseline: What Does "Normal" Look Like?

Before you can identify the abnormal, you need a mental model of typical corporate network traffic:

  • DNS queries: Frequent, short, to internal DNS servers or well-known resolvers (8.8.8.8, 1.1.1.1)
  • HTTP/HTTPS: Web browsing generates bursts of requests to many different domains
  • DHCP: Periodic lease renewals from client machines
  • ARP: Broadcast requests for MAC address resolution on the local subnet
  • SMB: File share access typically between workstations and file servers — not between workstations
  • Traffic volume: Relatively symmetric for browsing; asymmetric (more download than upload) is normal for most users
ℹ️ Key Principle

Anomaly detection requires a baseline. In a real investigation, you would compare the suspect traffic against known-normal patterns for that specific network. In our lab exercises, we provide the context — but in practice, always ask: "What is normal for this environment?"

7.2 Reconnaissance and Scanning Indicators

Attackers typically begin by mapping the network. These patterns reveal scanning activity:

  • SYN scan (port scan): Many SYN packets from one source to many destination ports, with no completed handshakes. The attacker sends SYN, receives SYN-ACK (port open) or RST (port closed), but never sends the final ACK.
  • Wireshark filter: ip.src == [attacker IP] && tcp.flags.syn == 1 && tcp.flags.ack == 0
  • Visual indicator: In the Packet List, you see sequential destination ports (21, 22, 23, 25, 80, 443…) from the same source IP — often with RST responses.
  • ICMP sweep: Many ICMP Echo Request packets to sequential IP addresses — the attacker is discovering which hosts are alive.
  • Wireshark filter: icmp.type == 8 && ip.src == [attacker IP]
  • Nmap signatures: Nmap's default SYN scan produces a distinctive pattern of SYN packets with specific TCP window sizes and options. Experienced analysts can recognize Nmap traffic by these characteristics.

7.3 Command-and-Control (C2) Indicators

After gaining access, malware often establishes a C2 channel — a persistent communication link back to the attacker's infrastructure. Key indicators:

  • Beaconing: Regular-interval outbound connections (e.g., every 60 or 90 seconds) to the same external IP or domain. This periodic "check-in" is the hallmark of C2 malware.
  • How to detect: Use Statistics → I/O Graphs to visualize traffic over time. Beaconing appears as evenly spaced spikes. Filter for the suspect connection and examine timestamps.
  • HTTP-based C2: Regular POST requests to suspicious URLs with encoded or encrypted payloads in the body. The URLs may look like legitimate paths (/api/update, /status) but carry encoded commands.
  • Unusual outbound connections: Internal hosts connecting to external IPs on non-standard ports (e.g., port 8443, 4444, 1337) or to IP addresses with no associated DNS name.
🔗 Cross-Module Connection — Week 11 Memory Forensics

If Volatility's netscan showed a suspicious ESTABLISHED connection from a process like svchost.exe to an external IP, you can now search for that same IP in the PCAP. Finding matching traffic in both memory and network evidence corroborates the finding and strengthens your case.

7.4 Data Exfiltration Indicators

Exfiltration — the unauthorized transfer of data out of the network — is often the attacker's ultimate goal. Look for:

  • Large outbound transfers: Use Statistics → Conversations and sort by bytes. An internal host sending gigabytes to an external IP is a red flag, especially outside business hours.
  • Asymmetric traffic: Normal browsing downloads more than it uploads. A workstation with more outbound data than inbound is suspicious.
  • Encrypted traffic to non-standard ports: TLS on port 443 is normal. TLS on port 8443 or an unrecognized port to an unknown IP warrants investigation.
  • FTP/SCP uploads from internal systems: Outbound FTP STOR commands or SCP transfers to external hosts — especially from servers that should not be initiating outbound file transfers.
  • HTTP POST with large bodies: Filter with http.request.method == "POST" && http.content_length > 100000 to find large data submissions.

7.5 DNS Tunneling Indicators

DNS tunneling encodes data within DNS queries and responses, using a legitimate protocol on a legitimate port (53) to bypass firewalls. Indicators include:

  • Unusually long subdomain strings: Normal DNS queries are short (www.example.com). Tunneling queries look like aGVsbG8gd29ybGQ.data.evil.com — the subdomain contains Base64 or hex-encoded data.
  • High volume of TXT record requests: TXT records can carry arbitrary text data. A host making hundreds of TXT queries to the same domain is suspicious. Filter: dns.qry.type == 16
  • High query volume to a single domain: Normal DNS queries are distributed across many domains. Hundreds of queries to one domain in a short period suggests tunneling.
  • Large DNS responses: Normal DNS responses are small (a few hundred bytes). Tunneling responses carrying exfiltrated data can be much larger.
🔬 Examiner Note

Attackers hide in plain sight. DNS tunneling uses a legitimate protocol on a legitimate port — but the content is anything but normal. Always examine DNS query names and response sizes, not just the fact that DNS traffic exists.

7.6 Lateral Movement Indicators

After compromising one system, attackers move laterally through the network to reach higher-value targets. Network indicators include:

  • SMB traffic between workstations: In most environments, SMB (port 445) traffic flows between workstations and file servers. SMB traffic between two workstations is unusual and may indicate file copying, credential harvesting, or remote execution.
  • Wireshark filter: tcp.port == 445 && ip.src == [workstation A] && ip.dst == [workstation B]
  • RDP between internal hosts: Remote Desktop (port 3389) connections between internal systems — especially from a compromised workstation to a server — suggest lateral movement.
  • Wireshark filter: tcp.port == 3389 && ip.src == 192.168.1.0/24 && ip.dst == 192.168.1.0/24
  • PsExec-style remote execution: PsExec creates an SMB connection, copies a service binary, and starts a remote service. Look for SMB traffic followed by service creation (which would also appear in Windows Event Logs as Event ID 7045).
  • Pass-the-hash / pass-the-ticket: These attacks use stolen credential hashes or Kerberos tickets to authenticate without knowing the plaintext password. Network indicators include NTLM authentication traffic and Kerberos ticket requests from unexpected hosts.
🔗 Cross-Module Connection — Week 8 Windows Event Logs

Lateral movement in PCAPs correlates directly with Windows Event Logs: an RDP connection (port 3389 in the PCAP) should produce Event ID 4624 Logon Type 10 on the destination system. A PsExec execution should produce Event ID 7045 (service installed). Correlating network evidence with host logs strengthens both.

7.7 Putting It Together: The Acme Corp PCAP

Returning to our scenario, here is what a systematic analysis of the Acme Corp PCAP might reveal:

  1. Protocol Hierarchy shows HTTP, DNS, TLS, FTP, and SMB traffic — FTP is unexpected on a web server segment.
  2. Conversations sorted by bytes reveals a large outbound transfer from 192.168.1.105 (web server) to 198.51.100.5 (external IP) over FTP.
  3. Display filter ip.addr == 203.0.113.42 isolates the attacker's traffic: a SYN scan at 2:15 AM, followed by HTTP requests starting at 2:22 AM.
  4. Following the HTTP stream reveals SQL injection payloads in GET parameters and a successful webshell upload via POST.
  5. DNS queries from the web server show a lookup for data.evil-c2.com at 2:35 AM — a domain not in any legitimate business records.
  6. Beaconing pattern: The web server sends small HTTPS requests to 198.51.100.5 every 60 seconds starting at 2:40 AM — classic C2 behavior.
  7. FTP export: At 2:47 AM (the time the SOC flagged), an FTP STOR command transfers a 23 MB file named customer_db.sql to 198.51.100.5 — data exfiltration confirmed.

This is the story the PCAP tells. In Module 12.3 you will examine the Apache access logs that show the same attack from the web server's perspective, and in Module 12.4 you will correlate all evidence sources.

8 Knowledge Check

Test your understanding of the concepts covered in this module. These questions are ungraded — use them to identify areas that need review before Lab 11 and Quiz 3.

Question 1

Which Wireshark display filter would show only HTTP POST requests originating from the IP address 203.0.113.42?

✅ Correct! The && (AND) operator ensures both conditions must be true: the request must be an HTTP POST and the source IP must match. Option A uses || (OR), which would show all POST requests from any IP plus all traffic from 203.0.113.42. Option C filters port 80 traffic but does not specifically isolate POST requests. Option D uses invalid syntax — http.post is not a valid Wireshark filter field.
❌ Not quite. The correct answer is B: http.request.method == "POST" && ip.src == 203.0.113.42. The && (AND) operator ensures both conditions must be true simultaneously. Option A's || (OR) would be too broad. Option C doesn't filter by HTTP method. Option D uses invalid syntax.

Question 2

You follow a TCP stream in Wireshark and see the following exchange:

USER admin
331 Password required for admin
PASS S3cur3P@ss!
230 User admin logged in

What protocol is this, and what is the forensic significance?

✅ Correct! The USER and PASS commands followed by numeric response codes (331, 230) are the signature of the FTP protocol. FTP transmits credentials in complete cleartext — no encoding, no encryption. This is a critical forensic finding: you now have the exact username (admin) and password (S3cur3P@ss!) the attacker used. This credential may be reused elsewhere in the environment.
❌ Not quite. The correct answer is C — FTP. The USER/PASS commands and numeric response codes (331 = password required, 230 = login successful) are characteristic of the FTP protocol. Unlike SSH (encrypted) or HTTP Basic Auth (Base64-encoded), FTP transmits credentials in complete cleartext. This gives the examiner the exact username and password used.

Question 3

An internal workstation (192.168.1.50) sends a small HTTPS request to the external IP 198.51.100.5 every 90 seconds, around the clock, including weekends and overnight. What does this pattern most likely indicate, and what would be your next investigative step?

✅ Correct! Regular-interval outbound connections (every 90 seconds, 24/7) to the same external IP are the hallmark of C2 (Command-and-Control) beaconing. The malware "checks in" with its controller at fixed intervals.
