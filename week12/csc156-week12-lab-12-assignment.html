1 Lab Overview and Objectives

In this lab you will act as a forensic examiner investigating a network security incident at Acme Corp. You have been provided with two primary evidence sources:

  1. A PCAP file captured from the network segment of Acme Corp's web server during the incident window.
  2. A set of Apache/Nginx access logs from the same web server covering the same timeframe.

Your objectives are to:

  • Apply Wireshark display filters to isolate and examine suspicious network traffic (Module 12.2 skills).
  • Reconstruct TCP streams and extract artifacts from packet data.
  • Parse web server logs to identify attack indicators — SQL injection, directory traversal, XSS, scanner signatures, and brute-force patterns (Module 12.3 skills).
  • Correlate findings across PCAP and log evidence to build a unified attack timeline (Module 12.4 skills).
  • Document your investigation with professional forensic rigor (CSLO 4 connection).
🔗 Scenario Connection

This lab continues the Acme Corp investigation introduced in Module 12.1. The SOC flagged unusual outbound traffic from the web server at 2:47 AM. You now have the evidence — determine what happened, who did it, and what data (if any) left the building.

Phase 1: Packet Capture Analysis

Wireshark PCAP Investigation • Execution points: part of 18 pts

Open the provided PCAP file in Wireshark. Follow the tasks below in order. Record every filter you apply and every observation you make.

Task 1.1

Initial PCAP Integrity Verification

Before analysis, verify the integrity of the PCAP file. Compute both MD5 and SHA-256 hashes of the file and record them below. Compare your hashes against the values provided by the instructor (if applicable).

Example tools: md5sum, sha256sum, Get-FileHash (PowerShell), FTK Imager

Task 1.2

Protocol Hierarchy and Traffic Overview

Use Wireshark's Statistics → Protocol Hierarchy to get an overview of the traffic in the capture. Also check Statistics → Conversations and Statistics → Endpoints.

  1. What protocols are present in the capture? List the top 5 by percentage of packets.
  2. How many unique source IP addresses are in the capture?
  3. How many unique destination IP addresses?
  4. Which IP pair has the highest packet count?
Task 1.3

Identify Suspicious Hosts

Based on your Protocol Hierarchy and Conversations analysis, identify any IP addresses that appear suspicious. Consider:

  • External IPs communicating with the web server on unexpected ports.
  • IPs with unusually high packet counts or data volumes.
  • IPs initiating connections on multiple ports (scanning behavior).

Apply display filters to isolate traffic from each suspicious IP. Document the filters used.

Task 1.4

Port Scan Detection

Apply a display filter to detect potential port scanning activity (SYN-only packets). Record the filter and your findings.

Task 1.5

TCP Stream Reconstruction

Select the most suspicious HTTP conversation you identified and use Right-click → Follow → TCP Stream to reconstruct the session. Examine the full request/response exchange.

  1. What HTTP method was used (GET, POST, etc.)?
  2. What URI was requested?
  3. What was the server's response code and content?
  4. Does the request contain any attack payload? If so, what type?
  5. Were any credentials visible in the stream?

Take a screenshot of the TCP stream window and include it in your lab notebook.

Task 1.6

Artifact Extraction from PCAP

Use File → Export Objects → HTTP to examine files transferred during the capture.

  1. List all files available for export (filename, content type, size).
  2. Export any suspicious files. Compute their hashes.
  3. Examine file content — do any appear to be web shells, scripts, or exfiltrated data?
Task 1.7

Beaconing and C2 Detection

Examine the PCAP for indicators of Command-and-Control (C2) communication:

  • Look for periodic outbound connections at regular intervals.
  • Check for DNS queries to unusual domains or high-volume TXT record requests.
  • Identify any outbound connections on non-standard ports.

Phase 2: Web Server Log Investigation

Apache / Nginx Access Log Analysis • Execution points: part of 18 pts

Open the provided web server access log file(s) in a text editor or use command-line tools (grep, awk, sort, uniq). Work through the tasks below systematically.

Task 2.1

Log Overview and Statistics

Gather baseline statistics from the log file:

  1. Total number of log entries.
  2. Time range covered (earliest and latest timestamps).
  3. Top 5 source IP addresses by request count.
  4. Top 5 most-requested URIs.
  5. Distribution of HTTP status codes (count of 200s, 301s, 403s, 404s, 500s).

Useful commands: wc -l access.log   awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -5

Task 2.2

Scanner and Automated Tool Detection

Search the logs for User-Agent strings associated with known scanning tools (e.g., sqlmap, Nikto, DirBuster, Gobuster, Nmap).

Example: grep -i "sqlmap\|nikto\|dirbuster\|gobuster\|nmap" access.log

Task 2.3

SQL Injection Indicator Detection

Search the logs for SQL injection indicators: UNION, SELECT, OR 1=1, single quotes (' or %27), semicolons (%3B), and comment sequences (-- or %2D%2D).

Task 2.4

Directory Traversal and XSS Detection

Search for directory traversal patterns (../, ..%2F, %2e%2e) and XSS payloads (<script, %3Cscript, onerror=, alert().

Task 2.5

Brute-Force and Authentication Attack Detection

Look for indicators of brute-force attacks against login endpoints: many POST requests to authentication URIs (e.g., /login, /admin, /wp-login.php) with high rates of 401 or 403 responses followed by a 200.

Task 2.6

Web Attack Timeline Reconstruction

Using the attacker IP address(es) you identified, filter and sort all log entries chronologically. Reconstruct the attacker's session from initial reconnaissance through exploitation.

Identify these phases in the log data:

  1. Reconnaissance: Directory enumeration, 404 responses, scanner activity.
  2. Weaponization / Exploitation: SQL injection attempts, traversal attempts, other attack payloads.
  3. Post-Exploitation: Successful access (200 codes), file uploads, large data responses.

Phase 3: Cross-Source Correlation

Connecting PCAP, Web Logs, and Other Evidence • Execution points: part of 18 pts

🔗 Why Correlation Matters

No single evidence source tells the complete story. Forensic strength comes from corroborating findings across multiple independent sources. In this phase you connect what you found in the PCAP (Phase 1) with what you found in the web server logs (Phase 2) — and any other evidence sources available — to build a unified attack narrative.

Task 3.1

IP Address Correlation

Compare the suspicious IP addresses identified in the PCAP (Phase 1, Task 1.3) with those found in the web server logs (Phase 2). Do the same attacker IPs appear in both sources?

Task 3.2

Timestamp Alignment

Compare timestamps between the PCAP and web server logs for the same events. Verify that the clocks are synchronized (or note any offset). Align key events across both sources.

Task 3.3

Attack Confirmation Across Sources

For each attack type identified in Phase 2 (SQL injection, traversal, brute force, etc.), find the corresponding packets in the PCAP that confirm the attack. Does the PCAP reveal anything the logs did not (e.g., response body content, additional requests not logged)?

Task 3.4

Unified Attack Timeline

Combine your findings from Phase 1 and Phase 2 into a single, chronologically ordered attack timeline. For each event, note the evidence source(s) that support it.

6 Hints and Common Pitfalls

  • Wireshark display filters are case-sensitive for string values. http.request.method == "POST" works; "post" does not.
  • Use == for exact match, contains for substring match. Example: dns.qry.name contains "acme".
  • Combine filters with && (AND), || (OR), ! (NOT). Parentheses help with complex expressions.
  • If your filter bar turns red, the syntax is invalid. Check field names in the Wireshark Display Filter Reference.
  • Remember: ip.addr matches both source and destination. Use ip.src or ip.dst for one direction.
  • Use grep -i for case-insensitive search: grep -i "union.*select" access.log.
  • Use grep -c to count matches: grep -c "403" access.log.
  • Chain grep with awk for field extraction: awk '{print $1}' access.log | sort | uniq -c | sort -rn gives IP frequency.
  • Filter by attacker IP first, then search for attack patterns: grep "203.0.113.42" access.log | grep -i "union".
  • URL-encoded characters are common in attacks. Search for both decoded and encoded forms: grep -E "(\.\./|%2e%2e)" access.log.
  • Forgetting to verify PCAP integrity: Always hash the PCAP before analysis. This is a forensic evidence file.
  • Ignoring time zones: PCAP timestamps may be in UTC while log timestamps may be in local time. Check and note any offset.
  • Focusing on one source only: The highest-scoring submissions will show clear correlation between PCAP and log findings.
  • Vague observations: "Traffic looks suspicious" is not sufficient. State what is suspicious, why, and cite the specific packet number or log line.
  • Missing documentation: Every filter you type and every observation you make should be recorded. 6 of 30 points depend on documentation quality.
  • Not checking the error log: If an Apache/Nginx error log is provided, it may reveal server-side errors triggered by attack payloads that the access log alone does not show.

A strong unified timeline follows this structure:

  1. List every significant event with its timestamp.
  2. Note which evidence source(s) confirm each event.
  3. Organize chronologically — the story should read like a narrative.
  4. Group events into phases: Reconnaissance → Exploitation → Post-Exploitation → Exfiltration.
  5. Note gaps — if there is a period with no evidence, say so. Gaps can indicate encrypted activity or log deletion.
