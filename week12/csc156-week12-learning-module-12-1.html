1 Week-Long Investigation Scenario — Acme Corp SOC

🔎 Scenario: Acme Corp — SOC Alert #4471

You are a forensic analyst on the Security Operations Center (SOC) team at Acme Corp, a mid-sized manufacturing company. At 02:14 UTC on Monday, the IDS flagged a burst of outbound traffic from the internal web server 10.10.50.20 to an external IP address 203.0.113.42. The SIEM correlated this with:

  • Unusual POST requests in the Apache access log with SQL injection signatures.
  • A spike in DNS TXT record queries to a domain registered only 48 hours earlier.
  • A new outbound connection on port 8443 that does not match any known application.

Your incident commander has authorized a full network forensic investigation. You have been handed:

  • A PCAP file captured by the network TAP between the DMZ and the firewall.
  • Apache access and error logs from 10.10.50.20.
  • Firewall logs (iptables) for the incident window.
  • DNS query logs from the internal resolver.

Your mission this week: Analyze these evidence sources to reconstruct the full attack timeline — from initial exploitation through command-and-control to data exfiltration — and prepare your findings for the incident report.

2 OSI Model for Forensic Examiners

The Open Systems Interconnection (OSI) model is a seven-layer framework that describes how data moves from an application on one machine to an application on another. As a forensic examiner you do not need to memorize every protocol at every layer, but you do need to know which layers produce evidence you can analyze and what that evidence looks like.

🧪 Forensic Examiner Note

For network forensics, your primary evidence lives at Layers 3, 4, and 7 — IP addresses, TCP/UDP ports, and application-layer content (HTTP requests, DNS queries, email headers). Keep this mental model as you analyze PCAPs and logs throughout Week 12.

In practice, most network forensics references use the four-layer TCP/IP model rather than the seven-layer OSI model. Here is how they map:

TCP/IP LayerOSI LayersKey Protocols
Application7, 6, 5HTTP, DNS, SMTP, FTP, SSH, TLS
Transport4TCP, UDP
Internet3IPv4, IPv6, ICMP
Network Access2, 1Ethernet, Wi-Fi, ARP

Both models describe the same reality. The OSI model is useful for precise discussion; the TCP/IP model reflects how protocols are actually implemented. You will encounter both in certification exams (CHFI, GCFE, GCFA).

When you type a URL in a browser, the HTTP request (Layer 7 data) is wrapped in a TCP segment (Layer 4), then in an IP packet (Layer 3), then in an Ethernet frame (Layer 2). Each layer adds a header containing metadata the forensic examiner can analyze.

Application Data  →  [HTTP GET /index.html]
       ↓
TCP Segment       →  [TCP Hdr | HTTP Data]
       ↓
IP Packet         →  [IP Hdr | TCP Segment]
       ↓
Ethernet Frame    →  [Eth Hdr | IP Packet | Eth Trailer]

A PCAP file captures at the frame level — meaning it contains all layers. Wireshark decodes each layer for you, which is why it is such a powerful forensic tool.

3 TCP/IP Protocol Suite Essentials

This section covers the protocols you will encounter most often in network forensic investigations. The focus is on what each protocol tells you as an examiner, not on network engineering.

TCP is a connection-oriented, reliable protocol. Before any data is exchanged, two hosts perform a three-way handshake:

Client → Server :  SYN           (I want to connect)
Server → Client :  SYN-ACK       (OK, acknowledged)
Client → Server :  ACK           (Connection established)

Why this matters forensically:

  • SYN-only packets with no ACK follow-up are a hallmark of port scanning (SYN scan / half-open scan). In the Acme Corp PCAP you may see hundreds of these from a single source IP.
  • Sequence and acknowledgment numbers let Wireshark reassemble a complete TCP stream — reconstructing an entire HTTP session, file transfer, or even a shell session.
  • RST (Reset) packets indicate a refused connection. A burst of RSTs from many ports suggests the target was scanned.
  • FIN packets indicate a graceful connection close. The absence of FINs (connections that just stop) can indicate a crash, kill, or network disruption.

Key TCP flags and their forensic meaning:

FlagNameForensic Significance
SYNSynchronizeConnection initiation. Mass SYN = port scan.
ACKAcknowledgeConfirms receipt. Present in almost all packets after handshake.
FINFinishGraceful close. Absence may indicate abnormal termination.
RSTResetAbrupt close. Closed port response in SYN scans.
PSHPushDeliver data immediately. Common in interactive sessions (SSH, shell).
URGUrgentRarely used legitimately. May appear in crafted attack packets.

UDP is connectionless — there is no handshake and no guaranteed delivery. Packets are sent and (hopefully) received.

Forensically important UDP protocols:

  • DNS (port 53) — The most forensically valuable UDP protocol. DNS queries reveal what domains a host resolved, and DNS can be abused for tunneling and exfiltration.
  • DHCP (ports 67/68) — DHCP lease logs map IP addresses to MAC addresses at specific times, critical for attribution.
  • NTP (port 123) — Time synchronization. NTP manipulation can affect timestamp reliability across evidence sources.
  • Syslog (port 514) — Remote log forwarding. If a compromised host forwards logs via syslog, those remote copies may survive log tampering on the host.

Because UDP has no connection state, you cannot "follow a UDP stream" the same way you follow a TCP stream. Instead, you filter by source/destination IP and port to group related UDP traffic.

HTTP is a request/response protocol. Every web interaction generates at least one request and one response — both of which appear in PCAPs and web server logs.

Key HTTP methods:

  • GET — Retrieve a resource. Parameters visible in the URL (and in logs). SQL injection payloads often appear in GET parameters.
  • POST — Submit data in the request body. Used for form submissions, file uploads, and C2 communication. POST bodies are not recorded in standard web server access logs — you need the PCAP to see them.
  • PUT / DELETE — Less common; may indicate web shell upload or API abuse.

Key HTTP status codes:

CodeMeaningForensic Context
200OKSuccessful request. A 200 to a SQL injection URL means the attack may have succeeded.
301/302RedirectMay indicate server-side redirect after successful login or exploitation.
403ForbiddenAccess denied. Many 403s from one IP = directory enumeration.
404Not FoundMany 404s from one IP = path/file scanning (DirBuster, Gobuster).
500Internal Server ErrorMay indicate a successful injection that caused an application error.

HTTPS: When traffic is encrypted with TLS, you cannot read the HTTP payload in a PCAP. However, you can still see: the TLS handshake, the Server Name Indication (SNI) field (which reveals the requested hostname), and the certificate details. Web server logs record the decrypted request because the server terminates TLS — so web logs remain valuable even when PCAP content is encrypted.

DNS translates domain names to IP addresses. For a forensic examiner, DNS is one of the most valuable network artifacts because nearly every network action begins with a DNS query.

Key DNS record types:

  • A / AAAA — Maps a domain to an IPv4 / IPv6 address.
  • MX — Mail exchange server. Relevant to email forensics (Week 16).
  • TXT — Text records. Used for SPF/DKIM/DMARC (email authentication) but also abused for DNS tunneling and exfiltration.
  • CNAME — Alias. Can reveal CDN or hosting infrastructure.
  • NS — Authoritative nameserver. Useful for investigating domain infrastructure.

DNS as an attack channel: In the Acme Corp scenario, the IDS flagged a spike in TXT record queries to a recently registered domain. This is a classic indicator of DNS tunneling — data encoded in DNS subdomain labels (e.g., dXNlcm5hbWU9YWRtaW4=.evil.com) to exfiltrate information through a protocol that is almost never blocked by firewalls.

IPv4 uses 32-bit addresses (e.g., 192.168.1.105) and remains the most common in forensic investigations. IPv6 uses 128-bit addresses (e.g., 2001:0db8:85a3::8a2e:0370:7334) and is increasingly present on modern networks.

Forensic implications of IPv6:

  • IPv6 addresses may be auto-generated from the MAC address (EUI-64 format), which can link a network connection directly to a physical NIC — powerful for attribution.
  • Many networks run dual-stack (IPv4 + IPv6). An attacker may use IPv6 to evade monitoring tools that only inspect IPv4 traffic.
  • IPv6 has no NAT by default — every device can have a globally routable address, simplifying attribution but expanding the attack surface.
  • Ensure your Wireshark filters account for both: ip.addr (IPv4) vs. ipv6.addr (IPv6).

🧪 Forensic Examiner Note

Ephemeral (source) ports are typically in the range 49152–65535 and are assigned dynamically by the operating system. In a PCAP, the destination port usually tells you the service; the source port tells you which client session. If you see a well-known service port (e.g., 443) used as a source port from an internal host, that is suspicious — the host may be running an unauthorized server.

IPv4 addresses are 32-bit numbers written in dotted-decimal notation: 192.168.1.105. For forensic work, the most important distinction is private vs. public.

RFC 1918 Private Address Ranges

RangeCIDRTypical Use
10.0.0.0 – 10.255.255.25510.0.0.0/8Large enterprises, cloud VPCs
172.16.0.0 – 172.31.255.255172.16.0.0/12Mid-size networks
192.168.0.0 – 192.168.255.255192.168.0.0/16Home/small office networks

Why this matters:

  • Private IPs appear in internal logs and PCAPs. They identify hosts within the network but are not routable on the Internet.
  • Public IPs are routable and can be traced to an ISP, organization, or geographic region via GeoIP and WHOIS.
  • When you see traffic from an internal private IP to an external public IP (like 10.10.50.20 → 203.0.113.42 in the Acme scenario), you know data is leaving the network.
  • NAT (Network Address Translation) maps private IPs to a public IP at the firewall. Firewall NAT logs are essential for linking an internal host to its external traffic.
💡 Quick Tip

The IP range 203.0.113.0/24 is reserved for documentation and examples (RFC 5737). In real investigations, you would run WHOIS and GeoIP lookups on the actual external IP. In our Acme scenario, treat 203.0.113.42 as the attacker's external IP.

GeoIP is the process of mapping an IP address to an approximate geographic location — typically a country, region, and city. Forensic examiners use GeoIP to:

  • Identify the likely origin of an attack. If your company operates only in the US and you see SSH brute-force attempts from an IP geolocated to a country where you have no business, that is an immediate red flag.
  • Detect impossible travel. If a user authenticates from New York at 10:00 AM and from Eastern Europe at 10:15 AM, the account is likely compromised.
  • Prioritize investigation leads. Connections to IPs in known hosting regions for C2 infrastructure (certain data centers, bulletproof hosting providers) warrant closer scrutiny.

Limitations of GeoIP:

  • GeoIP databases are approximate — accuracy varies from city-level to country-level depending on the ISP and region.
  • VPNs, Tor, and proxies can mask the true origin. An IP geolocated to the Netherlands may actually be a VPN exit node used by someone anywhere in the world.
  • GeoIP data changes over time. IP blocks are reassigned between ISPs and regions. Use a current GeoIP database for active investigations.

Common GeoIP tools include MaxMind GeoLite2 (free database), Wireshark's built-in GeoIP integration, and online lookup services. WHOIS queries provide registrant and ISP information that complements GeoIP data.

5 Types of Network Evidence

Not all network evidence is created equal. The following ranking orders evidence types from richest (most complete, most forensically valuable) to least rich (summarized, less detail). Understanding this hierarchy helps you know what to request first when responding to an incident.

  1. Full Packet Capture (PCAP)
    Complete copy of every packet — headers and payload. Allows full reconstruction of sessions, file extraction, credential recovery (if unencrypted), and protocol analysis. This is the gold standard of network evidence.
    Tool: Wireshark, tcpdump, NetworkMiner
  2. Session / Flow Data (NetFlow, sFlow, IPFIX)
    Summarized metadata about connections: source/destination IP, ports, protocol, byte count, duration, timestamps. No payload content. Useful for identifying communication patterns, beaconing, and data volume anomalies when full PCAP is unavailable.
    Tool: nfdump, SiLK, Wireshark Statistics
  3. Application Logs (Web Server, DNS, Email)
    Structured records of application-layer events. Web server logs capture every HTTP request (minus POST bodies). DNS logs capture every query. These are often the most available evidence because most servers enable logging by default.
    Tool: grep, awk, Splunk, ELK Stack
  4. Infrastructure Logs (Firewall, Router, DHCP)
    Record allow/deny decisions, routing events, and address assignments. Essential for correlating internal IPs with external traffic and establishing timelines. Quality varies by device configuration.
    Tool: grep, awk, Splunk, vendor-specific consoles
  5. IDS/IPS Alerts
    Pre-analyzed events triggered by signature matches. Useful as starting points but require validation — false positives are common. Alert data alone cannot reconstruct an attack.
    Tool: Snort, Suricata, vendor SIEM
  6. SIEM Aggregated Events
    Correlated, normalized events from multiple sources. Provides a high-level overview but abstracts away raw detail. Always go back to the original source (PCAP, raw logs) for forensic conclusions.
    Tool: Splunk, QRadar, Elastic SIEM
🧪 Forensic Examiner Note

In the Acme Corp scenario, you have evidence at three levels: full PCAP (richest), application logs (Apache), and infrastructure logs (firewall). This is an excellent evidence set. In many real investigations, you may only have flow data or logs — making PCAP preservation all the more critical when it is available.

6 Network Evidence Capture and Integrity

Just as you learned in Week 6 that disk images must be forensically sound, network evidence must also be captured, stored, and documented with integrity. A PCAP file is evidence — and it must be treated with the same chain-of-custody rigor as a forensic disk image.

Where you capture determines what you see. Placement of your capture point is a critical decision:

  • Network TAP (Test Access Point): A hardware device inserted inline on a network cable that copies all traffic to a monitoring port. TAPs are passive — they do not alter traffic and do not introduce a point of failure. This is the gold standard for forensic capture.
  • SPAN / Mirror Port: A switch feature that copies traffic from one or more ports to a designated monitoring port. Easier to set up than a TAP but can drop packets under heavy load. Acceptable for forensic use with documentation of limitations.
  • Host-based capture: Running tcpdump or Wireshark directly on a host. Captures only traffic to/from that host. Useful when network-level capture is unavailable, but the capture tool runs on the potentially compromised system — which raises integrity concerns.

Placement considerations for the Acme scenario:

  • A TAP between the DMZ and the firewall captures all traffic entering and leaving the web server segment — this is where the Acme PCAP was captured.
  • A TAP on the internal LAN side of the firewall would capture lateral movement traffic.
  • Capturing at both points provides the most complete picture.

Wireshark — GUI-based packet analyzer. Best for interactive analysis of existing PCAP files. Can also perform live capture, but is typically used on an analyst workstation rather than on the monitored host.

tcpdump — Command-line packet capture tool available on virtually every Linux/Unix system. Preferred for:

  • Capturing on headless servers (no GUI available).
  • Scripted or automated capture.
  • Remote capture via SSH.

Basic tcpdump examples:

# Capture all traffic on eth0, write to file
tcpdump -i eth0 -w /evidence/capture_20260413.pcap

# Capture only traffic to/from a specific host
tcpdump -i eth0 host 203.0.113.42 -w /evidence/suspect_host.pcap

# Read a PCAP and filter for HTTP traffic
tcpdump -r capture.pcap tcp port 80

PCAP vs. PCAPNG: PCAP is the classic format. PCAPNG (PCAP Next Generation) supports multiple interfaces, annotations, and richer metadata. Wireshark saves in PCAPNG by default. Both are widely supported by forensic tools.

A PCAP file is a forensic artifact. Apply the same integrity procedures you learned in Week 6 for disk images:

  1. Hash immediately after capture. Compute both MD5 and SHA-256 hashes of the PCAP file as soon as capture is complete: 
    md5sum /evidence/capture_20260413.pcap > capture_20260413.md5
sha256sum /evidence/capture_20260413.pcap > capture_20260413.sha256
  2. Document the capture in your chain of custody form. Record:
    • Who initiated the capture (analyst name, badge/ID).
    • When the capture started and stopped (UTC timestamps).
    • Where the capture point was (TAP location, SPAN port, host).
    • What was captured (interface, filter applied, file name, file size).
    • How (tool and version: e.g., tcpdump 4.99.4).
    • Hash values (MD5 and SHA-256).
  3. Store the original PCAP as read-only. Work on a copy for analysis. If your analysis tool modifies the file (e.g., adding annotations), the copy's hash will change — but the original's hash remains your integrity anchor.
  4. Verify hashes before analysis and before submission. Re-hash the original file at each stage to confirm it has not been altered.
⚠️ Critical Reminder

If you cannot demonstrate that a PCAP file has not been modified since capture, its evidentiary value is severely diminished. Defense attorneys will challenge the integrity of digital evidence. Your hash documentation and chain of custody form are your proof.

Capturing network traffic raises legal and ethical considerations you first encountered in Week 3:

  • Wiretap laws: In the United States, the federal Wiretap Act (18 U.S.C. § 2511) generally prohibits intercepting electronic communications. Key exceptions relevant to forensics:
    • Provider exception: An organization may monitor its own network to protect its rights and property.
    • Consent exception: If users have consented to monitoring (e.g., via an acceptable use policy or login banner), capture is permissible.
    • Law enforcement: Requires a court order or warrant.
  • Corporate policy: Ensure your organization has an acceptable use policy that notifies users their network activity may be monitored. This policy is your legal foundation for packet capture during incident response.
  • Scope limitation: Capture only what is necessary for the investigation. Avoid collecting unrelated personal communications when possible (minimization principle).

In the Acme Corp scenario, the incident commander has authorized the investigation, and Acme's acceptable use policy includes a monitoring notice. You are operating within legal bounds.

1. In the OSI model, which three layers are most important for network forensic analysis?

✅ Correct! Layers 3 (IP addresses for attribution), 4 (TCP/UDP ports and connection state), and 7 (application content like HTTP requests and DNS queries) provide the richest forensic evidence. This is the "forensic focus" highlighted in the OSI table above.
❌ Not quite. The correct answer is Layers 3, 4, and 7. Layer 3 (Network) gives you IP addresses for attribution. Layer 4 (Transport) gives you TCP/UDP ports and connection state (SYN scans, stream reconstruction). Layer 7 (Application) gives you the richest content — HTTP requests, DNS queries, email headers. While other layers have forensic value, these three are where most network forensic analysis occurs.

2. You capture a PCAP file during an incident response engagement. What is the first action you should take after the capture is complete?

✅ Correct! Just like forensic disk images (Week 6), the first step after capturing network evidence is to establish its integrity by computing hash values and documenting the capture in the chain of custody. You then work on a copy for analysis while preserving the hashed original.
❌ Not quite. The correct answer is B — compute hashes and document. Before you analyze, share, or process the PCAP in any way, you must establish its integrity baseline. This is the same principle from Week 6 (forensic imaging): hash → document → preserve original → work on copy. Without this step, the evidentiary value of the PCAP can be challenged.

3. In the network evidence hierarchy, which evidence type is the richest (most complete and most forensically valuable)?

✅ Correct! Full packet capture (PCAP) is the gold standard of network evidence. It contains complete copies of every packet — both headers and payload — allowing full session reconstruction, file extraction, and protocol analysis. All other evidence types are derived from or are summaries of what PCAP contains.
❌ Not quite. The correct answer is D — Full packet capture (PCAP). PCAP contains complete packet headers and payload data, making it the richest evidence source. NetFlow provides only metadata (no content). IDS alerts are pre-filtered and may contain false positives. SIEM events are aggregated summaries. When available, always prioritize PCAP as your primary evidence source.
