1. Firewall Log Analysis

1.1  What Firewalls Record and Why It Matters

Every enterprise network routes traffic through at least one firewall — a chokepoint that decides what enters and leaves the network. Because firewalls sit at the boundary, their logs provide a high-level chronological record of allowed and denied connections. While a PCAP captures full packet content, a firewall log captures decisions: was this traffic permitted or blocked, and by which rule?

Common Firewall Log Fields

Table 1 – Typical Fields in a Firewall Log Entry
FieldDescriptionForensic Value
TimestampDate and time the event was recordedAnchors events on the investigation timeline; must be normalized to UTC
ActionALLOW, DENY, DROP, REJECTShows whether traffic succeeded — a denied connection still proves the attempt
ProtocolTCP, UDP, ICMP, etc.Identifies service type; unexpected protocols (e.g., ICMP tunneling) are red flags
Source IPOriginating IP addressIdentifies attacker or compromised host
Source PortEphemeral port on the sourceCorrelates with PCAP session tracking
Destination IPTarget IP addressIdentifies the targeted internal asset
Destination PortService port (80, 443, 22, 3389, etc.)Reveals targeted service; unusual ports suggest backdoors or tunneling

Bytes / PacketsVolume transferred (if logged)Large outbound transfers may indicate exfiltration
Interface / ZoneWhich network segmentTracks lateral movement between zones (DMZ → internal)
1.2  Annotated Firewall Log Entry

Below is a single line from a pfSense / iptables-style firewall log. Each field is color-coded and labeled.

2026-04-12T03:14:22Z  ALLOW  TCP  198.51.100.47:49832 →  10.0.1.25:443  rule=12  bytes=234891
Timestamp Action Protocol Source IP Port Dest IP / Bytes Rule ID

Reading this entry: On April 12, 2026 at 03:14:22 UTC, the firewall allowed a TCP connection from external IP 198.51.100.47 (ephemeral port 49832) to internal host 10.0.1.25 on port 443 (HTTPS). Rule 12 matched. Approximately 235 KB were transferred.

Now Consider a Denied Entry

2026-04-12T03:14:58Z  DENY  TCP  198.51.100.47:49840 →  10.0.1.25:3389  rule=default-deny

Forensic insight: The same external IP, just 36 seconds later, attempted to reach port 3389 (RDP) on the same internal host — and was denied by the default-deny rule. This pattern (successful HTTPS followed by an RDP probe) suggests reconnaissance or lateral movement attempt after initial access. The denied connection proves the intent even though the traffic never reached the target.

🔍 Forensic Principle — Denied ≠ Irrelevant

Denied connections are some of the most forensically valuable entries. They reveal what the attacker tried to do — scanning activity, blocked lateral movement, and failed exfiltration attempts. Always include denied traffic in your analysis.

1.3  Common Forensic Patterns in Firewall Logs
  • Port scanning: Many denied connections from a single source to sequential ports on one or more targets, often within seconds.
  • Beaconing: Periodic outbound ALLOW entries to the same external IP at regular intervals (e.g., every 60 seconds) — characteristic of command-and-control (C2) callbacks.
  • Large outbound transfers: Unusually high byte counts on allowed outbound connections, especially to IP addresses not associated with normal business traffic — potential data exfiltration.
  • Protocol anomalies: Allowed traffic on unexpected protocols (DNS over TCP with large payloads, ICMP with unusual sizes) may indicate tunneling.
  • Rule-change artifacts: A connection that was previously denied suddenly being allowed may indicate the attacker modified firewall rules (check admin logs).

2. IDS/IPS Alert Analysis

2.1  IDS vs. IPS — Detection vs. Prevention

An Intrusion Detection System (IDS) passively monitors network traffic and generates alerts when patterns match known attack signatures or statistical anomalies. An Intrusion Prevention System (IPS) sits inline and can actively block traffic matching its rules. Many modern tools (Snort, Suricata) operate in either mode.

Table 2 – IDS vs. IPS Comparison
FeatureIDSIPS
PositionPassive tap / SPAN portInline (traffic flows through it)
Action on matchAlert onlyAlert + block / drop / reset
Risk of false positiveAlert noise, no traffic impactLegitimate traffic may be blocked
Forensic valueRecord of suspicious activityRecord + evidence that traffic was stopped
⚠️ Critical Concept — An IDS Alert Is a Hypothesis, Not a Conclusion

An IDS alert means the traffic matched a pattern — it does not confirm that an attack succeeded. Alerts must be validated against other evidence sources: Did the firewall allow the traffic? Does the target host show signs of compromise in its event logs or memory? Was the vulnerability even present on the target? Treating an unvalidated alert as proof of compromise is a common and serious analytical error.

2.2  Reading a Snort/Suricata Alert (Conceptual)

Snort and Suricata are the most widely deployed open-source IDS/IPS engines. Their alert format follows a consistent structure. You do not need to write rules for this course, but you must be able to read and interpret an alert.

Annotated Snort Alert

[1] [1:2024897:3] ET TROJAN Possible Cobalt Strike Beacon Activity [A Network Trojan was Detected] [Priority: 1] 04/12-03:14:25.003421 10.0.1.25:443 -> 198.51.100.47:49832
Severity Signature ID (SID) Alert Message Classification Timestamp Source Destination

Breaking Down the Fields

  • Signature ID (SID) 1:2024897:3: Generator ID 1 (Snort), rule 2024897, revision 3. The "ET" prefix indicates the rule comes from the Emerging Threats community ruleset.
  • Alert Message: Human-readable description — here, suspected Cobalt Strike C2 beacon traffic.
  • Classification: Broad category assigned by the rule author — "A Network Trojan was Detected."
  • Priority 1: Highest severity. Priorities typically range from 1 (critical) to 4 (informational).
  • Timestamp: When the packet that triggered the alert was observed. Note this is in the IDS sensor's local time — you must know the sensor's time zone to correlate with other logs.
  • Source → Destination: Here, the response traffic from the internal server (10.0.1.25:443) going back to the external IP. The IDS flagged the outbound payload as matching C2 beacon patterns.

True Positives, False Positives, and Validation

Table 3 – Alert Classification Matrix
Attack Actually OccurredNo Attack Occurred
Alert FiredTrue Positive (TP)False Positive (FP)
No AlertFalse Negative (FN)True Negative (TN)

Validation steps for an IDS alert:

  1. Check the firewall log — Was this traffic allowed or denied?
  2. Check the target host — Does the OS/application have the vulnerability the rule targets?
  3. Examine the PCAP — Does the full packet content confirm malicious payload, or was the signature match coincidental?
  4. Check host artifacts — Event logs, running processes (memory forensics), file system changes.

3. DNS Query Logs and DNS Forensics

3.1  Why DNS Matters in Forensics

The Domain Name System (DNS) is involved in virtually every network connection. Before a host connects to evil-c2-server.example.com, it must resolve that name to an IP address. DNS query logs — from internal DNS servers, endpoint DNS caches, or passive DNS databases — provide a map of every domain the network communicated with, often including traffic that was encrypted and invisible to content-level inspection.

Key DNS Record Types for Forensics

Table 4 – DNS Record Types and Forensic Relevance
Record TypePurposeForensic Relevance
AMaps domain → IPv4 addressCore resolution record; reveals which IPs a domain pointed to at query time
AAAAMaps domain → IPv6 addressSame as A record for IPv6; attackers may use IPv6 to bypass IPv4-only monitoring
MXMail exchange server for a domainIdentifies mail infrastructure; relevant in phishing and BEC investigations
TXTArbitrary text (SPF, DKIM, DMARC, etc.)SPF/DKIM/DMARC records validate email authenticity; also abused for DNS tunneling data exfiltration
CNAMECanonical name aliasCan chain to malicious infrastructure; used in domain fronting
NSAuthoritative name server for a zoneNS record changes may indicate domain hijacking
3.2  Suspicious DNS Activity and Tunneling Indicators

Red Flags in DNS Logs

  • Queries to newly registered domains (NRDs): Domains registered within the past 30 days are disproportionately associated with malicious activity.
  • High query volume to a single domain: Normal DNS involves diverse queries; hundreds of queries per minute to one domain may indicate C2 beaconing or tunneling.
  • Unusually long subdomain labels: DNS tunneling encodes data in subdomain names, producing queries like aGVsbG8gd29ybGQ.tunnel.evil.com. Normal subdomains rarely exceed 20–30 characters; tunneling labels often reach the 63-character per-label limit.
  • High volume of TXT record queries: TXT records can carry larger payloads than A records, making them preferred for DNS tunneling data return channels.
  • Queries to known-bad domains: Threat intelligence feeds maintain blocklists of domains associated with malware, phishing, and C2.
  • NXDOMAIN floods: Many queries returning "no such domain" may indicate a domain generation algorithm (DGA) — malware generating random domain names hoping one resolves to an active C2 server.

DNS Tunneling — Conceptual Overview

DNS tunneling encodes arbitrary data (commands, exfiltrated files) within DNS queries and responses. Because DNS traffic (port 53) is almost universally allowed through firewalls, it provides a covert channel even in tightly controlled networks.

Normal query:    www.example.com                    → A record → 93.184.216.34
Tunneling query: dGhpcyBpcyBhIHRlc3Q.c2.evil.com   → TXT record → [encoded response data]

The Base64-like subdomain label carries exfiltrated data; the TXT response carries C2 commands. Detection relies on statistical analysis: label length, query frequency, entropy of subdomain strings, and ratio of TXT queries.

WHOIS and Passive DNS

  • WHOIS: A protocol for querying domain registration databases. Returns registrant information (often privacy-protected), registration/expiration dates, and name servers. Forensically useful for identifying domain ownership and registration recency.
  • Passive DNS (pDNS): Databases that record historical DNS resolution data — "what IP did this domain resolve to on a given date?" Unlike active DNS queries that return only the current answer, passive DNS preserves the historical record, which is critical when attackers rotate infrastructure.
🔗 Cross-Module Connection — Windows DNS Cache (Week 8)

In Week 8 (Windows Forensics), you examined the local DNS resolver cache using ipconfig /displaydns as a host-based artifact. That cache provides a snapshot of recently resolved domains from the endpoint's perspective. Correlating the host DNS cache with the network DNS server logs lets you confirm which machine made a suspicious query — the server log shows the query source IP, and the host cache confirms the resolution was cached locally.

4. Router and Switch Logs

4.1  Infrastructure Logs — Syslog, MAC Tables, ARP, and DHCP

Routers and switches form the internal plumbing of the network. While they typically do not perform deep packet inspection, they generate syslog messages and maintain tables that are essential for one of the most fundamental forensic questions: Which physical device had this IP address at this time?

Key Evidence Sources from Network Infrastructure

Table 5 – Router/Switch Evidence Sources
SourceWhat It RecordsForensic Use
Syslog Timestamped messages: interface up/down, authentication events, configuration changes, routing changes Detects unauthorized configuration changes, identifies when links went down (anti-forensic activity such as disconnecting a device), admin login events
MAC Address Table Maps MAC addresses to switch ports (volatile — entries age out) Identifies which physical switch port (and therefore which network jack / location) a device was connected to
ARP Table Maps IP addresses to MAC addresses (volatile — entries expire) Resolves "which MAC address had IP 10.0.1.25?" — bridges the gap between Layer 3 (IP) and Layer 2 (MAC)
DHCP Lease Logs Records IP address assignments: MAC address, assigned IP, lease start/end time, hostname Critical for host identification: When a firewall log shows traffic from IP 10.0.1.25, the DHCP lease log tells you which device (by MAC address and hostname) held that IP at that specific time
⏳ Volatility Warning — Collect Early

MAC address tables and ARP tables are volatile — entries age out in minutes. DHCP lease logs are more persistent but may be overwritten by rotation. In an incident, capturing these tables early (or ensuring centralized syslog collection) is essential. By the time a forensic examiner arrives days later, the ARP and MAC tables will have changed.

DHCP Lease Log — The IP-to-Device Rosetta Stone

Consider this scenario: A firewall log shows suspicious outbound traffic from 10.0.1.25 at 03:14 UTC. But IP addresses are assigned dynamically — 10.0.1.25 could be a different device tomorrow. The DHCP lease log resolves this:

2026-04-12 02:58:11 DHCPACK  10.0.1.25  MAC=a4:83:e7:2b:11:f0  Host=ACCT-WS-042  Lease=3600s

This confirms that at 02:58 UTC (16 minutes before the suspicious traffic), IP 10.0.1.25 was assigned to MAC address a4:83:e7:2b:11:f0, hostname ACCT-WS-042, with a one-hour lease. The suspicious traffic at 03:14 falls within this lease period, so the device is identified.

5. Email Header Analysis (Introduction Only)

📋 Scope Notice — Introduction Only

This section provides a conceptual introduction to email header analysis. Full hands-on email forensics — including detailed header parsing, tracing email routing paths, and analyzing phishing campaigns — is covered in Week 16: Email and Messaging Forensics. The goal here is to understand what email headers contain and why they matter as a network evidence source.

5.1  Email Header Structure

Every email message contains headers — metadata prepended to the message body by each mail server that handles the message. While most email clients hide headers by default, they contain a wealth of forensic information.

Key Header Fields

Table 6 – Important Email Header Fields
Header FieldDescriptionForensic Note
From:Displayed sender addressEasily spoofed — do not trust at face value
Return-Path:Envelope sender (where bounces go)May differ from From: in spoofed emails
Received:Added by each mail server in the delivery chainRead bottom-to-top to trace the actual routing path; includes server IPs and timestamps
Message-ID:Unique identifier assigned by the originating serverUseful for correlating related messages and detecting forgeries
Date:Timestamp set by the sending clientCan be forged; compare with Received: timestamps for consistency
X-Originating-IP:IP of the sending client (if added by the mail service)May reveal the sender's actual IP address

The Received: Chain

The Received: headers form a chain that traces the email's path from origin to destination. Each server that handles the message prepends its own Received: header at the top. To trace the route chronologically, read from bottom to top:

Received: from mail-gateway.victim.com (10.0.2.5)
    by mail-store.victim.com; Sun, 12 Apr 2026 03:15:02 +0000
Received: from relay.external-provider.net (203.0.113.50)
    by mail-gateway.victim.com; Sun, 12 Apr 2026 03:14:58 +0000
Received: from sender-smtp.suspicious-domain.com (198.51.100.80)
    by relay.external-provider.net; Sun, 12 Apr 2026 03:14:44 +0000

Reading bottom-to-top: The email originated from 198.51.100.80 (suspicious-domain.com), passed through an external relay at 203.0.113.50, entered the victim's mail gateway, and was delivered to the mail store.

SPF, DKIM, and DMARC — Authentication Concepts

  • SPF (Sender Policy Framework): A DNS TXT record that lists which IP addresses are authorized to send email for a domain. If the sending server's IP is not in the SPF record, the email may be spoofed.
  • DKIM (DomainKeys Identified Mail): A cryptographic signature added to the email header by the sending server. The receiving server verifies it against a public key published in DNS. A valid DKIM signature confirms the message was not altered in transit.
  • DMARC (Domain-based Message Authentication, Reporting, and Conformance): A policy framework that combines SPF and DKIM results and tells receiving servers what to do with messages that fail authentication (none, quarantine, or reject).

Red Flags in Email Headers

  • From: domain does not match Return-Path: domain
  • SPF result: fail or softfail
  • DKIM result: fail or absent when the domain publishes DKIM
  • Inconsistent timestamps between Received: hops
  • X-Originating-IP from an unexpected geographic region
  • Unusually short or broken Received: chain

6. Host-to-Network Evidence Correlation

Individual evidence sources tell partial stories. The power of digital forensics lies in correlating evidence across sources to build a coherent, defensible narrative. This section is the capstone of the network forensics unit — it ties together everything from Weeks 8, 11, and 12.

6.1  Evidence Correlation Matrix

The matrix below maps host-based artifacts to their network-based counterparts. When both sides agree, confidence in the finding increases dramatically.

Table 7 – Host-to-Network Evidence Correlation Matrix
Host Artifact Network Artifact Correlation Value
Volatility netscan — active connections from memory image PCAP sessions showing matching src/dst IP:port tuples Confirms the connection existed on both the host and the wire; memory shows the process responsible
Windows Event Log 4624 Type 10 (Remote Interactive / RDP logon) Firewall log ALLOW entry for source IP → target:3389 Confirms the RDP session was permitted through the firewall and the logon succeeded on the host
Linux auth.log — SSH Accepted publickey entries PCAP SSH traffic (encrypted, but connection metadata visible) Correlates SSH session start times and source IPs between host authentication logs and network traffic
Windows DNS cache (ipconfig /displaydns) DNS server query logs showing the same domain resolution Confirms which endpoint queried a suspicious domain; cache provides the host-side view, server log provides the network-side view with timestamps
Browser history / web cache (host file system) Web server access logs (Module 12.3) Browser history shows user intent; server log shows what was actually served; differences may reveal MITM or redirect attacks
Prefetch / Shimcache (evidence of program execution) IDS alert for malware C2 traffic Host shows the malicious executable ran; IDS shows the resulting network communication — together they prove execution and communication
DHCP client lease (host-side, e.g., Windows Event 1020) DHCP server lease log Both sides confirm the IP assignment; essential when the host's IP is the only identifier in network logs
6.2  Timeline Integration and Time Zone Normalization

Correlation depends on accurate timestamps. Different evidence sources may record time in different formats and time zones:

Table 8 – Common Timestamp Formats Across Evidence Sources

⏰ Best Practice — Normalize Everything to UTC

Before correlating events across sources, convert all timestamps to UTC. Document the original time zone of each source and the conversion applied. A one-hour time zone error can make events appear to happen in the wrong order, destroying the investigative timeline.

6.4  Mini-Case Study — Tracing an Attack from Entry to Exfiltration
💡 Purpose

This mini-case demonstrates cross-source correlation by tracing a simulated attack through four evidence sources: DNS logs, firewall logs, an IDS alert, and web server logs. It directly scaffolds the skills you will apply in Lab 11.

Scenario

An analyst at Initech Corporation notices an IDS alert for suspected Cobalt Strike beacon activity. The SOC initiates an investigation. Available evidence sources: firewall logs, IDS alerts, DNS server logs, web server access logs, and DHCP lease records.

Correlated Timeline

03:12:05 UTC DNS Log

Internal DNS server records a query from 10.0.1.25 for update-service.suspicious-domain.com → resolves to 198.51.100.47. This domain was registered only 6 days ago (per passive DNS / WHOIS).

03:12:06 UTC Firewall Log

ALLOW TCP 10.0.1.25:49830 → 198.51.100.47:443 rule=12 bytes=1204 — Outbound HTTPS connection from the internal host to the newly resolved IP is permitted by the firewall.

03:14:22 UTC Firewall Log

ALLOW TCP 198.51.100.47:49832 → 10.0.1.25:443 rule=12 bytes=234891 — A second connection, this time inbound from the external IP. ~235 KB transferred. (Note: This is the return traffic of the C2 channel.)

03:14:25 UTC IDS Alert

[1:2024897:3] ET TROJAN Possible Cobalt Strike Beacon Activity — Priority 1. Source: 10.0.1.25:443 → Dest: 198.51.100.47:49832. The IDS flagged the outbound payload pattern as matching known Cobalt Strike beacon encoding.

03:14:58 UTC Firewall Log

DENY TCP 198.51.100.47:49840 → 10.0.1.25:3389 rule=default-deny — The external IP attempts RDP access to the same host. Blocked by default-deny. This suggests the attacker attempted to pivot to direct remote access after establishing the C2 channel.

03:18:44 UTC DNS Log

Burst of 47 TXT record queries from 10.0.1.25 for subdomains of exfil.suspicious-domain.com — subdomain labels are 50+ characters of Base64-encoded data. Consistent with DNS tunneling exfiltration.

03:19:02 UTC Web Server Log

Internal web server (10.0.2.10) logs: 10.0.1.25 - - "GET /internal/hr/salary-data.csv HTTP/1.1" 200 1482039 — The compromised host downloaded a 1.4 MB salary data file from an internal web application. This is the data being exfiltrated via DNS tunneling.

Cross-reference DHCP Lease

DHCP server log confirms: 10.0.1.25 was assigned to MAC a4:83:e7:2b:11:f0, hostname ACCT-WS-042 (Accounting workstation 042), lease active from 02:58 to 03:58 UTC. The compromised device is now identified.

Correlation Summary

Table 9 – Mini-Case Evidence Correlation Summary
FindingEvidence Sources
C2 domain resolved by compromised hostDNS log (query from 10.0.1.25) + passive DNS (domain age: 6 days)
C2 connection establishedFirewall log (ALLOW to 198.51.100.47:443) + IDS alert (Cobalt Strike signature)
Lateral movement attemptedFirewall log (DENY RDP from same external IP)
Internal data accessedWeb server log (GET salary-data.csv, 200 OK, 1.4 MB)
Data exfiltrated via DNS tunnelingDNS log (47 TXT queries with long Base64 subdomains)
Host identifiedDHCP lease log (10.0.1.25 = ACCT-WS-042)
🔍 Key Takeaway — No Single Source Tells the Whole Story

The IDS alert alone told us "possible Cobalt Strike." Only by correlating with the DNS log (newly registered domain), firewall log (allowed connection + blocked RDP pivot), web server log (internal data access), and DHCP lease (host identification) could we reconstruct the full attack chain: initial C2 establishment → failed lateral movement → internal data theft → DNS tunneling exfiltration.

🔗 Cross-Module Connection — Windows Event ID 4624 Type 10 (Week 8)

In Week 8 (Windows Forensics), you analyzed Windows Security Event Log entries including Event ID 4624, Logon Type 10 (Remote Interactive / RDP). In the mini-case above, the attacker's RDP attempt was blocked by the firewall — so no 4624 Type 10 event would appear on the target host. The absence of the host event, combined with the firewall DENY entry, confirms the lateral movement was prevented. If the firewall had allowed it, you would expect to see a 4624 Type 10 event on ACCT-WS-042 with the source IP 198.51.100.47.

Module 12.4 Summary

📗 Key Takeaways
  • Firewall logs record allow/deny decisions at the network boundary — denied entries prove attempted activity even when traffic was blocked.
  • IDS/IPS alerts are hypotheses based on signature matches — they must be validated against other evidence sources before drawing conclusions.
  • DNS logs reveal every domain the network communicated with; suspicious patterns include newly registered domains, long subdomain labels (tunneling), TXT query bursts, and DGA-style NXDOMAIN floods.
  • Router/switch logs and DHCP leases bridge the gap between IP addresses and physical devices — essential for host identification.
  • Email headers contain routing metadata (Received chain) and authentication results (SPF/DKIM/DMARC) that are foundational to email forensics (covered in depth in Week 16).
  • Cross-source correlation is the core forensic skill: no single evidence source tells the complete story. Normalize timestamps to UTC, map host artifacts to network artifacts, and build a unified timeline.
