Windows Forensics – Key Testable Concepts

  • Registry hive structure: Know the five root hives (HKLM, HKU, HKCR, HKCU, HKCC), their on-disk file locations (e.g., SAM, SYSTEM, SOFTWARE, NTUSER.DAT), and what artifact categories each stores.
  • User activity artifacts: ShellBags, UserAssist, RecentDocs, and MRU lists — what each records and where it resides in the registry.
  • Windows Event Logs: EVTX format, key log channels (Security, System, Application), and critical Event IDs (e.g., 4624/4625 logon events, 4688 process creation, 7045 service install).
  • NTFS artifacts: MFT structure, $STANDARD_INFORMATION vs. $FILE_NAME timestamps, Alternate Data Streams (ADS), and their forensic significance for timestomping detection.
  • Prefetch & AmCache: Purpose of Prefetch files (.pf), execution count and timestamps, AmCache.hve as evidence of program execution.
  • RegRipper & Zimmerman Tools: When and how each tool is used to parse registry hives and Windows artifacts; command-line basics.

✎ Self-Check – Windows Domain

An investigator discovers that the $STANDARD_INFORMATION Modified timestamp on a suspicious executable is 2026-01-15, but the $FILE_NAME Created timestamp is 2026-04-02. What does this discrepancy most likely indicate, and why is the $FILE_NAME attribute more trustworthy in this scenario?

Answer: The discrepancy is a strong indicator of timestomping — an anti-forensic technique where an attacker deliberately modifies $STANDARD_INFORMATION timestamps to blend the file with legitimate system files or hide its true creation date.

Explanation: The $STANDARD_INFORMATION attribute can be modified by user-level APIs (e.g., SetFileTime), making it easy for attackers to alter. The $FILE_NAME attribute, however, is maintained by the NTFS kernel-mode driver and cannot be changed by standard user-level tools. Because the $FILE_NAME Created date (April 2, 2026) is later than the $STANDARD_INFORMATION Modified date (January 15, 2026), the file could not logically have been modified before it existed — confirming tampering with $STANDARD_INFORMATION.

Linux Forensics – Key Testable Concepts

  • Filesystem layout & evidence locations: Key directories (/var/log, /etc, /home, /tmp) and what forensic artifacts each contains.
  • Log analysis: auth.log / secure for authentication events, syslog / journalctl for system events, wtmp/btmp/lastlog binary logs, and Apache/Nginx access logs.
  • User artifacts: Bash history (.bash_history), .bashrc/.profile persistence mechanisms, cron jobs (/etc/crontab, user crontabs), and SSH keys (~/.ssh/).
  • File system timestamps: MAC times (Modified, Accessed, Changed) in ext4, the stat command, and how noatime mount options affect evidence.
  • Command-line forensic tools: grep, find, strings, md5sum/sha256sum, dd for imaging, and lsof for open-file analysis.

✎ Self-Check – Linux Domain

During a Linux server investigation, you need to find every file under /var/www that was modified within the last 3 days and contains the string eval(base64_decode. Write the command pipeline you would use, and explain why this string pattern is suspicious.

Command:

find /var/www -type f -mtime -3 -exec grep -l "eval(base64_decode" {} \;

Explanation: The find command locates regular files (-type f) under /var/www modified within the last 3 days (-mtime -3). For each match, grep -l prints the filename if it contains the target string.

Why it's suspicious: eval(base64_decode(...)) is a classic PHP webshell/backdoor pattern. Attackers encode malicious payloads in Base64 to evade simple signature detection, then use eval() to execute the decoded string at runtime. Finding this pattern in recently modified web files is a strong indicator of compromise — likely a webshell planted after initial exploitation.

Memory Forensics – Key Testable Concepts

  • Why memory forensics matters: Volatile data is lost at power-off; memory captures running processes, network connections, decrypted data, and malware that never touches disk (fileless attacks).
  • Memory acquisition: Tools and methods (e.g., FTK Imager, WinPMEM, LiME for Linux), raw vs. crash dump formats, and order-of-volatility considerations.
  • Volatility framework: Profile selection, core plugins — pslist/psscan/pstree (process enumeration), netscan (network connections), malfind (injected code), dlllist, handles, cmdline, and filescan.
  • Detecting process injection & hiding: Comparing pslist vs. psscan to find hidden/unlinked processes (DKOM); malfind detecting PAGE_EXECUTE_READWRITE memory regions indicative of injection.
  • Indicators in memory: Suspicious parent-child process relationships (e.g., svchost.exe not spawned by services.exe), unusual DLLs, and extracted C2 IP addresses from network connection artifacts.
  • Integration with other domains: Memory artifacts corroborate disk-based evidence (e.g., a process visible in memory but absent from Prefetch may indicate fileless execution).

✎ Self-Check – Memory Domain

You run volatility -f memory.raw --profile=Win10x64 pslist and then psscan on the same image. psscan reveals a process named svch0st.exe (PID 4812) that does not appear in the pslist output. What technique does this suggest, and what should your next analysis steps be?

Answer: This suggests Direct Kernel Object Manipulation (DKOM) — the attacker unlinked the process's EPROCESS structure from the kernel's active process list so it is invisible to pslist (which walks that linked list). psscan, which scans memory using pool-tag signatures, finds it regardless.

Additional red flag: The name svch0st.exe (zero instead of the letter "o") is a common masquerading technique designed to look like the legitimate svchost.exe.

Next steps:

  • Run dlllist -p 4812 and handles -p 4812 to inspect loaded modules and open handles.
  • Run malfind -p 4812 to check for injected or executable memory regions.
  • Run netscan and filter for PID 4812 to identify any command-and-control (C2) connections.
  • Run cmdline -p 4812 to see the process's command-line arguments.
  • Dump the process executable with procdump -p 4812 and submit it for hash lookup / malware analysis.

Network Forensics – Key Testable Concepts

Week 12 covers seven learning objectives. The review below addresses all of them, including email header analysis, DNS forensics (Objective 6), and Quiz 3 readiness (Objective 7).

  • Packet capture fundamentals: PCAP vs. PcapNG formats, full packet capture vs. NetFlow/metadata, legal considerations for network monitoring, and capture placement (span port, tap, inline).
  • Wireshark analysis skills: Display filters (e.g., ip.addr==, tcp.port==, http.request.method==POST), following TCP streams, protocol hierarchy statistics, and exporting objects from HTTP/SMB streams.
  • TCP/IP & protocol analysis: TCP three-way handshake, HTTP request/response forensics, identifying cleartext credentials, and recognizing anomalous traffic patterns (beaconing intervals, unusual ports).
  • Detecting C2 & data exfiltration: Beaconing behavior (regular-interval callbacks), DNS tunneling indicators (high-entropy subdomain labels, TXT record abuse, abnormal query volume), large outbound transfers to unknown IPs, and encrypted channels on non-standard ports.
  • Network-based IOCs: Suspicious IP addresses, domain reputation, JA3/JA3S TLS fingerprints, User-Agent anomalies, and correlation with threat intelligence feeds.
  • Email header analysis & DNS forensics (Objective 6): Tracing email origin via Received: headers (bottom-up reading), SPF/DKIM/DMARC validation fields, identifying spoofed sender addresses. DNS forensic techniques — analyzing query logs, identifying fast-flux domains, and using passive DNS databases to track infrastructure changes.
  • Quiz 3 readiness (Objective 7): Ability to integrate network evidence with Windows, Linux, and memory artifacts to reconstruct attack timelines in scenario-based questions.

✎ Self-Check – Network Domain

In a PCAP, you observe a compromised workstation making DNS queries for subdomains like a3f7b2e1c9d4.data.evil-domain.com every 30 seconds, with the responses containing long TXT records. What attack technique does this indicate, and what Wireshark filter would you use to isolate this traffic for further analysis?

Answer: This is DNS tunneling — a technique where an attacker encodes data (commands or exfiltrated information) within DNS queries and responses to bypass firewalls that allow outbound DNS traffic.

Key indicators:

  • High-entropy subdomain labels (a3f7b2e1c9d4) — these are hex/base-encoded data, not legitimate hostnames.
  • Regular 30-second intervals — consistent beaconing cadence.
  • TXT record responses — TXT records can carry arbitrary data and are commonly abused for C2 return channels.

Wireshark filter:

dns.qry.name contains "evil-domain.com"

To further narrow to TXT queries specifically:

dns.qry.name contains "evil-domain.com" and dns.qry.type == 16

(DNS type 16 = TXT record.)

Cross-Domain Correlation: Connecting the Evidence

🔗 How the Four Domains Connect in a Real Investigation

Digital forensic investigations rarely involve a single evidence source. Quiz 3 may include scenario-based questions that require you to trace an attack across multiple domains. Here is a representative attack chain showing how evidence flows across all four areas:

  1. Initial Exploitation (Network): An attacker exploits a web application vulnerability. Network evidence: Wireshark captures show malicious HTTP POST requests containing exploit payloads; Apache access logs (Linux) record the initial request with a suspicious User-Agent string.
  2. Webshell Deployment (Linux): The exploit drops a PHP webshell on the Linux web server. Linux evidence: A new .php file appears in /var/www/html with eval(base64_decode content; auth.log may show no corresponding legitimate login — the file appeared via the web application, not SSH.
  3. Lateral Movement (Windows): The attacker pivots to an internal Windows workstation. Windows evidence: Security Event Log shows Event ID 4624 (Type 3 — network logon) from the web server's IP; a new service is installed (Event ID 7045); Prefetch files confirm execution of attacker tools like mimikatz.exe.
  4. In-Memory Persistence (Memory): Malware runs entirely in memory to evade disk-based detection. Memory evidence: psscan reveals a hidden process not in pslist; malfind identifies injected code with PAGE_EXECUTE_READWRITE permissions; netscan shows an active connection to an external C2 IP on port 443.
  5. Command & Control / Exfiltration (Network): The compromised workstation beacons to a C2 server and exfiltrates data. Network evidence: PCAP analysis reveals periodic DNS TXT queries (DNS tunneling) or HTTPS beaconing at regular intervals to a low-reputation domain; large outbound data transfers are visible in NetFlow data.
⚡ Key Takeaway for Quiz 3

When answering scenario-based questions, think across domains. Ask yourself: "What would this event look like in the registry? In memory? On the network?" The strongest forensic conclusions are those corroborated by multiple, independent evidence sources.

✎ Cross-Domain Self-Check

During an incident response, memory analysis of a Windows workstation reveals a process named outlook_update.exe (not a legitimate Microsoft process) with an active network connection to 198.51.100.47:8443. The process does not appear in Prefetch files or AmCache. On the network side, the firewall logs show the workstation has been connecting to 198.51.100.47 every 60 seconds for the past 5 days.

Question: (a) What type of malware behavior do these combined artifacts suggest? (b) Why is the absence of Prefetch/AmCache entries significant? (c) Name one additional evidence source from each of the other two domains (Linux and Windows Event Logs) that could further corroborate this finding.

(a) Behavior: This is a fileless malware / in-memory-only implant acting as a C2 beacon. The regular 60-second connection interval is classic beaconing behavior, and the process name mimics a legitimate application (masquerading).

(b) Absence of Prefetch/AmCache: Prefetch and AmCache record evidence of program execution from disk. Their absence means the malicious code was likely injected into a running process or executed entirely in memory (e.g., via PowerShell reflective loading, process hollowing, or DLL injection) — it was never written to disk as a standalone executable. This is the hallmark of fileless malware and makes disk-based detection extremely difficult.

(c) Additional corroborating evidence:

  • Windows Event Logs: Check for Event ID 4688 (Process Creation) with a suspicious parent process, or PowerShell Script Block Logging (Event ID 4104) that might capture the in-memory loading command. Also look for Event ID 7045 if a malicious service was installed for persistence.
  • Linux (if the C2 server or a pivot host is Linux-based): Examine firewall/iptables logs or tcpdump captures on the network boundary for the same 198.51.100.47 traffic; if a Linux server was the initial entry point, check auth.log for lateral-movement evidence and .bash_history for attacker commands.

Quiz 3 Strategy Tips

🎯 Approach the Quiz with a Plan
  • Read scenario questions carefully: Scenario-based questions often contain specific details (timestamps, process names, Event IDs) that point directly to the correct answer. Underline or note key facts before selecting your response.
  • Budget your time: Answer the questions you are most confident about first. Flag uncertain items and return to them. Do not spend disproportionate time on a single question.
  • Think cross-domain: Some questions will ask you to connect evidence from multiple domains (e.g., a memory artifact that explains a network observation). Use the attack-chain mental model from the Cross-Domain Correlation section above.
  • Watch for anti-forensic indicators: Several questions may test your ability to recognize timestomping, log clearing, process hiding (DKOM), or encrypted exfiltration. Know the detection method for each technique, not just the technique name.
  • Remember pre-Week 7 foundations: Expect 2–3 questions covering foundational topics such as evidence handling, chain of custody, imaging/hashing, and legal/ethical frameworks. Review your notes from Weeks 1–6 briefly.
