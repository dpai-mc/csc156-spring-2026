User-level Registry artifacts are among the most directly relevant evidence in digital forensic investigations because they tie specific actions to specific user accounts. While system hives tell you about the machine, NTUSER.DAT tells you about the person.

📖 1. Introduction & Context

In Module 7.2, you extracted system-level artifacts from the SYSTEM, SOFTWARE, and SAM hives. Those artifacts answered questions about the machine: What is its name? What time zone is it in? Who has accounts on it? What devices have been connected?

Now, in Module 7.3, the focus shifts to the person. The NTUSER.DAT hive file stores per-user configuration data and — critically for forensics — records of what each user did on the system:

  • What programs did they run? (UserAssist)
  • What files did they open? (RecentDocs)
  • What paths did they type in Explorer? (TypedPaths)
  • What did they search for? (WordWheelQuery)
  • What commands did they execute? (RunMRU)
  • What programs start automatically when they log in? (Per-user autoruns)

These artifacts are often the most directly relevant to investigations because they tie specific actions to a specific user account. A system-level artifact might tell you that a USB device was connected to the computer, but a user-level artifact tells you which user was interacting with that device.

🔍 From Machine to Person

Consider this progression across Modules 7.1–7.3:

  • Module 7.1: "The Registry is a hierarchical database stored in hive files." (Structure)
  • Module 7.2: "This computer is named DESKTOP-A1B2C3D, runs Windows 10, and has three user accounts." (Machine identity)
  • Module 7.3: "User jsmith ran CCleaner.exe 5 times, last on February 28 at 3:15 PM, and searched for 'how to delete browsing history'." (User behavior)

Module 7.3 is where forensic analysis becomes most investigatively powerful — and most directly useful in building a case.

📁 2. NTUSER.DAT Overview

NTUSER.DAT
A per-user Registry hive file that stores the settings and activity data for a specific Windows user account. Each user profile on the system has its own NTUSER.DAT file. When a user logs in, Windows loads their NTUSER.DAT into the Registry under HKEY_USERS\{SID}, and it becomes accessible as HKEY_CURRENT_USER (HKCU) for that session.

2.1 File Location

C:\Users\<username>\NTUSER.DAT

The file is marked with the Hidden and System file attributes, so it does not appear in standard Windows Explorer views. Forensic tools display all files regardless of attributes.

2.2 Associated Files

File Location Purpose
NTUSER.DAT C:\Users\<username>\ Primary user hive file — contains most per-user Registry data
NTUSER.DAT.LOG1 C:\Users\<username>\ Transaction log — contains uncommitted changes
NTUSER.DAT.LOG2 C:\Users\<username>\ Secondary transaction log
UsrClass.dat C:\Users\<username>\AppData\Local\Microsoft\Windows\ User-specific file associations and Shellbag data (covered in Module 7.4 / Week 8)

2.3 Forensic Extraction Strategy

⚠️ Critical — Extract ALL User Profiles

Extract NTUSER.DAT from every user profile in the forensic image — not just the suspected user. Other accounts may contain:

  • Relevant evidence: A suspect may have used a different account to perform certain actions
  • Baseline comparison: Normal activity patterns on other accounts help you identify what is unusual about the suspect's account
  • Unauthorized accounts: An attacker may have created a new account (identified in Module 7.2's SAM analysis) and used it for malicious activity

2.4 Connecting SAM to NTUSER.DAT

In Module 7.2, you enumerated user accounts from the SAM hive and mapped SIDs to profile paths using the ProfileList key in the SOFTWARE hive. Now you use that mapping to locate each user's NTUSER.DAT:

--- From Module 7.2: ProfileList mapping --- SID: S-1-5-21-3456789012-1234567890-9876543210-1001 ProfileImagePath: C:\Users\jsmith → NTUSER.DAT: C:\Users\jsmith\NTUSER.DAT SID: S-1-5-21-3456789012-1234567890-9876543210-1002 ProfileImagePath: C:\Users\temp_user → NTUSER.DAT: C:\Users\temp_user\NTUSER.DAT

🚀 3. UserAssist — Program Execution History

UserAssist

High Forensic Value NTUSER.DAT
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\{GUID}\Count
UserAssist
A Registry artifact that records which programs a user launched through the Windows graphical user interface (GUI), how many times each program was executed, the total focus time, and the timestamp of the last execution. UserAssist data is stored under GUID-identified subkeys within the user's NTUSER.DAT hive, with program paths encoded using ROT13 substitution.

3.1 Structure

UserAssist data is organized under two GUIDs that categorize the type of execution:

GUID Tracks
{CEBFF5CD-ACE2-4F4F-9178-9926F41749EA} Executable files (.exe) launched by the user through the GUI
{F4E57C4B-2036-45F0-A9AB-443BCFE33D9F} Shortcut files (.lnk) opened by the user — includes Start Menu and taskbar shortcuts

Under each GUID's Count subkey, each value represents one program or shortcut. The value name is the program path encoded in ROT13, and the value data is a binary structure containing execution metadata.

3.2 ROT13 Encoding

ROT13 (Rotate by 13)
A simple letter substitution cipher that replaces each letter with the letter 13 positions after it in the alphabet. A→N, B→O, C→P, ..., N→A, O→B, etc. Applying ROT13 twice returns the original text. ROT13 is not encryption — it provides no security whatsoever. Microsoft uses it in UserAssist purely as an obfuscation technique to prevent the casual observer from reading program paths in the Registry. Only alphabetic characters are rotated; numbers, symbols, backslashes, and other characters remain unchanged.

ROT13 Alphabet Mapping

Original ABCDEFGHIJKLM
ROT13 NOPQRSTUVWXYZ
Original NOPQRSTUVWXYZ
ROT13 ABCDEFGHIJKLM

Example Decoding

ROT13 encoded: P:\Hfref\wfzvgu\Qbjaybnqf\frghc.rkr
Decoded:       C:\Users\jsmith\Downloads\setup.exe

Letter mapping:
  P → C    (P is 13 letters after C)
  H → U    (H is 13 letters after U... or U is 13 after H)
  f → s    (lowercase also rotates: f → s)
  e → r
  r → e
  \, :, . → unchanged (non-alphabetic characters are not rotated)

3.3 UserAssist Value Data Structure

The binary data for each UserAssist entry (on Windows 7 and later) is a 72-byte structure containing:

Offset Size Field Description
0x00 4 bytes Session ID Session identifier
0x04 4 bytes Run Count Number of times the program was executed
0x08 4 bytes Focus Count Number of times the application gained focus
0x0C 4 bytes Focus Time Total milliseconds the application had focus
0x3C 8 bytes Last Execution Time Windows FILETIME — when the program was last run (UTC)

3.4 Example UserAssist Output

--- Registry Explorer: UserAssist entries for user jsmith --- Program: C:\Program Files\Mozilla Firefox\firefox.exe Run Count: 142 Focus Count: 387 Focus Time: 12:34:56 (total) Last Run: 2026-02-28 14:22:08 UTC Program: C:\Windows\System32\cmd.exe Run Count: 23 Focus Count: 45 Focus Time: 01:12:33 (total) Last Run: 2026-02-27 22:15:44 UTC Program: C:\Users\jsmith\Downloads\CCleaner64.exe Run Count: 5 Focus Count: 8 Focus Time: 00:04:12 (total) Last Run: 2026-02-28 15:30:17 UTC ← Anti-forensics tool — high investigative interest

3.5 Forensic Significance & Limitations

🔍 Forensic Significance

UserAssist is one of the most valuable per-user artifacts because it provides:

  • Program execution evidence: Proves a specific user launched a specific program
  • Frequency data: Run count shows how often a program was used — a single execution vs. habitual use
  • Temporal data: Last execution timestamp places the activity at a specific point in time
  • Anti-forensics detection: Execution of privacy tools (CCleaner, BleachBit, Eraser) is recorded here even if those tools later cleaned other artifacts
  • Usage patterns: Focus time reveals how long the user actually interacted with the application
⚠️ Limitations of UserAssist
  • GUI launches only: UserAssist records programs launched through the Windows GUI (double-clicking, Start Menu, taskbar). Programs launched from the command line (cmd.exe, PowerShell) or by scripts are not recorded in UserAssist (though cmd.exe itself would be recorded if launched via GUI)
  • No command-line arguments: UserAssist records the executable path but not the arguments passed to it
  • Only last execution time: Only the most recent execution timestamp is stored — previous execution times are overwritten
  • Can be cleared: A knowledgeable user can delete UserAssist entries, though the deletion itself may leave traces

📄 4. RecentDocs — Recently Accessed Files

RecentDocs

High Forensic Value NTUSER.DAT
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs
RecentDocs
A Registry artifact that records files recently opened by the user through Windows Explorer or the standard Open/Save dialog boxes. Entries are organized both as a master list and by file extension, providing a comprehensive record of what files the user has accessed. Each entry contains the filename and a link to the corresponding .lnk (shortcut) file in the user's Recent folder.

4.1 Structure

The RecentDocs key has a hierarchical structure:

HKCU\...\Explorer\RecentDocs\
  ├── (Default values) — Master MRU list of all recently opened files
  ├── .docx\            — Recently opened .docx files
  ├── .pdf\             — Recently opened .pdf files
  ├── .xlsx\            — Recently opened .xlsx files
  ├── .jpg\             — Recently opened .jpg files
  ├── .txt\             — Recently opened .txt files
  ├── .zip\             — Recently opened .zip files
  └── Folder\           — Recently opened folders

Each extension subkey contains MRU (Most Recently Used) values:

Value Name Type Description
MRUListEx REG_BINARY An ordered list of DWORD indices indicating the order in which files were accessed (most recent first)
0, 1, 2, ... REG_BINARY Each numbered value contains the filename (Unicode string) and a reference to the corresponding .lnk file

4.2 Example RecentDocs Output

--- Registry Explorer: RecentDocs for user jsmith --- RecentDocs (Master List): Key Last Write: 2026-02-28 16:45:12 UTC MRU Order: 7, 5, 3, 12, 0, 8, ... RecentDocs\.pdf: Key Last Write: 2026-02-28 16:45:12 UTC Entry 0: Q4_Financial_Report.pdf Entry 1: Employee_Handbook_2026.pdf Entry 2: Network_Diagram_Confidential.pdf RecentDocs\.xlsx: Key Last Write: 2026-02-27 11:30:55 UTC Entry 0: Customer_Database_Export.xlsx Entry 1: Salary_Schedule_2026.xlsx RecentDocs\.zip: Key Last Write: 2026-02-28 15:22:03 UTC Entry 0: backup_all_data.zip ← Potentially suspicious: large data backup on the same day as other suspicious activity

4.3 Forensic Significance & Limitations

🔍 Forensic Significance
  • File access evidence: Proves a user opened specific files — even if the files themselves have been deleted from the system
  • File type analysis: Extension subkeys reveal patterns of file access (e.g., accessing many .xlsx files could indicate data exfiltration from spreadsheets)
  • Filename evidence: Even without the file content, filenames like "Confidential_Client_List.xlsx" or "backup_all_data.zip" can be highly relevant
  • Temporal context: The key last-write timestamp on each extension subkey indicates when the most recent file of that type was accessed
  • Linked .lnk files: RecentDocs entries reference .lnk shortcut files in the user's Recent folder, which contain additional metadata (target path, timestamps, volume serial number, MAC address)
⚠️ Limitations of RecentDocs
  • Only Windows Explorer / Open-Save dialogs: Files opened directly by applications (without going through the standard dialog) may not be recorded
  • Filenames only: RecentDocs records the filename but not the full path — the full path is in the associated .lnk file
  • MRU list size: The list has a maximum size; older entries are pushed out as new ones are added
  • Can be cleared: Clearing "Recent Files" in Windows or running privacy tools can remove these entries

⌨️ 5. TypedPaths & RunMRU

TypedPaths

Medium Forensic Value NTUSER.DAT
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths
TypedPaths
A Registry artifact that records file system paths and UNC (Universal Naming Convention) network paths that the user manually typed into the Windows Explorer address bar. Unlike paths navigated to by clicking through folders, TypedPaths specifically captures what the user typed — indicating intentional, directed navigation to specific locations.

5.1 TypedPaths Structure

HKCU\...\Explorer\TypedPaths
  url1 (REG_SZ) = "C:\Users\jsmith\Documents\Confidential"
  url2 (REG_SZ) = "\\fileserver01\shared\accounting"
  url3 (REG_SZ) = "D:\Backup"
  url4 (REG_SZ) = "E:\"

Values are named url1 through urlN, with url1 being the most recently typed path. The list typically stores the last 25 entries.

🔍 Forensic Significance
  • Intentional navigation: Typing a path indicates the user knew the location and deliberately navigated there — this is stronger evidence of intent than clicking through folders
  • Network paths: UNC paths (e.g., \\fileserver01\shared\accounting) reveal network resources the user accessed, potentially on other machines
  • Removable media: Paths to drive letters like E:\ or D:\Backup may indicate access to USB devices or external drives
  • Sensitive locations: Typed paths to restricted directories demonstrate the user's awareness of and interest in those locations

RunMRU (Run Dialog History)

High Forensic Value NTUSER.DAT
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU
RunMRU
A Registry artifact that records commands typed into the Windows Run dialog (Win+R). Each entry stores the exact command string the user typed, providing direct evidence of command execution. The MRUList value tracks the order in which commands were entered.

5.2 RunMRU Structure

HKCU\...\Explorer\RunMRU
  MRUList (REG_SZ) = "dcba"  ← Order: d=most recent, a=oldest
  a (REG_SZ) = "cmd\1"
  b (REG_SZ) = "regedit\1"
  c (REG_SZ) = "notepad C:\Users\jsmith\Desktop\notes.txt\1"
  d (REG_SZ) = "\\192.168.1.50\c$\1"

Each value ends with \1 — this is a delimiter appended by Windows and is not part of the original command. The MRUList value indicates the order of entries, with the first character being the most recently typed command.

🔍 Forensic Significance
  • Direct command evidence: Shows exactly what the user typed in the Run dialog — including commands to launch programs, access network shares, or run system tools
  • Administrative tool access: Commands like cmd, regedit, powershell, mmc indicate the user was accessing administrative tools
  • Network access: UNC paths (e.g., \\192.168.1.50\c$) reveal attempts to access administrative shares on remote systems — a common lateral movement technique
  • Intent evidence: Using the Run dialog requires deliberate action — the user had to press Win+R and type a specific command

🔎 6. WordWheelQuery — Explorer Search History

WordWheelQuery

High Forensic Value NTUSER.DAT
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\WordWheelQuery
WordWheelQuery
A Registry artifact that records search terms typed into the Windows Explorer search box (the search field in the upper-right corner of Explorer windows). Each entry stores the exact search string the user typed, and the MRUListEx value tracks the order in which searches were performed. This artifact captures what the user was looking for on the local system.

6.1 Structure

HKCU\...\Explorer\WordWheelQuery
  MRUListEx (REG_BINARY) = Order of entries (most recent first)
  0 (REG_SZ) = "confidential client list"
  1 (REG_SZ) = "salary"
  2 (REG_SZ) = "password"
  3 (REG_SZ) = "how to encrypt files"
  4 (REG_SZ) = "backup"
🔍 Forensic Significance

WordWheelQuery is extremely powerful forensic evidence because it reveals the user's intent and interests:

  • What were they looking for? Search terms directly reveal what information the user was trying to find on the system
  • Knowledge and intent: Searching for "confidential client list" demonstrates the user knew such a document existed and was actively seeking it
  • Anti-forensics awareness: Searches for terms like "how to delete history," "wipe hard drive," or "encryption tool" may indicate the user was attempting to cover their tracks
  • Investigation keywords: Search terms can guide the examiner to look for specific files or data that the user was interested in
  • Temporal context: The key last-write timestamp indicates when the most recent search was performed
💡 Important Distinction — Explorer Search vs. Web Search

WordWheelQuery records searches performed in Windows Explorer (the file manager), not web browser searches. Web search history is stored in browser-specific locations (browser databases, typed URLs, etc.). Both types of search evidence are forensically valuable but come from different sources.

6.2 Forensic Scenario

--- WordWheelQuery for user jsmith --- --- Key Last Write: 2026-02-28 15:18:44 UTC --- MRU Order (most recent first): 0: "how to permanently delete files" 1: "ccleaner" 2: "Q4 financial report" 3: "customer database" 4: "salary schedule 2026" 5: "network diagram" Analysis: The two most recent searches (entries 0-1) indicate the user was looking for file deletion methods and a specific privacy tool. Earlier searches (entries 2-5) targeted sensitive corporate documents. This pattern suggests the user first searched for and accessed sensitive files, then sought ways to cover their tracks.

🔄 7. Per-User Startup Programs (Persistence)

Per-User Autorun Keys

High Forensic Value NTUSER.DAT
HKCU\Software\Microsoft\Windows\CurrentVersion\Run HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce

In Module 7.2, you examined the system-wide autorun keys under HKLM\SOFTWARE. The per-user equivalents under HKCU serve the same purpose but apply only to a specific user's login session.

Per-User Autorun (Run Key)
Registry values under HKCU\...\Run that specify programs to execute automatically each time the specific user logs into Windows. Unlike system-wide autoruns (HKLM), per-user autoruns affect only one account and do not require administrator privileges to create — making them a common persistence mechanism for both legitimate software and malware.

7.1 Run vs. RunOnce

Key Behavior Forensic Relevance
Run Programs execute every time the user logs in Persistent programs — legitimate software (cloud sync, chat clients) or malware maintaining persistence
RunOnce Programs execute once at next login, then the entry is deleted One-time execution — software updates, post-install configuration, or malware that executes once and removes its trace

7.2 Example

--- Per-User Run key for user jsmith --- --- Key Last Write: 2026-02-25 08:12:33 UTC --- HKCU\...\Run OneDrive REG_SZ "C:\Users\jsmith\AppData\Local\Microsoft\OneDrive\OneDrive.exe" /background Spotify REG_SZ "C:\Users\jsmith\AppData\Roaming\Spotify\Spotify.exe" /minimized SysUpdate REG_SZ "C:\Users\jsmith\AppData\Local\Temp\svchost.exe" ↑ SUSPICIOUS: legitimate svchost.exe runs from System32, not from a user's Temp folder. This is a common malware technique — naming malware after legitimate system files.
🔍 Forensic Significance
  • Malware persistence: Per-user Run keys are a top persistence mechanism because they do not require admin privileges
  • Suspicious indicators: Programs running from Temp folders, AppData, or Downloads are suspicious — legitimate software typically installs to Program Files
  • Name masquerading: Malware often uses names of legitimate Windows processes (svchost.exe, explorer.exe, csrss.exe) but runs from non-standard locations
  • Timestamp: The key last-write timestamp shows when the most recent autorun entry was added or modified

🔧 9. Tool Walkthroughs

  1. Load the NTUSER.DAT hive: File → Load Hive → select the extracted NTUSER.DAT (with .LOG1/.LOG2 in the same directory)
  2. Navigate to UserAssist: Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist
  3. Select the executable GUID: Click on {CEBFF5CD-ACE2-4F4F-9178-9926F41749EA}\Count
  4. View decoded entries: Registry Explorer automatically decodes ROT13 and parses the binary data — you will see program paths, run counts, focus times, and last execution timestamps in the values panel
  5. Bookmark suspicious entries: Right-click → Add Bookmark for any anti-forensics tools, unusual executables, or programs relevant to the investigation
  6. Export findings: Use the export function to save selected entries for your report
💡 Tip

Registry Explorer's UserAssist parsing is superior to manual analysis because it handles the ROT13 decoding, binary structure parsing, and FILETIME conversion automatically. Always verify a few entries manually to confirm the tool is parsing correctly.

RegRipper provides targeted plugins for each NTUSER.DAT artifact:

rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p userassist
rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p recentdocs
rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p typedpaths
rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p runmru
rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p wordwheelquery
rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p run
rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -p mountpoints2

Run all applicable plugins at once:

rip.exe -r C:\Evidence\jsmith_NTUSER.DAT -a > jsmith_ntuser_report.txt

The -a flag runs all plugins applicable to the detected hive type and outputs the combined results. Redirecting to a text file (> report.txt) creates a comprehensive artifact report.

Example userassist plugin output:

--- RegRipper: userassist plugin --- --- NTUSER.DAT (user: jsmith) --- UserAssist - GUID: {CEBFF5CD-ACE2-4F4F-9178-9926F41749EA} 2026-02-28 15:30:17 UTC C:\Users\jsmith\Downloads\CCleaner64.exe (Count: 5) 2026-02-28 14:22:08 UTC C:\Program Files\Mozilla Firefox\firefox.exe (Count: 142) 2026-02-27 22:15:44 UTC C:\Windows\System32\cmd.exe (Count: 23) 2026-02-27 19:05:12 UTC C:\Program Files\7-Zip\7zFM.exe (Count: 8) 2026-02-25 10:30:00 UTC C:\Windows\explorer.exe (Count: 312)

Autopsy's Recent Activity ingest module automatically parses NTUSER.DAT artifacts:

  • Data Artifacts → Recent Documents: RecentDocs entries with filenames and timestamps
  • Data Artifacts → Run Programs: UserAssist entries with decoded program paths
  • Data Artifacts → USB Device Attached: MountPoints2 data correlated with system-level USB artifacts

Advantage: Autopsy correlates user-level artifacts with file system data — if a RecentDocs entry references a file, Autopsy can link directly to that file (or its deleted remnant) in the forensic image.

Limitation: Autopsy may not parse all NTUSER.DAT artifacts (e.g., WordWheelQuery, RunMRU, TypedPaths may require manual examination with Registry Explorer or RegRipper).

✅ 11. Knowledge Check — Module 7.3

📝 Test Your Understanding

Answer the following questions in your own words before revealing the model answers. These questions align with the Lab Activity 7 requirements for user-level artifact extraction.

1. Explain the forensic significance of UserAssist artifacts. What specific data points does UserAssist provide, and what are its key limitations?

Significance: UserAssist records which programs a user launched through the Windows GUI, providing program execution evidence tied to a specific user account. It captures the program path (ROT13 encoded), run count, focus count, total focus time, and last execution timestamp.

Data points: (1) Executable path, (2) Number of executions, (3) Focus time showing actual usage duration, (4) Last execution timestamp in UTC.

Limitations: (1) Only records GUI-launched programs — command-line executions are not captured; (2) Only stores the last execution timestamp — previous execution times are overwritten; (3) Does not record command-line arguments; (4) Can be deleted by a knowledgeable user or privacy tools.

2. What is ROT13, and why does Microsoft use it in UserAssist? Is it a form of encryption?

ROT13 is a simple letter substitution cipher that replaces each letter with the letter 13 positions after it in the alphabet (A→N, B→O, etc.). Applying ROT13 twice returns the original text. Only alphabetic characters are rotated; numbers, symbols, and special characters remain unchanged.

ROT13 is NOT encryption. It provides zero security — it is trivially reversible by anyone who knows what it is (or even by someone who doesn't, since forensic tools decode it automatically). Microsoft uses it purely as obfuscation to prevent casual observation of program paths in the Registry using tools like regedit.exe. It was never intended to hide data from forensic examiners.

3. You are analyzing a suspect's NTUSER.DAT and find the following WordWheelQuery entries (most recent first): "how to permanently delete files", "eraser download", "customer database export", "salary information". What narrative might these search terms suggest?

The search history suggests a progression of activity:

  1. Earlier searches ("salary information", "customer database export") indicate the user was looking for sensitive corporate data on the system
  2. Later searches ("eraser download", "how to permanently delete files") indicate the user then sought ways to securely delete files — potentially to cover their tracks after accessing or exfiltrating the sensitive data

Investigative next steps: Check UserAssist for execution of Eraser or similar tools, check RecentDocs for access to files matching the search terms, examine USB device history for potential data exfiltration, and check file system artifacts for evidence of file deletion.

4. Why should a forensic examiner extract NTUSER.DAT from every user profile on the system, not just the suspected user's profile?

Extracting all user profiles is important for several reasons:

  • Multiple account usage: A suspect may have used a different account to perform certain actions to obscure attribution
  • Baseline comparison: Activity patterns on other accounts help establish what is "normal" on this system, making it easier to identify anomalous behavior on the suspect's account
  • Unauthorized accounts: An attacker may have created a new account (identified in SAM analysis) and used it for malicious activity — you won't find this evidence if you only examine the primary user's profile
  • Completeness: Forensic thoroughness requires examining all available evidence. Failing to examine other profiles could be challenged in court as an incomplete investigation
5. Describe how you would use RecentDocs artifacts to support an investigation into suspected data theft. What information does RecentDocs provide, and what are its limitations?

RecentDocs provides: Filenames of recently opened files organized by extension, MRU ordering showing which files were accessed most recently, key last-write timestamps showing when the most recent file of each type was opened, and links to .lnk shortcut files that contain additional metadata (full path, timestamps, volume information).

In a data theft investigation: RecentDocs can show that the suspect accessed specific sensitive files (e.g., "Customer_Database_Export.xlsx", "Salary_Schedule_2026.xlsx"). Even if the files themselves have been deleted, the RecentDocs entries prove they were opened. The extension subkeys can reveal patterns — accessing many .xlsx files from a financial database, for example.

Limitations: (1) Records filenames but not full paths (full paths are in linked .lnk files); (2) Only captures files opened through Explorer or standard Open/Save dialogs; (3) Has a maximum list size — older entries are pushed out; (4) Can be cleared by privacy tools or manual deletion.

6. What are the key elements that must be documented for each artifact finding to ensure it is legally defensible?

Each artifact finding must be documented with:

  • Source evidence: The forensic image filename and verified hash
  • Hive file: Which hive file the artifact was extracted from (e.g., C:\Users\jsmith\NTUSER.DAT)
  • Full Registry path: The exact key and value path (e.g., HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\{GUID}\Count)
  • Value name and data: The specific value examined and its contents
  • Timestamp: Key last-write timestamp and/or artifact-specific timestamps, with time zone notation (UTC)
  • Tool used: Which forensic tool was used for extraction and analysis (name and version)
  • Examiner interpretation: What the artifact means in the context of the investigation
  • Screenshots: Visual evidence of the tool output showing the artifact

📋 Module 7.3 Summary

In this module, you learned to extract and interpret per-user forensic artifacts from NTUSER.DAT:

  • NTUSER.DAT is a per-user hive file that maps to HKCU — extract it from every user profile, not just the suspect's
  • UserAssist records GUI program launches with ROT13-encoded paths, run counts, focus times, and last execution timestamps
  • RecentDocs records recently opened files organized by extension, with MRU ordering and linked .lnk files
  • TypedPaths captures paths typed into the Explorer address bar — showing intentional navigation to specific locations
  • RunMRU records commands typed in the Run dialog (Win+R) — revealing administrative tool access and network share connections
  • WordWheelQuery captures Explorer search terms — directly revealing user intent and interests
  • Per-User Run/RunOnce keys identify programs that auto-start at user login — a primary persistence mechanism for malware

📖 Key Terms — Module 7.3

Term Definition
NTUSER.DAT Per-user Registry hive file located in each user's profile directory; maps to HKCU when the user is logged in
UserAssist Registry artifact recording GUI-launched program execution history including run count, focus time, and last execution timestamp
ROT13 A simple letter substitution cipher (rotate by 13 positions) used to obfuscate program paths in UserAssist entries; not encryption
RecentDocs Registry artifact recording recently opened files, organized by file extension with MRU ordering
MRU (Most Recently Used) An ordering scheme where items are ranked by how recently they were accessed; the most recent item appears first
TypedPaths Registry artifact recording file system and UNC paths typed into the Windows Explorer address bar
RunMRU Registry artifact recording commands typed into the Windows Run dialog (Win+R)
WordWheelQuery Registry artifact recording search terms typed into the Windows Explorer search box
Persistence Mechanism Any technique that ensures a program (legitimate or malicious) survives system reboots and continues executing; Run keys are a common example
Name Masquerading A malware technique where malicious executables are given the same name as legitimate system files (e.g., svchost.exe) but placed in non-standard locations
UNC Path Universal Naming Convention path format (\\server\share) used to access network resources; presence in TypedPaths or RunMRU indicates network activity
