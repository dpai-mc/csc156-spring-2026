System-level Registry artifacts provide the foundational context for any Windows forensic examination: identifying the machine, establishing the time reference, enumerating users, and documenting connected devices. These artifacts are typically the first data points an examiner extracts when beginning a Windows forensic case.

📖 1. Introduction & Context

In Module 7.1, you learned the structure of the Windows Registry — its five root keys, keys, subkeys, values, data types, hive files, and transaction logs. Now you will apply that foundational knowledge to extract specific system-level forensic artifacts from three critical hive files:

Hive File Location on Disk What It Contains
SYSTEM C:\Windows\System32\config\SYSTEM Hardware configuration, services, drivers, boot settings, USB history, time zone, computer name
SOFTWARE C:\Windows\System32\config\SOFTWARE Installed software, OS version, network profiles, application settings, autorun programs
SAM C:\Windows\System32\config\SAM Local user accounts, password hashes, account policies, login history, account status

Why System-Level Artifacts Come First

When beginning a Windows forensic examination, system-level artifacts are always analyzed first because they provide the essential context needed to correctly interpret all other evidence:

  1. Machine identification: Computer name confirms you are examining the correct system
  2. Time zone: Required to convert UTC timestamps to local time — without this, all timestamps in your report could be wrong
  3. User account enumeration: Identifies all accounts on the system, their SIDs, and their status — this determines which NTUSER.DAT files to examine in Module 7.3
  4. OS version: Different Windows versions store artifacts in different locations; knowing the version ensures you look in the right places
  5. Device history: USB and storage device connections establish what external media was connected to the system
🔍 Forensic Workflow

Think of system-level artifacts as the "case setup" phase. Just as you would not begin reading a book in the middle, you should not begin analyzing user activity (Module 7.3) without first establishing the system context. Every forensic report begins with system identification data before presenting user activity findings.

🖥️ 2. SYSTEM Hive Artifacts

The SYSTEM hive (HKLM\SYSTEM) is one of the most forensically valuable hive files. It contains hardware configuration, service settings, and critical system identification data. All paths in this section are relative to HKLM\SYSTEM.

Artifact: Computer Name

SYSTEM Hive
HKLM\SYSTEM\CurrentControlSet\Control\ComputerName\ComputerName Value: ComputerName (REG_SZ)

What it tells you: The NetBIOS name assigned to the computer. This is the human-readable name used to identify the machine on a network and in forensic reports.

Example value: DESKTOP-A1B2C3D

🔍 Forensic Significance

The computer name is the first artifact extracted in most forensic examinations. It confirms you are analyzing the correct machine and is used throughout the forensic report to identify the evidence system. It also appears in network logs, event logs, and authentication records on other systems — enabling correlation across multiple evidence sources.

Artifact: Time Zone Configuration

SYSTEM Hive
HKLM\SYSTEM\CurrentControlSet\Control\TimeZoneInformation Key Values: TimeZoneKeyName (REG_SZ) — e.g., "Pacific Standard Time" Bias (REG_DWORD) — Offset from UTC in minutes ActiveTimeBias (REG_DWORD) — Current active offset including DST DaylightBias (REG_DWORD) — DST adjustment in minutes StandardBias (REG_DWORD) — Standard time adjustment

What it tells you: The time zone configured on the system and whether Daylight Saving Time (DST) adjustments apply.

⚠️ Critical — Time Zone Is Essential

The time zone setting is arguably the most important system artifact for forensic accuracy. All Registry key timestamps and most Windows file system timestamps are stored in UTC. To present timestamps in your forensic report as local times (which courts and investigators expect), you must know the system's time zone. An incorrect time zone conversion could place events at the wrong time of day — potentially undermining your entire analysis.

Understanding the Bias Value

The Bias value represents the number of minutes to subtract from UTC to get the local standard time. This is stored as a REG_DWORD:

Time Zone Bias Value (decimal) UTC Offset
Eastern Standard Time (EST) 300 UTC−5:00
Central Standard Time (CST) 360 UTC−6:00
Mountain Standard Time (MST) 420 UTC−7:00
Pacific Standard Time (PST) 480 UTC−8:00
Greenwich Mean Time (GMT) 0 UTC±0:00

The ActiveTimeBias value reflects the current offset, including any DST adjustment. During Pacific Daylight Time (PDT), the ActiveTimeBias would be 420 (UTC−7:00) instead of the standard 480 (UTC−8:00).

Artifact: Last Shutdown Time

SYSTEM Hive
HKLM\SYSTEM\CurrentControlSet\Control\Windows Value: ShutdownTime (REG_BINARY) — 8-byte Windows FILETIME in UTC

What it tells you: The date and time the system was last cleanly shut down. This is stored as a 64-bit Windows FILETIME value.

🔍 Forensic Significance

The last shutdown time helps establish when the computer was last used. If the shutdown time is significantly earlier than when the evidence was seized, it may indicate the system was turned off before the seizure. If the shutdown time is very recent (close to seizure time), it may indicate someone attempted to shut down the system — potentially to prevent live forensic capture or to trigger anti-forensics mechanisms. Compare this timestamp with file system activity and event logs for corroboration.

Artifact: Last Known Good Configuration

SYSTEM Hive
HKLM\SYSTEM\Select Values: Current (REG_DWORD) — Currently active ControlSet number Default (REG_DWORD) — Default ControlSet to use at boot LastKnownGood (REG_DWORD) — Last ControlSet that booted successfully Failed (REG_DWORD) — ControlSet that failed (if any)

What it tells you: Which ControlSet is currently active and which was last known to work. This is essential for knowing which ControlSet00X to examine (covered in detail in Section 5).

📦 3. SOFTWARE Hive Artifacts

The SOFTWARE hive (HKLM\SOFTWARE) contains system-wide application and operating system configuration data. It is typically the largest hive file on a Windows system, often exceeding 100 MB.

Artifact: Operating System Version

SOFTWARE Hive
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion Key Values: ProductName (REG_SZ) — e.g., "Windows 10 Pro" DisplayVersion (REG_SZ) — e.g., "22H2" CurrentBuild (REG_SZ) — e.g., "19045" CurrentBuildNumber (REG_SZ) — e.g., "19045" EditionID (REG_SZ) — e.g., "Professional" InstallDate (REG_DWORD) — Unix epoch timestamp of OS installation RegisteredOwner (REG_SZ) — Registered owner name RegisteredOrganization (REG_SZ) — Registered organization

What it tells you: The exact version, edition, and build number of the Windows installation, along with when it was installed and who it is registered to.

🔍 Forensic Significance

The OS version is critical because different versions of Windows store artifacts in different locations and formats. For example, Windows 10 version 1803+ stores timeline activity data that earlier versions do not. The InstallDate value (stored as a Unix epoch timestamp — seconds since January 1, 1970 UTC) tells you when the OS was installed, which helps establish the system's history. The RegisteredOwner may identify who set up the computer.

💡 Note — Unix Epoch vs. Windows FILETIME

The InstallDate value uses a Unix epoch timestamp (seconds since January 1, 1970), which is different from the Windows FILETIME format (100-nanosecond intervals since January 1, 1601) used by Registry key timestamps. Forensic tools handle both formats, but understanding the difference prevents misinterpretation. A Unix epoch value of 1709568000 converts to March 4, 2024.

Artifact: Installed Software

SOFTWARE Hive
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\{GUID or Name} Per-application values: DisplayName (REG_SZ) — Application name DisplayVersion (REG_SZ) — Version number Publisher (REG_SZ) — Software publisher InstallDate (REG_SZ) — Date installed (YYYYMMDD format) InstallLocation (REG_SZ) — Installation directory UninstallString (REG_SZ) — Uninstall command Also check for 32-bit applications on 64-bit systems: HKLM\SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\Uninstall\

What it tells you: A comprehensive list of software installed on the system, including application names, versions, publishers, and installation dates.

🔍 Forensic Significance

Installed software analysis can reveal:

  • Anti-forensics tools: Disk wiping software (e.g., Eraser, BleachBit), encryption tools (e.g., VeraCrypt), privacy cleaners (e.g., CCleaner)
  • Hacking tools: Network scanners, password crackers, exploitation frameworks
  • Relevant applications: Peer-to-peer file sharing, cloud storage clients, messaging applications, remote access tools
  • Timeline data: Installation dates help establish when applications were added to the system

Remember to check both the standard and WOW6432Node Uninstall paths on 64-bit systems to capture all installed software.

Artifact: Autorun / Startup Programs

SOFTWARE Hive
System-wide autoruns: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce Per-user autoruns (in NTUSER.DAT — covered in Module 7.3): HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce

What it tells you: Programs configured to run automatically when any user logs in (system-wide) or when a specific user logs in (per-user).

🔍 Forensic Significance

Autorun entries are a primary persistence mechanism for malware. If a system is suspected of being compromised, examining autorun locations can reveal malicious programs that execute automatically at startup. Legitimate software also uses these keys (antivirus, cloud sync clients, etc.), so the examiner must distinguish between normal and suspicious entries by researching each program found.

👤 4. SAM Hive Artifacts

The SAM (Security Account Manager) hive stores information about every local user account on the system. This is one of the first hives examined in any investigation because it tells you who has accounts on the machine.

Security Account Manager (SAM)
A database file in Windows that stores local user account information including usernames, relative identifiers (RIDs), password hashes, account creation dates, last login times, login counts, and account flags (enabled/disabled/locked). The SAM is stored at C:\Windows\System32\config\SAM and maps to HKLM\SAM.

Artifact: User Account Enumeration

SAM Hive
HKLM\SAM\SAM\Domains\Account\Users\Names\ (Each subkey = one local user account name) HKLM\SAM\SAM\Domains\Account\Users\{RID}\ Value: F (REG_BINARY) — Account metadata Value: V (REG_BINARY) — Username, full name, comment, password hashes

What it tells you: Every local user account that exists (or has existed) on the system. The F value contains structured binary data encoding critical account metadata.

Account Metadata in the F Value

The F binary value contains the following forensically relevant data (decoded by forensic tools):

Field Offset Description
Last Login Time 0x08 Windows FILETIME — last successful interactive login
Password Last Set 0x18 Windows FILETIME — when the password was last changed
Account Expiration 0x20 Windows FILETIME — when the account expires (0 = never)
Last Failed Login 0x28 Windows FILETIME — last unsuccessful login attempt
RID 0x30 Relative Identifier — unique numeric ID for the account
Account Flags 0x38 Bit flags indicating account status (enabled, disabled, locked, password required, etc.)
Login Count 0x42 Number of successful logins
Failed Login Count 0x44 Number of failed login attempts since last successful login
🔍 Forensic Significance

SAM account enumeration reveals:

  • All user accounts: Including accounts that may have been created by an attacker or unauthorized user
  • Account activity: Last login time and login count show which accounts are actively used
  • Suspicious accounts: Newly created accounts (recent password-set time with low login count) may indicate unauthorized access
  • Account status: Disabled or locked accounts may indicate administrative action or brute-force attempts
  • Password age: When the password was last changed can correlate with security incidents

4.1 Well-Known RIDs

Certain account RIDs are predefined by Windows and always have the same values:

RID (Hex) RID (Decimal) Account Notes
0x1F4 500 Administrator Built-in admin account; often disabled by default on Windows 10/11 but still present
0x1F5 501 Guest Built-in guest account; disabled by default
0x1F7 503 DefaultAccount System-managed default account
0x1F8 504 WDAGUtilityAccount Windows Defender Application Guard account
0x3E9+ 1001+ User-created accounts The first user-created account typically gets RID 1001

4.2 Mapping RIDs to SIDs

Security Identifier (SID)
A unique, variable-length identifier assigned to each security principal (user, group, or computer) in Windows. A local user SID follows the format: S-1-5-21-{machine-specific-values}-{RID}. For example, S-1-5-21-3456789012-1234567890-9876543210-1001 identifies the first user-created account on a specific machine. The SID is used as the subkey name under HKEY_USERS and in the path to the user's NTUSER.DAT profile.
💡 Connecting SAM to NTUSER.DAT

The SAM hive gives you the username → RID → SID mapping. Once you have the SID, you can identify which NTUSER.DAT file belongs to which user account. The profile path is stored at:

HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList\{SID}
Value: ProfileImagePath (REG_EXPAND_SZ)
Example: C:\Users\jsmith

This connection between the SAM hive and the SOFTWARE hive's ProfileList is essential for Module 7.3, where you will examine per-user artifacts in NTUSER.DAT.

⚙️ 5. Understanding CurrentControlSet

Many of the most important SYSTEM hive artifacts are located under CurrentControlSet. Understanding what this key represents and how to identify the correct one is essential.

ControlSet
A complete set of system configuration data including device drivers, services, and hardware settings. Windows maintains multiple ControlSets (typically ControlSet001 and ControlSet002) as a recovery mechanism. If a configuration change causes a boot failure, Windows can revert to the previous ControlSet. CurrentControlSet is a volatile symbolic link that points to whichever ControlSet is currently active — it exists only while Windows is running and does not exist in offline hive files.
⚠️ Critical — CurrentControlSet Does Not Exist Offline

When you open a SYSTEM hive file in a forensic tool, you will see ControlSet001, ControlSet002, etc. — but you will not see CurrentControlSet. That key is created dynamically by Windows at boot time. To determine which ControlSet was active when the system was last running, you must check:

HKLM\SYSTEM\Select
Value: Current (REG_DWORD)

If Current = 1, then ControlSet001 was the active ControlSet. If Current = 2, then ControlSet002 was active. Always verify this before examining ControlSet data.

5.1 Example: Determining the Active ControlSet

--- Registry Explorer: HKLM\SYSTEM\Select --- Key: HKLM\SYSTEM\Select Last Write: 2026-02-28 09:15:32 UTC Values: Current REG_DWORD 0x00000001 (1) ← ControlSet001 is active Default REG_DWORD 0x00000001 (1) LastKnownGood REG_DWORD 0x00000002 (2) Failed REG_DWORD 0x00000000 (0) ← No failed ControlSet

In this example, Current = 1 tells us that ControlSet001 was the active configuration. Therefore, when looking for artifacts like the computer name, time zone, or USB history, we should examine paths under ControlSet001.

5.2 Quick Reference: Common ControlSet Paths

Artifact Path (under active ControlSet)
Computer Name ControlSet001\Control\ComputerName\ComputerName
Time Zone ControlSet001\Control\TimeZoneInformation
Last Shutdown ControlSet001\Control\Windows
USB Device History ControlSet001\Enum\USBSTOR
Mounted Devices MountedDevices (root of SYSTEM, not under ControlSet)
Network Interfaces ControlSet001\Services\Tcpip\Parameters\Interfaces
Services ControlSet001\Services\

🔌 6. USB Device History

USB device tracking is one of the most powerful forensic capabilities of the Windows Registry. When a USB storage device is connected to a Windows system, the operating system records detailed information in multiple Registry locations. By correlating data from these locations, an examiner can determine:

  • What device was connected (vendor, product, serial number)
  • When it was first and last connected
  • Which drive letter it was assigned
  • Which user was logged in when it was connected

6.1 Primary USB Artifact Locations

USBSTOR — USB Storage Device Entries

HKLM\SYSTEM\CurrentControlSet\Enum\USBSTOR

Hive: SYSTEM

What it contains: An entry for every USB mass storage device ever connected to the system. Each device is identified by a subkey in the format:

Disk&Ven_{Vendor}&Prod_{Product}&Rev_{Revision}
  └── {Serial Number or Unique ID}
        Values:
          FriendlyName — Human-readable device name
          HardwareID — Device identification strings
          ContainerID — Links to other device entries

Example:

USBSTOR\Disk&Ven_SanDisk&Prod_Ultra_USB_3.0&Rev_1.00 └── 4C530001140818109484 FriendlyName: "SanDisk Ultra USB 3.0 USB Device" Key Last Write: 2026-02-20 16:42:18 UTC

The key last-write timestamp on the serial number subkey indicates the last time the device was connected to the system.

USB — USB Controller Entries

HKLM\SYSTEM\CurrentControlSet\Enum\USB

Hive: SYSTEM

What it contains: Lower-level USB device entries organized by Vendor ID (VID) and Product ID (PID). This key records all USB devices, not just storage devices (keyboards, mice, printers, etc.).

USB\VID_{VendorID}&PID_{ProductID}
  └── {Serial Number}
        Values:
          DeviceDesc — Device description
          LocationInformation — USB port information

Forensic use: Cross-reference VID/PID with the USB ID Database to identify the exact device manufacturer and model. The VID/PID values are assigned by the USB Implementers Forum and uniquely identify device types.

MountedDevices — Drive Letter Assignments

HKLM\SYSTEM\MountedDevices

Hive: SYSTEM (at the root level, not under a ControlSet)

What it contains: A mapping between drive letters (and volume GUIDs) and the physical devices they were assigned to. Each value name represents a mount point, and the data contains the device identifier.

Values:
  \DosDevices\C: — (REG_BINARY) Internal hard drive
  \DosDevices\D: — (REG_BINARY) Optical drive
  \DosDevices\E: — (REG_BINARY) USB device signature
  \??\Volume{GUID} — Volume GUID mount points

Forensic use: Determines which drive letter was assigned to a USB device. The binary data for removable devices contains a device signature that can be correlated with USBSTOR entries to establish: "SanDisk Ultra USB 3.0 was mounted as drive E:."

MountPoints2 — Per-User Device Connections

HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\MountPoints2

Hive: NTUSER.DAT (per-user — covered fully in Module 7.3)

What it contains: A record of every volume mounted during a specific user's session. Each subkey represents a volume GUID, and the key's last-write timestamp indicates when the user last accessed that volume.

Forensic use: This is the per-user link in the USB device chain. While USBSTOR and MountedDevices tell you a device was connected to the machine, MountPoints2 tells you which user was logged in and interacting with the device. This is critical for attributing device usage to a specific person.

🔍 Key Connection

To build a complete USB device timeline, you need data from all four locations: USBSTOR (device identity), USB (VID/PID), MountedDevices (drive letter), and MountPoints2 (user attribution). No single location tells the complete story.

6.2 USB Device Forensic Timeline

The following table summarizes how to determine when a USB device was connected:

Timestamp Source Meaning
First connection USBSTOR device subkey — key last-write timestamp (first install) and 0064 property value When the device was first connected to this computer
Last connection USBSTOR serial number subkey — key last-write timestamp When the device was most recently connected
Last removal 0067 property value under the device's Properties key When the device was last safely removed or disconnected
User interaction MountPoints2 subkey — key last-write timestamp When a specific user last accessed the device's volume
🔍 Forensic Scenario — USB Evidence

Scenario: An employee is suspected of copying proprietary files to a personal USB drive before resigning. By examining the SYSTEM hive's USBSTOR entries, you identify a SanDisk USB drive with serial number 4C530001140818109484 that was last connected on the employee's final day. MountedDevices shows it was assigned drive letter E:. The employee's NTUSER.DAT MountPoints2 key confirms they interacted with the volume. File system artifacts on the forensic image show files were accessed in the proprietary data directory during the same timeframe. This multi-source correlation builds a compelling evidence chain.

🌐 7. Network Artifacts

The Registry records information about network connections, providing evidence of what networks the system has connected to and when.

Artifact: Network Connection History

SOFTWARE Hive
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Signatures\Unmanaged\{GUID} HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Signatures\Managed\{GUID} Key Values: FirstNetwork (REG_SZ) — Network name (SSID for Wi-Fi) DefaultGatewayMac (REG_BINARY) — MAC address of the default gateway/router ProfileGuid (REG_SZ) — Links to the Profiles subkey for more details HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Profiles\{GUID} Key Values: ProfileName (REG_SZ) — Network name DateCreated (REG_BINARY) — SYSTEMTIME structure: when first connected DateLastConnected (REG_BINARY) — SYSTEMTIME structure: when last connected Category (REG_DWORD) — 0=Public, 1=Private, 2=Domain

What it tells you: Every network the system has connected to, including the network name, the MAC address of the gateway/router, and timestamps for first and last connection.

🔍 Forensic Significance

Network connection history can:

  • Place the computer at a location: Wi-Fi network names (SSIDs) can be correlated with known locations (e.g., "Starbucks_WiFi", "HiltonGuest", a company's internal SSID)
  • Establish timelines: First and last connection dates show when the system was at a particular network location
  • Identify gateway devices: The gateway MAC address can be used to identify specific routers, further pinpointing location
  • Detect suspicious connections: Connections to unusual or unexpected networks may indicate unauthorized activity

Artifact: Network Interface Configuration

SYSTEM Hive
HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\{GUID} Key Values: DhcpIPAddress (REG_SZ) — IP address assigned by DHCP DhcpSubnetMask (REG_SZ) — Subnet mask DhcpDefaultGateway (REG_MULTI_SZ) — Default gateway DhcpServer (REG_SZ) — DHCP server address DhcpNameServer (REG_SZ) — DNS servers LeaseObtainedTime (REG_DWORD) — Unix epoch: when the DHCP lease was obtained LeaseTerminatesTime (REG_DWORD) — Unix epoch: when the lease expires

What it tells you: The IP address configuration for each network interface, including DHCP-assigned addresses and lease times.

🔍 Forensic Significance

IP address and DHCP lease data can correlate the system with network activity logs from routers, firewalls, and servers. If a network intrusion was traced to IP address 192.168.1.105 at a specific time, and the DHCP lease data shows the suspect's machine held that address during that timeframe, it provides strong evidence linking the machine to the network activity.

🔧 8. Tool Walkthroughs

This section demonstrates how to extract the system-level artifacts covered in this module using forensic tools. All examples assume you have already extracted the hive files from a forensic image (as described in Module 7.1, Section 9).

8.1 Registry Explorer — Extracting System Identification

  1. Load the SYSTEM hive: File → Load Hive → select the extracted SYSTEM file (include .LOG1/.LOG2 in the same directory for automatic transaction log replay)
  2. Determine the active ControlSet: Navigate to Select key → note the Current value (e.g., 1 = ControlSet001)
  3. Find the computer name: Navigate to ControlSet001\Control\ComputerName\ComputerName → the ComputerName value contains the machine name
  4. Find the time zone: Navigate to ControlSet001\Control\TimeZoneInformation → note TimeZoneKeyName and Bias values
  5. Find the shutdown time: Navigate to ControlSet001\Control\Windows → the ShutdownTime value contains a FILETIME timestamp (Registry Explorer decodes this automatically)
  6. Bookmark each finding: Right-click each key → Add Bookmark → add a note describing the artifact

Example output:

--- Computer Identification Summary --- Computer Name: DESKTOP-A1B2C3D Time Zone: Pacific Standard Time (UTC-8:00) Active Bias: 480 minutes Last Shutdown: 2026-02-28 17:45:23 UTC (= 2026-02-28 09:45:23 PST)

8.2 RegRipper — Automated System Artifact Extraction

RegRipper uses plugins to automatically extract known artifacts. Key plugins for system-level analysis:

SYSTEM Hive Plugins

rip.exe -r C:\Evidence\SYSTEM -p compname
rip.exe -r C:\Evidence\SYSTEM -p timezone
rip.exe -r C:\Evidence\SYSTEM -p shutdown
rip.exe -r C:\Evidence\SYSTEM -p usbstor
rip.exe -r C:\Evidence\SYSTEM -p mountdev
rip.exe -r C:\Evidence\SYSTEM -p nic2

SOFTWARE Hive Plugins

rip.exe -r C:\Evidence\SOFTWARE -p winver
rip.exe -r C:\Evidence\SOFTWARE -p uninstall
rip.exe -r C:\Evidence\SOFTWARE -p networklist
rip.exe -r C:\Evidence\SOFTWARE -p profilelist

SAM Hive Plugins

rip.exe -r C:\Evidence\SAM -p samparse

Example samparse output:

--- RegRipper: samparse plugin --- User: Administrator [500] Account Created: 2025-09-15 10:23:45 UTC Last Login: <Never> Login Count: 0 Password Last Set: 2025-09-15 10:23:45 UTC Account Flags: Disabled, Password Not Required User: jsmith [1001] Account Created: 2025-09-15 10:30:12 UTC Last Login: 2026-02-28 09:12:33 UTC Login Count: 247 Password Last Set: 2025-12-01 14:55:08 UTC Account Flags: Normal Account User: temp_user [1002] Account Created: 2026-02-25 22:14:07 UTC Last Login: 2026-02-27 03:41:55 UTC Login Count: 3 Password Last Set: 2026-02-25 22:14:07 UTC Account Flags: Normal Account
🔍 Analysis of Sample Output

The temp_user account is suspicious: it was created very recently (February 25), has only 3 logins, and was accessed at unusual hours (3:41 AM). This pattern could indicate an unauthorized account created by an attacker or insider. This finding would warrant further investigation in the user's NTUSER.DAT (Module 7.3) and correlation with event logs.

8.3 Autopsy — Integrated Registry Analysis

Autopsy automatically parses Registry hive files when you run the Recent Activity ingest module. System-level artifacts appear under:

  • Data Artifacts → Operating System Information: Computer name, OS version, install date
  • Data Artifacts → Operating System User Account: User accounts from SAM
  • Data Artifacts → USB Device Attached: USB device connection history
  • Data Artifacts → Installed Programs: Software from the Uninstall key

Advantages of using Autopsy:

  • Automatic hive detection — no manual extraction required
  • Results integrated with timeline analysis
  • Keyword search works across Registry data and all other data sources
  • Built-in reporting generates formatted output

Limitation: Autopsy's Registry parsing may not extract every artifact that specialized tools (Registry Explorer, RegRipper) can find. For thorough analysis, use Autopsy for initial triage and then supplement with specialized tools for deep-dive analysis.

📝 Test Your Understanding

Answer the following questions in your own words before revealing the model answers. These questions align with the Lab Activity 7 requirements for system-level artifact analysis.

1. Describe the step-by-step process to determine the computer name from an offline SYSTEM hive file. Include how you identify the correct ControlSet.

Step 1: Open the extracted SYSTEM hive file in Registry Explorer (with .LOG1/.LOG2 files in the same directory for transaction log replay).

Step 2: Navigate to the Select key and check the Current value to determine the active ControlSet number (e.g., if Current = 1, use ControlSet001).

Step 3: Navigate to ControlSet001\Control\ComputerName\ComputerName.

Step 4: Read the ComputerName value (REG_SZ) — this is the NetBIOS computer name.

Step 5: Document the finding including the full Registry path, value, key last-write timestamp, and the tool used.

2. Why is the time zone setting considered one of the most critical system-level artifacts? What could go wrong if an examiner fails to account for it?

The time zone is critical because nearly all Windows timestamps (Registry key last-write times, NTFS file timestamps) are stored in UTC. To present accurate local times in a forensic report, the examiner must convert UTC to the system's local time zone. Without this conversion — or with an incorrect conversion — all timestamps in the report could be wrong.

Consequences of error: An incorrect time zone could shift events by hours, potentially placing activity at the wrong time of day. This could undermine alibis, misrepresent the sequence of events, or cause the examiner's findings to be challenged in court. For example, if the system is in Pacific Time (UTC-8) but the examiner reports in Eastern Time (UTC-5), all events would appear to have occurred 3 hours later than they actually did.

3. You are examining the SAM hive and find an account named "svc_backup" with RID 1003, created 2 days before the evidence was seized, with 5 logins all occurring between midnight and 4:00 AM. What questions should this raise for the investigator?

This account raises several red flags that warrant further investigation:

  • Recent creation: Created only 2 days before seizure — was this account authorized? Who created it?
  • Naming convention: "svc_backup" mimics a service account naming convention, which could be an attempt to make the account appear legitimate
  • Unusual login times: All logins between midnight and 4 AM suggest activity outside normal business hours — was someone accessing the system covertly?
  • Low login count: Only 5 logins on a recently created account — consistent with a purpose-built account for a specific task

Next steps: Examine the account's NTUSER.DAT for activity artifacts (Module 7.3), check Windows Event Logs for account creation events (Event ID 4720) and logon events (Event ID 4624), and investigate what programs were executed during those login sessions.

4. Explain how you would build a complete picture of a USB device connection by correlating data from at least three different Registry locations.

A complete USB device picture requires correlating data from multiple Registry locations:

1. USBSTOR (HKLM\SYSTEM\CurrentControlSet\Enum\USBSTOR) — Identifies the device: vendor name, product name, revision, and serial number. The key last-write timestamp indicates the last connection time.

2. USB (HKLM\SYSTEM\CurrentControlSet\Enum\USB) — Provides the Vendor ID (VID) and Product ID (PID), which can be cross-referenced with USB ID databases to identify the exact manufacturer and model.

3. MountedDevices (HKLM\SYSTEM\MountedDevices) — Maps the device to the drive letter it was assigned (e.g., E:), connecting the physical device to a logical access point.

4. MountPoints2 (HKCU\...\Explorer\MountPoints2 in NTUSER.DAT) — Attributes the device connection to a specific user account, with a timestamp showing when that user last accessed the volume.

Combined narrative: "A SanDisk Ultra USB 3.0 (serial: 4C530001...) was connected to DESKTOP-A1B2C3D, assigned drive letter E:, and accessed by user jsmith on February 20, 2026 at 4:42 PM."

5. What is the difference between the Bias and ActiveTimeBias values in the TimeZoneInformation key, and why does this distinction matter forensically?

Bias represents the standard time offset from UTC in minutes. For Pacific Standard Time, the Bias is 480 (UTC-8:00).

ActiveTimeBias represents the current offset including any Daylight Saving Time adjustment. During Pacific Daylight Time, the ActiveTimeBias would be 420 (UTC-7:00) instead of 480.

Forensic importance: If Bias and ActiveTimeBias are different, DST was active when the system was last running. If they are the same, the system was on standard time. This distinction affects timestamp conversion accuracy. Using the wrong offset (standard vs. daylight) would shift all reported times by one hour — which could be significant in establishing when events occurred.

6. You need to determine what software is installed on a suspect's system. Describe the complete process, including which hive to extract, the specific Registry paths to examine, and what information you would document for each installed application.

Hive: Extract the SOFTWARE hive from C:\Windows\System32\config\SOFTWARE (plus .LOG1/.LOG2 files).

Primary path: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\ — each subkey represents an installed application.

Secondary path (64-bit systems): HKLM\SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\Uninstall\ — contains 32-bit applications installed on a 64-bit OS.

For each application, document:

  • DisplayName — application name
  • DisplayVersion — version number
  • Publisher — software publisher
  • InstallDate — date installed (YYYYMMDD format)
  • InstallLocation — where it was installed
  • Key last-write timestamp — when the Registry entry was last modified

Tools: Use RegRipper's uninstall plugin for automated extraction, or browse manually in Registry Explorer. Flag any anti-forensics tools, hacking tools, encryption software, or applications relevant to the investigation.

📖 Key Terms — Module 7.2

Term Definition
ControlSet A complete set of system configuration data in the SYSTEM hive; Windows maintains multiple ControlSets for recovery purposes
CurrentControlSet A volatile symbolic link that exists only on a running system, pointing to the active ControlSet; does not exist in offline hive files
SAM (Security Account Manager) The hive file storing local user account information including usernames, RIDs, password hashes, and account metadata
RID (Relative Identifier) A numeric identifier unique to each account within a Windows system; built-in accounts have well-known RIDs (500 = Administrator, 501 = Guest)
SID (Security Identifier) A unique identifier for a security principal in Windows, combining machine-specific values with the account's RID
Bias The standard time zone offset from UTC in minutes, stored in the TimeZoneInformation key
ActiveTimeBias The current time zone offset including DST adjustment; differs from Bias when Daylight Saving Time is active
USBSTOR The Registry key under HKLM\SYSTEM that records USB mass storage device connection history
MountedDevices A SYSTEM hive key that maps drive letters and volume GUIDs to physical device identifiers
MountPoints2 A per-user key in NTUSER.DAT recording which volumes a user has accessed, providing user attribution for device connections
ProfileList A SOFTWARE hive key mapping user SIDs to their profile directory paths (e.g., C:\Users\jsmith)
Unix Epoch Timestamp A timestamp format representing seconds elapsed since January 1, 1970 00:00:00 UTC; used by some Registry values (e.g., InstallDate)
WOW6432Node A Registry redirection node on 64-bit Windows that stores Registry entries for 32-bit applications
