📖 1. Introduction & Context

Modules 7.1–7.3 taught you to extract artifacts from individual Registry hive files. You now know how to identify the machine (SYSTEM, SOFTWARE), enumerate users (SAM), and trace user activity (NTUSER.DAT). But real forensic investigations require something more: correlation.

No single artifact tells the complete story. A UserAssist entry proves a program was launched but not what files it accessed. A RecentDocs entry shows a file was opened but not what program opened it. A USB device entry shows a drive was connected but not what files were copied to it. Only by correlating multiple artifact sources can you build the comprehensive narrative that answers investigative questions.

This module introduces four additional artifact types that exist outside the Registry hive files but within the user profile and system directories. Then it teaches you the methodology for bringing everything together into a unified timeline.

📌 2. Jump Lists

Jump Lists

High Forensic Value User Profile
AutomaticDestinations: C:\Users\<username>\AppData\Roaming\Microsoft\Windows\Recent\AutomaticDestinations\ CustomDestinations: C:\Users\<username>\AppData\Roaming\Microsoft\Windows\Recent\CustomDestinations\
Jump List
A Windows feature (introduced in Windows 7) that tracks files, folders, and URLs recently or frequently accessed by specific applications. Jump Lists appear when you right-click an application icon on the taskbar or Start Menu. Forensically, they are stored as structured files in the user's profile and contain detailed records of application-file relationships with timestamps.

2.1 Types of Jump Lists

TypeLocationContentForensic Value
AutomaticDestinations ...\AutomaticDestinations\ Files automatically tracked by the OS when opened by an application Very High — system-generated, difficult for user to manipulate
CustomDestinations ...\CustomDestinations\ Items pinned by the user or custom entries created by applications Moderate — shows user-initiated pinning actions

2.2 Jump List File Naming

Each Jump List file is named with a 16-character hexadecimal Application ID (AppID) that identifies the application. The file extension is .automaticDestinations-ms or .customDestinations-ms.

5f7b5f1e01b83767.automaticDestinations-ms  → Microsoft Word
9b9cdc69c1c24e2b.automaticDestinations-ms  → Notepad
1b4dd67f29cb1962.automaticDestinations-ms  → Windows Explorer (pinned)
12dc1ea8e34b5a6.automaticDestinations-ms   → Microsoft Paint

2.3 Data Within Jump Lists

Each Jump List entry (internally stored as an embedded .lnk structure) contains:

  • Target file path: Full path to the file that was opened
  • Target file MAC timestamps: Modified, Accessed, Created times of the target file at the time of access
  • Target file size: Size of the file when it was opened
  • Volume information: Volume name, serial number, and type (local, network, removable)
  • Machine ID / NetBIOS name: The computer where the target file resides
  • Entry timestamp: When the file was accessed by the application
🔍 Forensic Significance
  • Application-file pairing: Jump Lists uniquely link which application opened which file — something neither UserAssist nor RecentDocs can do alone
  • Full file paths: Unlike RecentDocs (which stores only filenames), Jump Lists record the complete path including drive letter and directory
  • Volume serial numbers: Can be correlated with MountedDevices and USBSTOR to prove a file was accessed from a specific USB device
  • Deleted file evidence: Jump List entries persist even after the target file is deleted — proving the file once existed
  • Network file access: UNC paths in Jump Lists prove the user accessed files on network shares via specific applications

2.4 Example Jump List Output

--- JLECmd output: Microsoft Excel (AppID: b8ab77100df80ab2) --- Entry #1: Target Path: E:\Exfil\Customer_Database_Export.xlsx Target Created: 2026-02-28 02:30:15 UTC Target Modified: 2026-02-28 02:30:15 UTC Target Size: 4,521,984 bytes Volume Type: Removable Volume Serial: A4F2-8B31 Entry Time: 2026-02-28 02:45:22 UTC Entry #2: Target Path: \\fileserver\hr\Salary_Schedule_2026.xlsx Target Modified: 2026-01-15 09:12:00 UTC Volume Type: Network Entry Time: 2026-02-27 14:30:55 UTC ← Entry #1 shows Excel opened a file on a REMOVABLE drive (E:\Exfil\) at 2:45 AM. The volume serial can be matched to USBSTOR data. ← Entry #2 shows Excel opened a salary file from a network share the previous afternoon.

🗑️ 3. Recycle Bin Artifacts

Recycle Bin

High Forensic Value System / User
C:\$Recycle.Bin\{User SID}\ $I{random}.{ext} — Metadata file (original path, deletion time, file size) $R{random}.{ext} — Actual file content (the deleted file itself)
Recycle Bin
A special system folder that temporarily stores files deleted by users through Windows Explorer. Rather than immediately removing data, Windows moves deleted files to the Recycle Bin, renaming them with $R prefixes and creating companion $I metadata files. Files remain recoverable until the user empties the Recycle Bin or the system reclaims the space.

3.1 Recycle Bin File Structure (Windows 10/11)

FileContentsForensic Value
$I{id}.{ext} Metadata: Original file path, deletion timestamp (Windows FILETIME), original file size Very High — proves where the file was located and when it was deleted
$R{id}.{ext} File content: The actual deleted file data, preserved intact Very High — allows recovery and examination of the deleted file itself

The {id} portion is a random identifier that matches between the $I and $R pair. The {ext} preserves the original file extension.

3.2 $I File Structure

OffsetSizeField
0x008 bytesHeader/Version (2 = Win10+)
0x088 bytesOriginal file size in bytes
0x108 bytesDeletion timestamp (Windows FILETIME, UTC)
0x184 bytesLength of file path string
0x1CVariableOriginal file path (Unicode string)

3.3 User Attribution

Recycle Bin contents are organized by user SID:

C:\$Recycle.Bin\S-1-5-21-...-1001\  → Files deleted by user with RID 1001
C:\$Recycle.Bin\S-1-5-21-...-1002\  → Files deleted by user with RID 1002

By matching the SID to the SAM account enumeration from Module 7.2, you can determine which user deleted each file.

3.4 Example Recycle Bin Output

--- RBCmd output: C:\$Recycle.Bin\S-1-5-21-...-1001\ --- $IXQF2A1.xlsx Original Path: C:\Users\jsmith\Documents\Customer_Database_Full.xlsx File Size: 4,521,984 bytes (4.3 MB) Deleted: 2026-02-28 15:35:42 UTC $R File: $RXQF2A1.xlsx (content recoverable) $IA8B3C2.pdf Original Path: C:\Users\jsmith\Downloads\Employee_SSN_List.pdf File Size: 1,245,184 bytes (1.2 MB) Deleted: 2026-02-28 15:36:01 UTC $R File: $RA8B3C2.pdf (content recoverable) ← Both files deleted within 19 seconds of each other at 3:35-3:36 PM Filenames suggest sensitive corporate data Files are still recoverable from $R files
🔍 Forensic Significance
  • Deletion evidence: Proves what files were deleted and exactly when
  • File recovery: The actual file content can be recovered from the $R file
  • Original path: The $I file records where the file was located before deletion
  • User attribution: The SID-based directory structure identifies who deleted the file
  • Anti-forensics awareness: If the Recycle Bin has been emptied, the $I/$R files are deleted — but may still be recoverable through file carving (Week 5 techniques)
⚠️ Important — Shift+Delete Bypasses the Recycle Bin

Files deleted using Shift+Delete bypass the Recycle Bin entirely — no $I/$R files are created. Similarly, files deleted from the command line (del command) or by programs typically bypass the Recycle Bin. In these cases, you must rely on file system artifacts (MFT entries, file carving from unallocated space) to recover evidence of the deleted files.

🖼️ 4. Thumbcache — Thumbnail Evidence

Thumbcache

Medium Forensic Value User Profile
C:\Users\<username>\AppData\Local\Microsoft\Windows\Explorer\ thumbcache_32.db thumbcache_96.db thumbcache_256.db thumbcache_1024.db thumbcache_sr.db thumbcache_idx.db (index file)
Thumbcache
A set of database files that store thumbnail images (small preview images) generated by Windows Explorer when a user browses folders containing images, videos, documents, or other files that support preview generation. Thumbnails are cached to improve performance when revisiting folders.

4.1 Forensic Significance

🔍 Why Thumbnails Matter
  • Visual proof of file existence: Even if a file has been deleted, its thumbnail may persist in the Thumbcache — providing visual evidence that the file existed on the system
  • Content preview: Thumbnails show a small version of the file content, which can be sufficient to identify the nature of an image or document without recovering the original file
  • Browsing evidence: Thumbnails are only generated when a user browses a folder in a view mode that displays previews — their existence proves the user navigated to that folder
  • Persistence: Thumbcache entries are not automatically removed when the source file is deleted, making them resistant to simple deletion-based anti-forensics
💡 Practical Application

In investigations involving image-based evidence (intellectual property theft involving design files, inappropriate content, etc.), Thumbcache analysis can prove that specific images existed on the system even if the original files have been securely deleted. The thumbnail provides a visual "fingerprint" of the file's content.

🔎 5. Windows Search Index

Windows Search Index (Windows.edb)

Medium Forensic Value System
C:\ProgramData\Microsoft\Search\Data\Applications\Windows\Windows.edb
Windows Search Index
An Extensible Storage Engine (ESE) database maintained by the Windows Search service that indexes file metadata and content across the system to enable fast searching. The index contains file paths, sizes, dates, authors, and — for supported file types — the actual text content of documents. This database can be a rich source of metadata about files that have existed on the system.

5.1 What the Search Index Contains

  • File paths: Full paths to every indexed file
  • File metadata: Size, dates (created, modified, accessed), attributes, author
  • File content: Extracted text from documents (.docx, .pdf, .txt, .xlsx, etc.)
  • Email metadata: Sender, recipient, subject, dates for indexed email
  • Deleted file remnants: Files that were indexed before deletion may retain entries
🔍 Forensic Significance
  • Content without the file: If a document has been deleted but was previously indexed, the Search Index may still contain the document's text content — allowing you to read a "deleted" document
  • Metadata for deleted files: File paths and properties for deleted files may persist in the index
  • Corroboration: Search Index entries can corroborate findings from RecentDocs, Jump Lists, and other artifacts
💡 Note — Analysis Complexity

The Windows.edb file is an ESE database that requires specialized tools to parse (e.g., ESEDatabaseView by NirSoft, or Python libraries like libesedb). The database can be very large (multiple gigabytes) and contains many tables. Full Search Index analysis is typically performed in complex investigations; for routine cases, the artifacts covered in Modules 7.2–7.3 and earlier in this module are usually sufficient.

  7. The original customer database file was deleted to the Recycle Bin (Recycle Bin $I file)

No single artifact proves data theft. Together, they build a compelling, corroborated narrative of deliberate data exfiltration followed by evidence destruction attempts.

⏱️ 7. Timeline Analysis Methodology

Timeline analysis is the process of collecting, normalizing, and analyzing timestamped events from multiple sources to reconstruct a chronological narrative of activity. This is the core analytical technique in digital forensics.

7.1 Four-Step Timeline Process

Step 1: Collect Timestamps from All Artifact Sources

Gather every timestamped data point from all artifacts analyzed in Modules 7.1–7.4:

  • Registry key last-write timestamps (all hives)
  • UserAssist last execution timestamps
  • SAM account timestamps (creation, last login, password change)
  • Jump List entry timestamps
  • Recycle Bin deletion timestamps ($I files)
  • File system timestamps (NTFS MAC times from Week 5)
  • USBSTOR connection timestamps
  • Network connection timestamps (NetworkList)

Step 2: Normalize All Timestamps to a Common Time Zone

⚠️ Critical — Time Zone Normalization

All Registry timestamps are stored in UTC. Your forensic report should present times in the system's local time zone (determined from the SYSTEM hive's TimeZoneInformation key in Module 7.2). Every timestamp must be converted consistently. Document which time zone you are using and the UTC offset applied.

Example: If the system time zone is Pacific Standard Time (UTC-8), a Registry timestamp of 2026-02-28 03:15:00 UTC becomes 2026-02-27 19:15:00 PST (7:15 PM the previous evening).

Step 3: Combine Into a Single Sorted Timeline

Merge all normalized timestamps into a single chronological timeline. For this course, use a spreadsheet format:

Step 4: Analyze the Timeline

  • Look for patterns: Clusters of activity, regular schedules, unusual hours
  • Identify gaps: Periods of no activity may indicate the system was off or logs were cleared
  • Detect anomalies: Activity outside normal patterns, rapid sequences of file access/deletion
  • Correlate with known events: Incident reports, access requests, employee schedules, badge access logs

⚠️ Document Limitations

Always note the limitations of your findings:

  • Registry timestamps reflect key modification, not necessarily the exact moment of user action
  • AppCompatCache/ShimCache does not definitively prove program execution — only that the OS was aware of the program
  • Artifacts may be manipulated by sophisticated adversaries (anti-forensics, covered in Week 16)
  • UserAssist only records GUI program launches — absence from UserAssist does not mean a program was never executed
  • Timestamps can be manipulated (timestomping) — always corroborate with multiple sources

🔧 9. Tool Walkthroughs

JLECmd (by Eric Zimmerman) parses both AutomaticDestinations and CustomDestinations Jump List files.

JLECmd.exe -d "C:\Evidence\jsmith_AutomaticDestinations" --csv "C:\Output" --csvf jsmith_jumplists.csv

Key flags:

  • -d — Directory containing Jump List files
  • --csv — Output directory for CSV results
  • --csvf — Output filename
  • -q — Quiet mode (suppress console output)

The CSV output includes columns for AppID, target path, target timestamps, volume serial number, volume type (local/network/removable), and entry creation time.

AppID lookup: Use Eric Zimmerman's AppID list to map hexadecimal AppIDs to application names.

RBCmd (by Eric Zimmerman) parses Recycle Bin $I files to extract deletion metadata.

RBCmd.exe -d "C:\Evidence\$Recycle.Bin\S-1-5-21-...-1001" --csv "C:\Output" --csvf jsmith_recyclebin.csv

Key flags:

  • -d — Directory containing $I files (the user's Recycle Bin SID folder)
  • --csv — Output directory for CSV results
  • --csvf — Output filename

The CSV output includes: original file path, file size, deletion timestamp, and the source ($I) file name. To recover the actual deleted file, locate the matching $R file with the same identifier.

AppCompatCacheParser (by Eric Zimmerman) extracts ShimCache / AppCompatCache entries from the SYSTEM hive.

AppCompatCacheParser.exe -f "C:\Evidence\SYSTEM" --csv "C:\Output" --csvf shimcache.csv

Key flags:

  • -f — Path to the SYSTEM hive file
  • --csv — Output directory
  • --csvf — Output filename
⚠️ ShimCache Limitation

ShimCache (AppCompatCache) records that the operating system checked a program for compatibility — it does not definitively prove the program was executed. A program can appear in ShimCache simply by being browsed in Explorer. Use ShimCache as supporting evidence alongside UserAssist, BAM/DAM, or Prefetch data, not as standalone proof of execution.

✅ 11. Knowledge Check — Module 7.4

📝 Test Your Understanding

Answer each question in your own words before revealing the model answer. These questions directly support Lab Activity 7.

1. What information do Jump Lists provide that RecentDocs and UserAssist cannot provide individually? Why are Jump Lists considered one of the most valuable user artifacts?

Jump Lists uniquely provide the application-to-file relationship — they record which specific application opened which specific file. UserAssist tells you a program was launched but not what files it accessed. RecentDocs tells you a file was opened but not which program opened it. Jump Lists connect both pieces.

Additionally, Jump Lists contain: (1) full file paths (RecentDocs only stores filenames), (2) volume serial numbers that can be correlated with USB devices, (3) volume type (local/network/removable), and (4) target file timestamps. They persist even after files are deleted, and they are system-generated (difficult for users to manipulate).

2. Describe the $I and $R file pair in the Recycle Bin. What does each contain, and how do you match them?

$I file: Contains metadata about the deleted file — the original file path, the file size, and the deletion timestamp (Windows FILETIME in UTC). This is the "information" file.

$R file: Contains the actual deleted file content — the complete file data, preserved intact. This is the "recovery" file.

Matching: The $I and $R files share the same random identifier. For example, $IXQF2A1.xlsx and $RXQF2A1.xlsx are a pair — the identifier "XQF2A1" matches, and both preserve the original file extension (.xlsx).

User attribution: Recycle Bin contents are organized by user SID under C:\$Recycle.Bin\{SID}\, allowing you to determine which user deleted each file.

3. Explain why no single artifact can tell the complete story of a forensic investigation. Give an example using at least three different artifact types to answer one investigative question.

Each artifact captures only one dimension of activity. A complete picture requires correlating multiple sources.

Example question: "Did user jsmith copy company data to a USB drive?"

  • USBSTOR (SYSTEM hive): Proves a specific USB device was connected to the computer and when
  • MountedDevices (SYSTEM hive): Shows the USB device was assigned drive letter E:
  • TypedPaths (NTUSER.DAT): Shows the user typed "E:\Backup" in Explorer — proving they navigated to the USB drive
  • Jump List (user profile): Shows Excel opened "E:\Backup\Customer_Data.xlsx" from a removable volume — proving a file was accessed on the USB drive via a specific application
  • MountPoints2 (NTUSER.DAT): Confirms this specific user account interacted with the volume

No single artifact proves data theft. USBSTOR alone proves a device was connected but not that files were copied. TypedPaths proves navigation but not file access. Only the combination builds a compelling narrative.

4. What are the four steps of timeline analysis methodology? Why is Step 2 (time zone normalization) especially critical?

Four steps:

  1. Collect timestamps from all artifact sources
  2. Normalize all timestamps to a common time zone
  3. Combine into a single chronologically sorted timeline
  4. Analyze the timeline for patterns, gaps, and anomalies

Why normalization is critical: Different artifact sources may store timestamps in different formats (UTC, local time, Unix epoch, Windows FILETIME). Without normalizing to a single reference (typically the system's local time zone, determined from the SYSTEM hive), events may appear in the wrong chronological order. An event at 3:00 AM UTC and an event at 7:00 PM PST the previous day are actually only 4 hours apart — but without normalization, they might appear 8 hours apart in the wrong direction. Incorrect time placement could misrepresent the sequence of events and undermine the entire analysis.

5. What is the key difference between ShimCache (AppCompatCache) and UserAssist in terms of what they prove about program execution?

UserAssist records programs that were actually launched by the user through the GUI. It proves execution occurred and provides run count, focus time, and last execution timestamp.

ShimCache (AppCompatCache) records programs that the operating system checked for compatibility. A program can appear in ShimCache simply because the user browsed a folder containing it in Explorer — without ever executing it. ShimCache proves the OS was aware of the program, not that it was executed.

Practical implication: If a program appears in both UserAssist AND ShimCache, you have strong evidence of execution. If it appears only in ShimCache, you can say the program existed on the system but cannot conclusively prove it was run based on ShimCache alone.

6. List all the documentation elements required for each artifact finding to ensure legal defensibility. Why is it important to show both raw data and interpreted data?

Required elements: (1) Source image hash, (2) Hive file path within the image, (3) Specific Registry key path, (4) Tool used with version number, (5) Raw data (hex/binary values), (6) Interpreted data (human-readable), (7) Timestamps with time zone notation, (8) Screenshots of tool output.

Why both raw and interpreted data: Showing raw data allows another examiner to independently verify your interpretation. If you only show "firefox.exe was run 142 times," there is no way to verify this claim. By also showing the ROT13-encoded value name and the hex bytes of the binary data structure, any qualified examiner can confirm your interpretation is correct. This reproducibility is essential for legal defensibility — opposing counsel may hire their own forensic expert to verify your findings.

📋 Module 7.4 Summary

  • Jump Lists link applications to specific files with full paths, timestamps, and volume information — uniquely connecting "which program" to "which file"
  • Recycle Bin artifacts ($I/$R file pairs) preserve deletion evidence including the original path, deletion timestamp, and the actual file content
  • Thumbcache databases retain thumbnail images that prove files existed even after deletion
  • Windows Search Index may contain metadata and text content of files that have been deleted
  • Artifact correlation is the key to building compelling forensic narratives — no single artifact tells the complete story
  • Timeline methodology follows four steps: Collect → Normalize → Combine → Analyze
  • Documentation must include source hashes, paths, tools, raw data, interpreted data, timestamps, and screenshots for legal defensibility

AppIDA 16-character hexadecimal identifier in Jump List filenames that maps to a specific application
Recycle BinSystem folder storing files deleted through Explorer; uses $I (metadata) and $R (content) file pairs organized by user SID
$I FileRecycle Bin metadata file containing the original path, file size, and deletion timestamp of a deleted file
$R FileRecycle Bin content file containing the actual data of the deleted file
ThumbcacheDatabase files storing thumbnail preview images generated by Windows Explorer; can prove file existence after deletion
Windows Search IndexESE database (Windows.edb) containing indexed file metadata and content; may retain data about deleted files
ShimCache / AppCompatCacheSYSTEM hive artifact recording programs the OS checked for compatibility; proves awareness but not necessarily execution
Artifact CorrelationThe process of combining evidence from multiple artifact sources to build a comprehensive investigative narrative
Timeline AnalysisForensic methodology of collecting, normalizing, combining, and analyzing timestamped events to reconstruct activity chronologically
Time Zone NormalizationConverting all timestamps to a common time zone reference to ensure accurate chronological ordering
Forensic DefensibilityThe quality of documentation and methodology that enables findings to withstand legal scrutiny and independent verification
