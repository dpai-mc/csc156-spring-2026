🗄️ 2. What Is the Windows Registry?

Windows Registry
A centralized, hierarchical database built into the Windows operating system that stores low-level settings, configuration data, and options for the operating system, hardware devices, installed applications, and individual user preferences. It serves as the primary repository for virtually all configuration information on a Windows system.

2.1 Purpose of the Registry

The Windows Registry serves as the single source of truth for system configuration. Nearly every action on a Windows system — from installing software to changing your desktop wallpaper — results in a Registry modification. The Registry stores:

  • Operating system configuration: Boot settings, service configurations, driver parameters, system policies
  • Hardware profiles: Device settings, USB device history, mounted volumes, network adapter configurations
  • Software settings: Installed application data, file associations, licensing information, application preferences
  • User preferences: Desktop settings, recently accessed files, application usage history, typed URLs, search history
  • Security settings: User account information, password policies, audit configurations, access control data

2.2 Historical Context

Before the Registry existed, Windows (versions 3.x and earlier) stored configuration data in scattered .INI files (initialization files) spread across the file system. This approach had significant limitations:

  • No standardized storage location — each application chose its own location
  • No hierarchical organization — flat text files with limited structure
  • No access control — any user or program could modify any .INI file
  • No data typing — all values stored as plain text strings
  • Size limitations — .INI files were limited to 64 KB

Microsoft introduced the Registry in Windows 95 and refined it significantly in Windows NT. The modern Registry (Windows 10/11) retains the same fundamental structure established in the NT era, though it has grown enormously in size and complexity.

2.3 Why Forensic Examiners Care About the Registry

The Registry is one of the most valuable sources of forensic evidence on a Windows system because:

  1. Comprehensive activity recording: The Registry records evidence of user actions, program executions, device connections, and network activity — often without the user's knowledge
  2. Timestamps: Every Registry key records a "last-write" timestamp, providing temporal evidence of when changes occurred
  3. Persistence: Registry data persists across reboots and often survives user attempts to delete evidence (deleted keys may be recoverable from unallocated hive space)
  4. Correlation potential: Registry artifacts can be correlated with file system timestamps, event logs, and other evidence sources to build comprehensive activity timelines
  5. Anti-forensics resistance: Most users (and even many technically sophisticated suspects) do not know how to thoroughly clean Registry artifacts
🔍 Forensic Significance

Consider this scenario: A suspect claims they never used a particular USB drive. However, the Registry contains entries under HKLM\SYSTEM\CurrentControlSet\Enum\USBSTOR documenting the device's vendor ID, product ID, serial number, and the timestamp of when it was first and last connected. The Registry provides objective, machine-generated evidence that contradicts the suspect's claim — and the suspect likely had no idea this evidence existed.

🔑 3. The Five Root Keys

The Windows Registry is organized into a hierarchical tree structure, similar to a file system. At the top of this hierarchy are five root keys (also called "hives" at the top level, though technically "hive" refers to the backing file — we will clarify this distinction shortly). Every piece of data in the Registry exists under one of these five root keys.

The root keys are commonly referred to by their abbreviations. Click on each root key below to learn more about it:

Computer (Registry Root)

HKEY_CLASSES_ROOT (HKCR)

Purpose: Contains file extension associations and COM (Component Object Model) class registration information. This key determines which application opens when you double-click a file.

Technical detail: HKCR is actually a merged view of two other locations: HKLM\SOFTWARE\Classes (system-wide defaults) and HKCU\SOFTWARE\Classes (per-user overrides). It is not backed by its own hive file.

Forensic relevance: Moderate. Can reveal what applications were associated with specific file types, which may indicate installed software or user-configured preferences.

Example path: HKCR\.docx → shows which application handles .docx files

HKEY_CURRENT_USER (HKCU)

Purpose: Contains configuration settings and preferences for the currently logged-in user. This includes desktop settings, environment variables, printer connections, application preferences, and — critically for forensics — records of user activity.

Technical detail: HKCU is a pointer to the appropriate user's subkey under HKEY_USERS. It is backed by the user's NTUSER.DAT file located in their profile directory.

Forensic relevance: Very high. This is one of the most forensically valuable root keys because it contains per-user evidence including: recently opened files (RecentDocs), program execution history (UserAssist), typed URLs, search history, and application settings.

Example path: HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs

HKEY_LOCAL_MACHINE (HKLM)

Purpose: Contains configuration data for the entire computer, regardless of which user is logged in. This includes hardware configuration, installed software, security settings, system services, and boot configuration.

Technical detail: HKLM is backed by multiple hive files: SAM, SECURITY, SOFTWARE, SYSTEM, and DEFAULT, all located in C:\Windows\System32\config\.

Forensic relevance: Very high. Contains system-wide evidence including: computer name, time zone, OS version, installed software, USB device history, user account information (SAM), network configurations, and system service data.

Example path: HKLM\SYSTEM\CurrentControlSet\Control\ComputerName\ComputerName

HKEY_USERS (HKU)

Purpose: Contains profile data for all user accounts that have been loaded on the system. Each user's profile is stored under their Security Identifier (SID).

Technical detail: Each subkey under HKU corresponds to a loaded user profile. The .DEFAULT subkey contains settings for the default profile (used before any user logs in). Each user's SID subkey is backed by that user's NTUSER.DAT file.

Forensic relevance: High. Allows forensic examiners to analyze Registry data for all user accounts on the system, not just the currently logged-in user. This is essential when investigating multi-user systems.

Example path: HKU\S-1-5-21-...-1001\Software\Microsoft\Windows\CurrentVersion\Explorer

HKEY_CURRENT_CONFIG (HKCC)

Purpose: Contains information about the hardware profile used by the computer at startup. This is the active hardware configuration.

Technical detail: HKCC is a pointer to HKLM\SYSTEM\CurrentControlSet\Hardware Profiles\Current. It is not backed by its own hive file.

Forensic relevance: Low. Rarely contains unique forensic evidence because the data it references is available through HKLM\SYSTEM. Forensic examiners typically focus on HKLM and HKCU instead.

Example path: HKCC\System\CurrentControlSet\Control\VIDEO

3.1 Summary Table: The Five Root Keys

Root Key Abbreviation Scope Backed By Forensic Value
HKEY_CLASSES_ROOT HKCR File associations / COM Merged view (not own file) Moderate
HKEY_CURRENT_USER HKCU Current user settings NTUSER.DAT Very High
HKEY_LOCAL_MACHINE HKLM System-wide settings SAM, SECURITY, SOFTWARE, SYSTEM, DEFAULT Very High
HKEY_USERS HKU All loaded user profiles Each user's NTUSER.DAT High
HKEY_CURRENT_CONFIG HKCC Current hardware profile Pointer to HKLM\SYSTEM Low
💡 Key Insight — Virtual vs. Physical Root Keys

Notice that not all five root keys have their own backing files on disk. HKCR is a merged view of data from HKLM and HKCU. HKCC is a pointer into HKLM\SYSTEM. HKCU is a pointer to the current user's subkey under HKU. Only HKLM and HKU are directly backed by physical hive files. This distinction is important for forensic examiners because you extract and analyze the physical hive files, not the virtual/merged views.

📂 4. Keys, Subkeys, Values, and Data Types

Understanding the internal structure of the Registry is essential for forensic analysis. The Registry uses a hierarchical structure similar to a file system, but with its own terminology.

4.1 Registry Hierarchy

Registry Key
A container object in the Registry hierarchy, analogous to a folder in a file system. Keys can contain subkeys (child keys) and values. Every key has a name, a class (optional), a security descriptor, and a last-write timestamp.
Registry Subkey
A key that exists within (beneath) another key, analogous to a subfolder. There is no structural difference between a key and a subkey — the term "subkey" simply indicates its position relative to a parent key.
Registry Value
A named data entry stored within a key, analogous to a file within a folder. Each value has three components: a name, a data type, and the data itself. A key can contain zero or more values.
(Default) Value
Every Registry key has one unnamed value called the "(Default)" value. This value may or may not contain data. It appears in Registry editors as "(Default)" and is distinct from any named values under the same key.

4.2 Visual Analogy: Registry vs. File System

This notation is similar to a file system path. Forensic reports always include the full Registry path when documenting artifact locations.

4.4 Registry Data Types

Each Registry value has an associated data type that determines how the data is stored and interpreted. The most common data types you will encounter in forensic analysis are:

Data Type Description Forensic Example
REG_SZ A fixed-length string (null-terminated Unicode text) Computer name, user name, application path
REG_EXPAND_SZ An expandable string containing environment variable references (e.g., %SystemRoot%) Program paths using environment variables
REG_BINARY Raw binary data in any format Timestamps, encrypted data, hardware identifiers
REG_DWORD A 32-bit unsigned integer (4 bytes) Configuration flags, counters, boolean settings (0/1)
REG_QWORD A 64-bit unsigned integer (8 bytes) Large counters, 64-bit timestamps
REG_MULTI_SZ An array of null-terminated strings Lists of services, multiple values in a single entry
REG_NONE No defined data type Varies by application
🔍 Forensic Significance — Data Types Matter

Understanding data types is critical for correctly interpreting Registry evidence. For example, a REG_BINARY value might contain a Windows FILETIME timestamp (a 64-bit value representing the number of 100-nanosecond intervals since January 1, 1601 UTC). If you don't recognize the data type, you might misinterpret raw binary data as meaningless when it actually contains a precise date and time. Forensic tools like Registry Explorer automatically decode common data types, but an examiner must understand the underlying structure to verify tool output and identify unusual or custom data formats.

⏰ 5. Key Last-Write Timestamps & Forensic Significance

One of the most forensically important features of the Windows Registry is that every key maintains a last-write timestamp.

Key Last-Write Timestamp
A 64-bit Windows FILETIME value associated with every Registry key that records the date and time when the key or any of its values was last modified. This timestamp is automatically updated by Windows whenever a value within the key is created, modified, or deleted. The timestamp is stored in UTC (Coordinated Universal Time).

5.1 How Timestamps Work

Critical rules about Registry timestamps that every forensic examiner must understand:

  1. Keys have timestamps; values do not. Individual values within a key do not have their own timestamps. Only the parent key's timestamp is updated when any value changes.
  2. Any change updates the timestamp. Creating, modifying, or deleting any value under a key updates that key's last-write timestamp.
  3. The timestamp reflects the most recent change only. Previous timestamps are overwritten — there is no history of prior modifications stored in the timestamp itself.
  4. Subkey changes do NOT update the parent key's timestamp. If a subkey is modified, only that subkey's timestamp changes — the parent key's timestamp remains unchanged.
  5. Timestamps are stored in UTC. The examiner must apply the correct time zone offset (which can be found in the SYSTEM hive) to convert to local time.

5.2 Forensic Implications

🔍 Forensic Significance — The "One Timestamp" Limitation

Because a key has only one timestamp that is updated whenever any value changes, you cannot determine which specific value was modified just by examining the timestamp. If a key contains 10 values and the timestamp shows a modification at 2:30 PM, you know something changed at 2:30 PM, but the timestamp alone does not tell you which value was affected.

This is why forensic examiners use multiple data sources (file system timestamps, event logs, other Registry keys) to corroborate findings and build a complete picture.

5.3 Timestamp Example

Consider the following Registry key:

Key:    HKCU\Software\Microsoft\Office\16.0\Word\File MRU
Last Write: 2026-02-15 14:23:47 UTC
Values:
  Item 1 = "C:\Users\jsmith\Documents\Report.docx"
  Item 2 = "C:\Users\jsmith\Desktop\Notes.docx"
  Item 3 = "C:\Users\jsmith\Downloads\Contract.docx"

The last-write timestamp tells us that something in this key changed on February 15, 2026 at 2:23:47 PM UTC. Combined with the MRU (Most Recently Used) list values, we can infer that the user most recently accessed one of these Word documents around that time — but we would need additional evidence to determine exactly which file triggered the update.

💡 Windows FILETIME Format

The raw timestamp is stored as a 64-bit integer representing the number of 100-nanosecond intervals since January 1, 1601 00:00:00 UTC. For example, the hex value 01D74A2B 3C8E4000 decodes to a specific date and time. Forensic tools automatically convert these values to human-readable dates, but understanding the underlying format helps when examining raw binary data or verifying tool output.

💾 6. Registry Hive Files on Disk

While the Registry appears as a single unified database when viewed through tools like regedit.exe, it is actually stored in multiple separate files on disk called hive files.

Registry Hive
A logical group of keys, subkeys, and values in the Registry that is backed by a single file on disk. Each hive file contains a discrete, self-contained tree of Registry data that is loaded into memory when the system boots or when a user logs in.
Hive File
The physical file on the NTFS volume that stores the data for a Registry hive. Hive files have no file extension (e.g., the file is simply named SAM, not SAM.dat). The exception is user-specific hives, which use the .DAT extension (e.g., NTUSER.DAT).

6.1 System Hive Files

The system-wide hive files are located in:

C:\Windows\System32\config\
Hive File Disk Location Registry Location Contents
SAM C:\Windows\System32\config\SAM HKLM\SAM Security Account Manager — local user accounts, password hashes, account policies, login history
SECURITY C:\Windows\System32\config\SECURITY HKLM\SECURITY Security policies, cached domain credentials, audit settings
SOFTWARE C:\Windows\System32\config\SOFTWARE HKLM\SOFTWARE Installed software, OS configuration, application settings, network profiles
SYSTEM C:\Windows\System32\config\SYSTEM HKLM\SYSTEM Hardware configuration, services, drivers, boot settings, USB device history, time zone
DEFAULT C:\Windows\System32\config\DEFAULT HKU\.DEFAULT Default user profile settings (used before any user logs in)

6.2 User-Specific Hive Files

Each user account on the system has its own set of hive files, located within the user's profile directory:

C:\Users\<username>\NTUSER.DAT
C:\Users\<username>\AppData\Local\Microsoft\Windows\UsrClass.dat
Hive File Disk Location Registry Location Contents
NTUSER.DAT C:\Users\<username>\NTUSER.DAT HKU\<SID> (and HKCU for the current user) User preferences, recently accessed files, program execution history, typed URLs, search queries
UsrClass.dat C:\Users\<username>\AppData\Local\Microsoft\Windows\UsrClass.dat HKU\<SID>_Classes User-specific file associations, COM registrations, shell bag data (folder access history)

6.3 Example: Locating a Specific User's Hive

If you are examining a forensic image and need to find the Registry data for a user named "jsmith", you would look for:

C:\Users\jsmith\NTUSER.DAT
C:\Users\jsmith\AppData\Local\Microsoft\Windows\UsrClass.dat
🔍 Forensic Significance — Hidden and System Files

NTUSER.DAT is marked with the Hidden and System file attributes in NTFS. It will not appear in a normal Windows Explorer view. Forensic tools (Autopsy, FTK Imager) display all files regardless of attributes, but it is important to understand that these files are intentionally hidden from casual observation. Similarly, the system hive files in C:\Windows\System32\config\ are locked by Windows while the OS is running — they cannot be copied or opened on a live system without special tools. This is one of the key reasons forensic analysis is performed on forensic images rather than live systems.

⚠️ Important — Hive Files Are Locked on Live Systems

When Windows is running, the system hive files are locked by the operating system kernel. You cannot simply copy C:\Windows\System32\config\SAM from a running Windows system — the OS will deny access. This is another critical reason why forensic examiners work from forensic images: the image captures the raw disk data including all locked files.

📋 7. Transaction Log Files

Each Registry hive file has associated transaction log files that play an important role in both system reliability and forensic analysis.

Registry Transaction Log
A file that records pending changes to a Registry hive before those changes are committed (written) to the main hive file. Transaction logs act as a journaling mechanism to protect against data corruption from unexpected shutdowns, crashes, or power failures. Each hive has two associated log files with extensions .LOG1 and .LOG2.

7.1 How Transaction Logs Work

The Registry uses a write-ahead logging mechanism:

  1. Change is requested — An application or the OS requests a modification to a Registry value
  2. Change is written to the transaction log — The modification is first recorded in the .LOG1 or .LOG2 file
  3. Change is committed to the hive file — Once the log entry is confirmed, the actual hive file is updated
  4. Log is marked as committed — The transaction log entry is marked as successfully applied

If the system crashes between steps 2 and 3, the transaction log contains the pending change that was never written to the main hive file. On the next boot, Windows uses the transaction log to either complete or roll back the pending operation.

7.2 Transaction Log File Locations

Transaction logs are stored alongside their parent hive files:

C:\Windows\System32\config\SAM.LOG1
C:\Windows\System32\config\SAM.LOG2
C:\Windows\System32\config\SYSTEM.LOG1
C:\Windows\System32\config\SYSTEM.LOG2
C:\Windows\System32\config\SOFTWARE.LOG1
C:\Windows\System32\config\SOFTWARE.LOG2
C:\Windows\System32\config\SECURITY.LOG1
C:\Windows\System32\config\SECURITY.LOG2

C:\Users\<username>\NTUSER.DAT.LOG1
C:\Users\<username>\NTUSER.DAT.LOG2

7.3 Forensic Significance of Transaction Logs

🔍 Forensic Significance — Why Transaction Logs Matter

Transaction logs are forensically valuable for several reasons:

  • Recovering uncommitted changes: If the system was improperly shut down (power failure, crash, forced shutdown), the transaction logs may contain Registry modifications that were never written to the main hive file. These "dirty" transactions can reveal the most recent activity on the system.
  • Reconstructing recent activity: Transaction logs can contain evidence of recent changes that supplements the data in the main hive file, providing additional temporal context.
  • Detecting anti-forensics: If a suspect attempted to modify or delete Registry entries to cover their tracks, the transaction logs may still contain evidence of the original values or the modification itself.
  • Hive recovery: If a hive file is corrupted, the transaction logs can be used to reconstruct the hive to a consistent state.
💡 Practical Tip — Always Extract Transaction Logs

When extracting Registry hive files from a forensic image, always extract the associated .LOG1 and .LOG2 files as well. Forensic tools like Registry Explorer can automatically apply ("replay") transaction logs to reconstruct the most complete and current version of a hive. Failing to include transaction logs may result in analyzing an incomplete or outdated version of the Registry data.

🔧 8. Forensic Tools for Registry Analysis

Forensic examiners use specialized tools to analyze Registry hive files offline — that is, by examining the extracted hive files from a forensic image rather than using the built-in Windows regedit.exe on a live system.

8.1 Why Not Use regedit.exe?

⚠️ Important — Never Use regedit.exe for Forensic Analysis

The built-in Windows Registry Editor (regedit.exe) is not suitable for forensic analysis because:

  • It shows the live system's Registry — not the evidence system's Registry
  • It does not display timestamps — key last-write timestamps are not shown
  • It cannot load offline hive files reliably — while it has a "Load Hive" feature, it modifies the hive when loading and unloading, potentially altering evidence
  • It merges virtual keys — HKCR and HKCC show merged/virtual data, not the raw hive contents
  • It cannot process transaction logs — dirty hive data may be incomplete
  • No forensic documentation features — no bookmarking, reporting, or hash verification

8.2 Recommended Forensic Registry Tools

Tool Type Primary Use Key Features
Registry Explorer (Eric Zimmerman) GUI browser Interactive browsing and analysis of offline hive files Displays timestamps, supports transaction log replay, bookmarking, data type decoding, search functionality
RegRipper (Harlan Carvey) Plugin-based extractor Automated extraction of specific forensic artifacts from hive files Hundreds of plugins targeting known forensic artifacts, batch processing, text output for reporting
Autopsy Integrated forensic suite Comprehensive forensic analysis including Registry parsing Built-in Registry ingest modules, timeline integration, keyword search across all data sources
FTK Imager Imaging/extraction tool Extracting hive files from forensic images Browse forensic images, export individual files, verify hashes

8.3 Tool Usage Examples

Registry Explorer by Eric Zimmerman is the gold standard for interactive Registry analysis. To use it:

  1. Extract the hive file(s) and associated .LOG1/.LOG2 files from the forensic image
  2. Launch Registry Explorer
  3. Go to File → Load Hive
  4. Select the hive file (e.g., NTUSER.DAT)
  5. Registry Explorer will automatically detect and offer to apply transaction logs if they are in the same directory
  6. Browse the key tree in the left panel; view values and timestamps in the right panel

Key advantages: Displays last-write timestamps for every key, decodes common data types (FILETIME, ROT-13, etc.), supports bookmarking important keys, and can export findings for reports.

RegRipper by Harlan Carvey automates the extraction of known forensic artifacts using a plugin system. Each plugin targets a specific Registry artifact.

Command-line usage:

rip.exe -r C:\Evidence\NTUSER.DAT -p userassist

This command runs the userassist plugin against the extracted NTUSER.DAT file, producing a decoded list of programs the user has executed along with run counts and timestamps.

Running all plugins for a hive type:

rip.exe -r C:\Evidence\NTUSER.DAT -a

The -a flag runs all applicable plugins for the detected hive type and outputs the combined results.

Key advantages: Fast, automated, consistent output format suitable for reporting, extensive plugin library covering hundreds of known artifacts.

Autopsy includes built-in ingest modules that automatically parse Registry hive files when processing a forensic image:

  1. Create a new case and add the forensic image as a data source
  2. During ingest module selection, enable "Recent Activity" and "Registry Analysis" modules
  3. Autopsy will automatically locate and parse all Registry hive files in the image
  4. Results appear in the Data Artifacts section of the tree view

Key advantages: Fully integrated with other forensic analysis (file system, timeline, keyword search), automatic hive detection, results correlated with other artifact types.

📤 9. Extracting Hive Files from a Forensic Image

In practice, the first step of any Registry-based forensic analysis is extracting the relevant hive files from the forensic image. Here is the standard procedure:

9.1 Step-by-Step Extraction Process

📋 Hive File Extraction Procedure
  1. Open the forensic image in Autopsy or FTK Imager
  2. Navigate to the system hive directory: C:\Windows\System32\config\
  3. Export individual system hive files: SAM, SYSTEM, SOFTWARE, SECURITY, DEFAULT
  4. Navigate to each user profile directory: C:\Users\<username>\
  5. Export user hive files: NTUSER.DAT and UsrClass.dat for each user
  6. Export associated transaction log files: .LOG1 and .LOG2 for each hive
  7. Document the extraction: Record file hashes (MD5/SHA-1) and source paths for each extracted file

9.2 Documentation Requirements

For each extracted hive file, your forensic documentation should record:

Documentation Element Example
Source forensic image evidence_workstation01.E01
Image hash (verified) SHA-256: a3f2b8c9...
File source path C:\Windows\System32\config\SOFTWARE
Extracted file hash MD5: 7e4a1d3f...
File size 67,108,864 bytes
Extraction date/time 2026-03-04 10:15:23 UTC
Examiner name J. Smith, Digital Forensic Examiner
Associated log files SOFTWARE.LOG1, SOFTWARE.LOG2
🔍 Forensic Significance — Chain of Custody

Thorough documentation of hive file extraction maintains the chain of custody and ensures that your findings are legally defensible (connecting back to Week 3's legal frameworks). Every step from the original evidence media → forensic image → extracted file → analysis finding must be documented and verifiable. Hash values provide mathematical proof that the extracted files are identical to those in the forensic image.

✅ 11. Knowledge Check — Module 7.1

📝 Test Your Understanding

Answer the following questions in your own words before revealing the model answers. These questions prepare you for Module 7.2 and the Week 7 assignments.

1. What are the five root keys of the Windows Registry, and which one contains per-user settings for the currently logged-in user?

The five root keys are: HKEY_CLASSES_ROOT (HKCR), HKEY_CURRENT_USER (HKCU), HKEY_LOCAL_MACHINE (HKLM), HKEY_USERS (HKU), and HKEY_CURRENT_CONFIG (HKCC).

HKEY_CURRENT_USER (HKCU) contains per-user settings for the currently logged-in user. It is backed by the user's NTUSER.DAT file and is actually a pointer to the current user's SID subkey under HKU.

2. If you modify a single value under a Registry key, what happens to the key's timestamp? What are the implications for forensic analysis?

The key's last-write timestamp is updated to reflect the time of the modification. This single timestamp is the only temporal indicator at the key level — individual values do not have their own timestamps.

Forensic implications: Because the key has only one timestamp that is updated when any value changes, the examiner cannot determine from the timestamp alone which specific value was modified. If a key contains multiple values, additional corroborating evidence (file system timestamps, event logs, other Registry keys) is needed to identify which value changed and what the previous value was.

3. Where on disk would you find the NTUSER.DAT file for a user named "jsmith"?

The NTUSER.DAT file for user "jsmith" is located at: C:\Users\jsmith\NTUSER.DAT

Note that this file is marked with the Hidden and System attributes, so it would not be visible in a standard Windows Explorer view. Forensic tools display all files regardless of attributes.

4. Why is offline analysis of Registry hive files preferred over live analysis using regedit.exe in a forensic investigation?

Offline analysis is preferred for several critical reasons:

  • Evidence preservation: regedit.exe can modify hive data when loading/unloading offline hives, potentially altering evidence
  • Timestamps: regedit.exe does not display key last-write timestamps, which are essential forensic evidence
  • Correct target: regedit.exe shows the examiner's system Registry, not the evidence system's Registry
  • Virtual key merging: regedit.exe shows merged virtual views (HKCR, HKCC) rather than raw hive contents
  • Transaction logs: regedit.exe cannot process transaction log files to recover uncommitted changes
  • Forensic features: Specialized tools provide bookmarking, reporting, decoded data types, and search capabilities that regedit.exe lacks
5. What is the forensic significance of Registry transaction log files (.LOG1, .LOG2)?

Transaction log files are forensically significant because they:

  • Contain uncommitted changes: If the system was improperly shut down, the logs may contain Registry modifications that were never written to the main hive file — potentially revealing the most recent activity
  • Supplement hive data: Provide additional temporal context about recent changes
  • Aid anti-forensics detection: May retain evidence of original values even if a suspect attempted to modify or delete Registry entries
  • Enable hive recovery: Can be used to reconstruct a corrupted hive to a consistent state

Forensic examiners should always extract the associated .LOG1 and .LOG2 files when extracting hive files from a forensic image.

6. You are examining a forensic image and need to determine what software is installed on the system. Which hive file would you extract, and from what disk location?

You would extract the SOFTWARE hive file from C:\Windows\System32\config\SOFTWARE.

The SOFTWARE hive (mapped to HKLM\SOFTWARE) contains system-wide installed software information, including entries under HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall which lists installed programs, their versions, install dates, and publisher information.

Don't forget to also extract the associated SOFTWARE.LOG1 and SOFTWARE.LOG2 transaction log files from the same directory.

📋 Module 7.1 Summary

In this module, you learned the foundational concepts needed for all Registry-based forensic analysis:

  • The Windows Registry is a centralized hierarchical database storing OS, hardware, software, and user configuration data
  • The Registry is organized under five root keys: HKCR, HKCU, HKLM, HKU, and HKCC — with HKCU and HKLM being the most forensically valuable
  • The hierarchy consists of keys (like folders), subkeys (like subfolders), and values (like files) with specific data types
  • Every key has a last-write timestamp — a single UTC timestamp updated whenever any value in the key changes
  • Registry data is physically stored in hive files on the NTFS volume — system hives in C:\Windows\System32\config\ and user hives in C:\Users\<username>\
  • Transaction log files (.LOG1, .LOG2) record pending changes and may contain evidence not yet committed to the main hive
  • Forensic examiners use specialized tools (Registry Explorer, RegRipper, Autopsy) for offline analysis — never regedit.exe

📖 Key Terms

Term Definition
Registry Hive A logical group of keys, subkeys, and values in the Registry that is backed by a single file on disk
Registry Key A container object in the Registry hierarchy (analogous to a folder) that can contain subkeys and values
Registry Value A named data entry within a key, consisting of a name, data type, and data (analogous to a file)
Root Key One of the five top-level keys in the Registry hierarchy (HKCR, HKCU, HKLM, HKU, HKCC)
Last-Write Timestamp A 64-bit FILETIME value on each key recording when the key or any of its values was last modified (stored in UTC)
Hive File The physical file on disk that stores Registry hive data (e.g., SAM, SYSTEM, SOFTWARE, NTUSER.DAT)
Transaction Log A file (.LOG1, .LOG2) recording pending Registry changes before they are committed to the main hive file
NTUSER.DAT The hive file storing per-user Registry data, located in each user's profile directory
UsrClass.dat The hive file storing user-specific file associations and shell bag data, located in the user's AppData directory
Offline Analysis Examining extracted hive files from a forensic image using specialized tools, rather than analyzing a live running system
SID (Security Identifier) A unique identifier assigned to each user account in Windows, used as the subkey name under HKU
