🎯 Learning Overview & Objectives

This module introduces the foundational concepts of file systems—the critical layer between operating systems and physical storage that organizes how data is stored, retrieved, and managed. Understanding file systems is essential for digital forensic examiners because it determines where evidence can be found and how it can be recovered.

🎓 Learning Objectives

Upon completing this module, you will be able to:

  • Define what a file system is and explain its purpose in computing
  • Identify the core components that make up a file system
  • Distinguish between different types of file systems and their characteristics
  • Explain why file system knowledge is critical for digital forensics
  • Describe the relationship between file systems and storage media
🔍 CSLO Alignment

This module supports CSLO 2: Describe the procedures for the collection, preservation, and analysis of digital evidence, including the use of forensic tools and techniques for data recovery and examination.

Why File Systems Matter in Forensics

Every piece of digital evidence—documents, images, emails, browser history, and application data—is stored within a file system. The file system acts as a librarian, organizing billions of bytes into meaningful files and folders. For forensic examiners, understanding this organization is crucial because:

  • Evidence Location: File systems determine where and how data is physically stored on media
  • Deleted File Recovery: Understanding file system structures enables recovery of deleted files
  • Timeline Analysis: File systems maintain timestamps that establish when events occurred
  • Hidden Data: Knowledge of file system slack space and unallocated areas reveals hidden evidence
  • Data Integrity: File systems provide mechanisms to verify data hasn't been tampered with

📖 Key Definitions & Terminology

Before diving deeper into file systems, let's establish the essential terminology you'll encounter throughout this module and the remainder of the course.

File System

A method and data structure that an operating system uses to control how data is stored, organized, and retrieved on a storage device. It provides a logical structure for organizing files and directories, manages space allocation, and maintains metadata about stored data.

Metadata

Data about data. In file systems, metadata includes information such as file names, sizes, creation dates, modification dates, access dates, permissions, and ownership. Metadata is stored separately from the file content itself and is crucial for forensic timeline analysis.

Cluster (Allocation Unit)

The smallest unit of disk space that can be allocated to a file. A cluster consists of one or more contiguous sectors. When a file is saved, the file system allocates whole clusters to it, even if the file doesn't fill the entire cluster.

Sector

The smallest physical storage unit on a disk, traditionally 512 bytes but now commonly 4,096 bytes (4K) on modern drives. Sectors are grouped into clusters by the file system.

Volume

A logical storage unit that contains a single file system. A volume can span an entire physical disk, a partition of a disk, or multiple physical disks (in RAID configurations). Each volume is formatted with a specific file system type.

Partition

A logical division of a physical storage device that creates separate volumes. Each partition can be formatted with a different file system and appears as a separate drive to the operating system.

Slack Space

The unused space within a cluster after a file's data ends. Because clusters are allocated in fixed sizes, files rarely fill their last cluster completely. This leftover space may contain remnants of previously deleted files—a valuable source of forensic evidence.

Unallocated Space

Clusters that are not currently assigned to any file. When files are deleted, their clusters are marked as unallocated but the data typically remains until overwritten by new files. Forensic tools can recover data from unallocated space.

💡 Real-World Analogy

Think of a file system like a library's organization system:

  • The building = Physical storage device (hard drive)
  • Floors/wings = Partitions
  • The card catalog = File system metadata structures
  • Bookshelves = Clusters
  • Individual shelf slots = Sectors
  • Books = Files
  • Empty space on a shelf = Slack space
  • Empty shelves = Unallocated space

Just as a librarian uses the card catalog to locate books, the operating system uses file system structures to locate files. And just as a removed card doesn't make the book disappear from the shelf, deleting a file's catalog entry doesn't immediately erase its data.

🎯 Purpose & Functions of File Systems

File systems serve as the critical interface between the operating system and physical storage media. Without a file system, a storage device would be nothing more than a vast collection of meaningless bytes. Let's explore the essential functions that file systems provide.

Core Functions of File Systems

📂
Organization
Provides hierarchical directory structures (folders within folders) to logically organize files. Creates a namespace for identifying and locating files through paths like C:\Users\Documents\report.docx.
💾
Space Management
Tracks which clusters are in use and which are available. Allocates space when files are created or expanded, and deallocates space when files are deleted or shrunk.
🏷️
Metadata Storage
Maintains essential information about each file: name, size, location on disk, timestamps (created, modified, accessed), permissions, and ownership attributes.
🔗
Data Retrieval
Provides mechanisms to locate and read file data from storage. Translates logical file requests into physical disk locations, handling fragmented files spread across multiple clusters.
🔐
Access Control
Implements security through permissions that control who can read, write, or execute files. Modern file systems support complex access control lists (ACLs) for granular permissions.
🛡️
Data Integrity
Protects against data corruption through journaling (logging changes before applying them), checksums, and recovery mechanisms that can repair inconsistencies after crashes.

🔍 Forensic Significance

Each layer in this abstraction provides forensic opportunities:

  • Application Layer: May leave temporary files, recent file lists, or application-specific metadata
  • OS API Layer: Generates system logs and registry entries
  • File System Layer: Creates timestamps, journal entries, and allocation records
  • Physical Layer: Data persists in unallocated space even after deletion

File System Operations

Understanding how common operations work helps forensic examiners know where to look for evidence:

Operation What Happens Forensic Artifacts
Create File Allocates clusters, creates directory entry, initializes metadata Creation timestamp, journal entry, directory record
Write Data Writes content to allocated clusters, updates size and modification time Modified timestamp, file content, possible previous versions in slack
Read File Locates clusters via metadata, retrieves data, updates access time Access timestamp (on some systems)
Delete File Marks directory entry as deleted, marks clusters as unallocated Data remains in unallocated space, deleted directory entry often recoverable
Rename/Move Updates directory entries, file data doesn't move (usually) New timestamps, possible old directory entries

Component Details

1. Boot Sector / Volume Boot Record (VBR) ▼

The boot sector is the first sector of a volume and contains critical information about the file system structure. It serves as the "table of contents" for the entire volume.

Contents of a Boot Sector:

  • Jump instruction: Directs the CPU to bootstrap code (if bootable)
  • OEM identifier: Identifies the formatting utility/OS
  • BIOS Parameter Block (BPB): Critical file system parameters
    • Bytes per sector
    • Sectors per cluster
    • Total number of sectors
    • Location of file system structures
  • Volume serial number: Unique identifier assigned during formatting
  • Bootstrap code: Initial loading code for bootable volumes
🔍 Forensic Value

The boot sector reveals when the volume was formatted, what tool was used, and provides parameters needed to interpret the rest of the file system. A corrupted boot sector can prevent normal access but forensic tools can often reconstruct it from backup copies or known parameters.

2. Allocation Structures ▼

Allocation structures track which clusters are in use and which are available. They're essential for the file system to know where it can write new data.

Types of Allocation Tracking:

File System Allocation Structure How It Works
FAT12/16/32 File Allocation Table Array of entries, one per cluster. Each entry points to next cluster in chain or marks end-of-file/free.
NTFS $Bitmap file Bitmap where each bit represents one cluster. 1 = allocated, 0 = free.
ext4 Block bitmaps Similar bitmap approach, organized by block groups for efficiency.
APFS Space manager B-tree structure tracking free space extents.
🔍 Forensic Value

Allocation structures show which clusters are currently "free" and may contain deleted file data. Comparing allocation status with actual cluster content can reveal recently deleted files. In FAT systems, the allocation table itself contains the cluster chains, making it possible to trace file fragments.

3. Metadata Structures ▼

Metadata structures store information about files rather than the file content itself. This is where file names, timestamps, sizes, and locations are recorded.

Common Metadata Elements:

  • File name: The human-readable identifier
  • File size: Exact number of bytes in the file
  • Timestamps:
    • Created (birth time)
    • Modified (content changed)
    • Accessed (last read)
    • Changed (metadata changed) - Unix systems
  • Location pointers: Which clusters contain the file data
  • Attributes: Read-only, hidden, system, archive flags
  • Permissions: Who can read, write, execute
  • Owner: User/group ownership information

Metadata Structure by File System:

File System Metadata Structure Key Characteristics
FAT Directory Entries 32-byte entries in directory clusters; limited metadata
NTFS Master File Table (MFT) 1024-byte records; extensive attributes; resident small files
ext4 Inodes Fixed-size structures; name stored separately in directories
APFS Inode records B-tree organized; supports snapshots and clones
🔍 Forensic Value

Metadata structures are goldmines for forensic analysis. Timestamps establish timelines of user activity. Deleted file entries often remain recoverable. NTFS MFT entries may contain entire small files within the metadata itself. Comparing metadata timestamps with file content can reveal anti-forensic manipulation attempts.

4. Data Area ▼

The data area comprises the majority of the volume and contains the actual content of files, organized into clusters.

Data Area Concepts:

  • Allocated clusters: Currently assigned to files; contain current file data
  • Unallocated clusters: Not assigned to any file; may contain deleted data
  • Slack space: Unused portions within allocated clusters
  • Fragmentation: Files split across non-contiguous clusters

🔍 Forensic Value

The data area is where most evidence resides. Unallocated space can contain recoverable deleted files. Slack space may hold fragments of overwritten data. File carving techniques can recover files from unallocated space even without metadata. Forensic imaging captures the entire data area, including all unallocated and slack space.

📊 Types of File Systems

Different operating systems use different file systems, each with unique characteristics that affect forensic analysis. A forensic examiner must be familiar with multiple file system types to effectively analyze evidence from various sources.

File System Categories

FAT (File Allocation Table)

The FAT file system family (FAT12, FAT16, FAT32) was developed by Microsoft starting in 1977. While largely replaced by NTFS on modern Windows systems, FAT remains important for removable media and legacy systems.

Variant Max Volume Max File Common Use
FAT12 16 MB 16 MB Floppy disks (legacy)
FAT16 2 GB 2 GB Small partitions, DOS
FAT32 2 TB 4 GB USB drives, SD cards

NTFS (New Technology File System)

NTFS is the primary file system for modern Windows operating systems (Windows NT and later). It offers significant advantages over FAT for forensic analysis.

Key NTFS Features:

  • Maximum volume size: 16 EB (exabytes)
  • Maximum file size: 16 EB
  • Journaling for crash recovery
  • Access control lists (ACLs) for security
  • File compression and encryption (EFS)
  • Alternate Data Streams (ADS)
  • Sparse files and hard links
🔍 NTFS Forensic Features
  • $MFT: Master File Table contains metadata for all files
  • $LogFile: Transaction journal for recovery and forensic timeline
  • $UsnJrnl: Change journal tracking file system modifications
  • Alternate Data Streams: Can hide data attached to files
  • Resident files: Small files stored entirely in MFT

exFAT (Extended FAT)

Developed by Microsoft for flash drives and SD cards, exFAT removes FAT32's 4GB file size limit while maintaining cross-platform compatibility.

  • Maximum volume size: 128 PB
  • Maximum file size: 16 EB
  • Optimized for flash storage
  • No journaling (less forensic metadata)

ext Family (Extended File System)

The ext file system family is the standard for Linux distributions. Each version has added features while maintaining backward compatibility.

Version Max Volume Max File Key Features
ext2 32 TB 2 TB No journaling; still used for flash
ext3 32 TB 2 TB Added journaling
ext4 1 EB 16 TB Extents, delayed allocation, modern default

ext4 Key Features:

  • Inode-based metadata storage
  • Extent-based allocation (contiguous blocks)
  • Journaling with multiple modes
  • Nanosecond timestamps
  • Extended attributes for additional metadata

XFS

High-performance journaling file system originally from Silicon Graphics, now common in enterprise Linux (default in RHEL/CentOS 7+).

  • Excellent performance with large files
  • Maximum volume: 8 EB
  • Allocation groups for parallelism
  • Online defragmentation and growth

Btrfs (B-tree File System)

Modern copy-on-write file system with advanced features:

  • Built-in snapshots and cloning
  • Checksums for data integrity
  • RAID support at file system level
  • Transparent compression
🔍 Linux Forensic Considerations
  • Inodes: Contain timestamps but not file names (stored in directories)
  • Journal: May contain recent file operations
  • Deleted inodes: Often recoverable with intact metadata
  • Extended attributes: May contain security labels, checksums

HFS+ (Hierarchical File System Plus)

Apple's file system from 1998-2017, still found on older Macs and some external drives.

  • B-tree based catalog and extents
  • Case-insensitive by default (case-preserving)
  • Resource forks for additional file data
  • Journaling added in Mac OS X 10.2.2
  • Maximum volume: 8 EB

APFS (Apple File System)

Modern file system introduced in 2017, now default on all Apple devices (macOS, iOS, iPadOS, watchOS, tvOS).

Key APFS Features:

  • Optimized for flash/SSD storage
  • Copy-on-write design
  • Native encryption support
  • Space sharing between volumes
  • Snapshots for backup (Time Machine)
  • Clones for instant file copies
  • Nanosecond timestamp precision
🔍 APFS Forensic Challenges
  • Encryption: FileVault encryption is default; requires password/key
  • Snapshots: May contain historical file states
  • Copy-on-write: Deleted data may persist longer
  • Container structure: Multiple volumes share space
  • Limited tool support: Newer than other file systems

Common Removable Media File Systems

File System Typical Use Compatibility Limitations
FAT32 USB drives, SD cards Universal (Windows, Mac, Linux) 4 GB max file size
exFAT Large USB drives, SDXC cards Windows, Mac, Linux (with driver) No journaling
UDF Optical media (DVD, Blu-ray) Universal Complex structure
ISO 9660 CD-ROMs Universal Read-only, 8.3 names (without extensions)

Optical Disc File Systems

ISO 9660: Standard file system for CD-ROMs

  • Originally limited to 8.3 filenames
  • Extensions: Joliet (Windows long names), Rock Ridge (Unix attributes)
  • Read-only by design

UDF (Universal Disk Format):

  • Used for DVDs, Blu-rays, and packet-written CDs
  • Supports large files and long filenames
  • Multiple versions with varying features
🔍 Removable Media Forensic Notes
  • FAT-based systems have simpler structures, easier recovery
  • No journaling means less metadata but also less overhead
  • Flash wear leveling may preserve deleted data longer
  • SD cards may have hidden areas (protected areas)
  • Optical media is write-once, preserving original evidence

Key Forensic Concepts

🗑️
Deleted File Recovery

When files are deleted, the file system typically only marks the space as available—the actual data remains until overwritten. Recovery depends on:

  • Time since deletion
  • Disk activity since deletion
  • File system type
  • SSD TRIM operations
⏰
Timeline Analysis

File system timestamps (MAC times) establish when events occurred:

  • Modified: Content changed
  • Accessed: File read/opened
  • Changed: Metadata altered
  • Birth: File created (some FS)
🔎
File Carving

Recovering files from unallocated space using file signatures (magic numbers) rather than file system metadata. Works even when:

  • File system is damaged
  • Metadata is destroyed
  • Volume is reformatted
📊
Metadata Analysis

File system metadata reveals user activity:

  • File ownership and permissions
  • Access patterns
  • Security identifiers (SIDs)
  • Extended attributes

Forensic Challenges by File System

⚠️ SSD and Modern Storage Challenges

Modern storage technologies present unique forensic challenges:

  • TRIM Command: SSDs may immediately zero deleted data blocks, preventing recovery
  • Wear Leveling: Data may be moved without file system knowledge
  • Encryption: Full-disk encryption (BitLocker, FileVault, LUKS) requires keys for access
  • Copy-on-Write: APFS and Btrfs may preserve or destroy data unpredictably

Practical Forensic Workflow

📋 File System Analysis Steps
  1. Identify the file system type - Examine boot sector/superblock
  2. Parse file system structures - Read allocation tables, metadata structures
  3. Enumerate allocated files - Build directory tree of existing files
  4. Recover deleted entries - Find and parse deleted metadata records
  5. Analyze unallocated space - Carve for files without metadata
  6. Extract slack space - Recover data fragments from cluster slack
  7. Build timeline - Correlate timestamps across all sources
  8. Document findings - Record all evidence with proper chain of custody
