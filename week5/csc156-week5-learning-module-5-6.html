GUI vs. Command-Line Tools

🖼️ Autopsy (GUI)
Advantages
User-friendly interface, automated processing, integrated timeline, reporting, supports many file systems.
When to use
Initial broad analysis, complex cases, presenting findings, quick overview.
💻 The Sleuth Kit (CLI)
Advantages
Granular control, scripting capabilities, lightweight, precise low-level analysis, often faster for specific tasks.
When to use
Detailed verification, automation, specific artifact extraction, when GUI tools fail.
⚠️ Important Workflow Reminder

Always perform forensic analysis on a forensic image (read-only copy) of the original evidence. Never work directly on the original device. Use a write-blocker when acquiring images.

🖼️ Autopsy for File System Analysis

Autopsy is a powerful, open-source digital forensics platform that provides a graphical interface to The Sleuth Kit and other tools. It's excellent for initial case review, automated analysis, and generating reports.

Autopsy Workflow Overview

1
Create New Case
Start Autopsy, create a new case, provide case name and base directory.
2
Add Data Source
Select "Disk Image" and browse to your forensic image file (.dd, .e01, .aff).
3
Configure Ingest Modules
Choose which analysis modules to run (File System Analysis, Keyword Search, Hash Lookup, Deleted File Finder, etc.). Enable relevant modules for your investigation.
4
Analyze Results
Once ingest completes, navigate the tree view to explore files, deleted files, artifacts, and generated timelines.
5
Generate Report
Export findings into a report format for documentation and presentation.

Key Autopsy Features for File System Analysis

📁
Tree View Navigation

Explore the file system hierarchically:

  • Data Sources: Your loaded disk images
  • File System: Browse logical file systems (FAT, NTFS, ext4)
  • Deleted Files: Dedicated section for recovered deleted files
  • Unallocated Space: View raw data in unallocated regions
🔎
Ingest Modules

Automated analysis for common tasks:

  • File System Analysis: Parses file systems
  • Deleted File Finder: Identifies deleted entries
  • File Type Identification: Uses signatures
  • Keyword Search: Indexes and searches text
  • Hash Lookup: Identifies known good/bad files
  • Recent Activity: Browser history, LNK files, jump lists
  • EXIF Parser: Extracts image metadata
📊
Result Interpretation

Visual presentation of evidence:

  • File Metadata: Timestamps, permissions, attributes
  • Content Viewer: Hex, text, image, video preview
  • Timeline: Visualizes events chronologically
  • Communications: Email, chat logs
  • Web Artifacts: Browser history, downloads
💡 Autopsy Tip: Deleted Files

Autopsy's "Deleted Files" section is one of the first places to look. It automatically attempts metadata-based recovery for supported file systems (FAT, NTFS, ext4). Pay close attention to the metadata presented for these files.

💻 The Sleuth Kit (TSK) Command-Line Tools

The Sleuth Kit (TSK) is a collection of command-line tools for low-level forensic analysis of disk images. Autopsy uses TSK in its backend, but using TSK directly provides granular control and is essential for scripting or detailed verification.

Core TSK Commands for File System Analysis

Command Purpose Example Usage
mmls Display partition tables (e.g., MBR, GPT) 
mmls image.dd
fsstat Display file system details (superblock, MFT, etc.) 
fsstat -o 2048 image.dd
fls List files and directories (including deleted) 
fls -r -d -o 2048 image.dd
istat Display metadata (inode, MFT entry) for a specific file/directory 
istat -o 2048 image.dd 12345
icat Copy file content by inode/MFT entry number 
icat -o 2048 image.dd 12345 > recovered.txt
blkcalc Calculate block/cluster addresses from file system offsets 
blkcalc -o 2048 image.dd 12345678
blkls List blocks/clusters (e.g., for unallocated space) 
blkls -o 2048 image.dd > unallocated.raw
img_stat Display information about a disk image file 
img_stat image.dd
💡 TSK Tip: The Offset Parameter (-o)

Many TSK commands require the -o parameter to specify the starting sector of the file system you are analyzing within the disk image. You can find this offset using mmls or fsstat.

# Find partition offset
mmls image.dd 
# Output might show a partition starting at sector 2048

# Use that offset for fsstat on the file system
fsstat -o 2048 image.dd

Using TSK for Deleted File Recovery (Lab Prep)

For Lab Activity 5, you'll specifically use fls and icat for FAT32 recovery:

💻 FAT32 Deleted File Recovery Example

Assume your FAT32 partition starts at sector 2048 in usb_image.dd.

# 1. List all files, including deleted ones (marked with '*')
#    -r: recursive listing
#    -d: show deleted files
#    -o 2048: specify the offset to the partition
fls -r -d -o 2048 usb_image.dd

# Expected output snippet:
# r/r * 12345:    _ecret_document.doc   (inode 12345 is deleted)
# r/r   12346:    important_report.pdf

# 2. Use 'istat' to inspect the deleted file's metadata (inode 12345)
istat -o 2048 usb_image.dd 12345

# Look for information like:
# - File size
# - Starting cluster (FAT) or block (ext4)
# - Timestamps (if available)

# 3. Use 'icat' to recover the content of the deleted file
#    Redirect output to a new file
icat -o 2048 usb_image.dd 12345 > recovered_secret_document.doc

# After recovery, remember to manually rename the file (e.g., Secret_document.doc)
# and validate its content.

📊 Advanced MFT Analysis (NTFS)

While Autopsy provides a good overview of NTFS, direct analysis of the Master File Table (MFT) using specialized tools can reveal deeper insights, particularly about deleted files and file system history that might be missed by higher-level tools.

Purpose of Direct MFT Analysis

🕵️
Deeper Insights

Examine raw MFT entries for:

  • Detailed attribute information
  • Multiple $FILE_NAME attributes
  • Resident data content
  • Run lists for non-resident data
👻
Deleted Records

Recover deleted MFT entries that haven't been overwritten:

  • Full metadata for deleted files
  • Original filenames and timestamps
  • Pointers to data clusters
⏱️
Timeline Reconstruction

Correlate MFT entry sequence numbers and LSNs with other artifacts like $UsnJrnl and $LogFile to build a comprehensive activity timeline.

MFTECmd (Eric Zimmerman's Tool)

Eric Zimmerman's MFTECmd is a highly regarded command-line tool for parsing the NTFS $MFT and related artifacts. It's incredibly fast and outputs data in various formats suitable for further analysis.

💻 MFTECmd Example Usage

Assuming you have extracted the $MFT and $UsnJrnl from your image (e.g., using Autopsy or icat from TSK).

# 1. Parse the $MFT file
#    -f: input MFT file
#    -csv: output to CSV
#    -d: output directory
#    --csvf: specify output filename
MFTECmd.exe -f C:\Evidence\ntfs_image\$MFT -csv C:\Results\MFT_Output.csv -d C:\Results\

# 2. Parse the $UsnJrnl (if available)
#    -f: input UsnJrnl file
#    -csv: output to CSV
#    -d: output directory
#    --csvf: specify output filename
MFTECmd.exe -f C:\Evidence\ntfs_image\$Extend\$UsnJrnl -csv C:\Results\UsnJrnl_Output.csv -d C:\Results\

# 3. Combine MFT and UsnJrnl for timeline analysis (e.g., with Timeline Explorer)
#    MFTECmd can also output to bodyfile format for mactime
MFTECmd.exe -f C:\Evidence\ntfs_image\$MFT --body C:\Results\mft_body.csv
MFTECmd.exe -f C:\Evidence\ntfs_image\$Extend\$UsnJrnl --body C:\Results\usnjrnl_body.csv
# Then use mactime to combine and sort:
# mactime -b C:\Results\mft_body.csv C:\Results\usnjrnl_body.csv > timeline.csv
📈 Interpreting MFTECmd Output

The CSV output from MFTECmd will include columns like:

  • EntryNumber: MFT record number (e.g., 0 for $MFT, 5 for root directory)
  • SequenceNumber: Increments each time MFT entry is reused (indicates reuse)
  • FileName: Full filename (from $FILE_NAME attribute)
  • ParentPath: Full path to the parent directory
  • FileSize: Size of the file
  • IsDeleted: Boolean (True/False) if MFT entry flags indicate deletion
  • Timestamps (Created, Modified, LastAccessed, MFTModified): From both $STANDARD_INFORMATION and $FILE_NAME attributes. Critical for timestomping analysis.
  • AttrType: Type of attribute (e.g., $DATA, $STANDARD_INFORMATION)

This detailed output allows for comprehensive analysis of file system activity and deleted file metadata, even for entries that Autopsy might aggregate.

🔪 File Carving Tools

When metadata-based recovery isn't possible (e.g., formatted drives, corrupted file systems, or when seeking files in unallocated space), file carving tools come to the rescue. They scan raw data for file signatures to extract potential files.

Scalpel

Scalpel is a powerful, open-source file carving tool based on Foremost. It's highly configurable and excellent for precise carving tasks.

📝 Scalpel Configuration & Usage

Scalpel uses a configuration file (typically scalpel.conf) to define file types to carve. Each line specifies the file extension, case sensitivity, maximum size, header, and optional footer.

# Example scalpel.conf entry for JPEG:
# extension  case_sensitive  max_size  header                      [footer]
# jpg        y               20000000  \xff\xd8\xff\xe0\x00\x10    \xff\xd9

# To run Scalpel:
# -c: specify config file
# -o: output directory (must be empty or non-existent)
# image.dd: input disk image
scalpel -c /etc/scalpel/scalpel.conf -o /home/analyst/carved_files image.dd

# Reviewing Output:
# Scalpel creates a folder for each file type (e.g., jpg/, pdf/)
# It also generates an 'audit.txt' file with details of each carved file
# (start offset, end offset, file type).

Autopsy Carving Module

Autopsy integrates file carving capabilities directly into its ingest pipeline, making it convenient for comprehensive analysis.

⚙️ Autopsy Carving Workflow
  1. Add Data Source: Load your disk image into Autopsy.
  2. Configure Ingest Modules: During data source addition, ensure the "File Carver" module is enabled.
  3. Settings: You can configure which file types to carve in the module settings. Autopsy has a built-in list of common signatures.
  4. Review Results: Carved files will appear under the "Carved Files" section in the tree view. You can then preview, extract, and analyze them like other files.

Advantage: Autopsy's carving is integrated, allowing you to combine carving results with other artifacts (e.g., keyword hits in carved files).

PhotoRec

PhotoRec is another popular open-source tool, often paired with TestDisk for partition recovery. It's known for its ability to recover a wide variety of file formats from various media types.

📸 PhotoRec Usage (Interactive)

PhotoRec is typically run interactively:

# Run PhotoRec on your disk image
photorec image.dd

# Follow the on-screen prompts:
# 1. Select the source disk/image.
# 2. Select the partition type (e.g., Intel/PC partition).
# 3. Select the source partition/unallocated space.
# 4. Choose file types to search for (or all).
# 5. Select the destination directory for recovered files.

# Validating Carved Files:
# - Check file headers: Use a hex editor to confirm the file signature.
# - Attempt to open: Use native applications to see if the file is intact.
# - Hash comparison: If you know the hash of an original file, compare.
# - Review for garbage: Look for unexpected data at the end of files.

Lab Activity 5: Deleted File Recovery and File System Analysis

📚 Lab Overview & Objectives

In this lab, you will analyze a provided disk image containing multiple file systems. Your primary goal is to identify file systems, recover deleted files using both metadata-based recovery and file carving, and document your findings comprehensively.

Total Points: 20 points

Deliverable: A single PDF document (LastName_FirstName_Lab5.pdf) containing all three parts, including screenshots and recovered file hashes, submitted via Canvas.

Part 1: File System Identification (6 points)

  • Examine multiple partitions within the disk image.
  • Identify the file system types (e.g., FAT32, NTFS, ext4) for each relevant partition.
  • Document key characteristics of each file system (e.g., volume label, size, block/cluster size, number of files/directories).

Part 2: Deleted File Recovery (8 points)

  • FAT32 Volume:
    • List deleted files using fls -d.
    • Recover at least two deleted files using icat.
    • Document recovery success/failure and reasons, noting any filename character loss.
  • NTFS Volume:
    • Analyze MFT for deleted entries using Autopsy.
    • Recover at least two deleted files using Autopsy.
    • Compare $STANDARD_INFORMATION and $FILE_NAME timestamps for a recovered file, discussing any discrepancies.
    • Identify any Alternate Data Streams (ADS) on a file.
    • Export and analyze $UsnJrnl entries related to a deleted file.
  • ext4 Volume (if provided):
    • Examine inode table for deleted files.
    • Attempt recovery using extundelete or similar tool.
    • Document challenges specific to ext4 recovery.

Deliverable: Deleted file recovery report with evidence of recovered files (screenshots, metadata, hashes).

Part 3: File Carving Exercise (6 points)

  • Configure Scalpel or Autopsy's carving module.
  • Carve the provided disk image for specific file types: JPEG images, PDF documents, Office documents (DOCX, XLSX), and ZIP archives.
  • Review carved files for:
    • Valid files vs. false positives.
    • Complete vs. partial recovery.
    • Evidence of fragmentation effects.
  • Document your carving methodology and results.
  • Compare carved files to metadata-based recovery results (if applicable).

Deliverable: File carving report with configuration used and results analysis.

Required Tools Setup

Ensure you have the following tools installed and accessible in your forensic workstation environment:

  • Autopsy: Primary GUI tool for comprehensive analysis.
  • The Sleuth Kit (TSK): Command-line tools (mmls, fsstat, fls, istat, icat) for low-level interaction.
  • Scalpel or PhotoRec: For file carving exercises.
  • MFTECmd (Eric Zimmerman): For advanced MFT and UsnJrnl analysis on NTFS volumes.
  • Hex Editor: (e.g., HxD, Bless, ghidra's hex editor) for examining raw data and file signatures.

Time Management Tips & Common Pitfalls

⏱️
Time Management
  • Start early: Don't wait until the last minute.
  • Break it down: Tackle one part/file system at a time.
  • Carving takes time: Plan for carving to run for a while on large images.
  • Document as you go: Don't try to remember everything at the end.
❌
Common Pitfalls
  • Not using a write-blocker/image: Modifying evidence.
  • Incorrect offsets: TSK commands need the right partition offset.
  • Ignoring error messages: Read and troubleshoot them.
  • Not validating recovered files: Always check if files open and are complete.
  • Poor documentation: Screenshots and clear explanations are key.
  • Expecting 100% recovery: Recognize when data is truly gone.

📚 Key Takeaways

  • Proper lab preparation involves understanding objectives, setting up tools, managing time effectively, and documenting all steps for thorough reporting.
  • Always work on forensic images and validate all recovered data.
