🗑️ Understanding File Deletion

The fundamental principle of file recovery is simple: deleted ≠ destroyed. When you delete a file, the operating system doesn't erase the data—it simply marks the space as available for reuse. Understanding this distinction is crucial for forensic recovery.

Logical Deletion vs. Physical Destruction

Logical Deletion

What happens when you "delete" a file: The file system updates its metadata to indicate the file no longer exists and marks the file's storage space as available. The actual data remains on the storage device until overwritten by new data.

Analogy: Removing a book's card from the library catalog doesn't remove the book from the shelf—it just means the library no longer tracks it.

Physical Destruction

What actually erases data: Overwriting the storage locations with new data, secure deletion tools that write patterns over the data, or physical destruction of the storage media.

Analogy: Actually removing the book from the shelf and replacing it with a different book.

What Happens During Deletion (By File System)

FAT Deletion Process

1
Directory Entry Marked
First byte of filename changed to 0xE5. Rest of entry (including timestamps, starting cluster, size) remains intact.
2
FAT Chain Zeroed
Cluster entries in the FAT are set to 0x0000 (free). The chain linking clusters is broken.
3
Data Unchanged
Actual file content in data clusters is NOT modified. Data persists until clusters are reallocated.
Recovery Potential

High if file was contiguous (not fragmented). Starting cluster and size are preserved in directory entry. Only first character of filename is lost.

NTFS Deletion Process

1
MFT Entry Flags Changed
Flags set to 0x0000 (not in use). Sequence number incremented. All attributes remain intact.
2
$Bitmap Updated
Cluster bitmap marks file's clusters as free. Clusters available for reallocation.
3
Directory Index Updated
File removed from parent directory's $INDEX. May remain in index slack space.
4
$UsnJrnl Entry Created
Deletion recorded in change journal with FILE_DELETE reason code.
Recovery Potential

Excellent. Complete filename preserved in $FILE_NAME attribute. Run list (cluster locations) often intact. Timestamps, permissions, and all metadata available. $UsnJrnl proves file existed.

ext4 Deletion Process

1
Directory Entry Removed
Filename removed from parent directory. Unlike FAT, entry is actually removed, not just marked.
2
Link Count Decremented
Inode's hard link count reduced. If count reaches 0 and no open handles, inode marked for reuse.
3
Block Pointers May Be Cleared
ext4 may zero extent/block pointers in inode. This is the key challenge for ext4 recovery.
4
Deletion Time Set
dtime field in inode records when file was deleted—useful forensic artifact.
Recovery Potential

Moderate. Filename stored in directory (not inode), so often lost. Block pointers may be cleared. Journal may help for recent deletions. File carving often necessary.

The Recovery Window

The time between deletion and data destruction is called the "recovery window." Several factors affect this window:

✅
Extends Recovery Window
  • Large free space on drive
  • Low disk activity after deletion
  • HDD (no TRIM)
  • Quick acquisition after incident
  • File was in less-used area of disk
❌
Shortens Recovery Window
  • Low free space (high reuse)
  • High disk activity
  • SSD with TRIM enabled
  • Defragmentation run
  • System updates/installations
  • Secure deletion tools used
🚨 Critical: SSD TRIM Impact

Remember from Week 4: SSDs with TRIM enabled may immediately and permanently destroy deleted data. The SSD's controller zeros blocks in the background, often within seconds of deletion. This is the single biggest threat to file recovery on modern systems.

Forensic implication: For SSDs, time is critical. Acquire images immediately. Don't boot suspect systems if possible.

📋 Metadata-Based Recovery

Metadata-based recovery uses the file system's own structures to locate and recover deleted files. This is the preferred method when metadata remains intact because it preserves filenames, timestamps, and can handle fragmented files.

Metadata-Based Recovery

A file recovery technique that examines file system metadata structures (directory entries, MFT entries, inodes) to find records of deleted files and use their stored information (starting location, size, cluster chain) to reconstruct the original file.

Recovery by File System

💾 FAT Recovery Technique ▼

Step-by-Step FAT Recovery

1
Scan for 0xE5 Entries
Search directory areas for entries where the first byte is 0xE5. These are deleted file markers.
2
Extract Metadata
Read the remaining 31 bytes: filename (minus first char), extension, attributes, timestamps, starting cluster, file size.
3
Verify Cluster Status
Check if the starting cluster (and subsequent clusters based on file size) are still unallocated in the FAT.
4
Calculate Cluster Range
Using file size and cluster size, calculate how many clusters the file occupied: clusters = ceil(size / cluster_size)
5
Extract Data
Read data from starting cluster for the calculated number of clusters. Trim to exact file size.
FAT Recovery Limitations
  • First character lost: Must guess or use LFN entries if available
  • Fragmentation: FAT chain is zeroed, so fragmented files can't be fully recovered
  • Assumption: Recovery assumes contiguous clusters from starting cluster
Sleuth Kit Commands for FAT Recovery
# List all files including deleted (marked with *)
fls -r -d /dev/sdb1

# Example output:
# r/r * 5:    _EPORT.TXT
#             ^ First char replaced with underscore

# Get inode/entry information
istat /dev/sdb1 5

# Extract file by inode number
icat /dev/sdb1 5 > recovered_file.txt
🖥️ NTFS Recovery Technique ▼

Step-by-Step NTFS Recovery

1
Scan MFT for Deleted Entries
Search $MFT for entries with flags = 0x0000 (not in use). These are deleted files.
2
Parse Attributes
Extract $STANDARD_INFORMATION (timestamps), $FILE_NAME (complete filename), and $DATA (content or run list).
3
Check for Resident Data
Small files may be entirely stored in the MFT entry's $DATA attribute. If resident, extract directly.
4
Follow Run List
For non-resident files, parse the run list to find cluster locations. Run list often survives deletion!
5
Verify and Extract
Check if clusters are still unallocated. Extract data from clusters specified in run list.
NTFS Recovery Advantages
  • Complete filename: $FILE_NAME attribute preserves entire filename
  • Run list survives: Cluster locations often intact, enabling fragmented file recovery
  • Rich metadata: All timestamps, permissions, and attributes available
  • Journal evidence: $UsnJrnl proves file existed and when it was deleted
Sleuth Kit Commands for NTFS Recovery
# List all files including deleted
fls -r -d /dev/sdc1

# Get detailed MFT entry information
istat /dev/sdc1 [MFT_entry_number]

# Extract file content
icat /dev/sdc1 [MFT_entry_number] > recovered_file

# Analyze $MFT directly
icat /dev/sdc1 0 > mft.raw
# Then use MFTECmd or analyzeMFT
🐧 ext4 Recovery Technique ▼

Step-by-Step ext4 Recovery

1
Scan for Deleted Inodes
Search inode tables for inodes with deletion time (dtime) set and link count = 0.
2
Check Journal
Examine ext4 journal for recent transactions that may contain deleted file metadata or data.
3
Examine Block Pointers
Check if extent/block pointers in inode are still populated. ext4 may zero these on deletion.
4
Attempt Recovery
If block pointers exist, extract data. If not, fall back to file carving.
ext4 Recovery Challenges
  • Block pointers cleared: ext4 often zeros extent information on deletion
  • Filename in directory: Filename not stored in inode; directory entry removed
  • Journal limited: Journal only helps for very recent deletions
  • Tool recommendation: Use extundelete or ext4magic for best results
ext4 Recovery Commands
# Using extundelete (specialized ext recovery)
extundelete /dev/sda1 --restore-all

# Using Sleuth Kit
fls -r -d /dev/sda1
istat /dev/sda1 [inode_number]

# Examine journal
jls /dev/sda1
jcat /dev/sda1 [journal_block]

When Metadata Recovery Works Best

✅ Ideal Conditions for Metadata Recovery
  • File was recently deleted
  • Low disk activity since deletion
  • File system structures are intact
  • Metadata entries haven't been overwritten
  • File was not fragmented (especially for FAT)
  • HDD rather than SSD (no TRIM)

🔪 File Carving Fundamentals

File carving is a powerful technique that recovers files based on their content structure rather than file system metadata. It's essential when metadata is unavailable—after formatting, on corrupted file systems, or in unallocated space.

File Carving

A data recovery technique that identifies and extracts files from raw data by searching for known file signatures (magic numbers) and file structures, without relying on file system metadata. Carving works on any storage media regardless of file system type or condition.

Carving Techniques

📍
Header-Footer Carving

Finds file header, then searches for corresponding footer to determine file end.

Best for: JPEG, PDF, ZIP, Office documents

Limitation: Fails if footer is missing or file is fragmented

📏
Header-Maximum Size

Finds header, extracts up to a maximum size limit when no footer exists.

Best for: File types without footers (BMP, some video formats)

Limitation: May include garbage data; requires post-processing

🔍
Structure-Based Carving

Uses internal file structure (chunks, records) to determine boundaries.

Best for: PNG, AVI, MOV (files with internal structure)

Limitation: More complex; requires format knowledge

🧩
Fragment Recovery Carving

Advanced technique that attempts to reassemble fragmented files.

Best for: Heavily fragmented files

Limitation: Computationally expensive; not always successful

Carving Advantages & Limitations

Advantages Limitations
✅ Works without file system metadata ❌ No filename recovery
✅ Recovers from formatted drives ❌ No timestamps or attributes
✅ Works on corrupted file systems ❌ Cannot handle fragmented files
✅ File system agnostic ❌ High false positive rate
✅ Can recover from raw/unallocated space ❌ Requires known file signatures
✅ Useful when metadata is destroyed ❌ May recover partial/corrupted files
⚠️ The Fragmentation Problem

File carving's biggest weakness is fragmented files. When a file is stored in non-contiguous clusters, carving will:

  • Find the header in the first fragment
  • Continue reading until it hits the end of that fragment
  • Include unrelated data from the next file's clusters
  • Miss the remaining fragments entirely

Result: Corrupted or partial file recovery. This is why metadata-based recovery is preferred when possible.

🔢 File Signatures & Magic Numbers

File signatures (also called "magic numbers") are specific byte sequences that identify file types. They're the foundation of file carving and file type identification. Most file formats begin with a distinctive header that identifies them.

File Signature (Magic Number)

A constant sequence of bytes at a specific location (usually the beginning) of a file that identifies the file format. File signatures are independent of file extensions and are used by forensic tools to identify true file types and perform file carving.

Configuring Carving Tools

Most carving tools use configuration files that define file signatures. Here's an example Scalpel configuration:

📝 Scalpel Configuration Example (scalpel.conf)
# Scalpel configuration file
# Format: extension  case_sensitive  size  header  [footer]

# JPEG images
jpg     y       200000000       \xff\xd8\xff\xe0\x00\x10    \xff\xd9
jpg     y       200000000       \xff\xd8\xff\xe1            \xff\xd9

# PNG images  
png     y       200000000       \x89\x50\x4e\x47\x0d\x0a\x1a\x0a    \x49\x45\x4e\x44\xae\x42\x60\x82

# PDF documents
pdf     y       50000000        %PDF    %%EOF

# ZIP archives (and DOCX, XLSX, PPTX)
zip     y       500000000       PK\x03\x04      \x50\x4b\x05\x06

# Microsoft Office legacy
doc     y       50000000        \xd0\xcf\x11\xe0\xa1\xb1\x1a\xe1

# SQLite databases
sqlite  y       50000000        SQLite format 3\x00

File Type Identification vs. Extension

🔍 Why Signatures Matter More Than Extensions

File extensions can be easily changed to hide file types:

  • malware.exe → document.pdf (extension changed)
  • illegal_image.jpg → homework.doc (extension changed)

Forensic tools use file signatures to determine the true file type regardless of extension. This is why:

  • Autopsy shows "Extension Mismatch" alerts
  • The file command in Linux identifies files by content
  • Carving ignores extensions entirely
💻 Identifying True File Types
# Linux/macOS: Use 'file' command
$ file suspicious_document.pdf
suspicious_document.pdf: PE32 executable (GUI) Intel 80386, for MS Windows

# The file claims to be PDF but is actually a Windows executable!

# Using xxd to view hex header
$ xxd suspicious_document.pdf | head -1
00000000: 4d5a 9000 0300 0000 0400 0000 ffff 0000  MZ..............
#         ^^^^
#         MZ = Windows executable, NOT PDF!

⚠️ Recovery Challenges by Storage Type

The type of storage device dramatically affects recovery success. Understanding these differences helps set realistic expectations and choose appropriate techniques.

HDD vs. SSD Recovery Comparison

💾 HDD (Hard Disk Drive)
✅ Excellent recovery potential - Data persists until overwritten
✅ No automatic destruction - No TRIM or garbage collection
✅ Predictable behavior - Well-understood, mature tools
✅ Long recovery window - Days, weeks, or months
✅ Carving effective - Unallocated space contains deleted data
⚠️ Fragmentation - May affect carving success
⚡ SSD (Solid State Drive)
❌ TRIM destroys data - Deleted blocks zeroed automatically
❌ Garbage collection - Background process erases data
❌ Short recovery window - Seconds to minutes
❌ Wear leveling - Data location unpredictable
⚠️ Carving limited - Unallocated space often zeroed
✅ Over-provisioning - May contain remnant data

Other Recovery Challenges

🔐 Encryption ▼

Encrypted storage presents unique challenges:

  • Full Disk Encryption (BitLocker, FileVault, LUKS): Without the key, data appears as random bytes—carving is impossible
  • File-level encryption (EFS): Individual files encrypted; may be recoverable if keys are available
  • Self-encrypting drives (SEDs): Hardware encryption; data inaccessible without authentication

Forensic approach: Acquire keys through memory analysis, key escrow, or user cooperation before imaging if possible.

📝 Overwriting ▼

Once data is overwritten, recovery is generally impossible:

  • Single overwrite: Original data destroyed on modern drives
  • Secure deletion tools: Multiple overwrites with patterns
  • Defragmentation: Moves data, overwrites original locations
  • System updates: May overwrite large areas of disk

Myth busted: The idea that data can be recovered after overwriting (using electron microscopes, etc.) is largely a myth for modern drives. A single overwrite is sufficient to prevent recovery.

💥 Physical Damage ▼

Physical damage requires specialized recovery:

  • HDD head crash: Requires cleanroom, platter transplant
  • HDD motor failure: Requires donor parts
  • SSD controller failure: Direct chip reading may be possible
  • Water/fire damage: Depends on extent; professional recovery needed

Forensic approach: Do not attempt DIY recovery on physically damaged drives. Send to professional data recovery lab.

🕵️ Anti-Forensics ▼

Deliberate attempts to prevent recovery:

  • Secure deletion tools: Eraser, BleachBit, sdelete
  • Full disk wipe: DBAN, nwipe
  • Timestomping: Modifying timestamps to confuse timeline
  • Log clearing: Removing evidence of activity
  • Steganography: Hiding data within other files

Detection: Look for secure deletion tool artifacts, unusual timestamp patterns, gaps in logs, and use steganalysis tools.

🎉 Module 5.5 Summary

📚 Key Takeaways
  • Logical deletion ≠ physical destruction: Data persists until overwritten.
  • Metadata-based recovery uses file system structures to recover files with full metadata (filenames, timestamps). It's best when metadata is intact.
  • File carving scans raw data for file signatures (magic numbers) and works when metadata is corrupted or missing (e.g., formatted drives). It recovers content but not metadata.
  • Fragmentation is the biggest challenge for file carving, often leading to partial or corrupted files.
  • File signatures are crucial for identifying file types and configuring carving tools (e.g., Scalpel, Autopsy).
  • Storage type matters: HDDs offer excellent recovery potential; SSDs with TRIM enabled significantly reduce recovery chances.
  • Always use a forensic image and never work on original evidence.
