🎯 Learning Overview & Objectives

NTFS (New Technology File System) is the dominant file system for modern Windows operating systems and the most frequently encountered file system in Windows forensic investigations. Its sophisticated architecture provides rich forensic artifacts that can reveal detailed user activity, file history, and potential evidence of tampering.

🏗️ NTFS Architecture Overview

NTFS uses a fundamentally different architecture than FAT. While FAT separates metadata into distinct areas (FAT table, directory entries), NTFS treats everything as a file—including the metadata structures themselves. This unified approach provides consistency and flexibility but adds complexity.

NTFS History

Version Windows Version Key Features Added
NTFS 1.0 Windows NT 3.1 (1993) Initial release with MFT, security, journaling
NTFS 1.1 Windows NT 3.5 (1994) Compression support
NTFS 1.2 Windows NT 4.0 (1996) Security descriptors, disk quotas
NTFS 3.0 Windows 2000 (2000) Encryption (EFS), reparse points, $UsnJrnl
NTFS 3.1 Windows XP (2001) Self-healing, improved performance
NTFS 3.1 Windows 10/11 (current) Same version, minor improvements

Key NTFS Features

📋
Master File Table (MFT)
Central database containing a record for every file and directory. The MFT is itself a file ($MFT) and can be analyzed directly.
📝
Journaling
$LogFile provides crash recovery; $UsnJrnl tracks all file changes. Both are invaluable for forensic timeline reconstruction.
🔐
Security
Access Control Lists (ACLs) on every file. Encrypting File System (EFS) for file-level encryption. Stored in $SECURITY_DESCRIPTOR attribute.
📎
Alternate Data Streams
Files can have multiple data streams. Hidden streams can contain evidence or be used to conceal data.
🗜️
Compression
Transparent file and folder compression. Compressed data stored in sparse clusters.
📏
Large Volume Support
Maximum volume size: 16 EB (exabytes). Maximum file size: 16 EB. Far exceeds practical limits.

NTFS Metadata Files

NTFS stores its metadata in special files that begin with $. These files are hidden from normal directory listings but are accessible to forensic tools:

MFT Record Filename Purpose Forensic Value
0 $MFT Master File Table itself ⭐⭐⭐ Contains all file metadata
1 $MFTMirr Backup of first 4 MFT records ⭐⭐ Recovery if MFT damaged
2 $LogFile Transaction journal ⭐⭐⭐ Recent file operations
3 $Volume Volume information ⭐ Volume name, version, flags
4 $AttrDef Attribute definitions ⭐ Attribute type mappings
5 . Root directory ⭐⭐ Directory structure root
6 $Bitmap Cluster allocation bitmap ⭐⭐ Shows allocated/free clusters
7 $Boot Boot sector ⭐ Volume parameters
8 $BadClus Bad cluster list ⭐ May hide data (anti-forensics)
9 $Secure Security descriptors ⭐⭐ File permissions, ownership
10 $UpCase Uppercase conversion table ⭐ Case-insensitive comparisons
11 $Extend Extensions directory ⭐⭐ Contains $UsnJrnl, $ObjId, etc.
🔍 Key Forensic Files in $Extend

The $Extend directory contains additional metadata files crucial for forensics:

  • $UsnJrnl — Update Sequence Number Journal tracking all file changes
  • $ObjId — Object ID index for tracking files across moves/renames
  • $Quota — Disk quota tracking per user
  • $Reparse — Reparse point index (symbolic links, mount points)

📋 Master File Table (MFT)

The Master File Table (MFT) is the heart of NTFS—a relational database containing one record for every file and directory on the volume. Understanding MFT structure is essential for NTFS forensics because it contains all file metadata and can even store small file contents directly.

Master File Table (MFT)

A file ($MFT) that serves as a database of all files and directories on an NTFS volume. Each file has at least one 1,024-byte MFT entry (record) containing the file's attributes—metadata and potentially data. The MFT is the primary source of file system metadata for forensic analysis.

MFT Entry Structure

Each MFT entry is exactly 1,024 bytes (1 KB) and follows a consistent structure:

MFT Entry Layout (1,024 bytes)
┌─────────────────────────────────────────────────────────────────────────────┐
│                           MFT ENTRY HEADER (48 bytes)                        │
├─────────────────────────────────────────────────────────────────────────────┤
│ Offset │ Size │ Field                    │ Description                       │
├────────┼──────┼──────────────────────────┼───────────────────────────────────┤
│ 0x00   │ 4    │ Signature                │ "FILE" (0x454C4946) or "BAAD"     │
│ 0x04   │ 2    │ Fixup Array Offset       │ Offset to update sequence array   │
│ 0x06   │ 2    │ Fixup Array Count        │ Number of fixup entries           │
│ 0x08   │ 8    │ $LogFile Sequence Number │ LSN for journaling                │
│ 0x10   │ 2    │ Sequence Number          │ Incremented when entry reused     │
│ 0x12   │ 2    │ Hard Link Count          │ Number of hard links to file      │
│ 0x14   │ 2    │ First Attribute Offset   │ Offset to first attribute         │
│ 0x16   │ 2    │ Flags                    │ 0x01=In use, 0x02=Directory       │
│ 0x18   │ 4    │ Used Size                │ Bytes used in this entry          │
│ 0x1C   │ 4    │ Allocated Size           │ Total allocated (usually 1024)    │
│ 0x20   │ 8    │ Base Record Reference    │ Base record if extension          │
│ 0x28   │ 2    │ Next Attribute ID        │ ID for next attribute added       │

MFT Entry Flags

Flag Value Meaning Forensic Significance
0x0000 Entry not in use (deleted) ⭐⭐⭐ Deleted file—metadata may be recoverable
0x0001 Entry in use (file) Active file
0x0002 Directory Entry is a directory (combined with 0x01 = 0x03)
0x0003 In use + Directory Active directory
0x0004 Unknown/Extension Rarely seen
0x0008 Unknown/Index Rarely seen
🔍 Forensic Significance of MFT Flags

When a file is deleted in NTFS:

  • The flags field is set to 0x0000 (not in use)
  • The sequence number is incremented (helps track reuse)
  • The attributes remain intact until the entry is reused
  • The $Bitmap is updated to mark clusters as free

This means deleted files can often be recovered by finding MFT entries with flags = 0x0000 and intact attributes.

Resident vs. Non-Resident Data

NTFS has a unique optimization: small files can be stored entirely within the MFT entry itself, without using separate clusters. This is called "resident" data.

📦
Resident Data

File data stored directly in the MFT entry's $DATA attribute.

  • Typical for files under ~700 bytes
  • Very fast access
  • Entire file recoverable from MFT alone
  • Common for small text files, shortcuts
🔗
Non-Resident Data

File data stored in clusters outside the MFT. The $DATA attribute contains a "run list" pointing to the clusters.

  • Required for larger files
  • Run list describes cluster locations
  • Can be fragmented across volume
  • MFT contains metadata only
💡 Run List Example

A run list (also called "data runs") describes where non-resident data is stored. Each run specifies a starting cluster and length:

Run List: 21 18 34 12 01 00
Decoded:
  Run 1: Start at cluster 0x1234, length 0x18 clusters
  Run 2: (would continue if more data)

The first nibble indicates the size of the length and offset fields. Forensic tools parse these automatically, but understanding the concept helps when manually analyzing MFT entries.

🏷️ NTFS Attributes

In NTFS, everything about a file—its name, timestamps, permissions, and data—is stored as attributes within the MFT entry. Understanding these attributes is crucial for forensic analysis because different attributes contain different types of evidence.

Common NTFS Attribute Types

0x10$STANDARD_INFORMATION
Timestamps (MACE), file attributes, owner ID, security ID. Present in every file. Key forensic artifact.
0x30$FILE_NAME
Filename, parent directory reference, timestamps (MACE), file size. May have multiple (short + long name).
0x40$OBJECT_ID
Unique object identifier. Can track files across moves/renames. Used by Windows link tracking.
0x50$SECURITY_DESCRIPTOR
Access Control List (ACL), owner, group. Usually stored in $Secure file and referenced by ID.
0x60$VOLUME_NAME
Volume label. Only in $Volume metadata file.
0x70$VOLUME_INFORMATION
NTFS version, flags. Only in $Volume metadata file.
0x80$DATA
File content (default stream). Can be resident (in MFT) or non-resident (in clusters). Files can have multiple $DATA attributes (ADS).
0x90$INDEX_ROOT
Root of B-tree index. Used for directories (filename index) and other indexes.
0xA0$INDEX_ALLOCATION
Non-resident index data. For directories, contains additional filename entries. May contain deleted entries!
0xB0$BITMAP
Bitmap for indexes showing which entries are in use.
0xC0$REPARSE_POINT
Symbolic links, mount points, and other reparse data. Can indicate junction points or symlinks.
0xD0$EA_INFORMATION
Extended attribute information. Size of EA data.
0xE0$EA
Extended attributes. Used for OS/2 compatibility and some applications.
0x100$LOGGED_UTILITY_STREAM
EFS encryption information. Contains encryption keys and recovery data.

Attribute Header Structure

Every attribute starts with a header that describes the attribute:

Offset Size Field Description
0x00 4 Attribute Type Type code (0x10, 0x30, 0x80, etc.)
0x04 4 Length Total length of attribute including header
0x08 1 Non-Resident Flag 0 = Resident, 1 = Non-resident
0x09 1 Name Length Length of attribute name (for named streams)
0x0A 2 Name Offset Offset to attribute name
0x0C 2 Flags Compressed, encrypted, sparse flags
0x0E 2 Attribute ID Unique ID within this MFT entry

$STANDARD_INFORMATION Attribute (0x10)

This attribute is present in every file and directory and contains critical metadata:

Offset Size Field Description
0x00 8 Created Time File creation timestamp
0x08 8 Modified Time File content modification timestamp
0x10 8 MFT Modified Time MFT entry modification timestamp
0x18 8 Accessed Time Last access timestamp
0x20 4 Flags Read-only, hidden, system, archive, etc.
0x24 4 Max Versions Maximum versions (usually 0)
0x28 4 Version Number Current version (usually 0)
0x2C 4 Class ID Class ID (usually 0)
0x30 4 Owner ID Owner ID (NTFS 3.0+)
0x34 4 Security ID Security descriptor ID (NTFS 3.0+)
0x38 8 Quota Charged Bytes charged to quota (NTFS 3.0+)
0x40 8 USN Update Sequence Number (NTFS 3.0+)

$FILE_NAME Attribute (0x30)

Contains the filename and additional timestamps. A file may have multiple $FILE_NAME attributes (e.g., long name and 8.3 short name):

Offset Size Field Description
0x00 8 Parent Directory Reference MFT reference to parent directory
0x08 8 Created Time File creation timestamp
0x10 8 Modified Time File modification timestamp
0x18 8 MFT Modified Time MFT entry modification timestamp
0x20 8 Accessed Time Last access timestamp
0x28 8 Allocated Size Allocated size of file
0x30 8 Real Size Actual size of file
0x38 4 Flags File attribute flags
0x3C 4 Reparse Value Reparse point tag
0x40 1 Name Length Length of filename in characters
0x41 1 Namespace 0=POSIX, 1=Win32, 2=DOS, 3=Win32+DOS
0x42 varies Filename Unicode filename (2 bytes per char)
🔍 Namespace Values
  • 0 (POSIX): Case-sensitive, allows almost any character
  • 1 (Win32): Standard Windows long filename
  • 2 (DOS): 8.3 short filename for compatibility
  • 3 (Win32 & DOS): Name fits both Win32 and DOS requirements

Files with long names typically have two $FILE_NAME attributes: one with namespace 1 (Win32) and one with namespace 2 (DOS).

⏰ NTFS Timestamps & Forensic Analysis

NTFS stores eight timestamps per file—four in $STANDARD_INFORMATION and four in $FILE_NAME. Understanding the differences between these timestamps is critical for forensic analysis, particularly for detecting timestamp manipulation (timestomping).

NTFS Timestamp Format

FILETIME Format

NTFS timestamps use the Windows FILETIME format: a 64-bit value representing the number of 100-nanosecond intervals since January 1, 1601 UTC.

Resolution: 100 nanoseconds (0.0000001 seconds)

Range: 1601 to approximately 30828

This is vastly more precise than FAT's 2-second resolution!

The MACE Timestamps

Both $STANDARD_INFORMATION (SI) and $FILE_NAME (FN) contain four timestamps, often called MACE:

Letter Timestamp Updated When
M Modified File content is changed (data written)
A Accessed File is read (may be disabled for performance)
C Changed (MFT Modified) MFT entry is modified (metadata change)
E Entry Created (Birth) File is first created

$STANDARD_INFORMATION vs. $FILE_NAME Timestamps

This is one of the most important forensic concepts in NTFS analysis:

📊 $STANDARD_INFORMATION (SI)
Updated by: User-level APIs
Modifiable: Yes (easily)
Tools can change: SetFileTime(), touch, timestomp
Forensic trust: ⚠️ Lower (can be manipulated)
Shown by: Windows Explorer, dir command
📁 $FILE_NAME (FN)
Updated by: Kernel only
Modifiable: No (not by normal APIs)
Updated when: File created, renamed, moved
Forensic trust: ✅ Higher (harder to fake)
Shown by: Forensic tools only
🚨 Timestomping Detection

Timestomping is an anti-forensic technique where attackers modify file timestamps to hide malicious activity or make files appear older/newer than they are.

Detection method: Compare SI and FN timestamps!

  • If SI timestamps are older than FN timestamps → Suspicious!
  • If SI Created time is before FN Created time → Likely timestomped
  • If all SI timestamps are identical → Suspicious (tools often set all four the same)
  • If timestamps show impossible precision patterns → May indicate manipulation

🔍 Timestamp Comparison Scenarios

✅ Normal File Creation and Modification

A file created on March 1, 2025 and modified on March 15, 2025:

$STANDARD_INFORMATION
Created: 2025-03-01 09:15:32
Modified: 2025-03-15 14:22:18
MFT Changed: 2025-03-15 14:22:18

Modified: 2025-03-01 09:15:32
MFT Changed: 2025-03-01 09:15:32
Accessed: 2025-03-01 09:15:32
Analysis

Normal behavior: FN timestamps match the original creation time. SI timestamps show subsequent modifications. This is expected—FN is only updated on file creation, rename, or move.

📋 File Copied from Another Location

A file originally created January 10, 2025, copied to this location on March 20, 2025:

$STANDARD_INFORMATION
Created: 2025-03-20 11:45:03
Modified: 2025-01-10 08:30:15
MFT Changed: 2025-03-20 11:45:03
Accessed: 2025-03-20 11:45:03
$FILE_NAME
Created: 2025-03-20 11:45:03
Modified: 2025-03-20 11:45:03
MFT Changed: 2025-03-20 11:45:03
Accessed: 2025-03-20 11:45:03
Analysis

Copy behavior: Notice SI Modified is older than SI Created! This is normal for copied files—Windows preserves the original Modified time but sets Created to the copy time. FN timestamps all match the copy time.

🚨 Timestomped File (Manipulation Detected)

An attacker created malware on March 25, 2025, then used timestomping to make it appear old:

$STANDARD_INFORMATION ⚠️
Created: 2019-06-15 00:00:00
Modified: 2019-06-15 00:00:00
MFT Changed: 2019-06-15 00:00:00
Accessed: 2019-06-15 00:00:00
$FILE_NAME (Truth!)
Created: 2025-03-25 02:14:37
Modified: 2025-03-25 02:14:37
MFT Changed: 2025-03-25 02:14:37
Accessed: 2025-03-25 02:14:37
🚨 Red Flags Detected!
  • SI Created is BEFORE FN Created — Impossible without manipulation!
  • All four SI timestamps are identical — Common timestomping artifact
  • Timestamps at exactly midnight — Suspicious precision
  • FN reveals true creation date — File actually created March 25, 2025

📎 Alternate Data Streams (ADS)

Alternate Data Streams (ADS) are a unique NTFS feature that allows multiple data streams to be associated with a single file. While designed for legitimate purposes, ADS can be used to hide data, making them an important forensic consideration.

Alternate Data Stream (ADS)

An additional named $DATA attribute attached to a file or directory. Every file has a default (unnamed) data stream containing its normal content. Additional named streams can store extra data that is invisible to standard Windows tools like Explorer and dir.

How ADS Works

File with Multiple Data Streams
┌─────────────────────────────────────────────────────────────────────────────┐
│                         MFT Entry for "document.txt"                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  $STANDARD_INFORMATION (0x10)                                               │
│  └── Timestamps, flags, etc.                                                │
│                                                                              │
│  $FILE_NAME (0x30)                                                          │
│  └── "document.txt"                                                         │
│                                                                              │
│  $DATA (0x80) - UNNAMED (Default Stream)                                    │
│  └── "This is the normal file content that everyone sees..."               │
│      Size: 1,234 bytes                                                      │

│                                                                              │

ADS Syntax

Alternate Data Streams are accessed using the filename:streamname syntax:

# Create a normal file
C:\>
echo "Normal content" > document.txt
# Create an alternate data stream
C:\>
echo "Hidden content" > document.txt:secret
# View the file normally (ADS not shown)
C:\>
dir document.txt
03/25/2025 10:30 AM 17 document.txt
# View with /R flag to show streams
C:\>
dir /R document.txt
03/25/2025 10:30 AM 17 document.txt 16 document.txt:secret:$DATA
# Read the alternate stream
C:\>
more < document.txt:secret
Hidden content

Legitimate Uses of ADS

🌐
Zone.Identifier

Windows uses ADS to track where files came from:

  • ZoneId=0: Local machine
  • ZoneId=1: Local intranet
  • ZoneId=2: Trusted sites
  • ZoneId=3: Internet
  • ZoneId=4: Restricted sites

This triggers the "This file came from another computer" warning.

🖼️
Thumbnails & Summaries

Some applications store metadata in ADS:

  • Image thumbnails
  • Document summaries
  • Custom properties
  • Application-specific data
🔐
Encryption Metadata

EFS (Encrypting File System) stores encryption information in ADS, including recovery certificates and encryption keys.

Malicious Uses of ADS

⚠️ ADS as a Hiding Technique

Attackers may use ADS to:

  • Hide malware: Store executable code in an ADS attached to a legitimate file
  • Exfiltrate data: Hide stolen data in ADS before extraction
  • Evade detection: ADS content is invisible to basic file listings
  • Persist on system: Some malware uses ADS for persistence mechanisms

Note: Modern Windows and security tools are ADS-aware, but it remains a technique worth checking.

Forensic Detection of ADS

Tool/Command How to Detect ADS
dir /R Lists alternate streams in directory listing
streams.exe (Sysinternals) Recursively finds all ADS on a volume
PowerShell Get-Item -Stream * Lists all streams for a file
Autopsy / FTK Automatically extracts and displays ADS
MFT analysis tools Show all $DATA attributes including named ones
🔍 Zone.Identifier Forensic Value

The Zone.Identifier ADS is particularly valuable for forensics:

  • Proves a file was downloaded from the internet (ZoneId=3)
  • May contain the original URL (ReferrerUrl field)
  • May contain the downloading application (HostUrl field)
  • Survives file moves within the same NTFS volume
  • Removed when file is copied to non-NTFS volume
[ZoneTransfer]
ZoneId=3
ReferrerUrl=https://example.com/downloads/
HostUrl=https://example.com/files/malware.exe

📝 NTFS Journaling ($LogFile & $UsnJrnl)

NTFS maintains two journals that record file system activity. These journals are invaluable for forensic investigations because they can reveal file operations even after files have been deleted or modified.

The Two NTFS Journals

📋
$LogFile (Transaction Log)

Purpose: Crash recovery and file system integrity

  • Records metadata changes before they're committed
  • Enables rollback of incomplete operations
  • Circular log (overwrites old entries)
  • Typically 64 MB
  • Contains detailed operation data
📜
$UsnJrnl (Change Journal)

Purpose: Track all file changes for applications

  • Records what changed, not how
  • Used by backup, search, antivirus
  • Located in $Extend\$UsnJrnl
  • Can grow very large (32+ MB)
  • Survives longer than $LogFile

$LogFile Details

The $LogFile is a transaction journal that records all metadata operations. It's designed for crash recovery but has forensic value:

📋 $LogFile Forensic Value
  • Recent operations: Shows file system changes from the last few hours/days
  • Deleted file evidence: May contain MFT entries for recently deleted files
  • Timestamp history: Can reveal original timestamps before modification
  • Filename history: May show previous filenames after rename

Limitation: Circular buffer means old data is overwritten. Active systems may only have minutes of history.

$UsnJrnl (Update Sequence Number Journal)

The $UsnJrnl is the primary forensic journal, recording a history of all file system changes:

Field Size Description
Record Length 4 bytes Size of this record
Major Version 2 bytes USN journal version (usually 2)
Minor Version 2 bytes Minor version number
File Reference 8 bytes MFT reference number
Parent Reference 8 bytes Parent directory MFT reference
USN 8 bytes Update Sequence Number (offset in journal)
Timestamp 8 bytes When the change occurred (FILETIME)
Reason 4 bytes Flags indicating what changed
Source Info 4 bytes Source of the change
Security ID 4 bytes Security identifier
File Attributes 4 bytes File attribute flags
Filename Length 2 bytes Length of filename in bytes
Filename Offset 2 bytes Offset to filename
Filename varies Unicode filename

$UsnJrnl Reason Codes

The "Reason" field contains flags indicating what type of change occurred:

Flag Value Meaning
DATA_OVERWRITE 0x00000001 File data was overwritten
DATA_EXTEND 0x00000002 File data was extended
DATA_TRUNCATION 0x00000004 File data was truncated
NAMED_DATA_OVERWRITE 0x00000010 ADS was overwritten
NAMED_DATA_EXTEND 0x00000020 ADS was extended
FILE_CREATE 0x00000100 File was created
FILE_DELETE 0x00000200 File was deleted
EA_CHANGE 0x00000400 Extended attributes changed
SECURITY_CHANGE 0x00000800 Security descriptor changed
RENAME_OLD_NAME 0x00001000 Old name before rename
RENAME_NEW_NAME 0x00002000 New name after rename
INDEXABLE_CHANGE 0x00004000 Indexing-relevant change
BASIC_INFO_CHANGE 0x00008000 Timestamps or attributes changed
HARD_LINK_CHANGE 0x00010000 Hard link added/removed

🔍 $UsnJrnl Forensic Power

The example above shows how $UsnJrnl can reveal the complete history of a malicious file:

  1. File was downloaded as "malware.exe"
  2. Attacker renamed it to "svchost.exe" to disguise it
  3. Attacker modified timestamps (timestomping)
  4. File was later deleted to cover tracks

Even though the file is deleted, the $UsnJrnl preserves the entire activity history!

🗑️ NTFS File Deletion & Recovery

Understanding how NTFS handles file deletion is crucial for forensic recovery. Like FAT, NTFS deletion doesn't immediately destroy data, but the process is more complex due to NTFS's sophisticated architecture.

What Happens When You Delete a File in NTFS

NTFS Deletion Process
STEP 1: MFT Entry Flag Changed
┌─────────────────────────────────────────────────────────────────────────────┐
│  MFT Entry Header                                                            │
│  ├── Flags: 0x0001 (In Use) ──────────► 0x0000 (Not In Use)                │
│  ├── Sequence Number: 5 ──────────────► 6 (Incremented)                     │
│  └── Attributes: UNCHANGED (still contain all metadata!)                    │
└─────────────────────────────────────────────────────────────────────────────┘

STEP 2: $Bitmap Updated
┌─────────────────────────────────────────────────────────────────────────────┐
│  Cluster Bitmap ($Bitmap)                                                    │
│  Clusters 1000-1005: 111111 ──────────► 000000 (Marked as free)            │
│  (File's clusters are now available for reuse)                              │
└─────────────────────────────────────────────────────────────────────────────┘

STEP 3: Parent Directory Updated
┌─────────────────────────────────────────────────────────────────────────────┐
│  Parent Directory $INDEX_ROOT / $INDEX_ALLOCATION                           │
│  └── Index entry for file: REMOVED from B-tree index                        │
│      (But may remain in slack space of index!)                              │
└─────────────────────────────────────────────────────────────────────────────┘

STEP 4: $UsnJrnl Entry Created
┌─────────────────────────────────────────────────────────────────────────────┐
│  $UsnJrnl Record                                                             │
│  ├── Reason: FILE_DELETE (0x00000200)                                       │
│  ├── Timestamp: When deletion occurred                                      │
│  └── Filename: Name of deleted file                                         │
└─────────────────────────────────────────────────────────────────────────────┘

WHAT REMAINS AFTER DELETION:
✅ MFT entry (with flag = 0x0000) - All attributes intact!
✅ File data in clusters (until overwritten)
✅ $UsnJrnl record of deletion
✅ Possibly: Index slack space with filename
✅ Possibly: $LogFile entries

NTFS vs. FAT Deletion Comparison

Aspect FAT NTFS
Deletion marker First byte of filename → 0xE5 MFT flags → 0x0000
Filename preserved? Partially (first char lost) Completely (in MFT)
Timestamps preserved? Yes (in directory entry) Yes (in MFT attributes)
Cluster chain preserved? No (FAT entries zeroed) Yes (run list in MFT)
Deletion logged? No Yes ($UsnJrnl)
Recovery difficulty Moderate Often easier (more metadata)

Recovery Scenarios

✅ Best Case: Recent Deletion, Low Activity ▼

Conditions:

  • File deleted recently
  • Little disk activity since deletion
  • MFT entry not yet reused
  • Data clusters not overwritten

Recovery method:

  1. Parse MFT for entries with flags = 0x0000
  2. Extract $FILE_NAME attribute for filename
  3. Extract $DATA attribute (resident) or run list (non-resident)
  4. Read data from clusters specified in run list
  5. Verify with $UsnJrnl for deletion timestamp

Result: Complete file recovery with full metadata

⚠️ Moderate Case: MFT Entry Reused ▼

Conditions:

  • MFT entry has been reused for new file
  • Original metadata lost
  • Data clusters may still be intact

Recovery method:

  1. Check $UsnJrnl for deletion record (filename, timestamp)
  2. Check $LogFile for recent MFT changes
  3. Use file carving on unallocated clusters
  4. Match carved data with $UsnJrnl records

Result: Possible data recovery, limited metadata

❌ Worst Case: Data Overwritten ▼

Conditions:

  • MFT entry reused
  • Data clusters overwritten by new files
  • Significant time/activity since deletion

What may still exist:

  • $UsnJrnl record (filename, deletion time)
  • Slack space fragments
  • $INDEX_ALLOCATION slack (directory index)
  • Volume shadow copies (if enabled)

Result: Evidence of existence, but data likely unrecoverable

🔍 Key NTFS Recovery Advantages
  • Complete filenames: Unlike FAT, NTFS preserves the entire filename
  • Cluster locations: Run list often survives, showing exact data locations
  • Multiple timestamps: Both SI and FN timestamps may be available
  • Deletion evidence: $UsnJrnl proves file existed and when it was deleted
  • Resident files: Small files stored in MFT may be fully recoverable from MFT alone

🎉 Module 5.3 Summary

📚 Key Takeaways
  • NTFS is the dominant Windows file system and the most important for Windows forensics
  • The Master File Table (MFT) contains 1,024-byte entries for every file with rich metadata
  • NTFS attributes store all file information: $STANDARD_INFORMATION, $FILE_NAME, $DATA, etc.
  • Two sets of timestamps (SI and FN) enable detection of timestomping
  • Alternate Data Streams (ADS) can hide data and should always be checked
  • $LogFile and $UsnJrnl provide historical records of file operations
  • NTFS deletion preserves more metadata than FAT, aiding recovery
  • Zone.Identifier ADS reveals download sources for files
