🐧 The Extended File System Family

The Extended File System (ext) family is the standard file system for Linux. Understanding its evolution helps explain the features available in the current ext4 version and why older systems might still use ext2 or ext3.

Evolution of ext File Systems

1992 ext (First Extended File System)

First file system created specifically for Linux. Replaced the Minix file system. Maximum size: 2 GB. Now obsolete.

1993 ext2 (Second Extended File System)

Major rewrite with improved performance and larger limits. Maximum volume: 32 TB. Maximum file: 2 TB. No journaling—still used for flash drives where journaling causes wear.

2001 ext3 (Third Extended File System)

Added journaling for crash recovery. Backward compatible with ext2 (can mount ext2 as ext3). Same size limits as ext2. Three journaling modes: journal, ordered, writeback.

2008 ext4 (Fourth Extended File System)

Current standard. Added extents for better large file handling. Maximum volume: 1 EB. Maximum file: 16 TB. Nanosecond timestamps. Backward compatible with ext3. Default on most Linux distributions.

Key Differences Summary

Feature ext2 ext3 ext4
Year Introduced 1993 2001 2008
Journaling ❌ No ✅ Yes ✅ Yes (improved)
Max Volume Size 32 TB 32 TB 1 EB
Max File Size 2 TB 2 TB 16 TB
Max Filename 255 bytes 255 bytes 255 bytes
Timestamp Resolution 1 second 1 second 1 nanosecond
Block Allocation Indirect blocks Indirect blocks Extents
Creation Time (crtime) ❌ No ❌ No ✅ Yes
Current Usage Flash media, recovery Legacy systems Default Linux FS
🐧 Why ext2 Still Exists

You might wonder why ext2 (without journaling) is still used. The answer is flash storage:

  • Journaling causes additional writes, which wears out flash memory faster
  • SD cards and USB drives formatted on Linux often use ext2
  • Embedded systems with limited flash may use ext2
  • Recovery scenarios: ext2 is simpler to analyze and recover

Key ext4 Components

📋 Superblock ▼

The superblock contains critical file system metadata. It's located at byte offset 1024 (after the boot block) and has backup copies in certain block groups.

Key Superblock Fields:

Offset Size Field Description
0x00 4 s_inodes_count Total number of inodes
0x04 4 s_blocks_count_lo Total number of blocks (low 32 bits)
0x18 4 s_log_block_size Block size = 1024 << this value
0x28 4 s_blocks_per_group Blocks per block group
0x38 2 s_magic Magic number: 0xEF53
0x44 4 s_lastcheck Last file system check time
0x4C 4 s_creator_os OS that created the FS
0x58 4 s_inode_size Size of inode structure
0x78 16 s_uuid 128-bit UUID of volume
0x88 16 s_volume_name Volume label
🔍 Forensic Value
  • UUID: Uniquely identifies the volume—useful for tracking drives
  • Mount time/count: Shows when volume was last mounted
  • Last check time: When fsck was last run
  • Creator OS: Identifies what created the file system
  • Backup superblocks: Can recover from primary superblock corruption
📄 Inodes ▼

An inode (index node) is the fundamental metadata structure in ext file systems. Every file and directory has exactly one inode that stores all metadata except the filename.

Inode

A fixed-size data structure containing a file's metadata: permissions, ownership, timestamps, size, and pointers to data blocks. The inode number is the file's unique identifier within the file system. Filenames are stored in directory entries, not in inodes.

💡 Inode vs. NTFS MFT Entry

The inode serves a similar purpose to an NTFS MFT entry, but with key differences:

  • Filename location: Inodes don't store filenames (directories do); MFT entries contain $FILE_NAME
  • Size: Inodes are smaller (256 bytes vs. 1024 bytes)
  • Data storage: Inodes use extents or block pointers; MFT can store small files directly
  • Attributes: Inodes have fixed fields; MFT has flexible attributes
📊 Extents (ext4 Feature) ▼

Extents are an ext4 feature that replaces the traditional indirect block pointers used in ext2/ext3. An extent describes a contiguous range of blocks, making large file storage more efficient.

Indirect Blocks (ext2/ext3) vs. Extents (ext4)
ext2/ext3: INDIRECT BLOCK POINTERS
─────────────────────────────────────
Inode contains:
├── 12 direct block pointers → [Block 100][Block 101]...[Block 111]
├── 1 indirect pointer → [Pointer Block] → [Block 200][Block 201]...
├── 1 double indirect → [Ptr Block] → [Ptr Blocks] → [Data Blocks]
└── 1 triple indirect → [Ptr] → [Ptrs] → [Ptrs] → [Data Blocks]

Problem: Large files require many pointer blocks = overhead + fragmentation


ext4: EXTENTS
─────────────────────────────────────
Inode contains extent tree:
├── Extent 1: Start=100, Length=50   → Blocks 100-149 (contiguous!)
├── Extent 2: Start=500, Length=100  → Blocks 500-599 (contiguous!)
└── Extent 3: Start=1000, Length=200 → Blocks 1000-1199 (contiguous!)

Advantage: Fewer pointers, better performance, less fragmentation

Extent Structure:

Field Size Description
ee_block 4 bytes First logical block this extent covers
ee_len 2 bytes Number of blocks in extent (max 32768)
ee_start_hi 2 bytes High 16 bits of physical block
ee_start_lo 4 bytes Low 32 bits of physical block
🔍 Forensic Implications of Extents
  • Less fragmentation: Deleted file data more likely to be contiguous
  • Easier recovery: Extent information may survive in inode after deletion
  • Extent tree: Large files use B-tree of extents (more complex to parse)
  • Uninitialized extents: May contain data from previous files (forensic artifact)
📝 ext4 Journal ▼

ext4 uses a journal to ensure file system consistency after crashes. The journal records changes before they're committed to the main file system.

Journaling Modes:

Mode What's Journaled Performance Safety
journal Metadata + Data Slowest Highest
ordered (default) Metadata only, data written first Medium Good
writeback Metadata only, data order not guaranteed Fastest Lowest
🔍 Journal Forensic Value
  • Recent operations: Journal may contain recently deleted file metadata
  • Data recovery: In "journal" mode, actual file data may be in journal
  • Timeline: Journal transactions have sequence numbers for ordering
  • Tool: jls and jcat from Sleuth Kit parse ext journals

🔍 ext4 Forensic Analysis

ext4 provides rich forensic artifacts, including four timestamps per file and a journal that can reveal recent activity. Understanding these artifacts is essential for Linux forensics.

ext4 Timestamps (MAC + B)

ext4 stores four timestamps per file, similar to NTFS but with different names and behaviors:

Modify Time
mtime
When file content was last changed
Access Time
atime
When file was last read (may be disabled)
Change Time
ctime
When inode was last modified (metadata change)
Creation Time
crtime
When file was created (ext4 only!)
⚠️ ctime is NOT Creation Time!

A common mistake: ctime does NOT mean "creation time" in Unix/Linux. It means "change time" and records when the inode metadata was last modified. This includes:

  • Permission changes (chmod)
  • Ownership changes (chown)
  • Hard link creation/deletion
  • File content changes (also updates mtime)

crtime (creation time) was added in ext4 and is the actual birth time of the file.

Timestamp Comparison: ext4 vs. NTFS

Concept ext4 NTFS
Content Modified mtime $SI Modified / $FN Modified
Last Accessed atime $SI Accessed / $FN Accessed
Metadata Changed ctime $SI MFT Modified
File Created crtime (ext4 only) $SI Created / $FN Created
Resolution 1 nanosecond (ext4) 100 nanoseconds
Timezone UTC UTC
Easily Modified? mtime/atime: Yes; ctime: No* $SI: Yes; $FN: No

* ctime cannot be set directly by users but is updated automatically when inode changes.

🔍 Timestomping Detection in ext4

Like NTFS, ext4 timestamp manipulation can be detected:

  • mtime before crtime: Impossible without manipulation (file can't be modified before creation)
  • ctime after mtime: Normal if only metadata changed; suspicious if content should have changed
  • crtime much older than ctime: May indicate file was copied (crtime = copy time, but mtime preserved)
  • Journal analysis: Journal may contain original timestamps

ext4 File Deletion

When a file is deleted in ext4:

1️⃣
Directory Entry Removed
The filename is removed from the parent directory. Unlike FAT (0xE5 marker), the entry is actually removed from the directory structure.
2️⃣
Link Count Decremented
The inode's hard link count is reduced by 1. If it reaches 0 and no process has the file open, the inode is marked for reuse.
3️⃣
Blocks Deallocated
The block bitmap is updated to mark the file's blocks as free. The extent/block pointers in the inode may be cleared.
4️⃣
Deletion Time Set
ext4 sets a deletion time (dtime) in the inode, recording when the file was deleted—useful for forensics!
🔍 ext4 Recovery Challenges

ext4 file recovery is generally harder than NTFS because:

  • Extent pointers may be cleared: Unlike NTFS (run lists often survive), ext4 may zero extent data
  • No equivalent to $FILE_NAME: Filename is only in directory, not in inode
  • Journal helps: Recent deletions may be recoverable from journal
  • extundelete tool: Specialized tool for ext3/ext4 recovery

Useful Commands for ext4 Analysis

🍎 HFS+ (Mac OS Extended)

HFS+ (Hierarchical File System Plus) was Apple's primary file system from 1998 until 2017 when APFS was introduced. You'll still encounter HFS+ on older Macs, external drives, and Time Machine backups.

HFS+ (Mac OS Extended)

Apple's file system introduced in Mac OS 8.1 (1998). Uses B-tree structures for efficient file lookup. Supports journaling (HFS+ Journaled), case sensitivity options, and resource forks for storing additional file data.

HFS+ Forensic Features

📁
Catalog File

The Catalog File is a B-tree containing records for every file and folder. Each record includes:

  • Catalog Node ID (CNID) - unique file identifier
  • Parent CNID
  • File name
  • Timestamps (created, modified, accessed, attribute modified)
  • Data fork and resource fork info
🍴
Resource Forks

HFS+ files can have two "forks":

  • Data Fork: Normal file content
  • Resource Fork: Structured data (icons, metadata, code)

Resource forks are legacy but may contain hidden data.

🔤
Case Sensitivity

HFS+ has two variants:

  • Case-insensitive (default): "File.txt" = "FILE.TXT"
  • Case-sensitive: "File.txt" ≠ "FILE.TXT"

Most Macs use case-insensitive, but case-sensitive exists.

🔍 HFS+ Forensic Considerations
  • Catalog File: Provides complete file listing even for deleted files (if not overwritten)
  • Journal: May contain recently deleted file metadata
  • Resource forks: Check for hidden data in resource forks
  • Hard links: HFS+ supports hard links—check link counts
  • Timestamps: Stored in Mac absolute time (seconds since Jan 1, 1904)
  • Tools: Autopsy, BlackBag BlackLight, Cellebrite support HFS+

🍎 APFS (Apple File System)

APFS (Apple File System) replaced HFS+ starting in 2017 and is now the default file system for all Apple devices: macOS, iOS, iPadOS, watchOS, and tvOS. Understanding APFS is essential for modern Apple forensics.

APFS (Apple File System)

Apple's modern file system introduced in 2017, optimized for flash/SSD storage. Features include native encryption, space sharing between volumes, snapshots, clones, and nanosecond timestamps. Uses a copy-on-write design for data integrity.

APFS Key Features

📦
Container Architecture

APFS introduces "containers" that can hold multiple volumes sharing the same storage pool:

  • Container = physical partition
  • Volumes share free space dynamically
  • No need to pre-allocate volume sizes
📸
Snapshots

Point-in-time read-only copies of a volume:

  • Used by Time Machine
  • Created before macOS updates
  • May contain deleted file data!
  • Space-efficient (copy-on-write)
🔐
Native Encryption

Built-in encryption support:

  • FileVault uses APFS encryption
  • Per-file encryption keys possible
  • Hardware-accelerated on Apple Silicon
  • Forensic challenge: requires keys!
📋
Clones

Instant file/directory copies:

  • No data duplication initially
  • Copy-on-write when modified
  • Used by APFS for efficiency
  • May preserve deleted data longer
⏱️
Nanosecond Timestamps

High-precision timestamps:

  • 1 nanosecond resolution
  • Created, Modified, Changed, Accessed
  • Better than NTFS (100ns)
  • Excellent for timeline analysis
⚡
SSD Optimization

Designed for flash storage:

  • TRIM support built-in
  • Copy-on-write reduces writes
  • Space-efficient metadata
  • Challenge: TRIM may destroy evidence

APFS Forensic Challenges

⚠️ APFS Forensic Challenges
  • Encryption by default: FileVault encryption is standard on modern Macs; requires password or recovery key
  • T2/Apple Silicon security: Hardware encryption makes cold boot attacks ineffective
  • TRIM on SSDs: Deleted data may be immediately zeroed
  • Copy-on-write complexity: File history spread across multiple locations
  • Limited tool support: APFS is newer, fewer tools fully support it
  • Sealed System Volume: macOS Big Sur+ has read-only system volume
🔍 APFS Forensic Opportunities
  • Snapshots: May contain deleted files from before snapshot creation
  • Clones: Copy-on-write may preserve original data longer
  • Container metadata: Records of all volumes, even deleted ones
  • Timestamps: Nanosecond precision excellent for timeline analysis
  • Multiple volumes: User data often on separate volume from system
  • Tools: Autopsy (limited), BlackBag Mobilyze, Cellebrite UFED, mac_apt

macOS Volume Layout (Big Sur+)

Modern macOS uses a specific APFS volume layout:

Volume Purpose Forensic Notes
Macintosh HD System volume (read-only, sealed) OS files, applications; cryptographically sealed
Macintosh HD - Data User data volume ⭐⭐⭐ User files, documents, application data
Preboot Boot files Boot.efi, firmware data
Recovery macOS Recovery Recovery tools, reinstall capability
VM Virtual memory Swap files, sleep image (may contain RAM data)

📚 Key Takeaways
  • ext4 is the standard Linux file system, using inodes for metadata and extents for efficient storage
  • ext4 has four timestamps: mtime (modified), atime (accessed), ctime (changed), crtime (created)
  • ctime is NOT creation time—it's the inode change time; crtime is creation time
  • HFS+ was Apple's file system from 1998-2017; uses Catalog File B-tree for file metadata
  • APFS is Apple's modern file system with containers, snapshots, clones, and native encryption
  • APFS snapshots may contain deleted file data and are valuable for forensics
  • Encryption is a major challenge for APFS forensics—FileVault is common
  • Different tools support different file systems—choose appropriate tools for each platform
