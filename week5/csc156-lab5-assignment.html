✨ Lab Overview & Objectives

Welcome to Lab Activity 5! This lab is your opportunity to apply the theoretical knowledge gained in Week 5's modules on file systems, deleted file recovery, and file carving. You will work with four forensic disk images — three from the NIST CFReDS Deleted File Recovery test suite and one from the Naval Postgraduate School (NPS) Digital Corpora — to identify file systems, recover deleted data using metadata-based techniques, perform file carving, and thoroughly document your findings.

🎓
CSLO Alignment: This lab directly advances **CSLO 2: Conduct thorough examinations of computer hard disk drives, and other electronic data storage media to recover information and electronic data.**

**Course Objective Alignment:** This lab directly addresses the course objective: *"Analyze file systems and recover deleted files."*

💡
Required Tools: Ensure you have Autopsy, The Sleuth Kit (TSK) command-line tools (mmls, fsstat, fls, istat, icat), Scalpel (or PhotoRec), and MFTECmd installed and ready in your forensic workstation environment. A hex editor is also highly recommended.

This lab is divided into three parts, each contributing to your overall score.

Part 1: File System Identification (6 points)

In this section, you will load and examine all three NIST DFR-05 disk images — dfr-05-fat.dd, dfr-05-ntfs.dd, and dfr-05-ext.dd — to identify the partitions and file systems present in each. Each image is a single disk containing multiple partitions of the same file system family, giving you hands-on exposure to FAT variants, NTFS, and the ext family in a single lab. Proper identification is the first critical step in any forensic analysis.

Tasks:

  1. Add each of the three images as a separate data source in Autopsy, or examine them individually on the command line. Work through dfr-05-fat.dd, dfr-05-ntfs.dd, and dfr-05-ext.dd in turn.
    • Use mmls (TSK) to display the partition layout of each image. For example: mmls dfr-05-fat.dd
    • Use Autopsy's data source manager to add and view each image's partitions.
    • Note: The FAT image (dfr-05-fat.dd) contains three partitions (FAT12/16/32). The NTFS image contains one NTFS partition. The EXT image contains three partitions (ext2, ext3, ext4). You are not expected to document every partition — focus on the most forensically relevant one per image.
  2. Identify the file system type for the primary partition(s) in each image using fsstat (TSK) and Autopsy's file system information views. Example: fsstat -o [partition_offset] dfr-05-fat.dd
  3. For at least three distinct file systems (one FAT variant, NTFS, and one ext variant), document the following key characteristics:
    • File System Type (e.g., NTFS, FAT32, ext4)
    • Volume Label (if present)
    • Total Size
    • Block/Cluster Size
    • Total Number of Files and Directories
    • Journaling Status (e.g., enabled/disabled, type of journal for ext4)
    • Any other notable features relevant to forensics (e.g., NTFS version, ext4 journal type)
  4. Provide screenshots as evidence for each identified file system, clearly showing the mmls and fsstat tool output and the documented characteristics.
🔍
Tip: The partition offsets reported by mmls are in sectors. Pass the offset to subsequent TSK commands with the -o flag. For example, if mmls dfr-05-fat.dd shows a FAT32 partition starting at sector 2048, run: fsstat -o 2048 dfr-05-fat.dd. For images with only one partition, the offset is often 0 — confirm with mmls first.

Part 2: Deleted File Recovery (8 points)

This section focuses on recovering logically deleted files using metadata-based techniques across three different file systems. You will use dfr-05-fat.dd, dfr-05-ntfs.dd, and dfr-05-ext.dd — each image was specifically constructed by NIST with known deleted files and documented MAC times, making it possible to verify your recovery results. You will use both command-line TSK tools and Autopsy's GUI.

📋
About the DFR-05 Images: The DFR-05 test case was designed to practice recovering several fragmented files. Each image contains files that were deliberately created, fragmented across non-contiguous clusters/blocks, and then deleted. This means you will encounter cases where recovery is partial or where the file allocation chain must be carefully reconstructed — exactly the challenge discussed in Module 5.5.

Tasks:

1. FAT32 Volume — dfr-05-fat.dd:

  1. Run mmls dfr-05-fat.dd to identify the partition layout. Note the starting sector offset of the FAT32 partition (the largest partition — the FAT12 and FAT16 partitions are smaller and at lower offsets).
  2. List deleted files in the FAT32 partition using:
    fls -d -r -o [FAT32_partition_offset] dfr-05-fat.dd
  3. Select at least two deleted files from the fls output and attempt to recover each using:
    icat -o [FAT32_partition_offset] dfr-05-fat.dd [inode_number] > recovered_filename.ext
  4. Document the recovery success or failure for each file and explain why. Note: FAT deletes the first character of the filename (replaced with 0xE5) — document how you handled the missing character when naming recovered files.
  5. Because DFR-05 contains fragmented files, attempt recovery of at least one fragmented file. Document whether the recovered content is complete or truncated, and explain what fragmentation does to FAT recovery.
  6. Provide screenshots of the fls output showing deleted entries and the terminal output confirming at least one successful recovery.

2. NTFS Volume — dfr-05-ntfs.dd:

  1. Add dfr-05-ntfs.dd as a new data source in Autopsy. When configuring ingest modules, ensure Deleted File Finder and File System Analysis are enabled.
  2. Navigate to the Deleted Files section in Autopsy's tree panel and select at least two deleted files for recovery. Right-click and export these files to your working folder.
  3. For one of the recovered NTFS files, use Autopsy's file properties view to compare the $STANDARD_INFORMATION and $FILE_NAME timestamps. Discuss any discrepancies and their forensic implications (e.g., $SI timestamps are user-modifiable and commonly targeted in timestomping, while $FN timestamps are harder to alter). Provide screenshots of the timestamp comparison.
  4. Attempt to identify any Alternate Data Streams (ADS) on a file within the NTFS volume using Autopsy's file attributes view or the TSK command:
    fls -o [NTFS_partition_offset] dfr-05-ntfs.dd
    Document your findings — note that the DFR-05 NTFS image may not contain ADS, which is itself a valid forensic observation to record.
  5. Use MFTECmd to extract and analyze $UsnJrnl entries related to deleted file activity. Run MFTECmd against the image and filter for relevant entries. Document how the journal entries provide historical context about file operations beyond what the MFT alone can show.
  6. Provide screenshots of: Autopsy's deleted file view, the timestamp comparison, ADS findings, and MFTECmd output.

3. ext4 Volume — dfr-05-ext.dd:

  1. Run mmls dfr-05-ext.dd to identify the three ext partitions (ext2, ext3, ext4). Focus your work on the ext4 partition (typically the third and largest partition). Note its starting sector offset.
  2. List deleted files in the ext4 partition using:
    fls -d -r -o [ext4_partition_offset] dfr-05-ext.dd
  3. Examine inode details for a deleted file using:
    istat -o [ext4_partition_offset] dfr-05-ext.dd [inode_number]
    Note whether the block pointers have been cleared — this is the primary challenge for ext4 metadata recovery.
  4. Attempt to recover at least one deleted file using extundelete or Autopsy's recovery export. Because ext4 clears block pointers on deletion, success will depend on whether the journal still holds the original inode data.
  5. Document the specific challenges observed (e.g., cleared block pointers, loss of filename in directory entry, journal state). Record whether your recovery was successful or partial, and explain why.
  6. Provide screenshots of your fls output, istat output, and recovery attempt results.
⚠️
Important: Tools like extundelete may write changes when run against a disk image directly. Always work on a copy of the image file if using tools that modify the target. Keep your original downloaded images intact as your reference copies.

Part 3: File Carving Exercise (6 points)

In this section, you will perform file carving on nps-2009-canon2-gen6.E01 — a forensic image of a Canon PowerShot SD800IS camera memory card captured at the sixth stage of use. This image was created by the Naval Postgraduate School specifically for teaching file carving: it contains a mix of intact JPEG images still visible in the FAT32 file system, deleted JPEGs that are recoverable via metadata, and JPEGs that can only be recovered by carving because their file system metadata is gone — including at least two fragmented JPEGs whose fragments are interleaved on disk.

📷
About nps-2009-canon2-gen6.E01: This is an EnCase E01 format image. Autopsy opens E01 files natively — add it as a new data source just as you would a .dd file. The image is a FAT32 file system (Canon camera cards use FAT32 by convention). It contains JPEG image files only — do not expect PDFs, Office documents, or ZIP files to be present. The carving exercise for this lab focuses on JPEGs. The Scalpel configuration tip below has been updated accordingly.

Tasks:

  1. Add nps-2009-canon2-gen6.E01 as a new data source in Autopsy. This is a separate case from your Part 1/2 work — create a new Autopsy case or add it as an additional data source in your existing case.
  2. Before running any carving tools, first browse the active (non-deleted) files in Autopsy's file tree to note how many JPEG images are currently visible. This is your baseline — carving will attempt to recover images beyond what the file system currently shows.
  3. Configure Scalpel (or use Autopsy's built-in File Carver ingest module) to carve for JPEG images.
    • If using Scalpel: edit your scalpel.conf file to enable the JPEG rule (uncomment or add the line below). Run Scalpel against the image file directly:
      scalpel nps-2009-canon2-gen6.E01 -o ./carved_output/
    • If using Autopsy's File Carver: right-click the data source → Run Ingest Modules → enable File Carver and configure it for JPEG file types.
  4. Review the carved output. For at least 5 carved JPEG files, examine each and document:
    • Whether the file opens correctly (valid) or is corrupted (false positive or partial recovery).
    • Whether the recovery appears complete or partial — partial JPEGs will open but show truncation, color banding, or corrupted lower sections due to fragmentation.
    • Any observable fragmentation effects — for example, extra image data appended from a different file, or color block artifacts mid-image indicating where the carver lost track of the file's data.
  5. Document your carving methodology: which tool you used, your configuration settings, and the total number of files carved vs. files that opened successfully.
  6. Compare your carving results to the metadata-based recovery you performed in Part 2. Specifically: which method recovered more files? Which produced higher-quality (more complete) results? Under what circumstances would each method be the right choice for a forensic examiner?
  7. Provide screenshots of: your Scalpel configuration (or Autopsy File Carver settings), the carved output directory or Autopsy carved files view, and examples of both a valid carved JPEG and a partially corrupted one (if present).
🛠️
Scalpel Configuration for JPEGs: 
# scalpel.conf — enable this line for JPEG carving
# extension  case_sensitive  max_size  header            footer
jpg          y               10000000  \xff\xd8\xff      \xff\xd9
JPEG files begin with the magic bytes FF D8 FF and end with FF D9. The max size of 10MB is appropriate for camera JPEGs from this era. Note that JPEGs do not have a fixed-length footer — Scalpel will carve from each header to the next FF D9 marker it finds, which means fragmented JPEGs may include data from neighboring files between fragments.
