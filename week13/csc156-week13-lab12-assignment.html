🔎 Case Background — Continuing the svchost_update.exe Investigation

Over the past two weeks your forensic investigation of the Acme Corp incident has expanded significantly:

  • Week 11 (Memory Forensics): Volatility's malfind plugin flagged injected code in a process named svchost_update.exe (PID 4892). The netscan plugin revealed an ESTABLISHED connection from this process to 198.51.100.42:443. The pstree output showed that svchost_update.exe was spawned by explorer.exe — an abnormal parent for any svchost-named process.
  • Week 12 (Network Forensics): Wireshark analysis of the Acme Corp PCAP confirmed beaconing traffic every 90 seconds to 198.51.100.42:443. DNS query logs showed the workstation resolved evil-update.example.com (which resolves to 198.51.100.42) approximately 30 seconds before the first beacon. The web server access logs recorded a suspicious POST to /upload/exfil.php with an unusually large byte count from the compromised workstation's IP.

The incident response team has now extracted the suspect binary from the forensic image. The file, named svchost_update.exe, has been placed in a password-protected ZIP archive (malware_lab12_sample.zip, password: infected) and staged in the lab's shared evidence folder.

Your assignment: Perform a complete malware analysis — static and dynamic — on the provided sample. Classify the malware, extract IOCs, identify any anti-analysis techniques, and correlate your findings with evidence from prior weeks. Document everything in the structured report fields below.

💡 Hint — Forming a Classification Hypothesis

Think about the behavioral clues: a process with C2 beaconing suggests remote control capability (RAT). A process spawned by explorer.exe rather than services.exe suggests it was launched by a user action or trojan dropper, not a legitimate Windows service. Injected code (malfind) suggests the malware modifies other processes — a technique common in RATs and rootkits.

Phase 2: Static Analysis Obj 2

Examine the sample without executing it. Static analysis is safe, repeatable, and always performed first.

Task 2.1 — File Identification and Hashing

  1. Generate cryptographic hashes of the sample. In PowerShell: 
    Get-FileHash .\svchost_update.exe -Algorithm MD5 | Format-List
Get-FileHash .\svchost_update.exe -Algorithm SHA256 | Format-List
  2. Verify the file type using magic bytes. Check the first two bytes in a hex viewer (or use file on Linux). Expected for a PE: 4D 5A ("MZ").
  3. Record the file size in bytes.
  4. If authorized by your instructor, perform a hash-based reputation lookup (VirusTotal or local NSRL database). If not authorized, write "Not performed — OPSEC restriction."

Task 2.2 — String Extraction

  1. Run the strings utility on the sample: 
    strings svchost_update.exe > strings_output.txt
strings -el svchost_update.exe >> strings_output.txt

    (The -el flag extracts Unicode/little-endian strings.)

  2. Review the output and identify strings in the following categories:
    • IP addresses or domain names
    • URLs or URI paths
    • File paths or filenames
    • Registry key paths
    • Suspicious keywords (e.g., cmd.exe, powershell, CreateRemoteThread, VirtualAlloc)
    • Error messages or debug strings
💡 Hint — What Makes a String "Significant"?

Not every string is meaningful. Focus on strings that suggest capability: network addresses indicate C2, registry paths indicate persistence, API names indicate techniques (e.g., WriteProcessMemory → process injection). Ignore generic compiler strings, standard library names, and common Windows API names that appear in benign software.

Task 2.3 — PE Header Analysis

  1. Open the sample in pestudio (or an equivalent PE analysis tool such as PE-bear or CFF Explorer).
  2. Record the following from the PE headers:
    • Compile timestamp
    • Subsystem (GUI vs. Console)
    • Number of sections and their names
    • Entry point address
    • Any sections with entropy > 7.0 (indicating possible packing/encryption)
  3. Examine the Import Table. List imported DLLs and flag any suspicious API imports using the categories from Module 13.2 (file operations, registry, process manipulation, network, crypto, anti-debug).
  4. Check the Export Table (if present) for any exported functions.
💡 Hint — Interpreting Section Entropy

Entropy is measured on a 0–8 scale. Normal compiled code (.text) typically has entropy between 5.0 and 6.5. Entropy above 7.0 strongly suggests the section is packed, compressed, or encrypted. If you see section names like UPX0 or UPX1 combined with high entropy and a very small import table, the binary is almost certainly packed with UPX.

Phase 3: Dynamic Analysis and Behavioral Monitoring Obj 3

⚠️ Checkpoint — Before You Execute
  • Confirm your VM snapshot Lab12_Pre-Analysis exists and is current.
  • Confirm network isolation or simulation is active.
  • Start all monitoring tools before executing the sample.

Task 3.1 — Prepare Monitoring Tools

  1. Launch Process Monitor (ProcMon) and set a filter for the process name svchost_update.exe. Begin capturing.
  2. Launch Wireshark (or your network simulation tool's capture) on the VM's network interface. Begin capturing.
  3. Open Task Manager or Process Explorer for real-time process observation.
  4. (Optional) Start Regshot and take a "1st shot" of the registry to compare after execution.

Task 3.2 — Execute and Observe (Detonation)

  1. Double-click (or run from command prompt) svchost_update.exe.
  2. Observe for at least 3–5 minutes. Many malware samples have delayed behaviors or timed beaconing intervals.
  3. While observing, document any visible changes: new windows, error messages, CPU/memory spikes, new processes appearing.

💡 Hint — What If Nothing Seems to Happen?

Some malware performs environment checks before activating. If you see no activity after 5 minutes: (1) Check ProcMon — the process may have started and immediately exited (anti-VM detection). (2) Check if the sample queried any registry keys related to VMware/VirtualBox. (3) The sample may require internet connectivity — confirm your network simulation is responding to DNS queries. Document whatever you observe, including "no visible activity," as this itself is a finding.

💡 Hint — Prioritizing IOCs

The best IOCs for enterprise search are those that are unique to the threat and unlikely to produce false positives. A SHA-256 hash is highly specific but only matches the exact binary. A C2 domain or IP may detect variants that share infrastructure. A registry Run key value name may catch persistence across different binary versions. Consider a mix of network and host IOCs for comprehensive coverage.
