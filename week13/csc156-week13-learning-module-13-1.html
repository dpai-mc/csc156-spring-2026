The incident response team has isolated a Windows 10 workstation after Volatility's malfind plugin flagged injected code in a suspicious process. The network team confirms the workstation was beaconing to an external IP every 60 seconds — the same pattern you identified in last week's PCAP analysis.

A file named svchost_update.exe was found in C:\Users\jsmith\AppData\Local\Temp\.

Your mission: Determine what this file does, how it got there, and how to detect it across the enterprise.

🔗 Connecting the Threads

This scenario picks up where Weeks 11 and 12 left off. In Week 11 you used Volatility (malfind, netscan, pstree) to detect injected code and suspicious network connections in memory. In Week 12 you used Wireshark to identify C2 beaconing and data exfiltration in packet captures. This week you take the next logical step — analyzing the malicious tool itself.

This scenario will thread through all four Week 13 modules. By Module 13.4 you will have classified the malware, examined it statically and dynamically, extracted IOCs, and connected your findings back to memory and network evidence.

1. Malware Analysis in a Forensic Context

A Shift in Analytical Posture

Weeks 7–12 trained you to examine artifacts left behind by user activity, system events, and network traffic. Week 13 asks a fundamentally different question: instead of reconstructing what happened, you now determine what the tool does and how it can be identified across an enterprise.

The Dual Purpose of Malware Analysis

In a forensic investigation, malware analysis serves two parallel goals:

  1. Investigation support — answering key questions about the incident:
    • Capability: What can the malware do? (Steal data? Encrypt files? Open a backdoor?)
    • Attribution: What does the malware reveal about the attacker's tools and techniques?
    • Scope: What systems are affected and what data was compromised?
    • Initial access: How did the malware get on the system?
  2. Enterprise defense — producing actionable outputs:
    • IOCs (Indicators of Compromise): Hashes, IPs, domains, registry keys that can be searched across every endpoint.
    • Remediation guidance: What must be cleaned, rebuilt, or patched?
    • Detection rules: YARA signatures, IDS rules, or EDR queries to catch the same threat in the future.
🔬 Scope for This Course

Behavioral analysis (static + dynamic observation) is our focus. Full reverse engineering — assembly-level disassembly, decompilation, and code-level analysis — is beyond scope but represents an advanced career path in malware analysis. The techniques you learn this week are the essential first steps that even reverse engineers perform before opening a disassembler.

Classifying a malware sample is the first analytical step. Knowing what type of malware you are dealing with shapes every subsequent decision — which analysis tools to use, what behaviors to watch for, and which forensic artifacts to correlate.

Fileless malware is the hardest type to analyze because there may be no file to examine statically. Memory forensics (Week 11) becomes the primary evidence source. This is why memory acquisition early in an incident is critical.

Understanding how malware reaches a system helps you trace the initial access vector — a critical question in every investigation. Each mechanism connects to forensic techniques you have already practiced.

4. The Four-Phase Analysis Workflow

Malware analysis follows a structured, sequential workflow. Each phase builds on the previous one, and the order matters — always perform static analysis before dynamic analysis.

① Phase 1 — Triage

Goal: Quickly determine what you have and whether it is worth deeper analysis.

  • Generate file hashes (MD5, SHA-256) for identification.
  • Check hashes against reputation databases (VirusTotal, NSRL, threat intel feeds).
  • Identify file type using magic bytes (file command) — is it really what the extension claims?
  • Determine priority: Is this a known threat (hash match) or a novel sample requiring full analysis?

Module coverage: Module 13.2 (Static Analysis) covers triage techniques in depth.

② Phase 2 — Static Analysis

Goal: Learn everything possible about the file without executing it.

  • Extract readable strings (IP addresses, domains, registry paths, error messages).
  • Analyze PE headers: compile timestamp, section names, entropy levels.
  • Review import table: which Windows API functions does it call?
  • Check for packing indicators (high entropy, few imports, unusual section names).

Key advantage: Completely safe — the malware never runs.

Key limitation: Cannot observe runtime behavior; packed/encrypted samples reveal little.

Module coverage: Module 13.2 is entirely dedicated to static analysis.

③ Phase 3 — Dynamic Analysis

Goal: Execute the malware in a controlled, isolated environment and observe its behavior.

  • Monitor file system changes (files created, modified, deleted).
  • Monitor registry modifications (persistence keys, service creation).
  • Monitor network activity (DNS queries, C2 connections, data exfiltration).
  • Monitor process activity (child processes spawned, privilege escalation attempts).

Key advantage: Reveals actual behavior, defeats packing and obfuscation.

Key limitation: Malware may detect the analysis environment and alter behavior.

Module coverage: Module 13.3 covers dynamic analysis and sandbox configuration.

④ Phase 4 — IOC Extraction

Goal: Distill analysis findings into actionable Indicators of Compromise that enable enterprise-wide detection.

  • Document network IOCs: C2 IP addresses, domains, URLs, User-Agent strings.
  • Document host-based IOCs: file hashes, paths, registry keys, mutex names, service names.
  • Structure IOCs in a shareable format (IOC table, STIX/TAXII concepts).
  • Connect IOCs back to evidence from prior forensic domains (memory, network, OS artifacts).

Module coverage: Module 13.3 covers IOC extraction. Module 13.4 covers cross-domain correlation.

⚠️ Why "Static Before Dynamic"?

Static analysis is safe and non-destructive. It gives you hypotheses to test during dynamic analysis. If you skip straight to execution, you may trigger destructive behavior (ransomware encryption, data wiping) before you understand what you are dealing with. Additionally, static analysis findings — such as imported API calls and extracted strings — tell you what to watch for during dynamic analysis, making your observations far more targeted.

5. Safe Analysis Environment

⚠️ CRITICAL: Malware Analysis Safety Rules

Malware is live, dangerous software. Violating these rules can infect your host system, your home network, or your institution's network. Read and follow every rule before handling any sample.

  • NEVER analyze malware on a production system or any system connected to a production network.
  • NEVER use real credentials (personal or institutional) inside an analysis VM.
  • ALWAYS take a VM snapshot before executing any sample — revert after every analysis session.
  • ALWAYS use host-only networking or a network simulation tool (INetSim, FakeNet-NG) — never give the malware real internet access.
  • ALWAYS disable shared folders, clipboard sharing, and drag-and-drop between host and guest VM.
  • NEVER transfer malware samples to your host via shared folders — use a dedicated, isolated transfer method specified by your instructor.
  • If you suspect the malware has escaped the VM or your host is behaving unexpectedly, disconnect from the network immediately and contact your instructor.

Network Simulation Tools

Malware often requires network connectivity to reveal its full behavior (DNS resolution, C2 communication, payload download). Instead of providing real internet access, we use network simulation tools that fake common internet services inside the isolated environment.

INetSim (Linux-based)
  • What it does: Simulates DNS, HTTP, HTTPS, SMTP, FTP, and other common internet services on the analysis network.
  • How it works: Runs on a second VM (or the host in host-only mode) and responds to all DNS queries with its own IP, then serves fake responses for HTTP requests, accepts SMTP emails, etc.
  • Forensic value: Captures every DNS query and HTTP request the malware makes — revealing C2 domains and URLs without any real data leaving your lab.
  • Typical setup: Linux VM on the same host-only network as the analysis Windows VM; configure the Windows VM's DNS to point to the INetSim VM's IP.
FakeNet-NG (Windows-based)
  • What it does: Runs directly on the Windows analysis VM and intercepts all network traffic, responding to DNS, HTTP, and other protocols locally.
  • Advantage over INetSim: Single-VM setup — no need for a second VM. Easier for students new to malware analysis.
  • Forensic value: Same as INetSim — logs all network requests the malware attempts.
  • Limitation: Running on the same VM as the malware means the tool could theoretically be tampered with by sophisticated malware.
🔬 Why Simulate the Network?

Many malware samples check for internet connectivity before activating. Without a simulated network, the malware may sit dormant, and you will miss critical behaviors. Network simulation gives you the best of both worlds: the malware behaves normally, but all traffic stays inside your isolated lab and is fully captured for analysis.

👤 What Your Instructor Would Do

The safety rules above reflect responsible practice for a classroom or entry-level lab environment. In a real professional setting, I go significantly further — and I want you to understand why.

  • Air-gap the host machine entirely, not just the VM. I would never perform malware analysis on a machine connected to any network — not the internet, not a LAN, not even Wi-Fi that is merely "off." VM isolation is one layer of defense, but it is not a substitute for physically disconnecting the host machine from every network. The host itself must be off-network before any sample is handled.
  • Maintain a clean BIOS backup and a full machine image. Sophisticated malware can persist well below the operating system — in UEFI/BIOS firmware, in storage controller firmware, or in other places most analysts have never thought to look. I keep a verified clean copy of the BIOS firmware and a full, verified backup image of my analysis machine so I can restore it to a known-good state at any time.
  • Wipe the entire machine after every investigation. After completing any malware analysis, I would automatically wipe the entire physical machine — not just revert the VM snapshot — and restore from the verified clean backup image. This is extreme, but it is the only practice that gives me reasonable confidence that my analysis machine is uninfected before the next investigation begins.
  • AI-designed malware is raising the stakes considerably. We are seeing an increasing trend of AI-assisted and AI-designed malware. This new generation of malware is dramatically harder to analyze and defeat. It is entirely conceivable that future malware will be able to hide itself from detection and analysis in places we have never seen malware hide before — evading tools at the firmware level or in storage subsystems no current forensic software monitors. The defensive measures we take today must honestly account for threats that may exceed our current analytical capabilities.

This level of caution is beyond what is practical or expected in a student lab environment — but understanding why professionals operate this way is itself an important part of your forensic education. The gap between "safe enough for class" and "safe enough for a real investigation" is large, and closing it requires dedicated hardware, rigorous procedures, and a deeply cautious mindset.

Question 1: A forensic examiner discovers that a workstation is sending a small HTTPS request to the same external IP address every 60 seconds. Volatility's malfind plugin has flagged injected code in the process responsible. Which malware type is most consistent with these indicators?
✓ Correct! The combination of periodic beaconing (regular-interval outbound connections) and injected code detected by malfind is the signature pattern of a Remote Access Trojan (RAT). RATs maintain persistent C2 channels so the attacker can issue commands at will. Ransomware contacts payment infrastructure but does not typically beacon at regular intervals. Worms scan many hosts, not a single IP. Viruses modify files, not inject code into running processes.
✗ Not quite. Look at the two key indicators together: periodic beaconing to a single external IP and injected code in a process. This combination points to a Remote Access Trojan (RAT). RATs maintain a regular C2 heartbeat so the attacker knows the implant is alive. Ransomware, worms, and viruses each have different characteristic patterns — review the taxonomy cards above.

Skipping it means you are running blind — and if the sample is ransomware, you may trigger file encryption before you even know what you are dealing with. While VMs provide good isolation, "always safe" is never a valid assumption in malware analysis.
✗ Not quite. The core issue is skipping Phase 2 (Static Analysis) in the workflow. The rule is "static before dynamic" — examine the file without executing it first. This gives you hypotheses, reveals potential destructive capabilities, and ensures you know what to monitor. Option A has OPSEC concerns (uploading targeted malware to public services can alert the attacker). Option C is not standard practice. Option D is dangerously incorrect — VMs are not escape-proof. Question 3: Why do malware analysts use network simulation tools like INetSim or FakeNet-NG instead of giving the analysis VM real internet access?
✓ Correct! Network simulation serves two critical purposes: (1) Safety — the malware cannot reach real C2 infrastructure, download additional payloads, exfiltrate data, or alert the attacker that their sample is being analyzed. (2) Forensic capture — every DNS query, HTTP request, and connection attempt is logged by the simulation tool, giving you a complete record of the malware's intended network behavior. The malware "thinks" it has internet access and reveals its communication patterns, but nothing leaves your isolated lab.
✗ Not quite. Network simulation is about safety and evidence capture, not performance. Giving malware real internet access is extremely dangerous — it could download additional payloads, exfiltrate sensitive data, or signal the attacker. Simulation tools (INetSim, FakeNet-NG) respond to the malware's network requests with fake services, tricking it into revealing its C2 domains and URLs while keeping everything contained.
