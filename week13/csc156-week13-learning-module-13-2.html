Hashing, String Extraction, PE Header Analysis, and Import Table Review

1  Scenario Continuation — From Triage to Analysis

🔎 Continuing Investigation — svchost_update.exe

In Module 13.1 you classified the suspicious file svchost_update.exe, established your isolated analysis VM, and took a clean snapshot. The triage phase is complete. Now it is time to learn everything the file can tell you without ever running it.

Your goals for this module:

  1. Generate cryptographic hashes and check the file's reputation.
  2. Confirm the true file type using magic bytes.
  3. Extract readable strings for early intelligence.
  4. Examine the PE header structure, compile timestamp, and section entropy.
  5. Review the import table for suspicious Windows API calls.

Every finding you make here will become a hypothesis you test during dynamic analysis in Module 13.3.

What Is Static Analysis?

Static analysis is the examination of a file without executing it. You inspect the file's metadata, structure, and embedded content to form an initial understanding of its purpose and capabilities.

AspectAdvantagesLimitations
SafetyNo risk of infection — the file never runsCannot observe actual runtime behavior
RepeatabilityResults are deterministic; any analyst gets the same outputPacked or encrypted payloads hide their contents
SpeedQuick first-pass triage in minutesObfuscated strings and code require advanced tools

🔬 Forensic Principle

Always perform static analysis before dynamic analysis. Static findings generate hypotheses (e.g., "the import table suggests network communication"). Dynamic analysis then confirms or refutes those hypotheses. This ordered workflow is both safer and more methodical.

2  File Identification and Triage

🔗 Building on Week 6

You already know how to generate MD5 and SHA-256 hashes for evidence integrity verification. In malware forensics, hashing serves a second purpose — identification. The same hash algorithms you used to verify forensic images now help you identify known malware samples and look up their reputation in threat intelligence databases.

2.1  Hash Generation

Generating cryptographic hashes is the first thing you do with any suspicious file. The hash uniquely identifies the sample and allows you to search global databases for prior analysis.

Commands

# Linux / macOS
md5sum svchost_update.exe
sha256sum svchost_update.exe

# Windows PowerShell
Get-FileHash .\svchost_update.exe -Algorithm MD5
Get-FileHash .\svchost_update.exe -Algorithm SHA256

Record all three common hashes (MD5, SHA-1, SHA-256) in your analysis notes. Different databases index by different algorithms.

2.2  Hash-Based Reputation Lookup

ResourceTypeUse Case
VirusTotalOnline multi-scannerSubmit a hash (not the file) to check if any AV engine has flagged it
NSRL (National Software Reference Library)Known-good hash setIf the hash appears in NSRL, the file is a known legitimate application — not malware
Threat intelligence feedsIOC databasesOrganizational or commercial feeds that catalog known-bad hashes

⚠️ OPSEC Warning — VirusTotal

Uploading a file to VirusTotal makes it available to every subscriber, including the attacker. In sensitive investigations, submit only the hash, never the file itself. If the hash returns no results, you still learn something: the sample may be novel or targeted. Discuss with your team before uploading any sample to a public service.

2.3  File Type Identification — Magic Bytes

🔗 Building on Week 5

In Week 5 you learned that every file format begins with a characteristic byte sequence called a file signature or magic bytes. The file command on Linux reads these bytes to determine the true file type — regardless of the file extension.

file svchost_update.exe

Expected output for a Windows executable:

svchost_update.exe: PE32 executable (GUI) Intel 80386, for MS Windows

Common Magic Bytes Reference

Magic Bytes (Hex)ASCIIFile Type
4D 5AMZWindows PE executable (.exe, .dll)
7F 45 4C 46.ELFLinux ELF executable
50 4B 03 04PKZIP archive (also .docx, .apk)
25 50 44 46%PDFPDF document
D0 CF 11 E0—OLE2 Compound (legacy .doc, .xls)
FF D8 FF—JPEG image
89 50 4E 47.PNGPNG image

🔬 Mismatch Detection

A file named document.pdf whose magic bytes are 4D 5A (MZ) is not a PDF — it is a Windows executable masquerading as a document. Extension-to-magic-byte mismatches are a strong indicator of social engineering or deliberate deception. Always verify the true file type; never trust the extension alone.

2.4  Fuzzy Hashing with ssdeep

Cryptographic hashes are exact: change one byte and the hash changes completely. Fuzzy hashing (also called context-triggered piecewise hashing) produces a similarity score, allowing you to identify files that are nearly identical.

# Generate a fuzzy hash
ssdeep svchost_update.exe

# Compare two samples for similarity
ssdeep -d sample_a.exe sample_b.exe

Use case: An attacker recompiles malware with a minor change to evade signature detection. The SHA-256 hash is completely different, but ssdeep reports a high similarity score (e.g., 87%), linking the variant to the known family.

💡 Concept Check

Think of cryptographic hashing as asking "Is this the exact same file?" and fuzzy hashing as asking "Does this file look like a known file?" Both questions are valuable in malware forensics.

3  String Extraction

Strings are sequences of printable characters embedded in a binary file. Even compiled executables contain human-readable text — error messages, file paths, URLs, registry keys, and more. Extracting these strings is one of the fastest ways to gain intelligence about a sample's purpose.

3.1  The strings Command

# ASCII strings (default, minimum length 4)
strings svchost_update.exe > strings_ascii.txt

# Unicode (UTF-16LE) strings — common in Windows malware
strings -el svchost_update.exe > strings_unicode.txt

On Windows, the Sysinternals strings utility provides similar functionality:

strings.exe -n 6 svchost_update.exe

The -n 6 flag sets the minimum string length to 6 characters, reducing noise from random byte sequences that happen to be printable.

💡 Practical Tip

Always extract both ASCII and Unicode strings. Windows APIs heavily use Unicode (UTF-16LE), so many malware artifacts — registry paths, file paths, URLs — appear only in the Unicode output.

3.3  FLOSS — Defeating Obfuscated Strings

Concept Malware authors know that analysts use strings. To evade detection, they obfuscate strings using XOR encoding, Base64, stack-string construction, or custom encryption. The basic strings command cannot recover these.

FLOSS (FLARE Obfuscated String Solver), developed by Mandiant, automatically detects and decodes obfuscated strings by emulating the malware's decoding routines without full execution.

floss svchost_update.exe

FLOSS outputs three categories: static strings (same as strings), stack strings (built character-by-character at runtime), and decoded strings (XOR/Base64 deobfuscated). If basic strings output seems unusually sparse for the file's size, obfuscation is likely, and FLOSS is the next tool to try.

🔬 Forensic Insight

A binary with very few readable strings relative to its size is itself suspicious. Legitimate software typically contains abundant strings (error messages, UI text, help strings). A 500 KB executable with only a handful of strings likely uses packing or obfuscation — a finding you will explore further in Module 13.4 (Anti-Analysis Techniques).

4  PE Header Analysis

The Portable Executable (PE) format is the standard executable format for Windows. Every .exe and .dll file follows this structure. The PE header contains rich metadata that reveals when the file was compiled, what OS features it requires, and what Windows API functions it calls — all without executing a single instruction.

4.1  PE Structure Overview

┌──────────────────────────────────────────────────────────────┐
│  DOS Header                                                  │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ e_magic: 0x5A4D ("MZ")    ← Magic bytes (Week 5)      │  │
│  │ e_lfanew: offset to PE signature                       │  │
│  └────────────────────────────────────────────────────────┘  │
├──────────────────────────────────────────────────────────────┤
│  DOS Stub  ("This program cannot be run in DOS mode.")       │
├──────────────────────────────────────────────────────────────┤
│  PE Signature: 0x50450000 ("PE\0\0")                         │
├──────────────────────────────────────────────────────────────┤
│  COFF Header                                                 │
│    • Machine type (x86 / x64)                                │
│    • Number of sections                                      │
│    • ★ TimeDateStamp  ← Compile time (forensic gold)        │
├──────────────────────────────────────────────────────────────┤
│  Optional Header                                             │
│    • AddressOfEntryPoint                                     │
│    • ★ Subsystem (GUI / Console)                            │
│    • Data Directories (Import Table, Export Table, etc.)      │
├──────────────────────────────────────────────────────────────┤
│  Section Table                                               │
│    .text  │ .data  │ .rsrc  │ .reloc │ (or custom names)    │
├──────────┬────────┬────────┬────────┬───────────────────────┤
│ .text    │ .data  │ .rsrc  │ .reloc │  ...                  │
│ (code)   │ (data) │ (res.) │ (reloc)│                       │
└──────────┴────────┴────────┴────────┴───────────────────────┘
  ★ = Key forensic fields

Figure 1 — Simplified PE file structure with key forensic fields highlighted

4.2  Key Forensic Fields

The TimeDateStamp field in the COFF header records when the binary was compiled, stored as a Unix epoch value (seconds since January 1, 1970).

Forensic value:

  • A compile time that predates the investigation timeline may indicate a pre-staged tool.
  • A compile time far in the future (e.g., year 2038) is a strong indicator of timestamp manipulation.
  • Compile times in specific time zones can hint at the developer's geographic region.

Caveat: This field is trivially faked. Treat it as one data point, never as sole evidence.

The Subsystem field indicates whether the executable is a GUI application or a console application.

  • IMAGE_SUBSYSTEM_WINDOWS_GUI (2) — runs without a visible console window. Most malware uses this to remain invisible to the user.
  • IMAGE_SUBSYSTEM_WINDOWS_CUI (3) — opens a console window. Less common for malware unless it is a command-line tool or dropper.

A file named svchost_update.exe that is a GUI subsystem application is consistent with malware that wants to run silently.

PE executables are divided into sections, each with a name and characteristics:

SectionTypical Purpose
.textExecutable code
.dataInitialized data (global variables)
.rsrcResources (icons, dialogs, embedded files)
.relocRelocation information

Suspicious indicators:

  • Unusual section names — UPX0, UPX1, .ndata, or random-looking names indicate packing or custom tools.
  • High entropy — Entropy measures randomness on a 0–8 scale. Normal compiled code has entropy around 5.0–6.5. Entropy above 7.0 strongly suggests the section is encrypted or compressed (packed). A .text section at 7.8 entropy is a red flag.
  • Executable + writable sections — A section marked both writable and executable is unusual in legitimate software and common in self-modifying or unpacking code.

4.3  Import Table Analysis

The import table lists the Windows API functions the executable calls. This is one of the most powerful static analysis data points because it reveals the program's capabilities — what it can do to the system, the network, and other processes.

Suspicious API Categories Reference

CategorySuspicious APIsForensic Implication
File operationsCreateFile, WriteFile, DeleteFile, CopyFileFile manipulation, payload dropping, self-deletion
RegistryRegSetValueEx, RegCreateKeyExPersistence via Registry (Week 7 Run keys)
Process manipulationCreateProcess, VirtualAlloc, VirtualAllocEx, WriteProcessMemory, CreateRemoteThreadProcess injection — connects to Week 11 malfind detections
NetworkWSAStartup, connect, send, recv, InternetOpenUrlNetwork communication, C2 callback, data exfiltration
CryptographyCryptEncrypt, CryptDecrypt, CryptGenKeyRansomware encryption or encrypted C2 channel
Anti-debugIsDebuggerPresent, CheckRemoteDebuggerPresentAnti-analysis awareness (Module 13.4)
Service managementCreateService, StartServicePersistence via Windows service installation (Week 8 Event ID 7045)

🔬 Forensic Insight — Connecting Imports to Memory Forensics

If you see VirtualAllocEx and WriteProcessMemory in the import table, the malware likely performs process injection — allocating memory inside another process and writing code into it. This is exactly the behavior that Volatility's malfind plugin detects in memory dumps (Week 11). Static analysis of imports and memory forensics with malfind are two sides of the same coin.

⚠️ Small Import Table = Red Flag

If a PE file imports only LoadLibrary and GetProcAddress (and very little else), it is almost certainly packed. These two APIs allow the unpacking stub to dynamically load all other required functions at runtime, hiding the true import table from static analysis. You will explore this further in Module 13.4.

4.4  Tool Walk-Through: pestudio

Tool pestudio is a free Windows application purpose-built for initial static assessment of PE files. It automatically highlights suspicious indicators, saving analysts significant manual effort.

pestudio Key Views

pestudio assigns a risk score to the file and lists specific indicators that contributed to the score. Items flagged in red deserve immediate attention. For svchost_update.exe, you might see flags for: suspicious imports, high section entropy, and a compile timestamp inconsistency.

pestudio extracts strings and cross-references them against known blacklists (URLs, IP patterns, registry paths). Strings flagged as suspicious are highlighted, letting you quickly focus on the most forensically relevant entries rather than scrolling through thousands of lines.

The imports view lists every DLL and function the executable imports, with suspicious APIs highlighted in red. You can quickly identify whether the file imports networking functions, cryptographic functions, process injection APIs, or anti-debugging calls — all without manually parsing the PE header.

Displays each section's name, virtual size, raw size, entropy, and characteristics (readable, writable, executable). Sections with entropy above 7.0 are flagged. Sections that are both writable and executable are flagged. Unusual section names are flagged.

5  ELF Binary Analysis (Linux Context)

📌 Supplementary Section

The primary focus of this module is PE analysis for Windows executables. This section provides a brief parallel for Linux ELF binaries. If malware was persisted on a Linux system via cron, systemd, or a SUID binary (Week 9), the payload is likely an ELF executable, and these tools apply.

5.1  ELF Header Inspection

# Display ELF header
readelf -h suspicious_binary

# Display section headers
readelf -S suspicious_binary

Key fields to examine:

  • Magic bytes: 7f 45 4c 46 confirms ELF format.
  • Class: ELF32 vs. ELF64 — indicates target architecture.
  • Type: EXEC (standalone executable) vs. DYN (shared library / position-independent executable).
  • Entry point address: where execution begins.

5.2  Detailed Header Dump

# Comprehensive header information
objdump -x suspicious_binary | head -80

5.3  Shared Library Dependencies

# Static analysis of dependencies (safe — does not execute the file)
readelf -d suspicious_binary | grep NEEDED

⚠️ Do Not Use ldd on Untrusted Binaries

The ldd command may execute the binary to resolve dependencies. On untrusted files, always use readelf -d instead, which reads the ELF header statically without execution.

5.4  String Extraction for ELF

The strings command works identically on ELF binaries. The same categories from the reference card in Section 3.2 apply — look for IP addresses, file paths (now Linux paths like /tmp/, /etc/cron.d/), shell commands, and encoded data.

6  Knowledge Check

Self-Assessment — Static Analysis Fundamentals

These questions are ungraded. Use them to confirm your understanding before proceeding to Module 13.3 (Dynamic Analysis) and Lab 12.

1. You generate a SHA-256 hash of a suspicious file and submit it to VirusTotal. The hash returns zero results — no antivirus engine has ever seen this file. What does this tell you, and what should you do next?

✓ Correct. A zero-result hash on VirusTotal does not mean the file is safe — it means no AV vendor has analyzed it yet. This is common with targeted malware, newly compiled samples, or polymorphic variants. The correct response is to continue your static analysis workflow (strings, PE header, import table) and then proceed to dynamic analysis in your isolated VM. Never execute an unknown file on a production system based on the absence of AV detections.
✗ Not quite. The correct answer is B. A zero-result hash means no AV engine has seen this file before — it does not mean the file is safe. Targeted malware, newly compiled samples, and polymorphic variants routinely evade signature-based detection. The proper response is to continue your static analysis (strings, PE headers, imports) to build hypotheses, then test them in dynamic analysis within your isolated VM. Option A is dangerous; Option C discards useful data; Option D describes NSRL, not VirusTotal.

2. You examine a PE file with pestudio and observe the following: the .text section has an entropy of 7.8, and the import table contains only two functions — LoadLibraryA and GetProcAddress. What do these two findings, taken together, indicate?

✓ Correct. This is a classic indicator of a packed executable. Entropy above 7.0 (on a 0–8 scale) indicates the section content is encrypted or compressed — not normal compiled code. A minimal import table containing only LoadLibraryA and GetProcAddress is the hallmark of a packing stub: these two functions are sufficient to dynamically load any DLL and resolve any API function at runtime, effectively hiding the true import table from static analysis. When you encounter this pattern, note "likely packed" in your analysis and consider using a tool like UPX (upx -d) or proceeding to dynamic analysis where the unpacked code will be visible in memory.
✗ Not quite. The correct answer is C. High entropy (7.8 out of 8.0) in the .text section means the code is encrypted or compressed — not normal compiled instructions. Combined with an import table containing only LoadLibraryA and GetProcAddress, this is the classic signature of a packed executable. The packing stub uses those two functions to dynamically resolve all other API calls at runtime, hiding the malware's true capabilities from static analysis. This is not corruption (B), not a small utility (A), and definitely not low risk (D).

3. During static analysis of svchost_update.exe, you extract strings and find the text Software\Microsoft\Windows\CurrentVersion\Run and the API imports RegSetValueExA and InternetOpenUrlA. A colleague from Week 11 memory analysis found that Volatility's netscan plugin showed an established connection from a process named svchost_update.exe to IP 198.51.100.42. How do the static analysis findings and the memory forensics findings corroborate each other?

✓ Correct. This is a textbook example of cross-domain correlation — one of the most important principles in digital forensics. Static analysis reveals capability: the import table shows the malware can make network connections and can write to the Registry. Memory forensics reveals activity: the netscan output proves the malware did make a network connection to 198.51.100.42. Neither finding alone is as powerful as both together. The static analysis explains how the connection was made (via InternetOpenUrlA); the memory forensics proves that it was made. This corroboration strengthens your report and supports the IOC documentation you will create in Module 13.3.
✗ Not quite. The correct answer is B. Static analysis and memory forensics are complementary, not redundant or unrelated. Static analysis reveals capability — the malware's import table shows it can communicate over the network (InternetOpenUrlA) and can write registry persistence (RegSetValueExA targeting the Run key). Memory forensics reveals activity — Volatility's netscan proves the malware did establish a connection to 198.51.100.42. Corroborating findings across domains is a core forensic principle. Neither source alone tells the complete story.
