1. Scenario Continuation — From Hypotheses to Execution

🔎 Continuing Investigation — svchost_update.exe

Your static analysis of svchost_update.exe in Module 13.2 revealed:

  • Imported network APIs: WSAStartup, connect, send, recv
  • Imported registry API: RegSetValueEx
  • Imported process API: CreateProcess
  • Extracted string: http://198.51.100.42/beacon
  • Extracted string: Software\Microsoft\Windows\CurrentVersion\Run
  • Extracted string: %TEMP%\payload.dll
  • PE compile timestamp: 3 days before the incident
  • Section entropy: 7.4 for .text — possible packing

You have hypotheses. Now it is time to confirm or refute them by watching the malware run.

Static analysis tells you what a file could do. Dynamic analysis tells you what it actually does. The gap between the two is critical: imported APIs may never be called, strings may be decoys, and packed code may unpack into something entirely different from what the PE header suggested.

In this module you will learn to safely execute a suspect binary, monitor its behavior across four domains (file system, registry, processes, network), and then distill your observations into structured Indicators of Compromise (IOCs) — the actionable output that turns a single-sample analysis into an enterprise-wide hunt.

2. Hypothesis Framework — Bridging Static to Dynamic

Before you press “run,” document what you expect to see. This is not guessing — it is the scientific method applied to malware analysis. Your static analysis findings generate testable predictions; dynamic analysis is the experiment.

💡 Hypothesis Template — svchost_update.exe
Pre-Execution Hypothesis Table
Static Finding Hypothesis (Expected Dynamic Behavior) What to Monitor
WSAStartup, connect, send imports Malware will initiate outbound network connections Wireshark — TCP SYN to external IPs
String: http://198.51.100.42/beacon Malware will attempt DNS resolution and/or HTTP GET/POST to 198.51.100.42 Wireshark — DNS queries, HTTP requests; INetSim logs
RegSetValueEx import + Run key string Malware will write a value to HKCU\...\Run for persistence ProcMon — RegSetValue operations

Why this matters: If dynamic analysis confirms a hypothesis, your confidence in that finding is high. If it refutes one, you have learned something equally valuable — perhaps the API is imported but never called, or the string is a decoy. Either outcome advances the investigation.

3. Dynamic Analysis Setup — The Controlled Cage

3.1 What Is Dynamic Analysis?

Dynamic analysis is the process of executing malware in a controlled environment and systematically observing its runtime behavior. Where static analysis examines the file at rest, dynamic analysis watches it in motion.

Static vs. Dynamic Analysis Comparison
DimensionStatic AnalysisDynamic Analysis
ExecutionFile is not executedFile is executed
Risk levelMinimal — no infection possibleHigher — requires strict isolation
Defeats packing?No — packed code hides contentYes — code unpacks at runtime
Reveals actual behavior?Inferred from imports/stringsDirectly observed
ReproducibilityFully reproducibleMay vary by environment (anti-VM checks)
Typical orderAlways firstAlways second

3.2 Pre-Execution Checklist

Before detonating any sample, every item on this checklist must be confirmed. Skipping a step can compromise your host system, contaminate evidence, or destroy your analysis environment.

☑ Pre-Execution Safety Checklist

  • Isolated VM confirmed: Guest OS is running in VirtualBox or VMware with no shared folders, no shared clipboard, and no USB passthrough enabled.
  • Network isolation verified: VM is on host-only networking or connected only to a network simulation tool (INetSim or FakeNet-NG). No route to the real internet.
  • Clean snapshot taken: A “pre-detonation” snapshot exists so you can revert to a known-clean state after analysis.
  • Monitoring tools launched: ProcMon (with filters set), Process Explorer, and Wireshark are running and capturing before the sample is executed.
  • Hypothesis table documented: Static analysis predictions are recorded (Section 2 above) so you know what to watch for.
  • Timestamp noted: Record the exact time of execution in your analysis notes for timeline correlation.
  • No real credentials present: The analysis VM contains no real usernames, passwords, API keys, or sensitive data.
  • Host OS patched and hardened: The hypervisor and host OS are up to date to minimize VM escape risk.
⚠️ Safety Rule — Absolute

Never execute malware on a production system, your personal computer, or any machine connected to a real network. Even in a classroom lab, all malware execution occurs inside isolated virtual machines with snapshotted, disposable guest operating systems. If you are unsure whether your environment is properly isolated, stop and ask your instructor.

3.3 Network Simulation Tools

Malware frequently requires internet connectivity to function — it may attempt DNS lookups, contact C2 servers, or download second-stage payloads. If the network is completely dead, the malware may stall, fail to activate, or behave differently than it would in the wild. Network simulation tools solve this problem by faking internet services inside your isolated lab.

INetSim (Internet Services Simulation Suite)

  • Linux-based tool that simulates HTTP, HTTPS, DNS, SMTP, FTP, and other common services
  • Responds to any DNS query with its own IP, so the malware “thinks” it has resolved its C2 domain
  • Logs every request the malware makes — a rich source of network IOCs
  • Typical setup: INetSim runs on the host or a second VM; the analysis VM’s DNS is pointed to INetSim’s IP

FakeNet-NG

  • Windows-based alternative developed by Mandiant (FLARE team)
  • Runs directly on the analysis VM — intercepts all network calls locally
  • Supports DNS, HTTP, SMTP, and custom protocol simulation
  • Easier single-VM setup but slightly less realistic than INetSim
🔑 Key Concept — Why Simulate, Not Connect?

Connecting a malware analysis VM to the real internet creates two risks: (1) the malware could attack other systems on the internet, making you liable; (2) the malware could download updated payloads or receive C2 instructions that change its behavior unpredictably. Simulation gives you control while still triggering the malware’s network behavior.

3.4 Execution Methods

How you launch the sample depends on the file type:

  • PE executable (.exe): Double-click or run from the command line. Command-line execution lets you pass arguments if static analysis revealed expected parameters.
  • DLL: Use rundll32.exe malicious.dll,ExportName — requires knowing the export function name from static analysis.
  • Script (PowerShell, VBScript, JavaScript): Execute with the appropriate interpreter. For PowerShell: powershell -ExecutionPolicy Bypass -File script.ps1
  • Document with macro: Open in the target application (Word, Excel) with macros enabled.

For our scenario, svchost_update.exe is a PE executable. We will execute it from the command line after all monitoring tools are running.

4.1 Process Monitor (ProcMon) — The Primary Behavioral Tool

ProcMon (from Microsoft Sysinternals) captures real-time file system, registry, process, and network activity on a Windows system. It is the single most important tool for dynamic malware analysis on Windows.

Essential ProcMon Filters for Malware Analysis

Without filters, ProcMon captures thousands of events per second from all processes on the system. Effective filtering is critical:

Key ProcMon Filters for Malware Analysis
FilterColumnRelationValuePurpose
Isolate malware process Process Name is svchost_update.exe Show only events from the suspect process
Registry writes Operation is RegSetValue Detect persistence and configuration changes
File creates Operation is CreateFile Detect dropped files and payloads
Process creation Operation is Process Create Detect child processes spawned by malware
Write operations Operation is WriteFile Detect data being written to disk
Focus on Temp folder Path contains Temp Malware frequently drops files here
Focus on Run keys Path contains CurrentVersion\Run Detect autostart persistence
🔍 Forensic Tip — Filter Strategy

Start with a broad filter (Process Name = malware) and review the event stream. Then add targeted filters for specific operations. Save your filter set — you will reuse it across samples. Also include child processes: if the malware spawns cmd.exe or powershell.exe, add those process names to your filter.

Key ProcMon Events and What They Mean

When monitoring svchost_update.exe, here is what a typical malicious execution sequence looks like in ProcMon:

Simplified ProcMon output showing the behavioral sequence of svchost_update.exe

Reading this sequence, we can reconstruct the malware’s behavior:

  1. Drops a file: Creates and writes payload.dll to the Temp folder (245 KB)
  2. Establishes persistence: Writes a Run key so payload.dll loads at every login
  3. Launches payload: Spawns rundll32.exe to execute the dropped DLL
  4. Initiates C2: Connects to 198.51.100.42 on port 80
🔗 Connection to Weeks 7–8 (Windows Forensics)

The registry key HKCU\Software\Microsoft\Windows\CurrentVersion\Run is the same persistence location you analyzed with RegRipper in Week 7. In that context, you were examining artifacts after the fact. Here you are watching the malware create those artifacts in real time. Same evidence, different perspective — this is the power of connecting malware analysis to OS forensics.

4.2 Process Explorer — The Process Tree View

While ProcMon captures granular events, Process Explorer (also Sysinternals) provides a real-time process tree that shows parent-child relationships.

What to watch for during malware execution:

  • Child processes: Does the malware spawn cmd.exe, powershell.exe, rundll32.exe, or other executables?
  • Process names: Does it try to disguise itself (e.g., naming itself svchost.exe but running from the wrong directory)?
  • Resource usage: Sudden CPU or memory spikes may indicate encryption (ransomware) or cryptomining
  • Network connections: The TCP/IP tab in Process Explorer shows active connections per process
🔗 Connection to Week 11 (Memory Forensics)

In Week 11, you used Volatility’s pslist and pstree to examine process hierarchies in a memory dump. Process Explorer shows you the same information live. If svchost_update.exe spawns rundll32.exe, that parent-child relationship would also appear in a memory dump analyzed with Volatility. The skills transfer directly.

4.5 Registry Monitoring

Beyond the Run key, malware modifies the registry for many purposes:

  • Persistence: Run, RunOnce, Services, Winlogon\Shell
  • Configuration storage: Custom keys to store C2 addresses, encryption keys, or campaign IDs
  • Security weakening: Disabling Windows Defender (DisableAntiSpyware), disabling UAC, modifying firewall rules
🔍 Forensic Tip — Before and After

Export a baseline of key registry locations before execution using reg export. After execution, export again and use a diff tool to identify exactly what changed. This is especially useful when ProcMon captures too many events to review manually.

5.2 Applying the Template — svchost_update.exe Results

After executing svchost_update.exe in our isolated VM, here is what the monitoring tools revealed:

File System Observations (ProcMon)
  • Created: C:\Users\analyst\AppData\Local\Temp\payload.dll (245,760 bytes, SHA-256: d4e5f6...)
  • Created: C:\Users\analyst\AppData\Local\Temp\config.dat (1,024 bytes — likely encrypted configuration)
  • Deleted: svchost_update.exe deleted itself after dropping payload.dll — self-deletion is an anti-forensics technique

Hypothesis check: ✅ Confirmed — static analysis predicted a file drop to %TEMP%. The additional config.dat was not predicted (new finding).

Registry Observations (ProcMon)
  • Created: HKCU\Software\Microsoft\Windows\CurrentVersion\Run\WinUpdate
  • Value: "rundll32.exe C:\Users\analyst\AppData\Local\Temp\payload.dll,ServiceMain"

Hypothesis check: ✅ Confirmed — persistence via Run key exactly as predicted from the RegSetValueEx import and the Run key string.

Process Observations (Process Explorer + ProcMon)
  • svchost_update.exe (PID 4892) spawned rundll32.exe (PID 5104)
  • rundll32.exe loaded payload.dll and called the ServiceMain export
  • After spawning rundll32.exe, the parent process (svchost_update.exe) terminated and deleted itself

Hypothesis check: ✅ Confirmed — CreateProcess import predicted child process creation. The self-deletion was a new finding.

Network Observations (Wireshark + INetSim)
  • DNS query: evil-update.example.com → resolved by INetSim to its own IP
  • HTTP GET: http://198.51.100.42/beacon?id=DESKTOP-LAB01&v=1.2
  • Beaconing: HTTP GET repeated every 60 seconds with consistent URL pattern
  • HTTP POST: After 3rd beacon, a POST to /exfil with 4,096 bytes of base64-encoded data
  • User-Agent: Mozilla/5.0 (compatible; MSIE 10.0) — outdated, likely hardcoded

Hypothesis check: ✅ Confirmed — network communication to 198.51.100.42 as predicted. The 60-second beaconing interval and the exfiltration POST were new findings not visible in static analysis.

✍️ Scenario Exercise — Interpreting ProcMon Output

A colleague sends you the following ProcMon events captured during execution of a different suspicious file, update_helper.exe:

14:22:05  Process Create    update_helper.exe         PID: 3201
14:22:05  CreateFile        C:\Windows\System32\drivers\netmon.sys    SUCCESS
14:22:05  WriteFile         C:\Windows\System32\drivers\netmon.sys    SUCCESS (89,088 bytes)
14:22:06  RegSetValue       HKLM\SYSTEM\CurrentControlSet\Services\NetMonitor    SUCCESS
                            Type: REG_DWORD  ImagePath: "C:\Windows\System32\drivers\netmon.sys"
                            Start: 0x01 (SERVICE_KERNEL_DRIVER, auto-start)
14:22:06  RegSetValue       HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\NetMon    SUCCESS
                            Data: "rundll32.exe netmon.sys,Init"
14:22:07  TCP Connect       203.0.113.99:443    SUCCESS
14:22:07  Process Create    cmd.exe /c del update_helper.exe    PID: 3340
What type of malware is this, and what are the key behaviors?

Analysis:

  • Malware type: This appears to be a rootkit dropper. It drops a kernel-mode driver (.sys file) into the System32\drivers directory and registers it as a kernel driver service with auto-start — this will load at boot before most security tools.
  • Persistence (dual): Both a kernel driver service (HKLM\...\Services) and a Run key — redundant persistence mechanisms.
  • Network C2: Connects to 203.0.113.99 on port 443 (HTTPS) — encrypted C2 channel.
  • Anti-forensics: Self-deletes via cmd.exe /c del after dropping the payload.
  • Privilege level: Writing to System32\drivers and HKLM requires administrator privileges — this malware either exploited a vulnerability or was run with elevated credentials.

Connection to prior weeks: The service registration in HKLM\SYSTEM\CurrentControlSet\Services would appear as Windows Event ID 7045 (new service installed) — a key artifact from Week 8. The network connection to 203.0.113.99:443 would be visible in both a memory dump (netscan in Volatility, Week 11) and a PCAP (Wireshark, Week 12).

6. Indicators of Compromise (IOCs)

6.1 What Are IOCs?

Indicators of Compromise (IOCs) are observable artifacts — file hashes, IP addresses, domain names, registry keys, process names — that indicate a system has been compromised. They are the actionable output of malware analysis: once you have IOCs, you can search for them across every system in your environment to determine the scope of the compromise.

Think of IOCs as the malware’s fingerprints. Just as a crime scene investigator lifts fingerprints to identify a suspect across multiple crime scenes, a digital forensic analyst extracts IOCs to identify the same malware across multiple systems.

🔑 Key Concept — IOCs Bridge Analysis and Response

Malware analysis answers: “What does this file do?” IOCs answer: “Where else has this happened?” Without IOC extraction, your analysis helps only the single system you are examining. With IOCs, your analysis protects the entire organization.

6.2 IOC Categories

IOCs fall into three broad categories. The reference card below summarizes each with examples from our svchost_update.exe analysis.

📄 IOC Categories Reference Card

Network IOCs

Artifacts observable in network traffic (PCAPs, firewall logs, DNS logs, proxy logs).

  • IP addresses: C2 servers, exfiltration destinations (e.g., 198.51.100.42)
  • Domain names: C2 domains, staging servers (e.g., evil-update.example.com)
  • URLs: Specific paths contacted (e.g., /beacon?id=...&v=1.2)
  • User-Agent strings: Hardcoded or unusual agents (e.g., MSIE 10.0)
  • JA3/JA3S hashes: TLS client/server fingerprints (concept — unique hash of TLS handshake parameters that identifies specific malware families even over encrypted connections)

Host-Based IOCs

Artifacts observable on the compromised system (disk, registry, memory, event logs).

  • File hashes: MD5, SHA-1, SHA-256 of malware and dropped files
  • File names and paths: payload.dll in %TEMP%, config.dat
  • Registry keys/values: HKCU\...\Run\WinUpdate
  • Mutex names: Named mutexes malware creates to prevent multiple instances (e.g., Global\XYZ123)
  • Scheduled tasks / service names: Persistence mechanisms
  • Process names and command lines: rundll32.exe payload.dll,ServiceMain

Email IOCs

Artifacts from the delivery mechanism (email headers, attachments).

  • Sender addresses/domains: Phishing source
  • Subject lines: Lure text patterns
  • Attachment names and hashes: The initial delivery file
  • Embedded URLs: Links in the email body

Note: Email IOCs connect to Week 16 (Email Forensics), where you will analyze email headers and phishing campaigns in depth.

🔍 Forensic Tip — Confidence Levels

High confidence: The IOC is unique to this malware and unlikely to cause false positives (e.g., file hash, specific C2 URL path). Medium confidence: The IOC may appear in legitimate contexts (e.g., a common User-Agent string, a shared hosting IP). Low confidence: The IOC is generic and will generate many false positives (e.g., “connects to port 443”). Always document confidence so analysts using your IOCs can prioritize their searches.

6.4 YARA Rules — Concept Introduction

YARA is a pattern-matching tool used to identify and classify malware. Analysts write YARA rules that describe textual or binary patterns found in malware samples; the rules can then be used to scan file systems, memory dumps, or network traffic for matches.

You will not write YARA rules in this course, but you should understand their structure and purpose — they are a standard tool in the malware analyst’s toolkit.

rule svchost_update_trojan {
    meta:
        description = "Detects svchost_update.exe dropper"
        author      = "CSC 156 Analyst"
        date        = "2026-04-22"

    strings:
        $c2_url     = "http://198.51.100.42/beacon"
        $mutex      = "Global\\XYZ123"
        $reg_key    = "Software\\Microsoft\\Windows\\CurrentVersion\\Run"
        $ua_string  = "MSIE 10.0"

    condition:
        uint16(0) == 0x5A4D and   // MZ header (PE file)
        3 of ($c2_url, $mutex, $reg_key, $ua_string)
}

Example YARA rule — matches PE files containing at least 3 of the 4 defined strings

How to read this rule:

  • meta: Descriptive metadata (not used for matching)
  • strings: Patterns to search for in the file
  • condition: Logic that must be true for a match — here, the file must start with MZ (PE executable) and contain at least 3 of the 4 strings
🔑 Key Concept — YARA in Practice

YARA rules are the bridge between one analyst’s findings and automated detection at scale. After you extract IOCs and identify unique strings, a YARA rule encodes that knowledge into a machine-readable format that can scan thousands of files in minutes. Think of it as turning your analysis notes into a search warrant that a computer can execute.

6.5 STIX/TAXII — Concept Introduction

While YARA rules help you find malware, STIX and TAXII help you share threat intelligence about it.

  • STIX (Structured Threat Information eXpression): A standardized language (JSON format) for describing cyber threat information — including IOCs, threat actors, attack patterns, and campaigns. Your IOC table could be expressed as a STIX bundle.
  • TAXII (Trusted Automated eXchange of Intelligence Information): A transport protocol for sharing STIX data between organizations. Think of STIX as the language and TAXII as the delivery truck.

STIX/TAXII Conceptual Flow

1. You are about to execute a suspicious PE file in your analysis VM. Your ProcMon is running, Wireshark is capturing, and you have taken a snapshot. However, you realize the VM is configured with NAT networking (connected to the real internet through the host). What should you do?

✅ Correct. NAT networking routes the VM’s traffic through your host to the real internet. This means the malware could contact its actual C2 server, download additional payloads, attack other systems, or alert the attacker that analysis is underway. Always switch to host-only networking with a network simulator (INetSim or FakeNet-NG) before detonation. The simulator will capture the malware’s network requests without any traffic leaving your lab.
❌ Not quite. The correct answer is B. NAT networking gives the malware a path to the real internet, which creates serious risks: the malware could attack other systems (making you liable), download unpredictable payloads, or notify the attacker. Simply disconnecting the host or monitoring the firewall is insufficient — you need host-only networking with a network simulation tool to safely trigger and capture the malware’s network behavior.

2. During dynamic analysis of a trojan, ProcMon shows the following event:
RegSetValue   HKCU\Software\Microsoft\Windows\CurrentVersion\Run\SysHelper   Data: "C:\Users\victim\AppData\Roaming\helper.exe"
What does this event indicate, and which prior-week skill would you use to find this artifact on another compromised system?

✅ Correct. RegSetValue on a CurrentVersion\Run key is a classic persistence mechanism — the value ensures helper.exe launches every time the user logs in. To find this on other systems, you would examine the NTUSER.DAT registry hive using RegRipper or Registry Explorer, exactly as you learned in Week 7. This is a perfect example of how dynamic analysis (Week 13) and OS forensics (Week 7) connect: you see the malware create the artifact in real time, then use Week 7 skills to find it on other machines.
❌ Not quite. The correct answer is B. RegSetValue is a write operation, not a read or delete. The CurrentVersion\Run key is a well-known persistence location — any value placed here causes the referenced program to execute at user login. This is not a service installation (that would be under HKLM\SYSTEM\CurrentControlSet\Services and generate Event ID 7045). To hunt for this IOC on other systems, you would use RegRipper (Week 7) to examine the NTUSER.DAT hive for the SysHelper value.

3. After completing your analysis of a malware sample, you extract the following IOCs. For each row, identify whether the confidence level should be High, Medium, or Low, and briefly explain why.

IOC TypeValueYour Confidence Rating
SHA-256a1b2c3d4... (hash of the malware binary)_____
IP Address8.8.8.8 (Google DNS, queried during execution)_____
URL Path/gate.php?bot_id=DESKTOP-LAB01&cmd=get_____
💡 Explanation:
  • SHA-256 of the malware binary → High confidence. A cryptographic hash is unique to that exact file. Searching for this hash will match only this specific malware sample (or an identical copy). Virtually zero false positives.
  • IP 8.8.8.8 → Low confidence (likely not a useful IOC). 8.8.8.8 is Google’s public DNS server. Millions of legitimate systems query it every second. Using this as an IOC would generate overwhelming false positives. The malware likely used it for a connectivity check, not as a C2 server. Document it in your notes but do not include it in your IOC list for enterprise hunting.
  • URL path /gate.php?bot_id=...&cmd=get → High confidence. This URL path structure is highly specific to this malware’s C2 protocol. The parameter names (bot_id, cmd) and the filename (gate.php) are characteristic of botnet C2 panels. Searching proxy or web logs for this pattern is very likely to identify other infected systems with minimal false positives.

Key takeaway: Not every artifact observed during dynamic analysis is a useful IOC. The best IOCs are specific to the threat and unlikely to match legitimate activity. Always evaluate confidence before including an artifact in your IOC documentation.
