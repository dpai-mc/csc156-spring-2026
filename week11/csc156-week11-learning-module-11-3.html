1

Volatility Framework Overview

Volatility is an open-source, Python-based memory forensics framework and the industry standard for RAM analysis in both incident response and digital forensics. It operates entirely against a static memory image — no running operating system required. Volatility parses raw binary structures directly from the dump: kernel object pools, page tables, VAD trees, and linked lists to reconstruct evidence that exists only in RAM: the live process list, open network connections, in-memory code, and volatile registry data.

Two major versions are active in field documentation, certification materials, and online resources. Understanding both is necessary for reading industry write-ups. Volatility 3 is the primary tool for all course work.

Profile and Symbol Selection

Memory forensics requires parsing kernel data structures at exact memory offsets. Those offsets differ across OS versions and patch levels — a field in the Windows 10 Build 19041 EPROCESS structure sits at a different offset than the same field in Build 22621. Using the wrong offsets produces incorrect or empty output with no obvious error message.

Volatility 2 — Manual Profile Selection

In V2, the examiner determined the OS version independently — by running imageinfo or kdbgscan and interpreting suggestions — then supplied a profile name that bundled the correct offset table for that specific build. An incorrect profile produced silently wrong results. Profiles required separate downloads for patched systems and could become stale as Microsoft released updates.

Volatility 3 — Automatic Symbol Resolution

V3 replaces profiles with Intermediate Symbol Format (ISF) files — compact JSON files encoding kernel symbol offsets derived from Microsoft's public symbol server (PDB files). The workflow is automatic:

  1. V3 identifies the OS build from structures present in the memory image.
  2. V3 retrieves or uses a locally cached ISF file for that exact build.
  3. Analysis proceeds with correct offsets — no analyst action required.
⚙️
First-Run Behavior in Lab 10

When you analyze your own machine's memory dump in Lab Activity 10 Path B, V3 auto-resolves the ISF file for your exact Windows build. The first run against a new build may pause briefly to fetch symbol data. Subsequent runs use the local cache and run at full speed. All course-provided analysis images include pre-cached symbols.

Linux — ISF Files and Custom Kernels

Linux memory analysis in V3 requires an ISF file generated from the specific kernel of the target system. Unlike Windows, there is no centralized symbol server — Linux kernels vary by distribution, version, and compile-time flags. Examiners generate ISF files using the dwarf2json tool against the target kernel's debug symbol package.

Core Navigation Plugins

Six plugins form the foundation of every process-level Volatility investigation. Learn their purpose, syntax, and limitations before attempting Lab Activity 10. Click any card to expand its full reference entry.

⚠️
Best Practice — Always Run Both pslist AND psscan

windows.pslist walks the kernel's doubly-linked active-process list. A rootkit using Direct Kernel Object Manipulation (DKOM) can unlink its EPROCESS entry from that list, hiding the process from pslist entirely. windows.psscan scans raw memory for EPROCESS pool-tag signatures, bypassing the linked list and finding hidden processes. Any process visible in psscan but absent from pslist is a high-priority rootkit indicator. Comparing both outputs is standard practice on every investigation.

windows.info System identification and image orientation ▼

Purpose: Displays OS version, architecture, kernel build string, and key memory addresses. Run this plugin first on every image to confirm successful loading and to record the OS version in your case notes.

Syntax

Key output fields

DTB                0x187000
NtBuildLab         19041.1.amd64fre.vb_release...  ← confirms OS build
NtMajorVersion     10
NtMinorVersion     0
PE MajorOperSysVer 10
🐧
Linux Equivalent

linux.banners.Banners — extracts kernel version banner strings from a Linux memory image.

windows.pslist Active process list via EPROCESS linked list ▼

Purpose: Walks the kernel's doubly-linked list of EPROCESS structures to enumerate active processes. Returns PID, parent PID (PPID), process name, create time, and exit time. Fast and reliable for standard (non-rootkit) scenarios.

Limitation: A rootkit can unlink its EPROCESS entry via DKOM, hiding the process from pslist output. Always cross-reference with windows.psscan.

Syntax 
python3 vol.py -f memdump.mem windows.pslist
Sample output (truncated) 
PID   PPID  Name            Offset      Threads  Sessions  CreateTime
4     0     System          0xe000...   136      None      2024-11-01 09:12:03
556   496   lsass.exe       0xb501...   12       0         2024-11-01 09:12:15
1234  1340  powershell.exe  0xa800...   8        1         2024-11-01 14:32:01
🐧
Linux Equivalent

linux.pslist — walks the Linux task_struct doubly-linked list to enumerate processes.

windows.psscan Pool-tag scan for EPROCESS structures — rootkit-resistant ▼

Purpose: Scans physical memory for EPROCESS pool-tag signatures rather than following the linked list. Finds currently active processes, recently terminated processes still resident in memory, and DKOM-unlinked (hidden) processes. Slower than pslist but more complete and rootkit-resistant.

Key use: Run after pslist and compare outputs. Any process present in psscan but absent from pslist is a priority finding requiring immediate follow-up.

Syntax 
python3 vol.py -f memdump.mem windows.psscan
🐧
Linux Equivalent

linux.psscan — scans Linux memory pools for task_struct objects, finding processes that may not appear in the standard list.

windows.pstree Process hierarchy rendered as a parent–child tree ▼

Purpose: Renders the process list as an indented tree showing parent–child relationships visually. This view is critical for detecting wrong-parent anomalies — one of the most common structural indicators of process injection, privilege escalation, and living-off-the-land attacks.

Syntax 
python3 vol.py -f memdump.mem windows.pstree
Sample output with anomaly highlighted 
* 4      System
** 348   smss.exe
*** 448  csrss.exe
*** 496  wininit.exe
**** 548 services.exe
***** 688 svchost.exe
****** 1284 cmd.exe  ← cmd.exe child of svchost — suspicious wrong-parent
**** 556 lsass.exe
🐧
Linux Equivalent

linux.pstree — renders the Linux process hierarchy using task_struct parent pointers.

windows.cmdline Full command-line arguments for each process ▼

Purpose: Retrieves the full command-line string used to launch each process from its Process Environment Block (PEB). Exposes executable paths, launch flags, encoded payloads, and embedded URLs that a process name alone cannot reveal. Essential for confirming suspicious processes found in pslist or pstree.

Syntax 
python3 vol.py -f memdump.mem windows.cmdline
Suspicious example — Base64-encoded PowerShell payload 
PID   Process         Args
1234  powershell.exe  powershell.exe -enc SQBFAFgAKABOAGUAdwAtAE8A...
                      ↑ -enc flag + Base64 string = obfuscated command, major red flag
🐧
Linux Equivalent

linux.cmdline — retrieves command-line arguments from Linux process memory (equivalent of reading /proc/<pid>/cmdline).

windows.dlllist Loaded DLLs for each process — path and load order ▼

Purpose: Lists all DLLs loaded into each process's address space by walking the LDR_DATA_TABLE_ENTRY list in the PEB. Reveals injected DLLs, anomalous load paths, and name-masquerading DLLs (e.g., a malicious ntdll.dll loaded from C:\Users\Public\ alongside the real one from System32).

Syntax — all processes 
python3 vol.py -f memdump.mem windows.dlllist
Syntax — single process by PID 
python3 vol.py -f memdump.mem windows.dlllist --pid 688
Suspicious output — duplicate DLL name, wrong path 
PID  Name         Path
688  svchost.exe  C:\Windows\System32\ntdll.dll      ← expected
688  svchost.exe  C:\Users\Public\ntdll.dll          ← wrong path — injected DLL?
🐧
Linux Equivalent

linux.library_list — lists shared libraries loaded by Linux processes from their memory maps.

Normal Windows Process Baseline

No anomaly is detectable without a clear picture of what is normal. Before triaging any memory image, internalize the expected attributes of core Windows processes. Deviations in name spelling, executable path, parent process, or instance count are all investigative findings.

📋

Malware frequently impersonates these eight processes by misspelling names (scvhost.exe), running from non-standard paths (C:\Windows\Temp\lsass.exe), appearing with wrong parents, or spawning unexpected additional instances. Suspicious Process Detection

Malware authors study the same baseline table you just reviewed. Their evasion strategies are not random — they produce predictable anomaly patterns. Recognizing these patterns by name and example is a core analyst skill.

Detection Pattern Framework

Table 3 — Suspicious Process Detection Patterns
Pattern What to Look For Primary Plugin
Name masquerading Typosquatting: scvhost.exe, lsass.exe  (trailing space), svch0st.exe; correct-sounding names absent from the baseline; Unicode lookalike characters pslist, pstree
Wrong executable path Correct name but non-standard directory: svchost.exe from C:\Windows\Temp\, C:\Users\Public\, C:\ProgramData\, or any location other than System32 cmdline, dlllist
Wrong parent process Unexpected PPID: lsass.exe child of explorer.exe; cmd.exe child of svchost.exe; powershell.exe child of winword.exe (macro malware pattern) pstree

Wrong instance count Two copies of lsass.exe; zero explorer.exe on an active user session; unusually high svchost count on a simple workstation pslist

Process Injection Concepts

Sophisticated malware does not create a conspicuously named process — it hides inside a legitimate, trusted one. Process injection is a family of techniques that place malicious code into the address space of a running host process, allowing the injected code to execute under that process's identity, privileges, and security context while evading process-name-based detection entirely.

📌
Scope — Conceptual Overview

This section introduces four injection categories at the level needed for forensic analysis. Implementation-level reverse engineering is outside the scope of CSC 156. Your goal is to recognize the memory artifacts each technique leaves so that you can interpret Volatility plugin output correctly and document findings with appropriate technical precision.

DLL Injection

The attacker writes a malicious DLL path into a target process's memory and triggers LoadLibrary via a remote thread. The DLL executes inside the host process under its identity. Forensic signature: unexpected DLL in the process module list with an anomalous path — visible with windows.dlllist.

Process Hollowing

A legitimate process is created in a suspended state. Its original executable code is unmapped from memory and replaced with malicious code. Execution is then resumed. The process name in pslist looks legitimate; the running code is entirely attacker-controlled. Forensic signature: VAD entry mismatch between on-disk image and in-memory content — detectable with windows.malfind.

Reflective DLL Injection

The malicious DLL includes its own custom loader and maps itself directly into memory without invoking the standard Windows loader and without writing to disk. File-based detection misses it entirely. Forensic signature: anonymous executable memory regions containing a PE header (MZ signature) — the primary artifact targeted by windows.malfind.

Thread Hijacking

An existing thread in a target process is suspended. Its register context and instruction pointer are redirected to malicious shellcode. The thread is then resumed. No new thread or DLL is created. Forensic signature: thread instruction pointers referencing anomalous memory regions; indirect evidence from windows.malfind flagging the shellcode region.

Common Forensic Indicators of Injection

  • RWX memory regions (read + write + execute) — executable memory that is also writable is abnormal in legitimate code after process initialization.
  • PE headers in anonymous memory — MZ/PE magic bytes (4D 5A) at the start of a non-file-backed executable region indicate a reflectively loaded or hollowed PE image.
  • Module list vs. memory discrepancy — DLLs visible in memory but absent from the dlllist module list (reflective injection bypasses the standard loader and thus the LDR list).
  • On-disk vs. in-memory hash mismatch — the executable on disk hashes differently from its in-memory image, indicating process hollowing or patching.
7

Detecting Injected Code with windows.malfind

windows.malfind is Volatility's primary plugin for detecting code injection. It scans the Virtual Address Descriptor (VAD) tree of each process — the kernel structure recording the attributes of every memory region — and flags regions that combine suspicious permissions with suspicious content.

What malfind Flags

A region is reported when it satisfies both conditions simultaneously:

  1. Executable permissions — the region protection includes execute: PAGE_EXECUTE, PAGE_EXECUTE_READ, PAGE_EXECUTE_READWRITE, or PAGE_EXECUTE_WRITECOPY.
  2. Not file-backed — the region has no associated memory-mapped file; it is anonymous memory, not a loaded module from disk.

For each flagged region, malfind outputs: process name and PID, the virtual address range, VAD protection flags, a hex dump of the opening bytes, and a disassembly listing of those bytes.

Syntax — all processes 
python3 vol.py -f memdump.mem windows.malfind
Syntax — single process by PID 
python3 vol.py -f memdump.mem windows.malfind --pid 688
Syntax — save to CSV for triage 
python3 vol.py -f memdump.mem --output csv --output-file malfind.csv windows.malfind
Example output — PE header in anonymous executable memory (high-confidence injection) 
PID   Process      Start          End            Protection
688   svchost.exe  0x00d0000000   0x00d0001fff   PAGE_EXECUTE_READWRITE

Hexdump:
0x00d0000000  4d 5a 90 00 03 00 00 00  04 00 00 00 ff ff 00 00  MZ......  ........
              ↑ MZ at offset 0 = PE header start in anonymous executable memory = strong injection indicator

Malfind Triage Walkthrough

malfind output can be voluminous and includes legitimate false positives. Use this four-step workflow to prioritize findings systematically. Click each step to view the guidance.

Step 1 — Collect All Flagged Regions

Run windows.malfind --output csv and save to a file for easier review. Note the total number of findings, which processes appear, and which protection flags are present. Processes with many hits, or rare processes appearing in the output, are higher priority.

Focus first on regions flagged as PAGE_EXECUTE_READWRITE — combining write and execute permissions is almost never needed by legitimate code after process startup, and represents the highest-risk profile in malfind output.

Step 2 — Check for Embedded PE Headers

Examine the hex dump for each flagged region. Look for 4D 5A ("MZ") at offset 0 — the magic bytes that start every Windows PE executable. A PE header inside anonymous executable memory is a strong indicator of reflective DLL injection or process hollowing.

MZ signature identification 
4D 5A 90 00 03 00 00 00  ← "MZ" at offset 0 = PE file header
# In non-file-backed executable memory → high-confidence injection indicator
# No MZ present → may still be shellcode; check disassembly for position-independent code

Step 3 — Assess the Host Process Context

Apply your process baseline knowledge. A malfind hit in a process already flagged for a wrong path, wrong parent, or suspicious command-line arguments is a compounding indicator — treat it as high priority. A hit in lsass.exe or svchost.exe is also high priority: attackers target trusted system processes precisely because they are expected to be running and are less likely to be terminated by security tools.

Follow up every significant hit: run windows.cmdline --pid <PID> and windows.dlllist --pid <PID> on the same process before drawing conclusions.

Step 4 — Manage False Positives

malfind routinely flags legitimate anonymous executable memory in certain process categories. Recognizing these common sources prevents wasted triage time:

  • JIT-compiled code — Chrome (V8), Edge, Firefox (SpiderMonkey), .NET CLR, Node.js, and Java JVMs all allocate anonymous executable memory for just-in-time compiled code. Expect many hits in any browser or .NET application.
  • Self-modifying packers — some legitimate software temporarily uses RWX regions during unpacking or self-decryption at startup.
  • Anti-cheat engines — use intentionally obfuscated executable memory as a design feature.

Analyst guidance: False positives are classified and documented, not ignored. A malfind hit in chrome.exe with no MZ header = low priority / probable JIT. A hit in lsass.exe with an MZ header in a process that also has a suspicious parent = high priority regardless of alternative explanations. Never dismiss a finding in a core system process without a documented reason.

Summary and Look Ahead

Module 11.3 — Key Takeaways

  • Volatility 3 is the course tool. It auto-resolves symbols via ISF files, uses namespaced plugins (windows.pslist, etc.), and produces structured output. V2 uses flat plugin names and a required --profile flag — recognize it, do not use it.
  • Always run pslist AND psscan. A process in psscan but absent from pslist = DKOM rootkit indicator, high priority.
  • Six core plugins in sequence: windows.info → windows.pslist → windows.psscan → windows.pstree → windows.cmdline → windows.dlllist. Know the purpose, syntax, and limitation of each.
  • Baseline before anomaly. Memorize the expected name, path, parent, and instance count for the eight core Windows processes. Table 2 is your mental reference standard on every investigation.
  • Seven anomaly patterns: name masquerading, wrong path, wrong parent, suspicious command-line arguments, unusual timing, wrong instance count, psscan/pslist discrepancy. Any single pattern is a finding.
  • Four injection techniques: DLL injection (dlllist evidence), process hollowing (VAD content mismatch), reflective DLL injection (PE header in anonymous memory), thread hijacking (redirected thread context).
  • malfind triage: collect all flags → look for MZ headers → assess host process context → classify and document false positives (JIT, packers). Never dismiss a hit in a core system process without documentation.
