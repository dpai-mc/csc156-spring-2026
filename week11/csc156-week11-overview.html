Week 11

Volatile Memory Forensics

Welcome to Week 11 — our first new content week after the Week 10 review pause and a major turning point in the course. After six weeks of disk forensics, we shift our focus to volatile memory (RAM): live, temporary data that exists only while a system is powered on and vanishes the moment it is shut down. RAM can hold powerful evidence that never touches a hard drive — running processes, open network connections, decrypted data, encryption keys, and entire malware payloads residing only in memory — making it one of the most time-sensitive and investigatively rich sources available to a forensic examiner. This week you will learn how to capture RAM safely, verify its integrity, and use the Volatility Framework to extract meaningful artifacts before that evidence disappears forever.

Week 11 Learning Goals

By the end of this week you should be able to explain and demonstrate each of the following:

  1. 1

    Explain what volatile memory is, how it differs from non-volatile storage, and why its temporary nature makes it a high-priority target that must be collected before any other evidence in an active investigation.

  2. 2

    Apply the Order of Volatility to triage and sequence evidence collection, articulating why RAM sits at the top of that hierarchy above swap files, running processes logs, disk data, and archival media.

  3. 3

    Perform memory acquisition on both Windows and Linux systems using appropriate forensic tools, while applying proper chain-of-custody procedures and cryptographic hashing to verify the integrity of the captured image.

  4. 4

    Use the Volatility Framework to load and interrogate a memory image, executing core plugins to enumerate running processes, loaded DLLs, and open handles within a target system's memory snapshot.

  5. 5

    Compare the output of pslist and psscan to identify discrepancies that may reveal hidden, unlinked, or injected processes indicative of malicious activity or process-injection attacks.

  6. 6

    Extract and interpret network connection artifacts, registry hive data, and fileless malware indicators from a memory image, and integrate those findings into a structured investigation workflow and written summary.

Key Concepts This Week

After completing all four modules you should be able to define and explain each of these terms without referencing your notes:

Order of Volatility
The prioritized sequence for collecting digital evidence — from most perishable (CPU registers, RAM) to least volatile (archival media) — that governs acquisition order at any live scene.
Volatile vs. Non-Volatile Data
Volatile data (RAM, CPU cache, ARP tables) is lost on power-off; non-volatile data (disk, flash storage) persists. Understanding this distinction is foundational to evidence triage decisions.
Memory Acquisition
The process of capturing a bit-for-bit snapshot of live RAM using tools such as WinPmem (Windows) or LiME (Linux), preserving its contents for offline analysis while maintaining forensic integrity.
Volatility Framework
An open-source, plugin-based Python tool for parsing and analyzing memory image files across Windows, Linux, and macOS — the industry-standard platform for memory forensics.
pslist vs. psscan
pslist walks the OS-maintained active process list; psscan scans raw memory for EPROCESS structures. Comparing both can expose hidden, unlinked, or recently terminated rogue processes.
Fileless Malware
Malicious code that executes entirely in memory without writing a file to disk, leaving RAM acquisition as often the only forensic method able to recover evidence of the attack.
windows.netscan
A Volatility plugin that recovers active and recently closed network connection structures (sockets, endpoints) from a Windows memory image, revealing command-and-control or lateral-movement activity.
Process Injection
An attack technique in which malicious code is inserted into the address space of a legitimate running process — detectable through memory-region anomalies, unexpected DLL loads, and pslist/psscan discrepancies.
