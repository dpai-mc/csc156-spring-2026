Why RAM? Volatile Memory, the Order of Volatility, and the Forensic Inventory of a Running System

CSC 156 – Digital Forensic Fundamentals

⏱ Estimated time: 1.5–2.0 hours 🎯 Objective: Week 11 Obj. 1 📋 Supports: Lab Activity 10 · Quiz 3

🔍 Opening Scenario

📡 Active Scene — First Responder Decision Point
"You arrive at a suspect's active workstation. The screen is on, the system is running, and you can see what appears to be an encrypted file manager open on the desktop. Your supervisor says, 'Pull the power and bag the machine.'"
❓ Do you follow that instruction? What evidence might you lose forever if you do?

Hold that question. By the end of this module you will have a principled, standards-backed answer. Every concept introduced below feeds directly into the decision you must make at that workstation. We return to this scenario in the Summary.

🔗 Prior-Week Bridge — Week 6: Forensic Acquisition

In Week 6 you imaged non-volatile storage using hardware write-blockers, verified integrity with MD5/SHA-256 hashes, and documented chain of custody. This week extends that same discipline to a fundamentally different artifact: the live system's RAM. The acquisition window is measured in minutes, not hours — and there is no write-blocker equivalent for memory. Tool integrity verification (SHA-256 of the acquisition tool binary) replaces the write-blocker's role for volatile media.

⚡ Volatile vs. Non-Volatile Data

Digital evidence divides into two categories based on a single property: does the data survive a power cycle? This distinction determines when you collect, what you collect first, and what you can never recover once a machine is powered off.

Table 1 — Key properties and forensic examples of volatile and non-volatile data
Property ⚡ Volatile (RAM) 💾 Non-Volatile (Disk / Flash)
Survives power loss? ❌ No — lost immediately on shutdown ✅ Yes — persists indefinitely
Storage medium DRAM chips (charge-based capacitors) HDD platters, SSD NAND, optical, removable flash
Acquisition window Live system only; window closes on shutdown After power-off, using hardware write-blockers
Forensic examples Running processes, encryption keys, network connections, clipboard, injected code Files, registry hives on disk, browser history, event logs, MFT entries
Modification risk Every second the OS runs overwrites content Stable with write-blockers; low risk post power-off
Standard acquisition tools WinPmem, DumpIt, LiME (Linux) dd, FTK Imager, dcfldd
Integrity verification Hash the dump file; document tool artifacts Hash before and after; log write-blocker use
🔎 Forensic Tip — The Power-Off Decision

Powering off a live system before capturing RAM is the single most common first-responder mistake in computer investigations. The encrypted file manager in our opening scenario almost certainly holds its decryption key in RAM at this moment. Cutting the power destroys that key permanently. A RAM acquisition with WinPmem or DumpIt takes under five minutes. The disk will wait; RAM will not.

Why Volatile Evidence Is Uniquely Valuable

Sophisticated adversaries know how to defeat disk-based forensics: encrypted volumes, secure-deletion tools, fileless malware, and anti-forensic utilities all target the persistent layer. RAM does not lie in the same way. To execute, code must be loaded into memory — and memory does not honor disk encryption. Running processes, decrypted strings, live network connections, and injected shellcode all exist in RAM in an accessible form as long as the system has power.

⚠️ Warning — Investigator Contamination of Memory

Every action taken on a live system — opening a file, running a command, moving the mouse — writes data into RAM and may overwrite evidence. Run acquisition tools exclusively from write-protected external media. Document every command executed and every tool launched in your case notes. Contamination is unavoidable; concealing it is not acceptable.

🧠 The Forensic Inventory of RAM

A live system's RAM is not a passive buffer — it is a real-time snapshot of everything the operating system is doing. A forensically sound memory dump can yield the artifact categories below, most of which cannot be recovered from disk alone.

⚙️ Running Processes

Process name, PID, PPID, start time, virtual address space, and loaded DLLs for every active executable.

🔑 Encryption Keys & Credentials

Active encryption keys, plaintext passwords cached by applications, session tokens, BitLocker VMK, and LSASS credential caches.

🌐 Network Connections

Active and recently closed TCP/UDP connections with IP addresses, ports, states, and owning process — including connections absent from any log file.

📋 Clipboard Contents

The most recently copied data — passwords, PII, command output — persists in RAM until the clipboard is overwritten.

💬 Command History (In-Memory)

Recent shell commands (cmd.exe, PowerShell, bash) held in process memory before or instead of being flushed to a history file on disk.

🗂️ Volatile Registry Keys

HKCU and HKLM segments loaded in RAM, including volatile keys under HKLM\SYSTEM\CurrentControlSet that are never written back to disk in their live form.

🕸️ Injected Code & Malware Artifacts

Code injected into legitimate processes via DLL injection or process hollowing — leaving no trace on disk and detectable only through memory analysis.

📄 Open File & Object Handles

Files, sockets, and registry keys currently open — including handles to files that have been deleted from disk while still in use by a process.

🔄 Kernel Structures & Drivers

Loaded kernel modules, device drivers, and system call tables. Rootkits that manipulate these structures are often detectable only in RAM.

ℹ️ The "Ghost File" Scenario

Malware executes, completes its payload, and deletes its own binary from disk — all within seconds. A disk image finds nothing. But RAM still contains the process's allocated memory pages including code and strings, and the kernel's process list may still have an entry before the OS cleans it up. Memory forensics can reconstruct the malware binary from process memory even when disk analysis is entirely negative.

Deeper Dive — Select a Category

The expandable sections below provide forensic detail for three categories that appear directly in Modules 11.3 and 11.4 and in Lab Activity 10.

The Windows kernel maintains a doubly-linked list of EPROCESS structures, one per running process. Key fields include:

  • ImageFileName — truncated process name (15 characters)
  • UniqueProcessId — the PID
  • InheritedFromUniqueProcessId — PPID, enabling parent-child relationship analysis
  • CreateTime — kernel timestamp of process creation
  • VadRoot — the root of the Virtual Address Descriptor tree for this process's memory map

Volatility's pslist plugin walks this linked list. Malware authors sometimes unlink their EPROCESS node — a technique called Direct Kernel Object Manipulation (DKOM) — to hide from list-walking tools. Volatility's psscan counters this by scanning raw memory for EPROCESS pool tag signatures, independent of list integrity.

On Windows 7+, connection state is maintained by tcpip.sys using _TCP_ENDPOINT and _UDP_ENDPOINT structures. Each records:

  • Local and remote IPv4/IPv6 addresses and port numbers
  • Connection state: LISTENING, ESTABLISHED, CLOSE_WAIT, TIME_WAIT, etc.
  • A pointer to the owning EPROCESS — directly linking the connection to a process
  • Connection creation timestamp

Key forensic value: connections that have closed but not yet been garbage-collected by the kernel still appear in a memory dump, providing evidence of communication that would not appear in a live netstat run after the fact. The Volatility plugin for this is netscan.

Windows loads registry hives from disk into memory at boot and on user login. The live HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet is constructed from multiple backing hives and is never written back to disk as-is. Values modified since boot exist only in the in-memory form until a flush occurs.

HKEY_CURRENT_USER is the in-memory loaded form of the user's NTUSER.DAT. Pending writes not yet flushed to disk are visible only in a memory dump.

Volatility's hivelist locates all loaded hive structures; printkey and dumpregistry extract key data directly from RAM without touching the disk hive files at all.

📜 RFC 3227: The Order of Volatility

In 2002, the IETF published RFC 3227 — "Guidelines for Evidence Collection and Archiving." While not a law or a certification requirement, it is the universally cited forensic reference for evidence collection sequencing in incident response and digital forensics engagements worldwide. Its core mandate:

ℹ️ RFC 3227 Core Principle

"In general, order the collection of evidence from the most volatile to the least volatile." Collect what will disappear soonest, first. Stable evidence can wait.

Forensic Implications of the Order

  • RAM before disk, always. If you image the disk first and the system shuts down before memory capture, all RAM evidence is destroyed. The reverse is not true — with a write-blocker, a disk image taken after a RAM dump is byte-for-byte identical to one taken before it.
  • Live-response commands before the full dump. Documenting ARP cache, routing table, and running process list with individual commands is your Rank 2 collection step before launching a full acquisition tool.
  • The acquisition tool itself contaminates memory. Running WinPmem or DumpIt writes the tool's own EPROCESS entry into RAM and modifies kernel structures. This is unavoidable, acceptable, and must be disclosed in case notes — not hidden.
  • Time is the adversary. Malware, user activity, and OS background processes continuously overwrite RAM. Every minute of hesitation is a minute of potential evidence loss.
🔎 Forensic Tip — The Question You Will See on Quiz 3

"From a live Windows system, should you acquire disk or RAM first?" The answer is unambiguous under RFC 3227: RAM first. Disk is non-volatile and can wait. A disk image taken five minutes after a RAM dump is byte-for-byte identical to one taken five hours later (with a write-blocker). RAM will not be.

🔬 RAM as a Forensic Medium

Understanding how RAM is structured explains both what a memory dump contains and why some artifacts require specialized tooling to locate.

Physical Memory and Page Frames

The operating system presents each process with its own contiguous virtual address space — an abstraction. Underneath, physical RAM is divided into fixed-size page frames, typically 4 KB on x86/x64 systems. The Memory Management Unit (MMU) translates virtual addresses to physical addresses using page tables that are themselves stored in RAM.

A forensic memory dump captures physical memory — the raw page frames on the DRAM chips. Reconstructing the virtual address space of any given process requires walking that process's page table structures, which Volatility handles automatically once given a matching OS profile.

ℹ️ Page Size Quick Reference

Standard page size on x64 Windows and Linux: 4,096 bytes (4 KB). Large pages (2 MB) are also used for certain kernel structures. An 8 GB RAM system has approximately 2,097,152 standard page frames. Volatility processes these frames to reconstruct per-process memory maps during analysis.

Address Space Layout Randomization (ASLR)

Modern operating systems randomize the base addresses of the kernel, libraries, heap, and stack at each boot — a security mechanism called ASLR. Forensic implications:

  • Fixed-address assumptions are invalid; analysis tools must locate structures by scanning for pool tag signatures or following kernel data structure chains from known anchor points.
  • Volatility's profile system handles ASLR automatically for supported OS versions — it locates EPROCESS lists, pool allocations, and network endpoint structures regardless of boot-time randomization.
  • ASLR does not prevent memory forensics; it requires tools that are address-agnostic by design. This is exactly what Volatility provides.

The Pagefile as a Supplement

When physical RAM fills up, the OS moves infrequently-used pages to the pagefile (pagefile.sys on Windows; swap partitions on Linux). Pages swapped out at the moment of capture will appear as not present in the memory dump. However:

  • The pagefile is a non-volatile artifact on disk and can be imaged with standard acquisition tools alongside the disk image.
  • Some Volatility 3 builds can incorporate a pagefile alongside a RAM image, reconstructing pages that were swapped out at capture time.
  • Encryption keys and active malware code are rarely paged out because the OS prioritizes keeping frequently-accessed pages in physical RAM — but this is not guaranteed.
🔎 Forensic Tip — Always Image the Pagefile

Best practice: acquire both the live RAM dump and a disk image that includes pagefile.sys. Together they provide the most complete reconstruction of system execution state. Neither artifact alone is sufficient for a thorough memory investigation.

Cold Boot Attack — A Physical Edge Case

DRAM cells lose their charge gradually after power-off — they do not clear instantly. In laboratory conditions, researchers have demonstrated that cooling RAM chips with compressed air preserves charge state long enough to transplant and read the modules in a separate system. This is the cold boot attack. It is rarely feasible in the field but illustrates an important concept: "volatile" describes a time-decay spectrum, not an instantaneous disappearance.

📌 Summary and Look Ahead

Returning to the Opening Scenario

Your supervisor says: "Pull the power and bag the machine."

You now have a principled, standards-backed reason to respond professionally:

"Under RFC 3227's Order of Volatility, we need to capture volatile memory before we power off. That encrypted file manager almost certainly holds its decryption key in RAM right now. Pull the power and that key — and everything it unlocks — is gone forever. I can run a memory acquisition tool from write-protected media in under five minutes. Then we image the disk with a write-blocker. We lose nothing, and we gain everything that's in RAM."

Module 11.1 Key Takeaways

  • Volatile data exists only while the system has power and is uniquely valuable because it cannot be hidden by encryption at rest.
  • A live system's RAM holds a rich forensic inventory: running processes, encryption keys, network connections, clipboard contents, volatile registry data, injected code, and open file handles.
  • RFC 3227 mandates collecting evidence from most to least volatile. Volatile OS data structures (Rank 2) and full RAM (Rank 3) must be acquired before disk (Rank 5) on any live system.
  • Physical RAM is organized as page frames (typically 4 KB each). Forensic tools reconstruct virtual process spaces by walking the page tables in the raw physical dump.
  • ASLR randomizes memory addresses at every boot but does not prevent forensics — it requires address-agnostic tools like Volatility, which handle this automatically via OS profiles.
  • The pagefile is a non-volatile supplement to a RAM dump; acquiring both provides the most complete picture of execution state.
