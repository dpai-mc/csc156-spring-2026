Network & Registry Artifacts in RAM — and the Six-Phase Investigation Workflow

Complete the memory forensics picture: recover network connections and registry evidence directly from a RAM image, then unify all Week 11 tools into a structured, repeatable investigation methodology.

Course: CSC 156 – Digital Forensic Fundamentals Week: 11 Module: 11.4 Prerequisite: Modules 11.1–11.3

1 Network Connections in Memory

When a process opens a TCP or UDP socket, the Windows kernel maintains that connection's complete metadata — source address, destination address, port numbers, state, and owning process — entirely in physical RAM. Because this data lives only in memory and is never written to disk in a structured forensic-friendly format, a RAM image is often the only place an investigator can recover active and recently-closed connections.

The windows.netscan Plugin

Volatility 3's windows.netscan plugin carves the memory image for network connection and socket objects. It works by scanning pool tags associated with _TCP_ENDPOINT, _TCP_LISTENER, and _UDP_ENDPOINT kernel structures, then parsing each structure for address and process-ownership data.

# Volatility 3 — scan for all network artifacts in a RAM image python3 vol.py -f memory.raw windows.netscan # Filter output to a specific destination port (e.g., 443) python3 vol.py -f memory.raw windows.netscan | grep ":443" # Export to CSV for timeline correlation with other artifact sources python3 vol.py -f memory.raw windows.netscan --output csv -o netscan_results.csv

Output Columns Explained

TCP Connection States

Understanding TCP state helps distinguish active exfiltration from residual artifacts of past activity.

StateMeaningForensic Interpretation
ESTABLISHED Active, bidirectional data transfer in progress Connection was live at capture time — highest priority for investigation
LISTEN Process waiting for incoming connections on a port Unexpected listeners (e.g., ports 4444, 5900, 3389 on non-RDP systems) suggest backdoors or remote access tools
CLOSE_WAIT Remote side closed the connection; local side has not yet May indicate a completed C2 session or interrupted data transfer — destination IP is still recoverable
TIME_WAIT Connection closing; waiting to ensure remote received the final ACK Residual of a recently completed session — timestamps and addresses remain valid evidence
SYN_SENT Local host sent SYN; awaiting remote response Attempted outbound connection — may indicate scanning or a C2 check-in that has not yet succeeded
CLOSED Connection terminated; structure still resident in RAM Historical evidence; timestamps and addresses document past activity

What to Look For

  • Process–port mismatch: A process like notepad.exe or calc.exe that owns any network socket is immediately suspicious.
  • Non-standard ports: Common C2 frameworks use ports 4444, 1234, 8080, or high-range ephemeral ports for reverse shells. Port 4444 is the Metasploit meterpreter default.
  • Multiple simultaneous ESTABLISHED connections to the same foreign IP from a single process — a potential data exfiltration pattern.
  • Connections to RFC 1918 private ranges from a server that has no business communicating internally — a lateral movement indicator.
  • Loopback abuse: Some malware communicates over 127.0.0.1 between co-resident injected components, avoiding network visibility.
  • UDP sockets with no state field: DNS over unusual ports, ICMP-tunneling artifacts, or data exfiltration using raw UDP streams.

Legacy Volatility 2 Plugin Reference

If you encounter Volatility 2 plugin names in practitioner write-ups, the equivalents are:

  • connections — active TCP connections (Windows XP / Server 2003 only)

2 Registry Artifacts in Memory

The Windows Registry is not a single file on disk. Active hives are loaded into kernel memory and managed by the Windows Registry Manager. Volatility reads these in-memory hive structures directly, giving examiners access to the full registry state at the exact moment of capture — including volatile keys that are built dynamically at runtime and are never written to any disk file.

🔗
Prior-Week Bridge — Week 7: Windows Registry Forensics

In Week 7 you examined registry hive files on disk — NTUSER.DAT, SAM, SYSTEM, SOFTWARE — using tools such as Registry Explorer and RegRipper. The plugins below are the in-memory counterpart: the same registry data, accessed through RAM rather than a mounted disk image. Many of the same key paths you studied in Week 7 (persistence, user activity, credentials) appear in both contexts, making cross-source comparison a powerful consistency check.

Plugin: windows.registry.hivelist

Before querying any specific registry key, use hivelist to enumerate all hives currently loaded in memory. The output provides the virtual memory offset of each hive — which most other registry plugins require as input.

# Enumerate all registry hives currently loaded in memory python3 vol.py -f memory.raw windows.registry.hivelist # Typical output (truncated): Offset FileFullPath 0xe38ba800 \REGISTRY\MACHINE\SYSTEM 0xe38d3800 \REGISTRY\MACHINE\SAM 0xe3901800 \REGISTRY\MACHINE\SOFTWARE 0xe3970800 \Device\HarddiskVolume2\Users\jsmith\NTUSER.DAT 0xf80001234800 \REGISTRY\MACHINE\HARDWARE # volatile — never exists on disk
⚠️
Volatile Hive: HARDWARE

The HARDWARE hive (highlighted above) is built dynamically at boot time by the Windows kernel. It is never saved to any disk file. It contains the current hardware configuration, device mappings, interrupt assignments, and firmware-provided ACPI table data. This hive is accessible only through a RAM image — it is completely invisible to disk forensics.

Plugin: windows.registry.printkey

printkey displays the contents of a specific registry key and all of its values. It can be invoked with a key path directly; specifying the hive offset from hivelist speeds up the search.

# Print the HKLM Run key — a primary persistence location python3 vol.py -f memory.raw windows.registry.printkey \ --key "SOFTWARE\Microsoft\Windows\CurrentVersion\Run" # Example output: Last Write Time 2026-03-21 14:22:07 Key: Run Value: updater Type: REG_SZ Data: C:\ProgramData\upd.exe --silent --connect 185.220.101.9 Value: OneDrive Type: REG_SZ Data: C:\Users\jsmith\AppData\Local\Microsoft\OneDrive\OneDrive.exe

In the example above, the entry named updater is suspicious: it points to a file in ProgramData (not a standard software installation path) and passes a remote IP address as a command-line argument. This is a classic persistence-via-Run-key pattern used by many RATs (Remote Access Trojans).

High-Value Key Paths for printkey Investigation

Key PathForensic Purpose
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run Programs that auto-launch at boot for all users — system-wide persistence
HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run Programs that auto-launch for a specific user — per-user persistence
HKLM\SYSTEM\CurrentControlSet\Services Installed services — malicious services or drivers register here for persistent execution
HKLM\SYSTEM\CurrentControlSet\Control\Session Manager BootExecute entries run before the Windows subsystem loads — high-privilege pre-boot persistence
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon Shell and notification package values; hijackable by malware to achieve persistence at user login

Plugin: windows.registry.userassist

UserAssist is a Windows feature that tracks recently executed programs to populate the Start menu's usage statistics. The data is stored in HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist and is ROT-13 encoded to obscure program paths from casual inspection. Volatility decodes it automatically.

python3 vol.py -f memory.raw windows.registry.userassist # Output includes decoded path, execution count, last run time, and focus time # Volatility handles ROT-13 decoding automatically: C:\Users\jsmith\Desktop\mimikatz.exe Count: 3 LastRun: 2026-03-21 14:19:04 C:\Windows\System32\cmd.exe Count: 8 LastRun: 2026-03-21 14:20:51
💡
UserAssist Forensic Value

UserAssist provides execution evidence — it demonstrates that a user (or malware running in user context) actually launched a program. This is especially valuable when the executable has since been deleted from disk, because the execution artifact survives in the registry and therefore in the RAM image. Cross-reference UserAssist timestamps against the logon/logoff windows captured in Windows Event Log data studied in Week 8.

Plugin: windows.hashdump

hashdump extracts NTLM password hashes from the SAM (Security Account Manager) hive loaded in memory. The SAM hive on disk is encrypted by the boot key stored in the SYSTEM hive. Volatility reconstructs this decryption using both hives as they exist in the RAM image.

python3 vol.py -f memory.raw windows.hashdump # Output format: username : RID : LM-hash : NTLM-hash ::: Administrator:500:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0::: jsmith:1001:aad3b435b51404eeaad3b435b51404ee:fc525c9683e8fe067095ba2ddc971889::: backdoor:1002:aad3b435b51404eeaad3b435b51404ee:e19ccf75ee54e06b06a5907af13cef42:::

The LM hash value aad3b435b51404ee is the "null" LM hash, indicating LM authentication is disabled on this system (expected on modern Windows). NTLM hashes can be cracked offline or compared against precomputed rainbow tables. The account named backdoor (RID 1002) is a strong indicator of unauthorized account creation by an attacker.

🔒
Legal and Ethical Use of hashdump

Hash extraction from a live system without authorization may violate the Computer Fraud and Abuse Act (CFAA) and similar statutes. In this course, windows.hashdump is applied only to memory images acquired as part of an authorized forensic investigation or to instructor-provided lab images. Never use this technique on systems you do not own or lack explicit written authorization to examine.

Volatile Registry Keys — Data That Never Reaches Disk

Several registry hives and key subtrees are built dynamically at runtime and are never flushed to a disk file. They are accessible only through a memory image captured from a running system:

HKLM\HARDWARE
Kernel-populated hardware device map rebuilt at every boot. Contains PnP device enumeration, IRQ assignments, memory-mapped I/O ranges, and firmware-provided ACPI tables. No corresponding disk file exists.
HKEY_PERFORMANCE_DATA
Performance counter data read directly from the kernel subsystem. Never stored in any hive file; only visible through a live system call or a RAM image.
HKEY_CURRENT_CONFIG (HKCC)
Runtime-computed hardware profile derived from the SYSTEM hive at boot. The in-memory version may differ from the on-disk SYSTEM hive if hardware changes occurred post-boot.
RunMRU (volatile in-session updates)
Recently-run entries from the Windows Run dialog are updated in memory before being flushed to NTUSER.DAT on disk. A RAM image may contain entries absent from the disk hive file.
Session-based hive modifications
Registry writes made by processes during a session are cached in memory and flushed periodically. A crash or forced acquisition may capture in-memory state that diverges from the last flushed disk state — this discrepancy is itself a forensic finding.

4 The Six-Phase Memory Analysis Workflow

Running plugins at random produces an overwhelming, unstructured mass of data with no defensible chain of reasoning. The six-phase workflow below provides a repeatable, structured methodology that moves from high-level image verification, through progressive triage, to deep analysis and documentation. Each phase builds on findings from the previous one and produces artifacts consumed in subsequent phases.

Goal: Confirm image integrity and identify the OS environment before any investigative plugin is run.

Before analysis begins, verify that the memory image is intact and identify the target operating system, architecture, and build number. Running plugins against an incorrect environment produces meaningless or misleading output. Record system time at acquisition — this anchors all subsequent timestamps.

Key Plugins

windows.info banners.Banners windows.kpcr
# Confirm Windows build, architecture, KPCR addresses, system time python3 vol.py -f memory.raw windows.info # Detect OS banner strings (useful for Linux / mixed images) python3 vol.py -f memory.raw banners.Banners

Record: OS name and build number, architecture (x64/x86), system time at acquisition, and number of processors. These facts are the foundation for all subsequent plugin selection and output interpretation.

Goal: Build a complete process inventory and identify anomalies by cross-referencing list-walk and pool-scan outputs.

Process triage is the first investigative step after profiling. Run both windows.pslist (walks the doubly-linked EPROCESS list) and windows.psscan (pool-tag carving). Compare their outputs: processes present in psscan but absent from pslist may have been unlinked via DKOM (Direct Kernel Object Manipulation) to hide from the operating system — and from defenders.

Key Plugins

windows.pslist windows.psscan windows.pstree windows.cmdline windows.dlllist

Process Triage Checklist

  • Verify expected parent–child relationships (all svchost.exe instances must be children of services.exe)
  • Flag process names that mimic system processes by one character — e.g., svch0st.exe, lssas.exe
  • Identify orphaned processes whose parent PID no longer exists
  • Run windows.cmdline on all processes — look for Base64-encoded arguments or unusual paths
  • Count singleton processes: only one lsass.exe, one wininit.exe, one csrss.exe per session
  • Note processes started after an unusual hour or outside typical usage windows
Goal: Identify all network activity captured at acquisition time and cross-reference every socket to the process list from Phase 2.

Network triage maps every socket to its owning process. Prioritize ESTABLISHED and LISTEN entries. Any connection whose owning process was already flagged in Phase 2 is immediately elevated to a high-priority indicator and drives the focus of Phase 4.

Key Plugin

windows.netscan

Network Triage Checklist

  • List all ESTABLISHED connections; note foreign IPs and ports for offline threat intelligence review
  • List all LISTEN entries; compare against the known service baseline for this system type
  • Cross-reference every PID in the netscan output with the flagged process list from Phase 2
  • Record all connection timestamps and add them to the growing investigation timeline
  • Flag any process that owns both an ESTABLISHED external connection AND an internal SMB/CIFS connection simultaneously — lateral movement pattern
Goal: Perform detailed examination of flagged processes for injected code, open file/handle artifacts, and behavioral evidence.

For each process flagged in Phases 2 or 3, apply injection-detection and artifact-recovery plugins. This phase deliberately narrows focus to flagged targets — running these plugins against all processes without prior triage generates excessive output and risks overlooking the critical findings.

Key Plugins

windows.malfind windows.handles windows.filescan windows.envars windows.memmap
  • malfind — flags memory regions with executable (PAGE_EXECUTE_*) permissions that have no backing disk file; a strong indicator of injected shellcode or a reflectively-loaded PE
  • handles — reveals mutex names (malware families often use hardcoded mutexes to prevent double-infection), open registry keys, and shared memory sections
  • filescan — recovers all open file handles, including those whose backing files were deleted after being opened (the handle keeps the inode alive)
  • envars — inspect environment variables for embedded payloads or encoded C2 addresses passed at process creation

With suspect processes and connections confirmed, pivot to the registry to determine how the attacker achieved persistence and what accounts may have been compromised or created.

  • Run hivelist first to obtain hive offsets needed for targeted queries

Goal: Produce a reproducible, court-defensible record of every finding, its source, and the commands used to obtain it.

Documentation is not an afterthought — it should occur throughout the investigation. By Phase 6, all findings are consolidated, cross-verified, and exported in formats suitable for the case report and potential legal proceedings.

Documentation Checklist

  • Record the SHA-256 hash of the memory image at the start and end of the analysis session and confirm they match
  • Log every command run, with full flags and output, to a timestamped log file
  • Export all plugin results using Volatility's --output csv flag and hash each export immediately
  • Construct a unified timeline: process start times + network connection timestamps + registry Last Write Times + Event Log entries from Week 8
  • Write a short narrative for each finding: what was found, where in memory, what it means, and what follow-up action it warrants
  • Cross-reference all memory findings against corresponding disk artifacts using the correlation table in Section 5
ℹ️
Chain of Custody for Exported Artifacts

Every file you export during analysis must be hashed immediately upon creation and that hash recorded in your notes. If an exported CSV is later modified — even accidentally — the hash changes, creating a chain-of-custody gap that can undermine the admissibility of your findings in a legal context.

5 Cross-Source Correlation — Memory Meets Disk

No single forensic source tells the complete story. Memory forensics excels at capturing the live state of a system — running processes, open connections, credentials in use — while disk artifacts provide a longer historical record. Conversely, an attacker who thoroughly sanitizes disk-based logs may leave extensive traces in RAM. Disagreement Between Sources Is Also a Finding

When memory and disk artifacts agree, confidence in the finding increases. When they disagree — for example, the in-memory Run key contains an entry absent from the disk hive file — that discrepancy is itself a forensic finding. It may indicate that malware modified the registry in memory without ever flushing to disk, or that the disk image was acquired before the change propagated. Document the discrepancy and its possible explanations in your report.

6 Limitations of Memory Forensics

Memory forensics is a powerful technique, but experienced examiners understand its inherent constraints. Knowing these limitations helps you avoid over-interpreting evidence, articulate caveats clearly in reports, and design acquisition workflows that minimize their impact.

1. Incomplete Captures — RAM Is a Snapshot, Not a Recording

A RAM image captures a single point in time. If a malicious process completed its work and exited five minutes before acquisition, it will not appear in a live process scan. Network connections that terminated before capture will not appear in windows.netscan output — though residual pool structures may remain in unallocated memory pools if pages have not been reused.

  • Processes that complete in milliseconds (common in multi-stage malware droppers) leave no EPROCESS trace.
  • Encryption keys loaded, used, and immediately discarded are irrecoverable from a post-use image.
  • The elapsed time between the incident and the acquisition decision directly determines the completeness of recoverable evidence.

2. Acquisition Tool Footprint — The Observer Effect

Any tool run on a live system to acquire memory necessarily modifies that system's RAM. The acquisition tool itself must be loaded into memory, overwriting a small portion of the very data it is trying to preserve. This is sometimes called the "Heisenberg problem" of live forensics.

⚠️
Minimize Footprint During Acquisition

Best practice is to run the acquisition tool from a write-blocked USB device or a trusted network share — never install it to the disk of the examined system. Use purpose-built memory acquisition utilities (e.g., Magnet RAM Capture, WinPmem) that are designed to minimize their kernel footprint and document clearly which memory regions they occupy.

3. Anti-Forensic Techniques Targeting RAM Analysis

Sophisticated attackers are aware of memory forensics methodologies and actively attempt to defeat them:

  • DKOM (Direct Kernel Object Manipulation): Unlinking a process's EPROCESS entry from the PsActiveProcessHead list makes it invisible to windows.pslist. However, windows.psscan may still recover it via pool-tag carving, which is why running both is essential.
  • Memory wiping: Malware can zero or encrypt its own heap regions before exiting, destroying string artifacts, shellcode, and configuration data.
  • Timestamp manipulation: Overwriting EPROCESS creation and exit timestamps obscures the process timeline.
  • Pagefile offloading: Malware that forces its own pages to be swapped out before acquisition will be absent from the RAM image — though the pages may then appear in a pagefile examination.

4. OS Hardening and Kernel Protections

Modern operating systems increasingly restrict third-party access to raw memory, which directly affects forensic acquisition and analysis:

  • Windows Kernel Patch Protection (PatchGuard): Prevents modification of kernel structures on 64-bit Windows, which also restricts some live acquisition methods.
  • Virtualization-Based Security (VBS) / Credential Guard: Introduced in Windows 10 and standard in Windows 11 enterprise configurations. LSASS and domain credentials are moved to an isolated VTL 1 (Virtual Trust Level 1) region managed by the hypervisor. This region is not accessible to acquisition tools operating at VTL 0. The windows.hashdump plugin will not recover domain credential hashes from a VBS-protected system.
  • Secure Boot: Prevents unsigned bootloader and kernel code from executing, limiting the effectiveness of kernel-mode acquisition drivers that lack a valid signature.

5. ASLR — Address Space Layout Randomization

ASLR randomizes the base addresses of the stack, heap, and loaded modules in virtual memory at each process launch. Volatility 3 accounts for ASLR by reading data from kernel structures (which maintain the actual base addresses) rather than relying on hardcoded values. Forensic implications include:

  • Analysis scripts that assume fixed module base addresses will fail silently on ASLR-enabled systems, producing incorrect or missing results.
  • Reconstructing injected shellcode's API calls requires resolving the randomized base addresses of target modules — a more complex analysis step.
  • ASLR combined with DEP (Data Execution Prevention) limits the reliability of simple heap-spray pattern matching in memory images.

6. Partial or Fragmented Memory Images

Some acquisition tools skip hardware-mapped, firmware-reserved, or paged-out memory regions. Some enterprise configurations perform partial captures to reduce acquisition time. Volatility may generate parsing warnings or produce incomplete output when applied to such images. Always verify acquisition completeness by comparing the image file size against the reported total physical RAM. If there is a significant discrepancy, document it and note which regions were not captured.
