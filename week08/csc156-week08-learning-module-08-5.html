2. The Super-Timeline Concept

What Is a Super-Timeline?

A super-timeline is a single, chronologically sorted dataset that merges timestamped entries from every available artifact source on a forensic image. Instead of examining Event Logs in one tool, Prefetch in another, and Registry artifacts in a third, the examiner views all events in one unified stream.

📖 Definition

Super-timeline: A consolidated, time-ordered compilation of artifacts from multiple forensic sources (file system metadata, event logs, application artifacts, registry hives, etc.) that enables an examiner to see everything that happened on a system in chronological context.

Why Merge Everything?

  • Context: A single Event 4624 (logon) is informative; seeing a Prefetch hit for PSEXEC.EXE three seconds later, followed by a $UsnJrnl FILE_CREATE for payload.exe, tells a complete story.
  • Gap detection: If Event Logs show a logon but no corresponding Prefetch or file-system activity, the examiner may suspect log injection or timestomping.
  • Efficiency: Switching between tools for each artifact type is slow and error-prone. A super-timeline lets you scroll through time and see correlated events side by side.
  • Reporting: A single timeline supports clear, chronological narratives in forensic reports—essential for court presentations.

Data Sources That Feed a Super-Timeline

The following diagram shows the artifact families from Weeks 7–8 flowing into a unified timeline:

Event Logs (.evtx)→ Prefetch (.pf)→ LNK / Jump Lists→ Shellbags→ $MFT / $UsnJrnl / $LogFile→ Registry Hives→ Plaso (log2timeline)→ Super-Timeline CSV

Additional sources (browser history, SRUM database, AmCache, etc.) can also be included. The more sources you feed in, the richer the timeline.

Deep Dive: Timestamp Normalization

Different artifacts store timestamps in different formats and time zones:

  • Event Logs: UTC (stored in SystemTime attribute).
  • NTFS $MFT: UTC (64-bit FILETIME).
  • Prefetch: Local time (depends on system time-zone setting at execution).
  • Registry last-write times: UTC (64-bit FILETIME).

A super-timeline tool like Plaso normalizes all timestamps to UTC before merging. This is critical—without normalization, events from different sources would appear out of order. Always verify the time zone setting of the source system (check SYSTEM\CurrentControlSet\Control\TimeZoneInformation in the Registry) and confirm your tool is applying the correct offset.

3. Tool Walk-Through: log2timeline / Plaso

Plaso (Plaso Langar Að Safna Öllu—Icelandic for "Plaso is meant to collect everything") is an open-source, Python-based super-timeline engine. It includes two primary command-line tools:

  • log2timeline — ingests a forensic image (or mounted volume) and extracts timestamped events into a Plaso storage file (.plaso).
  • psort — reads the .plaso storage file, applies filters, and exports the timeline to CSV, JSON, or other formats.

Step-by-Step Commands

Step 1: Run log2timeline Against a Forensic Image

# Full parse (all parsers — can be slow on large images)
log2timeline.py --storage-file case001.plaso  /evidence/disk_image.E01

# Targeted parse (Windows-only parsers for faster processing)
log2timeline.py --parsers "win7,winevtx,prefetch,lnk,mft,usnjrnl,winreg" \
    --storage-file case001.plaso  /evidence/disk_image.E01

Key flags:

  • --storage-file — path for the output Plaso storage file.
  • --parsers — comma-separated list of parsers to enable. Omit this flag to run all parsers (comprehensive but slower).
  • The final positional argument is the forensic image or mount point.
⚠️ Processing Time

A full log2timeline run on a 256 GB image can take several hours. In lab environments, you will typically work with smaller images or use the --parsers flag to limit processing to relevant artifact types. Plan accordingly.

Step 2: Export with psort

# Export entire timeline to CSV (l2tcsv format)
psort.py -o l2tcsv -w timeline_full.csv  case001.plaso

# Export with a date filter (e.g., events on Feb 14, 2026)
psort.py -o l2tcsv -w timeline_filtered.csv  case001.plaso \
    "date > '2026-02-14 00:00:00' AND date < '2026-02-15 00:00:00'"

Key flags:

  • -o l2tcsv — output format (log2timeline CSV). Other options include json, xlsx, elastic.
  • -w — output file path.
  • The quoted string at the end is a Plaso filter expression (optional) to narrow the export by date, source type, or keyword.

Step 3: Load in Timeline Explorer

  1. Open Timeline Explorer.
  2. Go to File → Open and select the exported CSV.
  3. The tool displays all events in a sortable, filterable grid with columns for timestamp, source, source type, description, and more.

4. Pivot-Point Analysis

A super-timeline for even a modest forensic image can contain millions of entries. You cannot read them all. Pivot-point analysis is the technique examiners use to navigate a large timeline efficiently.

What Is a Pivot Point?

📖 Definition

Pivot point: A known or suspected event of investigative significance that serves as an anchor in the timeline. The examiner starts at the pivot point and works forward and backward in time to discover related activity.

Common Pivot-Point Sources

  • Alert or incident report: "The IDS flagged outbound traffic at 02:47 UTC" → search the timeline for events at that timestamp.
  • Known-bad indicator: A malware filename (svch0st.exe), an IP address, or a username → search the timeline for all entries containing that string.
  • Account event: Event 4720 (account created) or 4624 with unusual logon type → anchor on that event and explore surrounding activity.
  • File-system event: A $UsnJrnl FILE_CREATE for a suspicious file → look backward to see how it arrived, forward to see what it did.

The Forward-Backward Technique

  1. Identify the pivot point in the timeline (e.g., filter for the known-bad filename).
  2. Work backward (earlier timestamps):
    • How did the attacker gain access? (Logon events, RDP connections)
    • Were any tools staged before the pivot event? (Prefetch, $UsnJrnl FILE_CREATE)
    • Was there reconnaissance? (Shellbag entries for browsed folders, Event Log queries)
  3. Work forward (later timestamps):
    • What did the attacker do after the pivot event? (New services, new accounts, file exfiltration)
    • Did they cover their tracks? (Event 4719 audit policy change, $UsnJrnl FILE_DELETE, VSS deletion)
    • When did they leave? (Event 4634 logoff)
  4. Identify new pivot points discovered during steps 2–3, and repeat the process for each.
💡 Pro Tip

Keep a running list of pivot points as you analyze. Each new discovery (a second IP address, a renamed file, a new user account) becomes a new pivot. Experienced examiners often maintain a simple spreadsheet: Pivot Point | Timestamp | Source | Notes.

Walkthrough: Building the Narrative

Follow along and answer each embedded question before revealing the analysis. This mirrors the pivot-point technique you just learned.

Step 1: Identify the Pivot Point

The IT security alert flagged after-hours activity on February 19. Our first pivot point is Entry A—the logon event at 18:32 UTC.

Q1: Why is an interactive logon (Type 2) at 18:32 UTC significant?

Analysis: Logon Type 2 means David physically sat down at the workstation (or used a VM console). At 18:32 UTC, normal business hours are over. Combined with the knowledge that this is his second-to-last day, an after-hours physical logon is a strong behavioral indicator. Entry B confirms he received SeBackupPrivilege, which allows reading any file regardless of permissions—unusual for a standard engineering account.

Step 2: Work Forward — What Happened Next?

Q2: Entries C and D occur about 2 minutes after logon. What do they tell us?

Analysis: Entry C shows a new drive letter (E:) appeared in MountedDevices, and Entry D records a SanDisk Ultra USB device in the USBSTOR registry key. Together, these prove David connected a USB storage device approximately 2 minutes after logging on. The USBSTOR key also records the device serial number, which could be used to identify the specific device if recovered.

Q3: Entries F and G are Shellbag entries. What do they reveal about David's actions?

Analysis: Shellbags record folders the user browsed in Windows Explorer. Entry F shows David opened the root of E:\ (the USB drive), and Entry G shows he navigated into E:\Transfer. This folder name suggests premeditation—the folder likely already existed on the USB drive, created specifically for this purpose.

Q4: Entries H and I are $UsnJrnl records occurring 3 seconds apart. What story do they tell together?

Analysis: Entry H shows turbine_v4_FINAL.dwg was read from C:\Projects\Designs\ (the source). Three seconds later, Entry I shows the same filename was created on E:\Transfer\ with DATA_EXTEND (data written) and CLOSE. This is direct evidence of a file copy from the internal drive to the USB device. Entry K shows a second file (motor_controller_schematic.pdf) was copied ~5 minutes later.

Q5: Entry J is a LNK file, and Entry N shows it being deleted. What is the forensic significance?

Analysis: When David accessed turbine_v4_FINAL.dwg, Windows automatically created a Recent LNK file (Entry J) pointing to the design file. Later (Entry N), David deleted that LNK file—an attempt to remove evidence of the file access. However, the $UsnJrnl recorded both the creation and the deletion, so the evidence persists even though the LNK file itself is gone. This is a textbook example of why NTFS journals are forensically valuable.

Q6: Entry M shows SDelete was executed. Why is this a critical finding?

Analysis: SDelete (Sysinternals) is a secure-deletion tool that overwrites file data before deleting it, making recovery extremely difficult. The Prefetch entry shows a run count of 1 and a first-execution timestamp of 18:55:45—meaning David ran this tool for the very first time immediately after unmounting the USB drive (Entry L at 18:55:12). This is strong evidence of anti-forensic activity: David attempted to destroy evidence of what he copied. The Prefetch file itself survives because SDelete does not delete its own Prefetch entry.

Q7: Summarize the complete narrative in chronological order.

Reconstructed Narrative:

  1. 18:32 UTC — David Chen logged on interactively to ENG-WS-14 after business hours, receiving elevated backup privileges.
  2. 18:34 UTC — He connected a SanDisk Ultra USB drive, which mounted as drive E:.
  3. 18:35–18:36 UTC — He browsed to E:\Transfer on the USB drive.
  4. 18:37–18:42 UTC — He copied at least two proprietary files (turbine_v4_FINAL.dwg and motor_controller_schematic.pdf) from C:\Projects\Designs\ to the USB drive.
  5. 18:55 UTC — He unmounted the USB drive and immediately ran SDelete for the first time, likely attempting to securely delete local traces of the copied files or the LNK shortcut.
  6. 18:56 UTC — The Recent LNK file for the design file was deleted (possibly by SDelete or manually).
  7. 19:02 UTC — David logged off. Total session: approximately 30 minutes.

Artifacts correlated: Event Logs (logon/logoff, privileges) + Registry (USB device, MountedDevices) + Shellbags (folder browsing) + $UsnJrnl (file copy, file deletion) + LNK (file access) + Prefetch (anti-forensic tool execution). No single artifact tells the full story—the super-timeline makes the correlation possible.

7. Practice Questions

The following 10 questions are styled like midterm questions. They cover Weeks 1–8 in mixed format (multiple-choice and scenario-based). Try each question, then click Reveal Answer to see the explanation.

1. (Week 2) An examiner arrives at a crime scene and must decide which evidence to collect first. According to the order of volatility, which of the following should be collected first?
  • A. Hard drive contents
  • B. Contents of RAM
  • C. Backup tapes stored offsite
  • D. Optical media (CDs/DVDs)
Answer: B. RAM is the most volatile—it is lost when power is removed. The order of volatility (RFC 3227) dictates collecting the most volatile evidence first: registers/cache → RAM → swap/pagefile → disk → removable media → backups.
2. (Week 3) A forensic examiner is asked to testify about their methodology in a federal court case. The judge will evaluate whether the examiner's techniques are scientifically valid. Which legal standard governs this determination?
  • A. The Exclusionary Rule
  • B. The Daubert Standard
  • C. The Best Evidence Rule
  • D. The Plain View Doctrine
Answer: B. The Daubert Standard (Daubert v. Merrell Dow Pharmaceuticals, 1993) requires the judge to act as a gatekeeper, evaluating whether expert testimony is based on sufficient facts, reliable methods, and proper application. Factors include testability, peer review, error rate, and general acceptance.
3. (Week 4) An examiner is imaging an SSD that supports TRIM. Why does TRIM pose a challenge for forensic recovery?
  • A. TRIM encrypts deleted file data, making it unreadable.
  • B. TRIM instructs the SSD controller to zero out blocks associated with deleted files, potentially destroying data before the examiner can image the drive.
  • C. TRIM increases the file system's cluster size, reducing slack space.
  • D. TRIM prevents write blockers from functioning correctly.
Answer: B. When a file is deleted on a TRIM-enabled SSD, the OS sends a TRIM command to the controller, which may zero out (or garbage-collect) the underlying NAND pages. Unlike HDDs where deleted data persists until overwritten, TRIM can cause data destruction almost immediately, making timely acquisition critical.
4. (Week 5) In NTFS, the $MFT stores two sets of timestamps: STANDARD_INFORMATION (SI) and FILE_NAME (FN). An examiner notices that the SI "Created" timestamp is newer than the FN "Created" timestamp for the same file. What does this most likely indicate?
  • A. The file was recently defragmented.
  • B. The file's timestamps were manipulated (timestomping).
  • C. The file was moved from a FAT32 partition.
  • D. Normal Windows behavior during a system update.
Answer: B. SI timestamps can be modified by user-level tools (timestomping), but FN timestamps can only be modified by the kernel. If SI "Created" is newer than FN "Created," the SI timestamp was likely altered after the file was created—a classic indicator of timestomping. This is why comparing SI and FN timestamps is a key anti-forensic detection technique (Module 8.4).
5. (Week 6) An examiner uses FTK Imager to create an E01 image of a suspect's hard drive. After imaging, the SHA-256 hash of the image does not match the SHA-256 hash of the original drive. What should the examiner do?
  • A. Proceed with the analysis using the image; hash mismatches are common.
  • B. Re-image the drive, verify the write blocker is functioning, and re-hash. Document the discrepancy.
  • C. Use the image anyway but note the mismatch in the report as a minor issue.
  • D. Switch to MD5 hashing, which is more tolerant of minor errors.
Answer: B. A hash mismatch means the image is not a bit-for-bit copy of the original—the evidence integrity is compromised. The examiner must investigate the cause (failing write blocker, bad sectors, imaging error), re-image with a verified write blocker, and document everything. Using a mismatched image risks having the evidence excluded in court.
6. (Week 7) An examiner finds the following Registry key path on a suspect's system: NTUSER.DAT\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist. What type of evidence does this artifact provide?
  • A. A list of all files on the system and their hash values.
  • B. A record of programs launched by the user via the GUI, including run count and last execution time.
  • C. A log of all network connections made by the system.
  • D. A list of USB devices connected to the system.
Answer: B. UserAssist records programs launched through the Windows shell (Explorer GUI). Each entry includes a ROT13-encoded program path, a run count, and a last-execution timestamp. It does not record command-line executions. This artifact is valuable for proving a user intentionally launched a specific application.
7. (Week 8) A Prefetch file named MIMIKATZ.EXE-1A2B3C4D.pf is found on a system. The run count is 3 and the last execution time is 2026-02-18 at 14:22 UTC. Which of the following statements is most accurate?
  • A. Mimikatz was executed exactly 3 times, and the Prefetch file proves which user ran it.
  • B. Mimikatz was executed at least 3 times, the last being at 14:22 UTC, but the Prefetch file alone does not identify which user ran it.
  • C. The Prefetch file only proves Mimikatz was present on the system, not that it was executed.
  • D. Prefetch files are unreliable and should not be used as evidence.
Answer: B. Prefetch files record execution (not just presence), run count, and the last 8 run times (on Windows 10/11). However, Prefetch does not record which user executed the program. To determine the user, you must correlate with Event Logs (logon sessions active at 14:22 UTC) or UserAssist entries. Option C describes ShimCache, which records file presence but not necessarily execution.
8. (Weeks 2 & 6 — Scenario) You are called to image a running server suspected of hosting stolen data. The server cannot be shut down because it provides critical services to the organization. Describe the type of acquisition you would perform and explain two precautions you must take.

Model Answer: You would perform a live acquisition (the system remains powered on). Two key precautions:

  1. Capture volatile data first (RAM, network connections, running processes) following the order of volatility, because this data will be lost if the system is later shut down.
  2. Use a write blocker or forensic imaging tool that operates in read-only mode (e.g., FTK Imager Lite from a USB) to minimize changes to the live system. Document every action taken, including the tools used and their hashes, because a live acquisition inherently modifies some system state (e.g., loading the imaging tool into RAM).
9. (Week 8 — Scenario) An examiner sees Event ID 7045 in the System log showing a new service named WinDefendHelper installed with an image path of C:\ProgramData\Temp\svc.exe running as LocalSystem. Event 4624 (LogonType 3) from IP 10.10.5.99 occurred 12 seconds earlier. What conclusions can you draw?

Model Answer:

  • Event 7045 indicates a new service was installed. The name mimics Windows Defender (deceptive naming), the executable is in a temp directory (unusual for legitimate services), and it runs as LocalSystem (maximum privileges)—all red flags for malware or attacker persistence.
  • Event 4624 with LogonType 3 (Network logon) from 10.10.5.99 twelve seconds prior suggests lateral movement—an attacker on another machine used network authentication (possibly PsExec or similar) to access this system and install the service remotely.
  • Next steps: Investigate 10.10.5.99; preserve svc.exe for malware analysis; check Prefetch for execution evidence; search $UsnJrnl for file-creation records; check for Event 4672 (elevated privileges) in the same logon session.
10. (Week 8 — Integration) Explain why a super-timeline is more valuable than examining each artifact type in isolation. In your answer, give one specific example of a correlation that would be invisible without a merged timeline.

Model Answer: A super-timeline merges all timestamped artifacts into a single chronological view, allowing the examiner to see relationships between events that occur seconds apart across different artifact sources. Without merging, an examiner might see a Prefetch entry for a malicious tool in one tool and an Event Log logon in another, without realizing they occurred within the same 10-second window.

Example: A $UsnJrnl FILE_CREATE record for payload.exe at 14:22:05 UTC, followed by a Prefetch hit for PAYLOAD.EXE at 14:22:08 UTC, followed by Event 7045 (new service installed pointing to payload.exe) at 14:22:10 UTC. This 5-second sequence—file dropped, executed, installed as a service—tells a complete attack story that would be nearly impossible to reconstruct by examining each artifact type separately in different tools.

3. Common Pitfalls to Avoid

  • Confusing ShimCache and Prefetch: ShimCache records file presence (the file was shimmed by the compatibility engine); Prefetch records execution. This distinction is a common exam question.

  • Overlooking SI vs. FN timestamps: Understand why comparing STANDARD_INFORMATION and FILE_NAME timestamps detects timestomping.

9. Integrative Knowledge Check

These ungraded questions require you to identify which artifact(s) would answer a given investigative question—drawing on all Week 8 modules and Week 7 concepts. Select the best answer and click Check Answer for immediate feedback.

1. An investigator needs to determine whether a specific program (data_dump.exe) was ever executed on a Windows 10 system. Which artifact provides the strongest direct evidence of execution?
✓ Correct! Prefetch is the strongest direct evidence of execution on Windows 10. It records the program name, run count, and last 8 execution times. ShimCache (A) records file presence, not necessarily execution. $UsnJrnl (C) records file creation, not execution. LNK files (D) record file access, not program execution.
✗ Not quite. The correct answer is B (Prefetch). Prefetch files are created when a program executes, recording run count and execution times. ShimCache records file presence (not necessarily execution), $UsnJrnl records file-system operations, and LNK files record file access.
2. An examiner wants to prove that a user browsed to a specific folder on a USB drive, even though the folder has since been deleted. Which artifact is most likely to retain this evidence?
✓ Correct! Shellbags record every folder a user browsed in Windows Explorer, including folders on removable drives and network shares. Crucially, Shellbag entries persist even after the folder is deleted or the drive is removed. They are stored in the user's UsrClass.dat (and NTUSER.DAT) registry hives.
✗ Not quite. The correct answer is C (Shellbags). Shellbags record folder navigation in Explorer and persist after deletion. Event Logs don't typically record folder browsing. Prefetch records program execution, not folder navigation. AmCache records program installation/execution metadata.
3. You need to determine the exact source IP address of a Remote Desktop session to a compromised workstation. Which artifact and field provide this information?
✓ Correct! Event 4624 with LogonType 10 (RemoteInteractive) records the source IP address in the IpAddress field. This is the definitive artifact for identifying where an RDP session originated. Prefetch for mstsc.exe would appear on the source machine (where the user initiated the connection), not the target.
✗ Not quite. The correct answer is A. Only the Security Event Log (Event 4624, LogonType 10) captures the source IP of an RDP session on the target machine. Prefetch for mstsc.exe would be on the source machine. $UsnJrnl and UserAssist do not record network connection information.
4. An attacker deleted a file and then used a timestomping tool to alter the STANDARD_INFORMATION timestamps of other files. Which combination of artifacts would best detect both the deletion and the timestomping?
✓ Correct! The $UsnJrnl records FILE_DELETE operations with timestamps, providing evidence of the deletion. Comparing $MFT STANDARD_INFORMATION timestamps against FILE_NAME timestamps reveals timestomping because SI timestamps can be modified by user tools, but FN timestamps are kernel-controlled. This combination directly addresses both anti-forensic techniques.
✗ Not quite. The correct answer is D. The $UsnJrnl records file deletions (FILE_DELETE reason code), and comparing $MFT SI vs. FN timestamps is the standard technique for detecting timestomping. The other combinations either don't detect deletions reliably (B) or don't detect timestomping (A, C).
5. A forensic examiner has a super-timeline with 2.3 million entries. The investigation concerns a suspected data exfiltration that occurred "sometime last Tuesday evening." What is the examiner's best first step for navigating this timeline?
✓ Correct! Pivot-point analysis is the standard technique for navigating large timelines. Filter to the approximate time window, identify a key event (pivot point), then work forward and backward to build the narrative. Reading from the beginning (A) is impractical with millions of entries. Deleting entries (C) or limiting to one source (D) destroys valuable correlation data.
✗ Not quite. The correct answer is B. Pivot-point analysis (Section 4 of this module) is the key technique: filter by date, find a pivot event, and work forward/backward. Never delete timeline entries (C)—filter instead. Limiting to one artifact type (D) defeats the purpose of a super-timeline.
6. An investigator needs to recover a previous version of a document that was modified by the suspect two days ago. The current version on disk has been altered. Assuming no backups exist, which artifact offers the best chance of recovering the original version?
✓ Correct! Volume Shadow Copies store point-in-time snapshots of files. If a VSS snapshot was taken before the modification, the examiner can mount the shadow copy and recover the original version of the document. $UsnJrnl (A) records that the file changed but does not store file content. Prefetch (B) and RecentDocs (D) do not store file content either.
✗ Not quite. The correct answer is C (Volume Shadow Copy). VSS stores differential snapshots that can contain previous versions of files. The $UsnJrnl records metadata changes but not file content. Prefetch and RecentDocs record execution/access metadata, not file content.
