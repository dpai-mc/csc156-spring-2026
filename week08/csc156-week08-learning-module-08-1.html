Event Log Architecture

Modern Windows systems (Vista / Server 2008 and later) use the Windows Event Log service (wevtsvc) to collect, store, and manage event data. Understanding the architecture is essential before diving into specific events.

The .evtx File Format

Event data is persisted in binary files with the .evtx extension. Each file is internally structured as a sequence of chunks (64 KB each), and each chunk contains individual event records stored in a binary XML representation.

📂 Default Storage Location

C:\Windows\System32\winevt\Logs\

This directory typically contains dozens of .evtx files—one per channel. In a forensic image, you will navigate to this path within the mounted volume to extract the logs.

Key structural facts about .evtx

  • File header: Contains a magic number (ElfFile\x00), chunk count, and flags indicating whether the log was "dirty" (not cleanly closed).
  • Chunk header: Each 64 KB chunk has its own header with first/last event record numbers and checksums.
  • Event records: Stored in Binary XML (BinXml)—a Microsoft-proprietary compact XML encoding. When you view an event in Event Viewer, the tool deserializes BinXml back into readable XML.
  • Circular logging: By default, once a log file reaches its maximum size, the oldest records are overwritten. This means critical evidence can be lost if logs are not collected promptly.
Deep Dive: Inside an Event Record

Each event record contains a standard set of system fields plus an event-specific data payload:

FieldDescription
EventIDNumeric identifier for the event type (e.g., 4624).
LevelSeverity: Information (4), Warning (3), Error (2), Critical (1).
TimeCreatedUTC timestamp of when the event was generated.
ProviderThe source component that logged the event (e.g., Microsoft-Windows-Security-Auditing).
ChannelThe log channel the event belongs to (e.g., Security).
ComputerThe hostname of the machine that generated the event.
EventRecordIDSequential record number within the log file.
EventData / UserDataEvent-specific payload (e.g., for Event 4624: target username, logon type, source IP).

Here is a simplified XML representation of a Security event record:

<Event xmlns="http://schemas.microsoft.com/win/2004/08/events/event">
  <System>
    <Provider Name="Microsoft-Windows-Security-Auditing" Guid="{...}" />
    <EventID>4624</EventID>
    <Level>0</Level>
    <TimeCreated SystemTime="2026-02-15T08:23:41.1234567Z" />
    <Computer>WORKSTATION-PC</Computer>
    <EventRecordID>108754</EventRecordID>
    <Channel>Security</Channel>
  </System>
  <EventData>
    <Data Name="TargetUserName">jsmith</Data>
    <Data Name="LogonType">10</Data>
    <Data Name="IpAddress">192.168.1.55</Data>
    <Data Name="IpPort">49823</Data>
    ...
  </EventData>
</Event>

The Three Main Channels

While Windows maintains many specialized log channels, forensic examiners focus primarily on three:

Security Log (Security.evtx)

Provider: Microsoft-Windows-Security-Auditing

The Security log is the single most important log for forensic investigations. It records events generated by the Windows Local Security Authority (LSA) based on the system's audit policy configuration.

Key event categories:

  • Logon / Logoff — successful and failed authentication attempts (Event IDs 4624, 4625, 4634).
  • Account Management — user/group creation, deletion, modification (4720, 4726, 4732, 4733).
  • Privilege Use — special privileges assigned at logon (4672).
  • Object Access — file/folder/registry access when auditing is enabled (4663).
  • Policy Changes — modifications to audit policy (4719).
⚠️ Audit Policy Dependency

The Security log only records events that the audit policy is configured to capture. If "Audit Logon Events" is not enabled, Event 4624/4625 will not appear. An attacker who disables audit policies (Event 4719) can effectively blind the Security log. Always check the audit policy configuration as part of your investigation.

System Log (System.evtx)

Provider: Various system components (kernel, drivers, Service Control Manager).

The System log records events from Windows system components, including hardware drivers, the kernel, and the Service Control Manager (SCM). Forensically relevant events include:

  • Service Control Manager events (7034–7036, 7045) — service start, stop, crash, and new service installation.
  • Windows Update events — patches installed (useful for establishing system state at a point in time).
  • Unexpected shutdown / blue screen events (6008, 1001) — can indicate system instability or forced shutdown.
  • Time change events (1) — system clock modifications that could indicate anti-forensic activity.

Event 7045 (new service installed) is particularly important: attackers frequently install malicious services for persistence or lateral movement. Tools like PsExec create a service (PSEXESVC) on the remote machine, which is logged here.

Application Log (Application.evtx)

Provider: Individual applications and services.

The Application log captures events from user-mode applications and services. While generally less critical than Security or System logs for forensics, it can provide valuable context:

  • Application crashes (1000, 1002) — may indicate exploitation attempts or malware instability.
  • Windows Error Reporting (WER) — crash dump references that can be analyzed separately.
  • Antivirus/EDR alerts — many security products log detections here.
  • MSI Installer events (11707, 11724) — software installation and removal records.

In malware investigations, correlating Application log crash events with Security log logon events can reveal when a malicious payload was executed and by which user.

Additional Channels of Forensic Interest

Beyond the big three, several specialized channels are valuable in certain investigations:

  • Microsoft-Windows-PowerShell/Operational — PowerShell script block logging (Event 4104) captures the full text of executed scripts. Critical for detecting fileless malware.
  • Microsoft-Windows-TaskScheduler/Operational — scheduled task creation and execution. Attackers use scheduled tasks for persistence.
  • Microsoft-Windows-TerminalServices-RemoteConnectionManager/Operational — RDP connection attempts (Event 1149).
  • Microsoft-Windows-Sysmon/Operational — if Sysmon is installed, provides detailed process creation, network connection, and file-creation events.
  • Microsoft-Windows-Windows Defender/Operational — malware detection and remediation events.

Tool Walk-Through: Windows Event Viewer

Windows Event Viewer (eventvwr.msc) is the built-in GUI tool for browsing event logs. While not the most efficient tool for large-scale forensic analysis, it is universally available on Windows systems and useful for quick triage.

Step-by-Step: Navigating Event Viewer

🛠️ Hands-On Walkthrough

Follow along on your forensic workstation or a Windows VM. You do not need a forensic image for this walkthrough—you can examine the live logs on your own system to practice navigation.

Step 1: Launch Event Viewer

  • Press Win + R, type eventvwr.msc, and press Enter.
  • Alternatively, right-click the Start button and select Event Viewer.

Step 2: Browse the Main Channels

  • In the left pane, expand Windows Logs.
  • You will see the three main channels: Application, Security, and System (plus Setup and Forwarded Events).
  • Click Security to load the Security log. Note: this may take a moment on systems with large logs.

Step 3: Examine an Individual Event

  • Click any event in the center pane to see its details in the lower preview.
  • Switch between the General tab (human-readable summary) and the Details tab (raw XML view).
  • On the Details tab, select XML View to see the full BinXml-deserialized record. This is the same structure parsed by forensic tools.

Step 4: Filter the Current Log

  1. With the Security log selected, click Filter Current Log… in the right-hand Actions pane.
  2. In the dialog, enter 4624 in the <All Event IDs> field.
  3. Click OK. The log now shows only successful logon events.
  4. To filter for multiple IDs, separate them with commas: 4624,4625,4672.

Step 5: Create a Custom View

  1. In the Actions pane, click Create Custom View….
  2. Set the time range using the Logged dropdown (e.g., "Last 24 hours" or a custom range).
  3. Under Event logs, check Security.
  4. Enter the Event IDs you want: 4624,4625,4672,4720,4726.
  5. Give the view a name (e.g., "Forensic – Logon Activity") and click OK.
  6. Your custom view now appears under Custom Views in the left pane and persists across sessions.

Step 6: Open a Saved .evtx File

In forensic work, you will often need to examine .evtx files extracted from a forensic image rather than the live system logs:

  1. Go to Action → Open Saved Log….
  2. Browse to the extracted .evtx file (e.g., Security.evtx from the forensic image).
  3. Event Viewer will load the file in a new node under Saved Logs.
  4. You can now filter and browse the extracted log exactly as you would a live log.
⚠️ Limitations of Event Viewer for Forensics
  • Slow with very large log files (millions of events).
  • No built-in export to CSV/JSON for timeline integration.
  • Cannot merge multiple log files into a single view.
  • Limited query capability compared to command-line tools.

For serious forensic work, use Event Viewer for quick triage and then switch to EvtxECmd for comprehensive parsing and export.

Tool Walk-Through: EvtxECmd

EvtxECmd is part of Eric Zimmerman's open-source forensic tool suite. It parses .evtx files from the command line and exports structured data to CSV, JSON, or XML—making it ideal for forensic analysis and timeline construction.

Why EvtxECmd?

  • Speed: Parses millions of events in seconds.
  • Structured output: CSV output loads directly into Timeline Explorer, Excel, or any data-analysis tool.
  • Event maps: Built-in "maps" that extract event-specific data fields (e.g., the source IP from Event 4624) into standardized columns.
  • Batch processing: Can process an entire directory of .evtx files in one command.

Step-by-Step: Parsing with EvtxECmd

Step 1: Parse a Single Log File

EvtxECmd.exe -f "C:\Evidence\Security.evtx" --csv "C:\Output" --csvf "Security_parsed.csv"

Flags explained:

  • -f — path to a single .evtx file.
  • --csv — output directory for the CSV file.
  • --csvf — filename for the CSV output.

Step 2: Parse an Entire Directory of Logs

EvtxECmd.exe -d "C:\Evidence\winevt\Logs" --csv "C:\Output" --csvf "AllLogs_parsed.csv"

The -d flag processes every .evtx file in the directory. This is the most common forensic workflow—you extract the entire winevt\Logs folder from the forensic image and parse everything at once.

Step 3: Load Results in Timeline Explorer

  1. Open Timeline Explorer (another Eric Zimmerman tool).
  2. Go to File → Open and select the CSV file generated by EvtxECmd.
  3. Timeline Explorer displays the data in a sortable, filterable grid.

Step 4: Key Columns in EvtxECmd Output

Important EvtxECmd CSV Columns
ColumnDescription
TimeCreatedUTC timestamp of the event.
EventIdThe numeric Event ID.
ChannelWhich log the event came from (Security, System, etc.).
ComputerHostname that generated the event.
MapDescriptionHuman-readable event name (from the event map, e.g., "Successful Logon").
UserNameExtracted target or subject username (when available from the map).
RemoteHostSource IP or hostname (for logon events).
PayloadData1–6Additional event-specific fields extracted by the map (e.g., LogonType, privilege list).
PayloadFull XML payload of the event (for fields not covered by the map).

Step 5: Filtering in Timeline Explorer

Timeline Explorer supports column-level filters. Common forensic workflows:

  • Click the filter icon on the EventId column → type 4624 to show only successful logons.
  • Filter RemoteHost to a specific IP address to see all activity from that source.
  • Sort by TimeCreated to establish chronological order.
  • Use Ctrl+F to search across all columns for a username, IP, or keyword.
Tip: Keeping Event Maps Updated

EvtxECmd uses event map files (YAML) to extract event-specific fields into the standardized columns. Eric Zimmerman regularly updates these maps. Before a forensic examination, update your maps:

EvtxECmd.exe --sync

This downloads the latest map definitions from the project repository, ensuring you get the best field extraction for newer Event IDs.

Mini-Case: Unauthorized Logon Investigation

🔍 Scenario: After-Hours Logon on ACCT-PC

You are a forensic examiner investigating a potential unauthorized access incident at a mid-size accounting firm. The IT department flagged an alert: the workstation ACCT-PC (assigned to employee Maria Torres, username mtorres) showed network activity at 02:47 AM on Saturday, February 14, 2026—well outside normal business hours. Maria was confirmed to be on vacation that week.

You have extracted the Security.evtx file from a forensic image of ACCT-PC and parsed it with EvtxECmd. Below are selected event records from the relevant time window.

Evidence: Parsed Event Log Entries

Record 1:
  TimeCreated:  2026-02-14T02:47:12Z
  EventId:      4624
  Channel:      Security
  Computer:     ACCT-PC
  UserName:     mtorres
  LogonType:    10
  RemoteHost:   10.0.50.200
  LogonId:      0x1A3F7C2

Record 2:
  TimeCreated:  2026-02-14T02:47:12Z
  EventId:      4672
  Channel:      Security
  Computer:     ACCT-PC
  UserName:     mtorres
  Privileges:   SeBackupPrivilege, SeRestorePrivilege, SeDebugPrivilege

Record 3:
  TimeCreated:  2026-02-14T02:52:38Z
  EventId:      7045
  Channel:      System
  Computer:     ACCT-PC
  ServiceName:  SysHealthCheck
  ImagePath:    C:\Windows\Temp\svch0st.exe
  AccountName:  LocalSystem

Record 4:
  TimeCreated:  2026-02-14T02:52:40Z
  EventId:      7036
  Channel:      System
  Computer:     ACCT-PC
  ServiceName:  SysHealthCheck
  State:        Running

Record 5:
  TimeCreated:  2026-02-14T03:15:22Z
  EventId:      4720
  Channel:      Security
  Computer:     ACCT-PC
  TargetUserName: svc_backup
  SubjectUserName: mtorres

Record 6:
  TimeCreated:  2026-02-14T03:15:45Z
  EventId:      4732
  Channel:      Security
  Computer:     ACCT-PC
  GroupName:    Administrators
  MemberName:   svc_backup
  SubjectUserName: mtorres

Record 7:
  TimeCreated:  2026-02-14T04:02:10Z
  EventId:      4634
  Channel:      Security
  Computer:     ACCT-PC
  UserName:     mtorres
  LogonId:      0x1A3F7C2

Investigative Questions

Use the event records above and your knowledge of Event IDs and Logon Types to answer the following questions. Click each question to reveal the answer and analysis.

Q1: How did the user access ACCT-PC?

Answer: Via Remote Desktop (RDP).

Analysis: Record 1 shows Event 4624 (successful logon) with LogonType: 10. Logon Type 10 is RemoteInteractive, which corresponds to an RDP session. The source IP is 10.0.50.200—this IP should be investigated to determine what device it belongs to (is it an internal workstation? a VPN endpoint? an unknown device?).

Q2: What privileges did the session receive, and is this concerning?

Answer: The session received SeBackupPrivilege, SeRestorePrivilege, and SeDebugPrivilege.

Analysis: Record 2 (Event 4672) shows special privileges assigned at logon. These are highly sensitive privileges:

  • SeBackupPrivilege — allows reading any file on the system, bypassing access controls (can exfiltrate any data).
  • SeRestorePrivilege — allows writing any file, bypassing access controls.
  • SeDebugPrivilege — allows attaching a debugger to any process, including LSASS (can dump credentials).

If mtorres is a standard accounting employee, she should not have these privileges. This strongly suggests her account has been compromised and either was already an administrator or was elevated by the attacker.

Q3: What happened approximately 5 minutes after logon?

Answer: A new service was installed and started.

Analysis: Record 3 (Event 7045) shows a new service named SysHealthCheck installed with the image path C:\Windows\Temp\svch0st.exe. Record 4 (Event 7036) confirms the service entered the Running state two seconds later.

Red flags:

  • The executable is in C:\Windows\Temp\—legitimate services are rarely installed from temp directories.
  • The filename svch0st.exe mimics svchost.exe (note the zero replacing the letter "o")—a classic attacker naming technique.
  • The service runs under LocalSystem, giving it the highest possible privileges.

This service should be treated as suspected malware and the executable preserved for analysis (see Week 13: Malware Forensics).

Q4: What account-management actions occurred, and why are they suspicious?

Answer: A new user account svc_backup was created and added to the local Administrators group.

Analysis: Record 5 (Event 4720) shows user mtorres created a new account named svc_backup. Record 6 (Event 4732) shows svc_backup was immediately added to the Administrators group.

This is a textbook persistence technique: the attacker creates a new admin account so they can regain access even if mtorres’s password is changed. The name svc_backup is designed to look like a legitimate service account.

Q5: How long was the session active, and how do you determine this?

Answer: Approximately 1 hour and 15 minutes (02:47 – 04:02 UTC).

Analysis: Record 1 (Event 4624) and Record 7 (Event 4634) share the same LogonId: 0x1A3F7C2. The LogonId is a unique session identifier that links logon and logoff events. By subtracting the timestamps: 04:02:10 – 02:47:12 = ~1 hr 15 min.

During this session, the attacker installed a malicious service, created a backdoor account, and added it to the Administrators group—all within 75 minutes.

Q6: What investigative steps would you recommend next?

Recommended next steps:

  1. Identify 10.0.50.200 — determine which device this IP belongs to (DHCP logs, ARP tables, asset inventory).
  2. Analyze svch0st.exe — extract the binary from the forensic image and perform static/dynamic malware analysis (Week 13 skills).
  3. Check for svc_backup logon events — search for Event 4624 with UserName: svc_backup to see if the attacker already used the backdoor account.
  4. Examine Prefetch files — look for a Prefetch entry for svch0st.exe to confirm execution and identify referenced files (Module 8.2).
  5. Check for audit policy changes — search for Event 4719 to see if the attacker attempted to disable logging.
  6. Build a timeline — correlate Event Logs with Prefetch, LNK files, and $UsnJrnl to construct a complete picture (Module 8.5).
  7. Preserve all evidence — ensure chain of custody documentation is complete (Week 2 procedures).

XPath Filtering

Both Windows Event Viewer and the wevtutil command-line tool support XPath 1.0 queries for filtering events. XPath queries operate on the XML structure of event records, giving you precise control over which events are returned.

Basic XPath Syntax for Event Logs

All Event Log XPath queries start from the <Event> root element. The general pattern is:

*[System[condition] and EventData[condition]]

Example 1: Filter by Event ID

*[System[EventID=4624]]

Returns all events with Event ID 4624 (successful logon).

Example 2: Filter by Multiple Event IDs

*[System[(EventID=4624) or (EventID=4625)]]

Returns both successful and failed logon events.

Example 3: Filter by Time Range

*[System[TimeCreated[@SystemTime>='2026-02-14T00:00:00Z' and @SystemTime<='2026-02-15T00:00:00Z']]]

Returns events from February 14, 2026 (UTC).

Example 4: Filter by Event Data Field

*[System[EventID=4624] and EventData[Data[@Name='LogonType']='10']]

Returns only RDP logon events (Event 4624 with LogonType 10).

Example 5: Combine Conditions

*[System[EventID=4624 and TimeCreated[@SystemTime>='2026-02-14T02:00:00Z']] and EventData[Data[@Name='TargetUserName']='mtorres']]

Returns successful logons by mtorres after 02:00 UTC on February 14, 2026.

Using XPath in Event Viewer

  1. Open Event Viewer and select a log (e.g., Security).
  2. Click Filter Current Log….
  3. Switch to the XML tab in the filter dialog.
  4. Check Edit query manually.
  5. Enter your XPath query inside the <Select> element:
<QueryList>
  <Query Id="0" Path="Security">
    <Select Path="Security">
      *[System[EventID=4624] and EventData[Data[@Name='LogonType']='10']]
    </Select>
  </Query>
</QueryList>

Using XPath with wevtutil

wevtutil qe Security /q:"*[System[EventID=4624] and EventData[Data[@Name='LogonType']='10']]" /f:text /c:25

Flags: qe = query events, /q: = XPath query, /f:text = text output format, /c:25 = return first 25 matches.

💡 When to Use XPath vs. EvtxECmd

XPath is best for quick, targeted queries on a live system or a single log file—especially during triage. EvtxECmd is better for comprehensive parsing of all logs with structured CSV output for timeline analysis. In practice, many examiners use both: XPath for initial triage, then EvtxECmd for full analysis.

Knowledge Check

Test your understanding of the concepts covered in this module. These questions are ungraded—use them to identify areas you may want to review before moving on to Module 8.2.

1. A forensic examiner finds Event ID 4625 repeated 847 times within a 10-minute window, all targeting the username administrator. What type of attack does this most likely indicate?
✓ Correct! Event 4625 is a failed logon attempt. Hundreds of failed attempts against a single account in a short time window is a strong indicator of a brute-force attack. If the attempts targeted multiple accounts with a few passwords each, it would be a password-spray attack. Either way, the high volume of 4625 events is the key indicator.
✗ Not quite. The correct answer is C. Event 4625 records failed logon attempts. 847 failures in 10 minutes against the administrator account is a classic brute-force pattern. Pass-the-hash (B) would typically show successful logons (4624) with Logon Type 3. Privilege escalation (A) would appear as 4672 events. Audit policy tampering (D) would be Event 4719.
2. You see Event ID 4624 with LogonType: 10 and a RemoteHost of 192.168.5.30. What does this tell you?
✓ Correct! Logon Type 10 is RemoteInteractive, which means a Remote Desktop (RDP) session. The RemoteHost field tells you the source IP of the RDP connection. Logon Type 3 would indicate a network share (A), Type 4 would be a scheduled task (C), and Type 2 would be an interactive console logon (D).
✗ Not quite. The correct answer is B. Logon Type 10 = RemoteInteractive (RDP). Network share access is Type 3, scheduled tasks are Type 4, and physical console logon is Type 2. Memorizing the logon type codes is essential for forensic Event Log analysis.
3. Which Event ID should be treated as a high-priority alert because it indicates the system's audit logging configuration was modified?
✓ Correct! Event 4719 ("System audit policy was changed") fires when the audit policy is modified. This is a high-priority event because an attacker who disables auditing can effectively blind the Security log. The event is generated before the change takes effect, so it serves as a last warning.
✗ Not quite. The correct answer is D (4719). Event 4624 is a successful logon, 4672 is special privilege assignment, and 7045 is a new service installation. While 7045 is also high-priority, the question specifically asks about audit logging configuration changes, which is Event 4719.
4. You need to parse all .evtx files from a forensic image and export them to CSV for timeline analysis. Which command is correct?
✓ Correct! The -d flag tells EvtxECmd to process an entire directory of .evtx files, and --csv with --csvf specifies CSV output. Option B uses -f (single file) with a directory path, which would fail. Option C uses wevtutil, which doesn't support CSV export. Option D exports to JSON, not CSV.
✗ Not quite. The correct answer is A. Key distinction: -f is for a single file, -d is for a directory. The question asks about parsing all .evtx files, so you need -d. The --csv and --csvf flags specify CSV output format and filename.
5. An investigator discovers Event ID 7045 showing a new service named WindowsUpdateHelper with an image path of C:\Users\Public\Downloads\payload.exe running under the LocalSystem account. Which statement best describes the forensic significance?
✓ Correct! Multiple red flags: (1) the service name mimics a legitimate Windows function, (2) the executable is in C:\Users\Public\Downloads\—a user-writable location where legitimate services are never installed, (3) it runs as LocalSystem (highest privileges), and (4) the filename payload.exe is not a standard Windows binary. This should be treated as a suspected malicious service and the executable preserved for malware analysis.
✗ Not quite. The correct answer is C. Legitimate Windows Update services do not run from C:\Users\Public\Downloads\, and the filename payload.exe is a clear indicator of malicious intent. Event 7045 (new service installed) is always worth investigating, and the combination of a deceptive name, user-writable path, and LocalSystem privileges makes this highly suspicious regardless of whether 4625 events are present.
