1. Introduction & Objectives

Welcome to the third module of Week 8: Windows Forensics II. In Module 8.1 you learned how Windows Event Logs record system-wide security events. In Module 8.2 you explored Prefetch, AmCache, and ShimCache—artifacts that prove what programs ran. This module shifts focus to artifacts that answer a different set of questions: Where did the user navigate? What files did they access? and Can we recover files that have been deleted or modified?

You will study three artifact families:

  1. LNK (shortcut) files — preserve metadata about files a user opened, including files that have since been deleted.
  2. Shellbags — Registry entries that record every folder a user browsed, even folders on removable or network drives.
  3. Volume Shadow Copies (VSS) — point-in-time snapshots of an entire volume, enabling recovery of prior file versions and historical Registry hives.
🎯 Why This Matters

Suspects often delete incriminating files and assume the evidence is gone. LNK files and Shellbags persist independently of the original files, proving that deleted content once existed and was accessed. Volume Shadow Copies go even further—they can contain the actual deleted or modified files in their original state. Together, these three artifact families are among the most powerful tools for defeating evidence destruction.

2. LNK File Forensics

Every time a user opens a file from Windows Explorer, the operating system automatically creates (or updates) a shortcut file with the .lnk extension. These LNK files are stored in the user’s Recent folder and contain a rich set of metadata about the target file—metadata that persists even if the original file is deleted.

📂 Default LNK File Location

C:\Users\<username>\AppData\Roaming\Microsoft\Windows\Recent\

Additional LNK files may be found in the Recent\AutomaticDestinations and Recent\CustomDestinations subdirectories (these are the Jump List databases you studied in Week 7).

LNK Internal Structure

A LNK file is a binary structure defined by the Microsoft Shell Link format. The fields most relevant to forensic examiners are:

Key Forensic Fields in a LNK File
FieldDescriptionForensic Significance
Target Path Full path to the file or folder the shortcut points to. Proves the file existed at that path at the time the LNK was created or last updated.
Target MAC Timestamps Modified, Accessed, and Created timestamps of the target file at the time the LNK was created. Preserves a "snapshot" of the target’s timestamps—invaluable if the original file is later deleted or timestomped.
Target File Size Size of the target file when the LNK was created. Can reveal file modifications if current size differs from the LNK-recorded size.
Volume Serial Number Serial number of the volume where the target file resided. Links the file to a specific drive or partition. Matches serial numbers found in Registry MountedDevices and MountPoints2 keys.
Drive Type Type of storage: fixed disk, removable media, network share, etc. Identifies whether the file was on a USB drive, local hard disk, or network location.
Volume Label The user-assigned label of the volume (e.g., "BACKUP_USB"). Helps identify specific external devices.
NetBIOS Name Hostname of the machine where the target file resided (for network or local targets). Proves access to files on a specific remote machine.
LNK File Own Timestamps Created, Modified, Accessed timestamps of the .lnk file itself. The LNK Created timestamp indicates when the user first opened the target; Modified indicates the most recent access.

Forensic Scenarios

Scenario: The Deleted Spreadsheet

An employee is suspected of exfiltrating a confidential customer list. The file customer_database_export.xlsx is not found anywhere on the system—the employee claims it never existed. However, a LNK file in the Recent folder tells a different story:

Target Path:    D:\Exports\customer_database_export.xlsx
Target Created: 2026-02-10 09:15:22 UTC
Target Modified:2026-02-10 09:15:22 UTC
Target Size:    4,812,288 bytes (4.6 MB)
Drive Type:     Removable Media
Volume Serial:  A4F3-9B21
Volume Label:   SANDISK_USB
LNK Created:    2026-02-10 09:18:45 UTC
LNK Modified:   2026-02-10 09:18:45 UTC

What this proves:

  • The file did exist—the LNK preserves its exact path, size, and timestamps.
  • It was on a removable USB drive (Drive Type: Removable Media) labeled "SANDISK_USB".
  • The user opened the file at 09:18:45 UTC on February 10 (LNK Created time).
  • The identical Created and Modified timestamps on the target suggest the file was freshly created (exported) and not subsequently edited before being accessed.

Scenario: Unauthorized External Device

Company policy prohibits USB storage devices. You find multiple LNK files referencing paths on drive E:\ with Drive Type: Removable Media and a consistent Volume Serial Number of 7C2D-1A88. Cross-referencing this serial number with the Registry keys SYSTEM\MountedDevices and NTUSER.DAT\Software\Microsoft\Windows\CurrentVersion\Explorer\MountPoints2 (from Week 7) confirms the device was a Kingston DataTraveler USB drive.

Forensic value: Even if the USB device has been physically removed and the files deleted, the LNK files prove the device was connected and specific files were accessed from it. The volume serial number ties all the LNK files to the same device.

Scenario: Accessing Files on a Remote Server

During an insider-threat investigation, you discover LNK files with target paths pointing to \\FILESERVER03\Finance\Q4_Projections.pptx. The NetBIOS Name field in the LNK records FILESERVER03, and the Drive Type is Network.

Forensic value: This proves the user accessed a file on a specific network server. Combined with Event Log data from Module 8.1 (Event 4624, Logon Type 3 showing a network logon to FILESERVER03), you can build a corroborated timeline of the user’s access to sensitive network resources.

Connection to Week 7: Jump Lists

🔗 Cross-Module Connection

In Week 7 you learned that Jump Lists (stored in AutomaticDestinations and CustomDestinations) record recently and frequently accessed files per application. Internally, Jump Lists are structured storage files (OLE compound files) that contain embedded LNK entries. Each entry within a Jump List is essentially a LNK file with the same metadata fields described above.

This means the LNK parsing skills you learn in this module apply directly to Jump List analysis. Tools like JLECmd (Jump List Explorer Command Line, also by Eric Zimmerman) parse Jump Lists by extracting and interpreting their embedded LNK data. If you understand LNK structure, you understand the core data within Jump Lists.

3. Tool Walk-Through: LECmd

LECmd (LNK Explorer Command Line) is part of Eric Zimmerman’s forensic tool suite. It parses .lnk files and outputs structured data to the console or to CSV for analysis in Timeline Explorer.

🛠️ Hands-On Walkthrough

Follow along on your forensic workstation. You can practice with LNK files from your own Recent folder or with files extracted from a forensic image.

Step-by-Step: Parsing a Single LNK File

Step 1: Parse and View Console Output

LECmd.exe -f "C:\Evidence\Recent\customer_database_export.lnk"

LECmd displays a detailed breakdown of the LNK file directly in the console. Key output sections to examine:

LECmd Console Output Sections
SectionWhat to Look For
HeaderLNK file creation time, attributes, and flags.
Target Created / Modified / AccessedThe MAC timestamps of the target file at the time the LNK was created.
Volume InformationDrive type (Fixed, Removable, Network), volume serial number, and volume label.
Local PathFull path to the target file on the local or mapped drive.
Network Share InfoUNC path and NetBIOS name (for network targets).
Machine ID / MAC AddressNetBIOS name of the machine that created the LNK; sometimes includes a MAC address in the tracker data block.

Step 2: Batch-Process the Entire Recent Folder

LECmd.exe -d "C:\Evidence\Users\mtorres\AppData\Roaming\Microsoft\Windows\Recent" --csv "C:\Output" --csvf "mtorres_lnk_files.csv"

Flags explained:

  • -d — process all .lnk files in the specified directory (and subdirectories).
  • --csv — output directory for the CSV file.
  • --csvf — filename for the CSV output.

Step 3: Analyze Results in Timeline Explorer

  1. Open Timeline Explorer and load the CSV file generated by LECmd.
  2. Key columns to examine:
    • SourceFile — the name of the .lnk file.
    • LocalPath / NetworkPath — where the target file was located.
    • TargetCreated, TargetModified, TargetAccessed — the preserved timestamps of the target.
    • VolumeSerialNumber, DriveType, VolumeLabel — storage device identification.
    • MachineID — the NetBIOS name embedded in the LNK.
  3. Filter by DriveType to quickly isolate files accessed from removable media or network shares.
  4. Sort by TargetModified or the LNK file’s own timestamps to establish a chronological access history.
Tip: Combining LECmd Output with Other Artifact Data

For maximum investigative value, load LECmd CSV output alongside your EvtxECmd output (Module 8.1) and PECmd output (Module 8.2) in separate Timeline Explorer tabs. You can then cross-reference:

  • A LNK file showing a user opened mimikatz.exe from a USB drive → Prefetch confirming mimikatz.exe executed → Event Log showing a logon with elevated privileges around the same time.

This multi-artifact correlation is the foundation of the super-timeline you will build in Module 8.5.

4. Shellbag Analysis

While LNK files record which files a user opened, Shellbags record which folders a user browsed—including folder window size, position, view settings, and sort order. Shellbags are Registry-based artifacts that persist even after the folders themselves are deleted.

What Are Shellbags?

Every time a user opens a folder in Windows Explorer, the operating system saves the user’s display preferences (icon size, sort column, window position) in Registry values collectively called Shellbags. From a forensic perspective, the display preferences themselves are uninteresting—what matters is that the existence of a Shellbag entry proves the user navigated to that folder, and the timestamps on the Registry keys indicate when.

Registry Locations

Shellbag Registry Locations
HiveRegistry PathWhat It Tracks
NTUSER.DAT Software\Microsoft\Windows\Shell\BagMRU
Software\Microsoft\Windows\Shell\Bags		 Folders accessed via the Desktop and classic shell paths. Primarily covers Desktop, My Computer, and Control Panel navigation.
UsrClass.dat Local Settings\Software\Microsoft\Windows\Shell\BagMRU
Local Settings\Software\Microsoft\Windows\Shell\Bags		 Folders accessed via Windows Explorer navigation (the majority of user browsing). This is the primary Shellbag location for forensic analysis.
📂 UsrClass.dat Location on Disk

C:\Users\<username>\AppData\Local\Microsoft\Windows\UsrClass.dat

This hive is separate from NTUSER.DAT and is often overlooked by less experienced examiners. Always extract both hives when performing Shellbag analysis.

What Shellbags Record

Each Shellbag entry contains:

  • Folder path — the full path the user navigated to (local, removable, or network).
  • First interacted timestamp — when the user first browsed to that folder (derived from the Registry key’s created time).
  • Last interacted timestamp — when the user most recently browsed to that folder (derived from the Registry key’s last-write time).
  • Folder MAC timestamps — the Created, Modified, and Accessed timestamps of the folder as they appeared when the Shellbag was created or updated.
  • Shell item details — folder type, GUID identifiers for special folders, and display settings.

Forensic Value of Shellbags

Proving Access to Deleted Folders

A user creates a folder D:\Project_X_Confidential, copies sensitive files into it, then deletes the entire folder. The folder and its contents are gone from the file system. However, the Shellbag entry for D:\Project_X_Confidential remains in UsrClass.dat, proving the folder existed and the user navigated to it.

Identifying Removable Drive Browsing

Shellbags record navigation to folders on removable drives (e.g., E:\Photos\Vacation on a USB drive). Even after the USB drive is removed, the Shellbag entries persist. This complements the LNK file evidence from Section 2—LNK files show which files were opened from the USB; Shellbags show which folders were browsed.

Tracking Network Share Navigation

Shellbags record navigation to network paths such as \\FILESERVER\HR\Payroll. This proves the user actively browsed the network share—not just that a file was accessed (which could be automated). Combined with Event Log Logon Type 3 events, Shellbags provide strong evidence of intentional network exploration.

Establishing User Awareness

In intellectual property theft cases, Shellbags can establish that a user actively navigated to folders containing sensitive data. This demonstrates awareness and intent—the user didn’t just accidentally open a file; they browsed through a folder hierarchy to find it. A Shellbag trail showing \\SERVER\Engineering → Projects → Classified → Designs demonstrates deliberate, multi-level navigation.

5. Tool Walk-Through: ShellBags Explorer

ShellBags Explorer is a GUI tool by Eric Zimmerman that parses Shellbag data from Registry hives and presents it as a navigable folder tree. It is the standard tool for Shellbag analysis in forensic investigations.

🛠️ Hands-On Walkthrough

You will need extracted Registry hives (NTUSER.DAT and UsrClass.dat) from a forensic image. If you are practicing on your own system, you can load your live hives, but note that some entries may be locked by the OS.

Step-by-Step: Analyzing Shellbags

Step 1: Load a Registry Hive

  1. Open ShellBags Explorer.
  2. Go to File → Load offline hive.
  3. Browse to the extracted UsrClass.dat file (this is the primary Shellbag hive).
  4. ShellBags Explorer parses the hive and displays a folder tree in the left pane.
⚠️ Load Both Hives

For complete analysis, load both UsrClass.dat and NTUSER.DAT. The majority of user browsing data is in UsrClass.dat, but NTUSER.DAT contains Desktop-level navigation that may be relevant.

Step 2: Navigate the Folder Tree

  • The left pane shows a hierarchical tree of every folder the user browsed.
  • Expand nodes to see subfolders. The tree mirrors the folder structure the user navigated.
  • Look for entries under My Computer for local and removable drives, and Network for UNC paths.

Step 3: Interpret Timestamps

  • Select a folder in the tree. The right pane displays detailed information.
  • Key fields:
    • First Interacted — derived from the Shellbag Registry key’s Created timestamp. Indicates when the user first navigated to this folder.
    • Last Interacted — derived from the Registry key’s Last Write timestamp. Indicates the most recent time the user browsed this folder.
    • Created / Modified / Accessed — the folder’s own MAC timestamps as recorded in the Shellbag entry.

Step 4: Filter and Search

  • Use the filter bar at the top to search for specific folder names or paths (e.g., type "USB" or a drive letter like "E:").
  • Look for folders with paths referencing removable drives, network shares, or suspicious directory names.

Step 5: Export Results

  • ShellBags Explorer can export the parsed data for documentation or further analysis.
  • The companion command-line tool SBECmd.exe can batch-process hives and output to CSV, which loads into Timeline Explorer for integration with other artifact data.
SBECmd.exe -d "C:\Evidence\RegistryHives" --csv "C:\Output" --csvf "shellbags_parsed.csv"

6. Volume Shadow Copy Forensics

The Volume Shadow Copy Service (VSS) is a Windows feature that creates point-in-time snapshots of an entire volume. These snapshots—called shadow copies—store differential data that allows the system (or a forensic examiner) to reconstruct the state of the volume at the time the snapshot was taken.

VSS Architecture

VSS operates at the block level, not the file level. When a shadow copy is created, VSS does not duplicate the entire volume. Instead, it uses a copy-on-write mechanism:

  1. A snapshot is triggered (see triggers below).
  2. VSS records the current state of all blocks on the volume.
  3. When a block is about to be modified after the snapshot, VSS copies the original block to a shadow storage area before the modification occurs.
  4. To reconstruct the volume as it appeared at snapshot time, VSS combines the current volume data with the preserved original blocks.
💡 Key Concept: Differential, Not Full Copy

Shadow copies are space-efficient because they only store blocks that have changed since the snapshot. A shadow copy of a 500 GB volume might only consume a few gigabytes of shadow storage. However, this also means that if the shadow storage area fills up, the oldest shadow copies are deleted automatically.

Snapshot Triggers

Shadow copies are created automatically by several mechanisms:

Common VSS Snapshot Triggers
TriggerDescription
System Restore PointsWindows creates restore points before significant system changes (driver installs, Windows Updates).
Windows BackupThe built-in Windows Backup utility triggers a shadow copy as part of its backup process.
Scheduled TaskBy default, Windows may create periodic shadow copies on a schedule (commonly daily or weekly, depending on OS version and configuration).
Application RequestSome applications (e.g., SQL Server, Exchange) request VSS snapshots for consistent backups.

Forensic Use Cases

Use Case: Recovering Deleted Files

A suspect deletes incriminating files from their system. Standard file recovery from unallocated space fails because the sectors have been overwritten by new data. However, a shadow copy taken before the deletion still contains the original blocks. By mounting the shadow copy, the examiner can browse the volume as it appeared at snapshot time and recover the deleted files in their entirety.

Example: A shadow copy from January 15 contains C:\Users\jsmith\Documents\financial_fraud_spreadsheet.xlsx. The file was deleted on January 20. The current volume no longer contains the file, but the January 15 shadow copy does.

Use Case: Recovering Prior File Versions

A suspect modifies a document to remove incriminating content before the investigation begins. The current version of the file appears innocent. However, a shadow copy from before the modification contains the original version with the incriminating content intact.

Example: An employee edits a memo to remove references to unauthorized transactions. A shadow copy from the previous week contains the unedited memo. Comparing the two versions reveals exactly what was changed and when (by correlating with $UsnJrnl data from Module 8.4).

Use Case: Accessing Historical Registry Hives

This is one of the most powerful forensic applications of VSS. Shadow copies contain snapshots of the entire volume—including the Registry hive files. By extracting Registry hives from a shadow copy, an examiner can analyze the system’s configuration and user artifacts as they existed at a prior point in time.

Example: An attacker creates a backdoor user account (svc_backup, from the Module 8.1 mini-case), uses it, and then deletes it before the investigation. The current Registry shows no trace of the account. But a shadow copy from the day the account existed contains a SAM hive with the account entry and an NTUSER.DAT hive with the account’s user profile artifacts.

You can load these historical hives into Registry Explorer (Week 7) or ShellBags Explorer (Section 5 above) to analyze the system’s state at the snapshot time.

7. Tool Walk-Through: Accessing VSS with Arsenal Image Mounter

Arsenal Image Mounter is a forensic tool that mounts forensic images (E01, raw/dd, VMDK, etc.) as physical or logical disks on Windows. Critically, it can expose Volume Shadow Copies contained within the mounted image, allowing examiners to browse and extract files from prior snapshots.

🛠️ Hands-On Walkthrough

This walkthrough describes the workflow for accessing shadow copies from a forensic image. You will practice this technique in Lab 8.

Step-by-Step: Mounting and Accessing Shadow Copies

Step 1: Mount the Forensic Image

  1. Open Arsenal Image Mounter with administrator privileges.
  2. Click Mount disk image.
  3. Browse to the forensic image file (e.g., evidence.E01).
  4. Select Write-temporary mount mode. This allows the operating system to interact with the image (required for VSS access) while keeping the original image forensically sound—all writes go to a temporary differential file.
  5. Check Launch Windows disk signature check / offline if prompted, to prevent Windows from modifying the volume.
  6. The image mounts as a physical disk with one or more volumes appearing in Windows Explorer (e.g., F:\).
⚠️ Write-Temporary Mode Is Essential

Never mount a forensic image in read/write mode. Write-temporary mode ensures all OS-initiated writes (drive letter assignment, VSS catalog access) are captured in a temporary overlay, preserving the integrity of the original evidence. Always verify your image hash after unmounting.

Step 2: List Available Shadow Copies

Once the image is mounted, open an elevated Command Prompt and run:

vssadmin list shadows /for=F:

Replace F: with the drive letter assigned to the mounted volume. The output lists all available shadow copies with their creation dates and Shadow Copy IDs:

vssadmin 1.1 - Volume Shadow Copy Service administrative command-line tool

Contents of shadow copy set ID: {a1b2c3d4-...}
  Contained 1 shadow copies at creation time: 1/15/2026 3:00:12 AM
    Shadow Copy ID: {e5f6a7b8-...}
      Original Volume: (F:)\\?\Volume{...}\
      Shadow Copy Volume: \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1

Contents of shadow copy set ID: {c9d0e1f2-...}
  Contained 1 shadow copies at creation time: 1/22/2026 3:00:08 AM
    Shadow Copy ID: {g3h4i5j6-...}
      Original Volume: (F:)\\?\Volume{...}\
      Shadow Copy Volume: \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy2

Step 3: Create a Symbolic Link to a Shadow Copy

To browse a shadow copy in Windows Explorer, create a symbolic link (symlink) that maps a folder to the shadow copy volume:

mklink /d C:\VSS_Jan15 \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\

Now you can browse C:\VSS_Jan15 as if it were the volume at the time of the January 15 snapshot.

💡 Trailing Backslash Required

The shadow copy volume path must end with a backslash (\). Without it, the symlink will fail silently.

Step 4: Browse and Extract Files

  1. Navigate to C:\VSS_Jan15 in Windows Explorer or the command line.
  2. You are now viewing the volume as it appeared on January 15. Deleted files are present; modified files show their prior content.
  3. Copy any relevant files to your output directory for analysis.
  4. To access Registry hives from the snapshot, navigate to C:\VSS_Jan15\Windows\System32\config\ for system hives or C:\VSS_Jan15\Users\<username>\NTUSER.DAT for user hives.

Step 5: Compare Shadow Copies

If multiple shadow copies exist, mount each one and compare file versions across snapshots. This can reveal:

  • When a file was first created (present in a later snapshot but not an earlier one).
  • When a file was deleted (present in an earlier snapshot but not a later one).
  • How a file changed over time (different content in different snapshots).

Step 6: Clean Up

  1. When finished, remove the symbolic link: rmdir C:\VSS_Jan15
  2. Unmount the forensic image in Arsenal Image Mounter.
  3. Verify the original image hash matches the acquisition hash to confirm no modifications.
Alternative: Using vshadowmount (Linux)

On a Linux forensic workstation, the libvshadow library provides vshadowinfo and vshadowmount commands for accessing shadow copies from a forensic image:

# List shadow copies in a raw image
vshadowinfo /path/to/evidence.raw

# Mount all shadow copies as virtual files
vshadowmount /path/to/evidence.raw /mnt/vss/

# Each shadow copy appears as a file: vss1, vss2, etc.
# Mount a specific shadow copy
mount -o ro,loop,show_sys_files,streams_interface=windows /mnt/vss/vss1 /mnt/shadow1/

This approach is common in Linux-based forensic environments and will be referenced again in Week 9 (Linux Forensics).

8. Anti-Forensic Awareness

Volume Shadow Copies are so forensically valuable that sophisticated attackers—and virtually all modern ransomware—actively target them for destruction. As a forensic examiner, you must be aware of these anti-forensic techniques and know what to look for when shadow copies are missing.

Ransomware and VSS Deletion

One of the first actions most ransomware families take after gaining execution is to delete all shadow copies. This prevents victims from using shadow copies to recover encrypted files without paying the ransom. The most common technique:

vssadmin delete shadows /all /quiet

This command silently deletes every shadow copy on the system. Variants include:

wmic shadowcopy delete
Get-WmiObject Win32_ShadowCopy | ForEach-Object { $_.Delete() }
vssadmin resize shadowstorage /for=C: /on=C: /maxsize=401MB
⚠️ Detecting VSS Deletion

Even when shadow copies have been deleted, evidence of the deletion attempt often remains:

  • Event Log: The System log records VSS-related events. Look for Event ID 8224 (VSS service events) and Service Control Manager events showing the VSS service being stopped.
  • Prefetch: A Prefetch file for VSSADMIN.EXE or WMIC.EXE (from Module 8.2) proves these tools were executed.
  • $UsnJrnl: The NTFS change journal (Module 8.4) may record the deletion of VSS-related files.
  • Command-line logging: If PowerShell script block logging is enabled, the exact commands used to delete shadow copies may be captured in the PowerShell Operational log.

SSD TRIM and VSS Reliability

Solid-State Drives (SSDs) use the TRIM command to proactively erase blocks that are no longer in use. This has implications for VSS:

  • TRIM can erase shadow copy data. Because VSS stores differential data on the same volume, TRIM may erase blocks that VSS needs to reconstruct a prior snapshot. This can cause shadow copies to become corrupt or incomplete even though they appear in the vssadmin list shadows output.
  • Impact varies by OS version and SSD firmware. Newer versions of Windows are better at protecting VSS storage areas from TRIM, but the risk remains, especially on older systems.
  • Forensic implication: On SSD-based systems, do not assume that a listed shadow copy is fully intact. Always attempt to mount and verify the contents. If a shadow copy fails to mount or shows corrupted files, TRIM may be the cause.
💡 Connection to Week 4

In Week 4 (Hardware and Storage Technologies) you learned about SSD architecture and the TRIM command. The VSS reliability issue is a practical consequence of the TRIM behavior discussed in that module. This is another example of why understanding the physical storage layer is essential for interpreting logical artifacts.

9. Knowledge Check

Test your understanding of the concepts covered in this module. These questions are ungraded—use them to identify areas you may want to review before moving on to Module 8.4.

1. A forensic examiner finds a LNK file in a user’s Recent folder that references E:\Confidential\merger_plan.docx with a Drive Type of "Removable Media" and a Volume Serial Number of 8A3F-12CD. The file merger_plan.docx does not exist anywhere on the system. What can the examiner conclude?
✓ Correct! The LNK file preserves the target path (E:\Confidential\merger_plan.docx), the Drive Type (Removable Media), and the Volume Serial Number. This proves the file existed on a removable device (likely a USB drive) and was opened by the user. The device has been disconnected, which is why the file is not currently on the system. The Volume Serial Number can be cross-referenced with Registry MountedDevices and MountPoints2 keys to identify the specific device.
✗ Not quite. The correct answer is B. The Drive Type field explicitly says "Removable Media," ruling out a network share (C). LNK files are automatically created by the OS when a user opens a file, so corruption (A) or malware planting (D) are unlikely explanations for a well-formed LNK with consistent metadata. The LNK file is strong evidence that the file existed on a USB device.
2. Which Registry hive is the primary location for Shellbag data recording a user’s folder navigation in Windows Explorer?
✓ Correct! UsrClass.dat is the primary Shellbag hive, storing folder navigation data from Windows Explorer. While NTUSER.DAT (C) also contains some Shellbag entries (primarily Desktop-level navigation), the vast majority of user browsing activity is recorded in UsrClass.dat. Always extract and analyze both hives for complete coverage.
✗ Not quite. The correct answer is D (UsrClass.dat). SYSTEM (A) contains hardware and service configuration. SAM (B) contains user account data. NTUSER.DAT (C) does contain some Shellbag entries, but the primary location for Explorer folder navigation is UsrClass.dat, located at C:\Users\<username>\AppData\Local\Microsoft\Windows\UsrClass.dat.
3. What mechanism does the Volume Shadow Copy Service use to minimize storage consumption?
✓ Correct! VSS uses a copy-on-write mechanism. When a block is about to be modified after a snapshot, VSS copies the original block to shadow storage before allowing the modification. This means only changed blocks consume shadow storage space, making snapshots very space-efficient. A 500 GB volume might only need a few gigabytes of shadow storage.
✗ Not quite. The correct answer is C. VSS does not compress the volume (A) or create full copies (B)—either approach would consume enormous storage. It does not store only metadata (D)—VSS preserves actual block-level data, which is why full file recovery is possible. The copy-on-write approach saves only the blocks that would be overwritten, making it efficient while preserving complete recoverability.
4. During a ransomware investigation, you discover that no Volume Shadow Copies exist on the victim system, even though VSS was enabled. You find a Prefetch file for VSSADMIN.EXE with a last-run time just minutes before the encryption began. What is the most likely explanation?
✓ Correct! This is a textbook ransomware technique. The Prefetch file for VSSADMIN.EXE with a last-run time immediately before encryption is strong evidence that the ransomware executed vssadmin delete shadows /all /quiet to prevent victims from recovering files via shadow copies.
