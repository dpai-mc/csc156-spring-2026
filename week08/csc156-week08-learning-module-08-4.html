1. Introduction & Objectives

In Module 5.3 you learned that NTFS maintains two journal structures—$LogFile and $UsnJrnl—that record metadata changes on the volume. In this module you will move from conceptual understanding to forensic application: parsing journal data, interpreting reason codes, and—critically—using journal timestamps to detect timestomping, one of the most common anti-forensic techniques an examiner will encounter.

🎯 Why This Matters

An attacker can easily change the dates on a malicious file to make it blend in with legitimate system files. Standard file-system timestamps (dir, Windows Explorer) will show the faked dates. But the NTFS journals tell a different story—one the attacker almost never thinks to alter. Journals are your ground-truth timeline when timestamps cannot be trusted.

2. $MFT Timestamp Review

Before diving into the journals, let’s revisit a critical concept from Week 5 (Module 5.3): every file on an NTFS volume has two separate sets of timestamps stored in its MFT entry, and they behave very differently from a forensic perspective.

Two Attribute Sets, Eight Timestamps

Each MFT entry contains (at minimum) two attributes that store MAC(b) timestamps—Modified, Accessed, Changed (MFT entry), and Born (created):

$STANDARD_INFORMATION (SI) Attribute — Type 0x10

Present in every MFT entry. Contains four timestamps:

  • SI_Modified — last time the file content was written.
  • SI_Accessed — last time the file was read (often disabled on modern Windows for performance).
  • SI_Changed (MFT Entry Modified) — last time the MFT record itself was changed (e.g., permission change, rename).
  • SI_Born (Created) — when the file was originally created on this volume.
⚠️ Key Forensic Fact

SI timestamps can be modified by any user-level process using the Windows API function SetFileTime(). This is exactly what timestomping tools exploit. When you view a file’s properties in Windows Explorer or run dir, you are seeing SI timestamps.

$FILE_NAME (FN) Attribute — Type 0x30

Also present in every MFT entry (at least one; files with both a short 8.3 name and a long name have two FN attributes). Contains its own four timestamps:

  • FN_Modified
  • FN_Accessed
  • FN_Changed (MFT Entry Modified)
  • FN_Born (Created)
✅ Key Forensic Fact

FN timestamps can only be modified by the NTFS kernel-mode driver. There is no documented user-level API to change them. This means FN timestamps are highly resistant to manipulation and serve as a trusted baseline when SI timestamps are suspect.

FN timestamps are updated by the kernel during specific operations: file creation, file rename, file move (to a different directory), and certain hard-link operations. They are not updated on every read or write, so they tend to reflect the file’s original creation or last structural change rather than its most recent content modification.

3. Detecting Timestomping

Timestomping is the anti-forensic technique of modifying a file’s timestamps to mislead investigators. Attackers use it to make a malicious file appear as though it was created months or years ago—blending it in with legitimate system files—or to push the creation date outside the investigation’s time window.

How Timestomping Works

Tools such as Timestomp (part of the Metasploit framework) or custom PowerShell scripts call the SetFileTime() Windows API to overwrite the four SI timestamps. The attacker can set any arbitrary date and time. However, the FN timestamps remain untouched because they are managed exclusively by the NTFS driver in kernel mode.

Detection: SI vs. FN Comparison

The primary detection method is straightforward: compare SI and FN timestamps for the same file. If the SI timestamps are significantly earlier than the FN timestamps, the file has almost certainly been timestomped.

Example: Timestomped Malware

Consider a file named C:\Windows\System32\healthsvc.dll found on a suspect system. The MFT entry reveals:

$STANDARD_INFORMATION Timestamps

SI_Born:     2024-04-12  09:15:33 UTC
SI_Modified: 2024-04-12  09:15:33 UTC
SI_Accessed: 2024-04-12  09:15:33 UTC
SI_Changed:  2024-04-12  09:15:33 UTC

All four timestamps are identical and date to April 2024—over a year before the investigation.

$FILE_NAME Timestamps

FN_Born:     2026-02-14  02:53:07 UTC
FN_Modified: 2026-02-14  02:53:07 UTC
FN_Accessed: 2026-02-14  02:53:07 UTC
FN_Changed:  2026-02-14  02:53:07 UTC

🚩 The FN timestamps show the file was actually created on February 14, 2026—during the incident window.

Red-Flag Patterns

When reviewing MFT data, watch for these timestomping indicators:

  • SI_Born earlier than FN_Born: This is logically impossible under normal NTFS operation. A file cannot have an SI creation date before its FN creation date. This is the strongest single indicator of timestomping.
  • All four SI timestamps identical: Legitimate files almost never have all four timestamps set to the exact same second. Timestomping tools often set all four to one value for simplicity.
  • SI timestamps with suspicious precision: Timestamps rounded to the exact second (e.g., 09:15:33.0000000) while surrounding files have normal nanosecond precision may indicate manual setting.
  • SI dates inconsistent with OS install date: A file in System32 with an SI_Born date predating the OS installation is suspicious.
💡 Investigative Significance

Detecting timestomping does more than just reveal the file’s true creation time. It demonstrates intent to deceive—the attacker deliberately tried to hide their activity. This is powerful evidence of malicious intent and can be presented in court to show the sophistication and deliberateness of the intrusion.

Advanced: Can FN Timestamps Be Manipulated?

In theory, a kernel-mode driver or a raw-disk editing tool could modify FN timestamps directly in the MFT on disk. However, this requires:

  • Administrator (or SYSTEM) privileges and the ability to load a kernel driver, or
  • Offline access to the raw NTFS volume (e.g., booting from a USB drive).

In practice, this level of sophistication is extremely rare in real-world cases. The vast majority of timestomping affects only SI timestamps. Even when an attacker modifies FN timestamps, the $UsnJrnl and $LogFile entries will still contain the original operation timestamps—providing yet another layer of evidence.

4. $LogFile — The NTFS Transaction Journal

The $LogFile is NTFS’s internal transaction journal. Its primary purpose is crash recovery: if the system loses power or crashes mid-operation, NTFS replays or rolls back incomplete transactions from $LogFile to restore the volume to a consistent state.

What $LogFile Records

$LogFile records metadata operations—changes to MFT entries, index entries (directory listings), and other NTFS metadata structures. It does not record file content changes (data writes to clusters).

Operations captured include:

  • File and directory creation
  • File and directory deletion
  • File rename and move
  • Attribute changes (timestamps, permissions, flags)
  • Index updates (adding/removing entries from directory indexes)
  • MFT record allocation and deallocation

Structure and Size

$LogFile Key Facts
PropertyDetail
MFT record number2 (always the third record in the MFT)
LocationRoot of the NTFS volume (hidden system file)
Typical size~64 MB (default on most volumes)
Logging behaviorCircular — oldest entries overwritten when full
Record formatRestart Area + Log Record Pages containing redo/undo operation pairs

Forensic Value

  • Short-term, high-detail history: Because $LogFile is relatively small and circular, it typically retains only hours to a few days of activity on an active system. But within that window, it provides granular, operation-level detail about every metadata change.
  • Deleted file evidence: Even after a file is deleted and its MFT entry reused, $LogFile may still contain the original creation, rename, or deletion records if they fall within the journal’s retention window.
  • Redo/Undo pairs: Each logged operation includes both a redo record (how to replay the operation) and an undo record (how to reverse it). Forensically, the undo data can reveal the previous state of an attribute before it was changed—including prior timestamps and filenames.
  • Timestomping corroboration: If an attacker modified SI timestamps, the $LogFile may contain the redo/undo records showing the original values and the new (faked) values.
⚠️ Limitations

Retention is short. On a busy system, $LogFile may wrap (overwrite old entries) in just a few hours. If you are investigating an event from weeks ago, $LogFile data is likely gone. This is why prompt evidence acquisition is critical—the longer you wait, the more journal data you lose.

Deep Dive: $LogFile Internal Structure

The $LogFile begins with two Restart Area pages (for redundancy) followed by Log Record Pages. Each log record page contains multiple individual log records.

Each log record includes:

  • LSN (Log Sequence Number): A monotonically increasing 64-bit number that uniquely identifies the record and establishes order.
  • Transaction ID: Groups related operations into atomic transactions.
  • Redo Operation: The operation code and data needed to replay the change (e.g., “write these bytes to offset X of MFT record Y”).
  • Undo Operation: The operation code and data needed to reverse the change.
  • Target MFT reference: Which MFT record and attribute were affected.

Common operation codes include: InitializeFileRecordSegment, DeallocateFileRecordSegment, CreateAttribute, DeleteAttribute, UpdateResidentValue, AddIndexEntryRoot, DeleteIndexEntryRoot, and many more.

5. $UsnJrnl — The Change Journal

The $UsnJrnl (Update Sequence Number Journal), also called the change journal, is a separate NTFS structure that records a high-level log of file and directory changes on the volume. While $LogFile is designed for crash recovery, $UsnJrnl is designed to help applications (such as backup software, search indexers, and antivirus) efficiently detect what has changed since they last checked.

Location and Structure

$UsnJrnl Key Facts
PropertyDetail
Location$Extend\$UsnJrnl (a system file in the $Extend directory)
Data streams$Max (configuration) and $J (journal entries)
Typical size32–64 MB default allocation; can be larger
Logging behaviorCircular within the allocated space; oldest entries overwritten
RetentionDays to weeks on a typical workstation (much longer than $LogFile)

The forensically important data stream is $J. Each record in $J contains:

  • USN (Update Sequence Number): Byte offset of the record within the journal; monotonically increasing.
  • Timestamp: When the change occurred (UTC, 100-nanosecond precision).
  • Filename: The name of the file or directory that changed.
  • MFT Reference Number: Links the entry back to the MFT record.
  • Parent MFT Reference: The directory containing the file.
  • Reason Code: A bitmask describing what changed (see below).
  • Source Info: Indicates if the change was from the OS, replication, etc.

Reason Codes

Reason codes are the heart of $UsnJrnl forensic analysis. Each record contains one or more reason flags that describe the nature of the change:

Common $UsnJrnl Reason Codes
Reason CodeHex ValueMeaningForensic Significance
FILE_CREATE0x00000100A new file or directory was created.Establishes when a file first appeared on the volume.
FILE_DELETE0x00000200A file or directory was deleted.Proves a file existed and when it was removed—even if the MFT entry has been reused.
DATA_EXTEND0x00000002Data was appended to the file (file grew).Indicates active writing; useful for tracking data exfiltration or file downloads.
DATA_OVERWRITE0x00000001Existing data was overwritten.Content modification; may indicate file tampering.
DATA_TRUNCATION0x00000004File was truncated (shrank).May indicate log clearing or evidence destruction.
RENAME_OLD_NAME0x00001000The file’s previous name (before rename).Paired with RENAME_NEW_NAME to track file renaming—a common anti-forensic technique.
RENAME_NEW_NAME0x00002000The file’s new name (after rename).See above. Together they reconstruct the full rename history.
SECURITY_CHANGE0x00000800File permissions (ACL) were modified.May indicate privilege escalation or evidence-access attempts.
BASIC_INFO_CHANGE0x00008000Basic attributes changed (timestamps, flags).Key timestomping indicator—if you see this without a corresponding content write, timestamps may have been manipulated.
CLOSE0x80000000The file handle was closed.Marks the end of an operation sequence. The CLOSE record typically carries a combined reason mask summarizing all changes made during that open-close cycle.
💡 Reading Combined Reason Codes

A single $UsnJrnl record often contains multiple reason flags OR’d together. For example, a record with reason FILE_CREATE | CLOSE (hex 0x80000100) means a new file was created and the handle was immediately closed. When you see DATA_EXTEND | CLOSE, it means data was written and the file was closed. Parsed output from MFTECmd displays these as comma-separated labels for readability.

Forensic Value of $UsnJrnl

  • Longer retention than $LogFile: Because the $J stream is typically larger, it retains entries for days to weeks—covering a much wider investigation window.
  • Evidence of deleted files: Even after a file is deleted and its MFT entry overwritten, the $UsnJrnl may still contain the FILE_CREATE, DATA_EXTEND, and FILE_DELETE records with the original filename and timestamps.
  • Rename tracking: The RENAME_OLD_NAME / RENAME_NEW_NAME pairs let you trace a file’s complete naming history—critical when attackers rename malware to mimic system files.
  • Timeline backbone: $UsnJrnl entries, with their precise timestamps and reason codes, form the backbone of a file-system timeline. In Module 8.5 you will merge this data with Event Logs, Prefetch, and Registry artifacts to build a super-timeline.

6. Tool Walk-Through: MFTECmd

MFTECmd is part of Eric Zimmerman’s forensic tool suite. It parses three NTFS structures: the $MFT, the $LogFile, and the $UsnJrnl:$J data stream. Output is exported to CSV for analysis in Timeline Explorer or any spreadsheet application.

🛠️ Hands-On Walkthrough

Follow along on your forensic workstation. You will need the extracted $MFT, $LogFile, and $UsnJrnl:$J files from a forensic image. If you do not have these yet, the commands below show the exact syntax you will use in Lab 8 and Homework 3.

Step 1: Parse the $MFT

MFTECmd.exe -f "C:\Evidence\$MFT" --csv "C:\Output" --csvf "MFT_parsed.csv"

This produces a CSV with one row per MFT entry, including both SI and FN timestamps. You can immediately filter for timestomping by comparing SI_Born and FN_Born columns.

Step 2: Parse the $LogFile

MFTECmd.exe -f "C:\Evidence\$LogFile" --csv "C:\Output" --csvf "LogFile_parsed.csv"

The $LogFile CSV contains columns for LSN, operation type, redo/undo data, and the affected MFT reference. Use this to investigate specific metadata changes within the journal’s retention window.

Step 3: Parse the $UsnJrnl ($J stream)

MFTECmd.exe -f "C:\Evidence\$J" --csv "C:\Output" --csvf "UsnJrnl_parsed.csv"

The $UsnJrnl CSV includes the following key columns:

Key MFTECmd $UsnJrnl Output Columns
ColumnDescription
UpdateTimestampUTC timestamp of the journal entry.
NameFilename of the affected file or directory.
UpdateReasonsComma-separated reason codes (e.g., FileCreate, Close).
EntryNumberMFT entry number of the affected file.
ParentEntryNumberMFT entry number of the parent directory.
UpdateSequenceNumberThe USN (byte offset in the journal).
FileAttributesFile attributes (Archive, Hidden, System, Directory, etc.).

Step 4: Load in Timeline Explorer

  1. Open Timeline Explorer.
  2. Go to File → Open and select UsnJrnl_parsed.csv.
  3. The data loads into a sortable, filterable grid.

Step 5: Filter by Reason Code

  1. Click the filter icon on the UpdateReasons column.
  2. Type FileCreate to see only file-creation events.
  3. Clear and type FileDelete to see deletions.
  4. Use RenameOldName to find files that were renamed—then look for the adjacent RenameNewName entry to see the new name.

Step 6: Filter by Time Range

  1. Click the UpdateTimestamp column header to sort chronologically.
  2. Use the column filter to specify a date range matching your incident window.
  3. This narrows thousands of entries down to the relevant time period.

Step 7: Cross-Reference with $MFT

When you find a suspicious $UsnJrnl entry, note the EntryNumber (MFT record number). Then open MFT_parsed.csv in Timeline Explorer and filter the EntryNumber column to that value. This gives you the full MFT record including both SI and FN timestamps—allowing you to check for timestomping on the same file.

Tip: Extracting $UsnJrnl:$J from a Forensic Image

The $UsnJrnl is an NTFS system file with an alternate data stream ($J). Extracting it requires a tool that understands NTFS alternate data streams:

  • Autopsy: Navigate to the root of the NTFS volume → $Extend → $UsnJrnl. Right-click the $J stream and select Extract File(s).
  • FTK Imager: Browse to the same path, select the $J data stream, and export.
  • icat (Sleuth Kit): Use the MFT entry number for $UsnJrnl with the ADS name: icat -o <offset> image.E01 <inode>-128-3 > $J.

Similarly, extract $MFT (MFT entry 0) and $LogFile (MFT entry 2) from the image root.

Investigative Questions

Use the journal entries and MFT data above to answer the following. Click each question to reveal the answer.

Q1: What was the original name of the file, and what was it renamed to?

Answer: The file was originally created as employee_salaries.xlsx (Entry 1) and renamed to Q4_budget_draft.xlsx (Entries 3 & 4).

Analysis: $UsnJrnl Entry 3 shows RenameOldName for employee_salaries.xlsx and Entry 4 shows RenameNewName for Q4_budget_draft.xlsx. Both share the same EntryNumber (48291), confirming they are the same file. The rename occurred at 01:15:22 UTC—about 3 minutes after creation. The new name appears designed to disguise the file’s true contents.

Q2: What evidence suggests the file was timestomped?

Answer: The SI_Born timestamp (2025-06-15) is 8 months earlier than the FN_Born timestamp (2026-02-14 01:12:05).

Analysis: Under normal NTFS operation, SI_Born cannot be earlier than FN_Born. The $UsnJrnl confirms the file was created at 01:12:05 on Feb 14, 2026 (Entry 1: FileCreate). The SI timestamps were set to June 2025—a date that would make the file appear to be a routine document from months ago. Additionally, all four SI timestamps being identical (both set to 2025-06-15 10:30:00) is another red flag.

Q3: Reconstruct the complete sequence of events in chronological order.
  1. 01:12:05 — File employee_salaries.xlsx was created (Entry 1: FileCreate).
  2. 01:12:06 — Data was written to the file (Entry 2: DataExtend).
  3. 01:15:22 — File was renamed from employee_salaries.xlsx to Q4_budget_draft.xlsx (Entries 3–4: RenameOldName / RenameNewName).
  4. Between 01:15 and 01:23 — SI timestamps were timestomped to June 2025 (evidenced by SI vs. FN mismatch; the exact journal entry for the BASIC_INFO_CHANGE may have been filtered out or combined with another record).
  5. 01:23:41 — File backup.zip was created (Entry 5: FileCreate).
  6. 01:23:42 – 01:23:58 — Data was written to backup.zip in two phases (Entries 6–7: DataExtend), suggesting the ZIP was being built incrementally.
  7. 01:31:09 — Q4_budget_draft.xlsx (the renamed salary file) was deleted (Entry 8: FileDelete).
  8. 01:31:15 — backup.zip was deleted (Entry 9: FileDelete).

Investigative narrative: Someone created a file containing employee salary data, wrote content to it, renamed it to disguise its contents, timestomped it to hide its true creation date, compressed it into a ZIP archive (likely for exfiltration), and then deleted both files to cover their tracks—all within a 19-minute window in the early morning hours.

Q4: Despite both files being deleted, what evidence remains? What would you investigate next?

Evidence that persists:

  • The $UsnJrnl entries documenting the entire create → write → rename → delete sequence.
  • The $MFT entry (48291) still exists (marked as deleted) with both SI and FN timestamps revealing the timestomping.
  • The $LogFile may contain additional detail if the events are within its retention window.
  • The file content may still be recoverable if the clusters have not been overwritten.

Next steps:

  1. Attempt file recovery: Use Autopsy or Sleuth Kit to recover the deleted employee_salaries.xlsx and backup.zip from unallocated space.
  2. Check Event Logs: Who was logged into HR-PC at 01:12 UTC? (Module 8.1 skills — look for Event 4624.)
  3. Check Prefetch: Was a ZIP utility or timestomping tool executed? (Module 8.2 skills.)
  4. Check LNK files / Jump Lists: Did the user open employee_salaries.xlsx? (Module 8.3 skills.)
  5. Network artifacts: Was backup.zip transmitted before deletion? (Weeks 11–12.)

8. Knowledge Check

Test your understanding of NTFS journal artifacts and timestomping detection. These questions are ungraded—use them to identify areas to review before Module 8.5.

1. An examiner finds a file where SI_Born is 2024-01-15 and FN_Born is 2026-02-14. What is the most likely explanation?
✓ Correct! When SI_Born is earlier than FN_Born, it is a strong indicator of timestomping. Under normal NTFS operation, SI_Born and FN_Born are set to the same value at file creation, and SI_Born can only move forward (or stay the same) through normal operations. An SI_Born date years before FN_Born means someone used SetFileTime() or a similar tool to backdate the SI timestamps. FN timestamps cannot be modified by user-level tools, so FN_Born reflects the true creation time.
✗ Not quite. The correct answer is B. SI_Born earlier than FN_Born is the classic timestomping signature. FN timestamps are managed by the NTFS kernel driver and cannot be changed by user-level APIs. While file copying (A) can affect SI timestamps, a copy operation creates a new FN_Born that matches or is close to SI_Born—it would not produce a two-year gap where SI is older. FN modification (C) requires kernel-level access and is extremely rare.
2. Which of the following best describes the difference between $LogFile and $UsnJrnl?
✓ Correct! $LogFile is the NTFS transaction journal designed for crash recovery, storing granular redo/undo operation pairs at the byte level. $UsnJrnl is the change journal designed for applications, storing higher-level records with reason codes (FileCreate, FileDelete, etc.). $LogFile actually has shorter retention (hours to days) than $UsnJrnl (days to weeks), and neither records file content directly.
✗ Not quite. The correct answer is C. $LogFile stores low-level redo/undo pairs for crash recovery; $UsnJrnl stores higher-level change notifications with reason codes. Neither records actual file content (A is wrong). $LogFile has shorter retention than $UsnJrnl (B is backwards). They are structurally different systems with different purposes (D is wrong).
3. You find the following consecutive $UsnJrnl entries for MFT entry 50123:
Entry A: RenameOldName — Name: malware.exe
Entry B: RenameNewName — Name: svchost.exe
What does this sequence indicate?
✓ Correct! RenameOldName records the file’s name before the rename, and RenameNewName records the name after. Both entries share the same MFT entry number (50123), confirming it is the same file. The attacker renamed malware.exe to svchost.exe to disguise it as a legitimate Windows system binary. This is a common evasion technique.
✗ Not quite. The correct answer is A. In $UsnJrnl, RenameOldName always contains the previous name and RenameNewName contains the new name. Since both entries reference the same MFT entry (50123), this is a single file rename operation: malware.exe → svchost.exe. This is not a deletion/creation (D) or two separate files (C).
4. Which MFTECmd command correctly parses the $UsnJrnl:$J data stream and exports to CSV?
✓ Correct! MFTECmd uses the -f flag to specify the input file (the extracted $J stream), --csv for the output directory, and --csvf for the output filename. MFTECmd automatically detects whether the input is an $MFT, $LogFile, or $J stream based on its internal structure. Option A uses -d (directory mode, wrong). Option B uses EvtxECmd (wrong tool—that’s for Event Logs). Option C points to the $MFT file, not $J.
✗ Not quite. The correct answer is D. MFTECmd parses NTFS structures ($MFT, $LogFile, $J); EvtxECmd is for Event Logs (B is wrong tool). The -f flag specifies a single file; -d is for directories (A is wrong). The input must be the extracted $J file, not the $MFT (C is wrong input).
5. Why is $UsnJrnl generally more useful than $LogFile for building a file-system timeline covering the past week?
✓ Correct! The key advantage of $UsnJrnl for timeline work is its longer retention.
