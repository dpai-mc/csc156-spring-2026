Program Execution Artifacts — Prefetch, AmCache, and ShimCache

⏱ 50–65 min CSLO 2 CSLO 1

1. Introduction & Objectives

In Module 8.1 you learned how Windows Event Logs record who logged on, when, and from where. But Event Logs rarely tell you what programs a user ran during that session. That is where program execution artifacts come in.

Windows maintains several independent mechanisms that track which executables have been launched on a system. Each artifact records slightly different information, has different retention characteristics, and carries different evidentiary weight. A skilled examiner uses them together—and cross-references them with Event Logs—to build an airtight picture of program activity.

🎯 Why This Matters

Consider these investigative questions:

  • Did the suspect run a data-exfiltration tool on this workstation?
  • Was the ransomware binary actually executed, or was it only downloaded?
  • Did the attacker use PsExec to move laterally to this machine?
  • Has an anti-forensic wiping utility ever been run on this system?

Execution artifacts—Prefetch, AmCache, and ShimCache—can answer every one of these questions, even after the executable itself has been deleted.

Module Objectives

By the end of this module you will be able to:

  1. Explain the purpose, location, and internal structure of Windows Prefetch files, including OS version differences.
  2. Parse Prefetch files using PECmd and interpret the output (run count, last-run timestamps, referenced files).
  3. Describe the AmCache.hve Registry hive—its location, key fields, and forensic significance.
  4. Explain what ShimCache (AppCompatCache) records and articulate the critical nuance that ShimCache proves file presence, not necessarily execution.
  5. Correlate execution artifacts with Event Log data to confirm who ran a program, when, and how.
  6. Compare Prefetch, AmCache, ShimCache, and UserAssist (from Week 7) in terms of what each records, its limitations, and OS support.

2. Prefetch Files

Windows Prefetch is a performance-optimization feature that monitors the first ~10 seconds of an application’s launch, records which files and directories the application accesses, and stores that data in a .pf file. On subsequent launches, the OS uses this data to pre-load resources into memory, reducing startup time.

For forensic examiners, Prefetch is a goldmine: it records evidence of execution that persists even after the executable is deleted.

📂 Location

C:\Windows\Prefetch\

This directory contains .pf files, one per unique executable (by name and path). The directory is limited to 1,024 entries on Windows 10/11 (128 on Windows 7/8), so older entries are eventually evicted.

File Naming Convention

Prefetch files follow a specific naming pattern:

EXECUTABLE_NAME-XXXXXXXX.pf

Where:

  • EXECUTABLE_NAME — uppercase name of the executable (e.g., CMD.EXE).
  • XXXXXXXX — an 8-character hexadecimal hash of the executable’s file path. This means the same executable launched from two different locations produces two separate Prefetch files.

Example: CMD.EXE-4A81B364.pf and CMD.EXE-AC113AA8.pf represent cmd.exe launched from two different paths.

What a Prefetch File Records

Key Data Fields in a Prefetch File
FieldDescriptionForensic Value
Executable name Name of the program that was run. Confirms the program existed and was executed on this system.
Run count Number of times the executable has been launched. Distinguishes a one-time event from habitual usage.
Last run timestamp(s) The most recent execution time. On Windows 8+, the last 8 run times are stored. Establishes when the program was used. Multiple timestamps show a pattern of usage over time.
Referenced files & directories List of files and folders accessed during the first ~10 seconds of execution. Reveals what the program interacted with—DLLs loaded, config files read, data files opened. Can link an executable to specific data files.
Volume information Volume serial number and creation timestamp of the volume the executable was on. Can tie execution to a specific drive, including removable media.

OS Version Differences

The Prefetch format has evolved across Windows versions. Understanding these differences is important when examining images from different OS versions:

Windows 7 (Prefetch Version 23)

  • Timestamps: Only the single most recent run time is stored.
  • Compression: Files are not compressed.
  • Max entries: 128 Prefetch files in the directory.
  • Forensic note: With only one timestamp and a lower entry limit, Windows 7 Prefetch provides less historical depth than later versions.

Windows 8 / 8.1 (Prefetch Version 26)

  • Timestamps: The last 8 run times are stored—a major improvement for forensic analysis.
  • Compression: Files are compressed using MAM (Microsoft Application Model) compression. Tools like PECmd handle decompression automatically.
  • Max entries: 1,024 Prefetch files.
  • Forensic note: The jump from 1 to 8 timestamps provides a usage pattern over time, not just the last event.

Windows 10 / 11 (Prefetch Version 30)

  • Timestamps: The last 8 run times are stored (same as Windows 8+).
  • Compression: Files are compressed (same MAM compression as Win 8).
  • Max entries: 1,024 Prefetch files.
  • Forensic note: Functionally similar to Windows 8 for forensic purposes. The internal format version changed (30 vs. 26) but the data fields are largely the same. PECmd handles both transparently.
⚠️ When Prefetch Is Disabled

Prefetch can be disabled via Group Policy or Registry settings. It is also disabled by default on Windows Server editions. Additionally, systems with SSDs may have the Superfetch/SysMain service configured differently, though Prefetch itself typically remains enabled on client editions of Windows 10/11 even with SSDs.

Always verify that Prefetch is enabled on the system under examination. Check the Registry key:
HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management\PrefetchParameters
Value EnablePrefetcher: 0 = disabled, 1 = application only, 2 = boot only, 3 = both (default on client editions).

Deep Dive: How the Path Hash Works

The 8-character hex value in the Prefetch filename is a hash of the full path to the executable (including the device path). For example:

  • C:\Windows\System32\cmd.exe → hash 4A81B364
  • C:\Users\jsmith\Desktop\cmd.exe → hash AC113AA8

This means:

  • If an attacker copies cmd.exe to a different directory and runs it from there, a new Prefetch file is created with a different hash.
  • Finding a Prefetch file for an executable in an unusual path (e.g., C:\Windows\Temp\) is a strong indicator of suspicious activity.
  • The hash algorithm is documented and deterministic, so you can compute the expected hash for a known path to verify which Prefetch file corresponds to which location.

3. Tool Walk-Through: PECmd

PECmd (Prefetch Explorer Command Line) is part of Eric Zimmerman’s open-source forensic tool suite. It parses individual .pf files or an entire Prefetch directory and exports structured data to CSV, JSON, or the console.

🛠️ Hands-On Walkthrough

Follow along with these commands on your forensic workstation. If you have a forensic image mounted, navigate to the C:\Windows\Prefetch directory within the image. Otherwise, you can practice on your own system’s Prefetch directory (run the tool as Administrator).

Step 1: Parse a Single Prefetch File

PECmd.exe -f "C:\Windows\Prefetch\CMD.EXE-4A81B364.pf"

This displays parsed output directly in the console. Key fields to look for:

  • Executable name and the path hash.
  • Run count — how many times the program has been launched.
  • Last run — the most recent execution timestamp (UTC).
  • Previous runs (Win 8+) — up to 7 additional historical run times.
  • Volume information — serial number and creation time.
  • Directories referenced and Files referenced — resources the program accessed at launch.

Step 2: Parse an Entire Prefetch Directory to CSV

PECmd.exe -d "C:\Windows\Prefetch" --csv "C:\Output" --csvf "Prefetch_parsed.csv"

Flags explained:

  • -d — path to a directory of .pf files (batch processing).
  • --csv — output directory for CSV files.
  • --csvf — filename for the main timeline CSV.

PECmd generates two CSV files:

  1. A timeline CSV (the filename you specified) — one row per Prefetch file with run count, timestamps, and volume info.
  2. A referenced-files CSV (automatically named with _Timeline suffix) — one row per referenced file/directory per Prefetch entry.

Step 3: Load and Analyze in Timeline Explorer

  1. Open Timeline Explorer and load the main CSV.
  2. Sort by LastRun to see the most recently executed programs.
  3. Filter the ExecutableName column to search for a specific program (e.g., PSEXEC).
  4. Check the RunCount column to see how many times each program was launched.

Step 4: Interpreting Sample Output

Here is an example of parsed PECmd console output for a suspicious executable:

Notice the red flags in this output:

  • It runs from C:\Windows\Temp\—not a normal location for system binaries.
  • It was run 3 times over approximately 8 hours.

4. AmCache.hve

The AmCache is a Windows Registry hive that tracks information about applications and executables that have been run or installed on the system. It was introduced in Windows 8 and is present on all subsequent versions (Windows 8, 8.1, 10, 11, and corresponding Server editions).

📂 Location

C:\Windows\AppCompat\Programs\Amcache.hve

This is a standalone Registry hive file (not loaded into the standard Registry editor view). It must be parsed with forensic tools such as Eric Zimmerman’s AmcacheParser or loaded manually in Registry Explorer.

What AmCache Records

AmCache stores metadata about executables and application installations. The data is organized under several key paths within the hive:

Key AmCache Data Fields
FieldDescriptionForensic Value
Full file path Complete path to the executable. Shows exactly where the program was located on disk.
SHA-1 hash Cryptographic hash of the executable file. Extremely valuable. Allows positive identification of the binary even after deletion. Can be searched against malware databases (e.g., VirusTotal).
File size Size of the executable in bytes. Supports identification and comparison with known samples.
Publisher / company name From the file’s version information (PE header). Helps distinguish legitimate software from suspicious binaries.
File description & version Product name, version number from PE metadata. Identifies the specific version of software that was present.
Link date (compile time) Timestamp from the PE header indicating when the binary was compiled. Can reveal when malware was built; anomalous compile times are suspicious.
Last modified timestamp Key last-write time in the AmCache hive. Approximates when the entry was created or updated—often close to first execution or installation time.

AmCache Structure

The hive is organized into several subkeys. The two most forensically relevant are:

  • Root\InventoryApplicationFile — records individual executable files with the fields described above. This is the primary location for execution/installation evidence on Windows 10+.
  • Root\InventoryApplication — records installed applications (from Add/Remove Programs), including install date, publisher, and version.
Historical Note: The File Key Structure (Windows 8 / Early Win 10)

On older systems (Windows 8 and early builds of Windows 10), AmCache used a different structure under Root\File\ with volume GUIDs as subkeys. Each entry was keyed by a sequence number and contained similar fields (path, SHA-1, size, etc.).

Modern versions of Windows 10 and all Windows 11 systems use the InventoryApplicationFile structure. Forensic tools like AmcacheParser handle both formats automatically, but it is useful to know that the structure changed in case you encounter older images.

Forensic Significance

💡 Why AmCache Is Uniquely Valuable

AmCache is the only standard Windows artifact that records a SHA-1 hash of executed or installed programs. This means:

  • Even if the attacker deletes the malware binary, the AmCache entry preserves the hash.
  • You can search the hash on threat intelligence platforms to determine if the binary is known malware.
  • You can compare the hash with other systems on the network to identify lateral movement (same malware deployed to multiple machines).

No other execution artifact (Prefetch, ShimCache, or UserAssist) provides a cryptographic hash of the executable.

Parsing AmCache with AmcacheParser

AmcacheParser.exe -f "C:\Evidence\Amcache.hve" --csv "C:\Output" --csvf "AmCache_parsed.csv"

The tool exports CSV files that can be loaded into Timeline Explorer for filtering and analysis. Key columns to examine: FullPath, SHA1, FileSize, Publisher, LinkDate, and KeyLastWriteTimestamp.

5. ShimCache / AppCompatCache

The Application Compatibility Cache—commonly called ShimCache—is a Windows mechanism that helps the OS determine whether an executable needs compatibility fixes (“shims”) to run properly. It is stored in the SYSTEM Registry hive and records metadata about executables the OS has encountered.

📂 Location

HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache

The data is stored as a binary blob within the AppCompatCache value. It must be parsed with a specialized tool—it is not human-readable in a Registry editor.

What ShimCache Records

ShimCache Data Fields
FieldDescription
File pathFull path to the executable file.
Last modified timeThe $STANDARD_INFORMATION last-modified timestamp of the file (from the file system, not the time it was added to ShimCache).
Cache entry positionThe order of the entry in the cache (most recent entries first). This is a relative ordering, not a timestamp.
Execution flag (Win 7/Server 2008 R2 only)A flag indicating whether the file was actually executed. Not present on Windows 8+.

The Critical Nuance: Presence ≠ Execution

⚠️ Key Forensic Distinction

On Windows 8 and later, ShimCache records that a file was present on the system, not that it was executed.

A file can appear in ShimCache simply because:

  • The user browsed to the directory containing the executable (e.g., opened the folder in Explorer).
  • The OS scanned the directory (e.g., during an antivirus scan or Windows Search indexing).
  • An application enumerated files in the directory.

On Windows 7 and Server 2008 R2, the execution flag can distinguish between presence and execution. On Windows 8+, this flag is absent, so ShimCache alone cannot prove execution—only that the file existed on disk at some point.

This is one of the most commonly misunderstood points in Windows forensics. Many inexperienced examiners incorrectly claim that a ShimCache entry proves a program was run. In court or in a professional report, this distinction matters enormously.

How ShimCache Works

  1. When an executable file is “encountered” by the OS, the Application Compatibility subsystem checks the ShimCache to see if it already has compatibility information for that file.
  2. If not, the file’s path and last-modified timestamp are added to the cache.
  3. The cache is maintained in memory during the current session and is only written to the Registry when the system shuts down.
⚠️ Shutdown Dependency

Because ShimCache is written to the Registry only at shutdown, entries from the current (still-running) session are only in memory. If a system is seized while running, the on-disk ShimCache may be missing the most recent entries. This is one reason memory forensics (Week 10) is important—the in-memory ShimCache can be extracted from a RAM dump.

Parsing ShimCache with AppCompatCacheParser

Step 1: Parse from a SYSTEM Hive

AppCompatCacheParser.exe -f "C:\Evidence\SYSTEM" --csv "C:\Output" --csvf "ShimCache_parsed.csv"

Flags:

  • -f — path to the SYSTEM Registry hive file extracted from the forensic image.
  • --csv / --csvf — CSV output directory and filename.

Step 2: Examine the Output

Load the CSV in Timeline Explorer. Key columns:

  • Path — full path to the executable.
  • LastModifiedTimeUTC — the file’s last-modified timestamp (filesystem metadata, not execution time).
  • CacheEntryPosition — lower numbers are more recent entries. Use this for relative chronological ordering.
  • Executed — (Windows 7 only) “Yes” or “No”.

Step 3: Investigative Use

ShimCache is most useful when combined with other artifacts:

  • ShimCache entry + Prefetch file → confirms both presence and execution.
  • ShimCache entry + AmCache entry → provides the SHA-1 hash of the binary that was present.
  • ShimCache entry alone (Win 8+) → proves the file existed on disk, but does not prove it was executed.

6. Correlation Exercise

🔍 Scenario: Did PsExec Run on FILESVR-01?

You are investigating a suspected lateral-movement incident. The security team believes an attacker used PsExec to remotely execute commands on the file server FILESVR-01. You have extracted artifacts from a forensic image of FILESVR-01 and parsed them with Eric Zimmerman’s tools. Your task: determine whether psexec.exe (or its service component PSEXESVC.exe) was executed on this system, when, and correlate with Event Log data.

Evidence Set A: Prefetch (PECmd Output)

---------- Prefetch Entry 1 ----------
Executable name: PSEXESVC.EXE
Hash: 7A2B3C4D
Run count: 2
Last run: 2026-03-01 14:32:18 (UTC)
  Other run times: 2026-03-01 09:15:44 (UTC)

Directories referenced:
  \VOLUME{...}\WINDOWS\
  \VOLUME{...}\WINDOWS\SYSTEM32\

Files referenced:
  \VOLUME{...}\WINDOWS\PSEXESVC.EXE
  \VOLUME{...}\WINDOWS\SYSTEM32\NTDLL.DLL
  \VOLUME{...}\WINDOWS\SYSTEM32\KERNEL32.DLL

Evidence Set B: Event Log Entries (EvtxECmd Output)

Record A:
  TimeCreated:  2026-03-01T09:15:30Z
  EventId:      4624
  Channel:      Security
  Computer:     FILESVR-01
  UserName:     admin_jdoe
  LogonType:    3
  RemoteHost:   10.0.10.55
  LogonId:      0x2B8F401

Record B:
  TimeCreated:  2026-03-01T09:15:42Z
  EventId:      7045
  Channel:      System
  Computer:     FILESVR-01
  ServiceName:  PSEXESVC
  ImagePath:    C:\Windows\PSEXESVC.exe
  AccountName:  LocalSystem

Record C:
  TimeCreated:  2026-03-01T14:32:05Z
  EventId:      4624
  Channel:      Security
  Computer:     FILESVR-01
  UserName:     admin_jdoe
  LogonType:    3
  RemoteHost:   10.0.10.55
  LogonId:      0x3C7A902

Record D:
  TimeCreated:  2026-03-01T14:32:16Z
  EventId:      7036
  Channel:      System
  Computer:     FILESVR-01
  ServiceName:  PSEXESVC
  State:        Running

Investigative Questions

Use the evidence above to answer each question. Click to reveal the analysis.

Q1: Was PsExec executed on FILESVR-01? What evidence confirms this?

Answer: Yes. Multiple independent artifacts confirm execution:

  1. Prefetch: A Prefetch file exists for PSEXESVC.EXE with a run count of 2 and specific timestamps. Prefetch files are created only when a program executes.
  2. Event Log (7045): A new service named PSEXESVC was installed (Record B). PsExec works by copying PSEXESVC.exe to the target system and installing it as a service.
  3. Event Log (7036): The PSEXESVC service entered the Running state (Record D), confirming the service binary executed.

Three independent artifact types (Prefetch, service installation event, service state-change event) all corroborate execution. This is strong, multi-source evidence.

Q2: How many times was PsExec used, and when?

Answer: At least 2 times.

  • First use: ~09:15 UTC on March 1, 2026 (Prefetch second run time: 09:15:44; service install event: 09:15:42).
  • Second use: ~14:32 UTC on March 1, 2026 (Prefetch last run: 14:32:18; service running event: 14:32:16).

The Prefetch run count of 2 matches the two sets of corroborating Event Log entries. The timestamps are within seconds of each other across artifacts, providing strong temporal correlation.

Q3: Who initiated the PsExec sessions? How do you know?

Answer: The account admin_jdoe, connecting from IP 10.0.10.55.

Records A and C show Event 4624 (successful logon) with LogonType: 3 (Network logon) from 10.0.10.55 under the admin_jdoe account. Both logon events occur seconds before the corresponding PSEXESVC service events, establishing a clear causal link:

  • 09:15:30 → network logon by admin_jdoe → 09:15:42 → PSEXESVC installed.
  • 14:32:05 → network logon by admin_jdoe → 14:32:16 → PSEXESVC running.

Note: Prefetch alone cannot tell you who ran a program—it records execution but not the user context. The Event Log provides the “who” and “from where.” This is why cross-artifact correlation is essential.

Q4: If you found a ShimCache entry for PSEXESVC.EXE but no Prefetch file, could you conclude it was executed?

Answer: Not definitively (on Windows 8+).

A ShimCache entry on Windows 8+ proves only that the file was present on disk—it could have been written to the system without being executed. You would need corroborating evidence from another artifact (Event Log service events, AmCache, UserAssist, or memory forensics) to confirm execution.

On Windows 7, you would check the ShimCache execution flag. If it shows “Yes,” that provides stronger (though not absolute) evidence of execution.

🧩 The Power of Correlation

No single execution artifact tells the complete story. The strongest forensic conclusions come from correlating multiple artifacts:

  • Prefetch tells you what ran and when.
  • AmCache tells you what binary it was (via SHA-1 hash).
  • ShimCache confirms the file was present on disk.
  • UserAssist tells you which user launched it (if GUI-based).
  • Event Logs (Module 8.1) tell you who was logged on and from where.

Together, these artifacts answer: What program ran? When? Who ran it? Where did it come from? What did it interact with? Is it known malware?

✓ Correct! A Prefetch file is created when a program executes. The run count of 1 and the last-run timestamp confirm a single execution at that time. However, Prefetch does not record who ran the program (eliminating C), and a Prefetch entry does prove execution (eliminating A and D—the “presence only” caveat applies to ShimCache, not Prefetch).
✗ Not quite. The correct answer is B. Prefetch files are created when a program executes, so the entry proves execution (not just presence—that’s ShimCache). The run count of 1 confirms a single execution. Prefetch does not record which user ran the program, so C overstates what Prefetch alone can prove.
2. Which execution artifact uniquely provides a SHA-1 hash of the executable, allowing the examiner to positively identify the binary and search malware databases?
✓ Correct! AmCache is the only standard Windows execution artifact that stores a SHA-1 hash of the executable file. This makes it uniquely valuable for malware identification—you can search the hash on platforms like VirusTotal even after the binary has been deleted from disk.
✗ Not quite. The correct answer is D (AmCache). Prefetch records run counts, timestamps, and referenced files but no hash. ShimCache records file paths and last-modified times but no hash. UserAssist records GUI execution counts and times but no hash. Only AmCache stores a SHA-1 hash of the executable.
3. An examiner finds a ShimCache entry for C:\Tools\wce.exe on a Windows 10 system. No Prefetch file or AmCache entry exists for this executable. What is the most accurate conclusion?
✓ Correct! On Windows 10, ShimCache records file presence, not execution. The file was encountered by the OS (e.g., the user browsed to the directory, or the system scanned it), but without corroborating evidence from Prefetch, AmCache, Event Logs, or memory forensics, you cannot conclude it was executed. This is the critical “presence ≠ execution” nuance.
✗ Not quite. The correct answer is A. On Windows 8 and later, ShimCache does not prove execution—only that the file was present on disk. Without supporting evidence from Prefetch, AmCache, or Event Logs, the examiner can only conclude the file existed, not that it ran. This is one of the most important nuances in Windows forensics.
4. On a Windows 10 system, how many historical run timestamps does a single Prefetch file store?
✓ Correct! Windows 8 and later (including Windows 10 and 11) store the last 8 run timestamps in each Prefetch file. Windows 7 stores only the single most recent run time. This is a significant forensic improvement—8 timestamps let you see a pattern of usage over time, not just the last event.
✗ Not quite. The correct answer is C (8). Windows 10/11 Prefetch files (version 30) store the last 8 run timestamps. Only Windows 7 (version 23) is limited to a single timestamp. The storage is not unlimited—only the 8 most recent runs are retained.
5. Why is ShimCache data only written to the SYSTEM Registry hive at system shutdown, and what is the forensic implication of this behavior?
✓ Correct! ShimCache is maintained in memory during the active session for performance reasons and is flushed to the SYSTEM Registry hive only at shutdown. The forensic implication is significant: if a system is seized while still running (e.g., during a live response), the on-disk ShimCache will be missing entries from the current session. To recover those entries, an examiner needs a memory dump (covered in Week 10: Memory Forensics).
✗ Not quite. The correct answer is B. ShimCache is kept in memory for performance and written to the Registry at shutdown. This means the on-disk version may lack entries from the current session.
