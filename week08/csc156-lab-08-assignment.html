Phase 1 — Event Log Analysis

Background

Windows Event Logs (.evtx files) are stored in C:\Windows\System32\winevt\Logs and record system, security, and application events. In this phase you will examine the Security and System log channels to identify a suspicious time window based on patterns you observe in the logs, then document the security-relevant events within that window.

Tools

ToolPurpose
Windows Event ViewerGUI-based browsing and filtering of .evtx files
EvtxECmd (Eric Zimmerman)Command-line parsing; export to CSV for Timeline Explorer

Key Event IDs to Investigate

Event IDChannelMeaning
4624SecuritySuccessful logon
4625SecurityFailed logon attempt
4672SecuritySpecial privileges assigned to new logon
4720SecurityUser account created
4726SecurityUser account deleted
7035SystemService sent a start/stop control
7036SystemService entered running/stopped state

Step-by-Step Tasks

  1. In Autopsy, open the vm32 image case from Lab 07. In the left-panel file tree, navigate to Windows/System32/winevt/Logs. Extract Security.evtx and System.evtx to a local output folder on your analysis machine.
  2. Parse the Security log using EvtxECmd: 
    EvtxECmd.exe -f "Security.evtx" --csv "C:\Output" --csvf Security_parsed.csv
    Open the resulting CSV in Timeline Explorer. Browse the full timeline to look for spikes or clusters of security-relevant events.
  3. Define your suspicious time window. Before filtering, scroll through the parsed log (or sort by time) to identify a 1–3 hour period showing unusual activity — for example, a cluster of logon failures, an after-hours successful logon, or unexpected account activity. Record the start and end time of your chosen window and explain why you selected it. Apply this filter for the steps that follow.
  4. Identify failed logon attempts (Event ID 4625) within your chosen time window. Record the number of failures, the target account names, source IP addresses, and logon types. Note any patterns (e.g., rapid sequential failures suggesting password guessing).
  5. Identify successful logons (Event ID 4624) within your chosen time window. For each successful logon, record the account name, logon type (e.g., Type 2 = Interactive, Type 3 = Network, Type 10 = RemoteInteractive), source IP, and timestamp.
  6. Check for privilege escalation (Event ID 4672) within your chosen time window. Determine whether any logon session was assigned special (administrative) privileges.
  7. Parse the System log using EvtxECmd and filter for Events 7035 and 7036 within your chosen time window. Identify any services that were started or stopped. Pay attention to unusual services (e.g., remote access tools, new services not part of the default Windows configuration).
  8. Check for account management events (Event IDs 4720, 4726) within your chosen time window. Determine whether any new accounts were created or deleted.
  • Brute-force pattern: Multiple 4625 events in rapid succession targeting the same account, followed by a 4624 success — suggests password guessing.
  • Logon Type 10 (RemoteInteractive): Indicates an RDP session — may indicate remote access activity.
  • New or unusual services: Attackers sometimes install services for persistence or remote access. Look for service names you don't recognize.
  • Account creation then deletion: A 4720 followed by a 4726 for the same account may indicate a temporary "backdoor" account.

Phase 1 — Response Fields

Identify the time window you selected as suspicious — give the start and end timestamps and explain your reasoning for choosing it. Then list the failed logon attempts (Event ID 4625) found within that window, noting timestamp, target account, source IP, and logon type. How many total failures did you observe?

List the successful logon events (Event ID 4624) within your chosen window. For each, note the timestamp, account name, logon type, and source IP. Which logon appears most suspicious and why?

Did you find any Event ID 4672 (special privileges) entries associated with the suspicious logon in your chosen window? Were any accounts created (4720) or deleted (4726)? Describe your findings.

List any services started or stopped (Event IDs 7035/7036) within your chosen window. Are any of these services unusual or potentially related to unauthorized activity? Explain.

In 3–5 sentences, summarize what the Event Log evidence tells you about what happened during your chosen window. What is your working hypothesis at this point?

Phase 2 — Prefetch File Analysis

Background

Windows Prefetch files (.pf) are stored in C:\Windows\Prefetch and are created when an application is executed. Each .pf file records the executable name, a run count, up to the last 8 execution timestamps (Windows 10), and the files/directories referenced during execution. Prefetch is one of the most reliable artifacts for proving program execution.

Tools

ToolPurpose
PECmd (Eric Zimmerman)Parse individual or batch Prefetch files; output to CSV or console
Timeline ExplorerView and filter PECmd CSV output

Step-by-Step Tasks

  1. In Autopsy, navigate to the Prefetch directory on the forensic image: Windows/Prefetch. Note the number of .pf files present. Extract the entire Prefetch folder to your local output directory.
  2. Run PECmd in batch mode to parse all Prefetch files: 
    PECmd.exe -d "C:\Output\Prefetch" --csv "C:\Output" --csvf Prefetch_parsed.csv
    Open the resulting CSV in Timeline Explorer. (Replace the -d path with wherever you extracted the Prefetch folder.)
  3. Filter by your Phase 1 suspicious window (or a nearby window you can justify). Identify all executables with a "last run" timestamp within or near that window.
  4. Identify suspicious executables. Look for command-line tools or utilities that are unusual on a standard workstation, such as:
    • CMD.EXE, POWERSHELL.EXE — command shells
    • PSEXEC.EXE, WMIC.EXE — remote execution tools
    • NET.EXE, NET1.EXE — network/account commands
    • RAR.EXE, 7Z.EXE — archiving tools (potential data staging)
    • Cloud sync or backup utilities (potential data exfiltration)
    • Any executable with an unfamiliar or random-looking name
  5. For each suspicious executable, record:
    • Executable name and full path (from the .pf file name and referenced files)
    • Run count (how many times it has ever been executed)
    • Last run timestamp(s)
    • Notable referenced files or directories (e.g., user profile paths, external drive letters, network paths, or sensitive-looking folder names)
  6. Correlate with Phase 1. Do the execution timestamps for any of these programs fall within or near your Phase 1 window? Note any correlations or gaps.
  • Low run count + window timestamp: A run count of 1 with a last-run time during your chosen window strongly suggests the program was executed for the first time by an unauthorized user.
  • Archiving tools: Execution of RAR.EXE or 7Z.EXE during the window may indicate data was being packaged for exfiltration.
  • Referenced files: PECmd output includes files and directories the executable touched. Look for references to sensitive-looking paths such as user profile directories, external drives, or network locations.
  • Sequence matters: If NET.EXE ran before RAR.EXE, which ran before a cloud sync tool, this suggests reconnaissance → staging → exfiltration.

Phase 2 — Response Fields

How many total Prefetch files were present in the directory? How many executables had a "last run" timestamp within or near your Phase 1 window? State the window you used for this filter.

Select 3–5 executables you consider suspicious or noteworthy (based on name, run count, timing, or referenced files). For each, provide: (a) executable name, (b) run count, (c) last-run timestamp(s), (d) notable referenced files or directories, and (e) a brief justification for why you flagged it.

Compare your Prefetch findings with the Event Log evidence from Phase 1. Do the execution timestamps align with the suspicious logon session you identified? Describe any correlations or discrepancies you observe.

In 3–5 sentences, summarize what the Prefetch evidence reveals about the activity on this system. What programs were run, in what apparent sequence, and what does this suggest about the objectives of whoever was active during your chosen window?

Phase 3 — Volume Shadow Copy Examination

Background

The Volume Shadow Copy Service (VSS) creates point-in-time snapshots of NTFS volumes, triggered by events such as system restore points, Windows Update, or scheduled tasks. These snapshots allow forensic examiners to recover deleted or modified files from prior system states. In this phase you will access shadow copies from the forensic image to determine whether files were altered or deleted during or after the activity you identified.

Tools

ToolPurpose
AutopsyBrowse Volume Shadow Copies from the forensic image via the left-panel VSC tree
File comparison utilityCompare current vs. historical file versions (e.g., WinMerge, fc, diff)

Step-by-Step Tasks

  1. In Autopsy, locate the Volume Shadow Copies. In the left-panel tree, look for the Volume Shadow Copies section listed under the image source. Autopsy surfaces available shadow copies automatically. The forensic image is already read-only in Autopsy — no additional mounting step is required.
    ⚠️ Important

    Autopsy opens forensic images in read-only mode. Do not export and re-open the image in a way that could modify it. Always preserve evidence integrity.

  2. Enumerate available shadow copies. Autopsy lists available shadow copies in the left-panel tree with their creation timestamps. Record each shadow copy's creation date and time directly from the Autopsy interface. Note how many shadow copies exist.
  3. Identify the most relevant shadow copy. Select the shadow copy with a creation date closest to but before the start of your Phase 1 window. This represents the system state before the activity you identified. Record why you chose this snapshot.
  4. Browse the shadow copy. Navigate to directories of interest, such as:
    • A user profile directory and any subfolders you consider sensitive or potentially relevant (e.g., Documents, Desktop, Downloads, AppData\Local\Temp)
    • Any paths identified from Phase 2 Prefetch referenced files
    • Any other locations you can justify as forensically significant
  5. Compare file versions. Select 2–3 files that exist in both the shadow copy (pre-activity) and the current image. Compare them:
    • Are any files present in the shadow copy but missing from the current image? (Deleted after activity)
    • Are any files present in the current image but absent from the shadow copy? (Created during/after activity)
    • Do any files have different content or sizes between the two versions? (Modified during activity)
  6. Extract relevant historical files. In Autopsy, right-click the file in the shadow copy tree and choose Extract File(s) to save it to your output folder. Record the hash (MD5 or SHA-256) of each extracted file.
  7. Check for anti-forensic activity. Look in the System Event Log (from Phase 1) for evidence that shadow copies were deleted or the Volume Shadow Copy Service was stopped unexpectedly. Also look for gaps in the shadow copy timeline — a large gap between creation dates that does not match normal system activity may indicate deliberate deletion.
  • Deleted files recovered: If a file exists in the shadow copy but not on the live volume, someone deleted it after the snapshot was taken. This is strong evidence of cover-up activity.
  • Log file tampering: Compare Event Log files between the shadow copy and the current image. If the current logs are smaller or missing entries, log clearing may have occurred.
  • New files post-activity: Files present on the current image but not in the pre-activity shadow copy were created during or after the intrusion — potential attacker tools or staged data.
  • Missing or sparse shadow copies: If shadow copies are absent or span an unexpected gap, this may indicate anti-forensic activity (common in ransomware and intrusion scenarios) or simply that VSS was disabled on this system. Note whichever you observe.

Phase 3 — Response Fields

How many Volume Shadow Copies were available on the forensic image? List the creation date and time of each. Which shadow copy did you select as your pre-activity baseline, and why? How does its creation date relate to your Phase 1 window?

Describe 2–3 files or folders you selected for comparison (explain why you chose them). For each, state: (a) file name and path, (b) presence/absence/modification status in each version, (c) any size or content differences, and (d) whether the difference appears normal or suspicious, and why.

List any files you extracted from the shadow copy. For each, provide the file name, path, hash value (MD5 or SHA-256), and explain why this file is relevant to the investigation.

Did you find any evidence that shadow copies were deleted or the VSS service was tampered with? Describe what you found (or note the absence of such evidence and what that suggests).

In 3–5 sentences, summarize what the Volume Shadow Copy evidence adds to your understanding of the activity on this system. How does it complement or expand upon the Event Log and Prefetch findings?
