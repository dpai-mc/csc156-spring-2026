Phase 1 — Event Log Analysis

Background

Windows Event Logs (.evtx files) are stored in C:\Windows\System32\winevt\Logs and record system, security, and application events. In this phase you will examine the Security and System log channels to identify suspicious logon activity and service changes during the incident window (Feb 28, 2026, 21:00–23:30 UTC).

Tools

ToolPurpose
Windows Event ViewerGUI-based browsing and filtering of .evtx files
EvtxECmd (Eric Zimmerman)Command-line parsing; export to CSV for Timeline Explorer

Key Event IDs to Investigate

Event IDChannelMeaning
4624SecuritySuccessful logon
4625SecurityFailed logon attempt
4672SecuritySpecial privileges assigned to new logon
4720SecurityUser account created
4726SecurityUser account deleted
7035SystemService sent a start/stop control
7036SystemService entered running/stopped state

Step-by-Step Tasks

  1. Mount the forensic image in your forensic workstation using Arsenal Image Mounter (read-only mode). Navigate to the Event Log directory: C:\Windows\System32\winevt\Logs.
  2. Open the Security log (Security.evtx) in Windows Event Viewer. Apply a time filter for Feb 28, 2026, 21:00–23:30 UTC. Alternatively, use EvtxECmd: 
    EvtxECmd.exe -f "Security.evtx" --csv "C:\Output" --csvf Security_parsed.csv
    Then open the CSV in Timeline Explorer and filter by the incident time window.
  3. Identify failed logon attempts (Event ID 4625). Record the number of failures, the target account names, source IP addresses, and logon types. Note any brute-force patterns (rapid sequential failures).
  4. Identify successful logons (Event ID 4624). For each successful logon during the window, record the account name, logon type (e.g., Type 2 = Interactive, Type 3 = Network, Type 10 = RemoteInteractive), source IP, and timestamp.
  5. Check for privilege escalation (Event ID 4672). Determine whether any logon session was assigned special (administrative) privileges.
  6. Open the System log (System.evtx) and filter for Events 7035 and 7036 in the same time window. Identify any services that were started or stopped during the incident. Pay attention to unusual services (e.g., remote access tools, new services not part of the default Windows configuration).
  7. Check for account management events (Event IDs 4720, 4726) in the Security log. Determine whether any new accounts were created or deleted during the window.
  • Brute-force pattern: Multiple 4625 events in rapid succession targeting the same account, followed by a 4624 success — suggests password guessing.
  • Logon Type 10 (RemoteInteractive): Indicates an RDP session — unusual for a clinical workstation after hours.
  • New or unusual services: Attackers sometimes install services for persistence or remote access. Look for service names you don't recognize.
  • Account creation then deletion: A 4720 followed by a 4726 for the same account may indicate a temporary "backdoor" account.

Phase 1 — Response Fields

List the failed logon attempts (Event ID 4625) you found within the incident window. For each, note the timestamp, target account, source IP, and logon type. How many total failures did you observe?

List the successful logon events (Event ID 4624) in the incident window. For each, note the timestamp, account name, logon type, and source IP. Which logon appears most suspicious and why?

Did you find any Event ID 4672 (special privileges) entries associated with the suspicious logon? Were any accounts created (4720) or deleted (4726)? Describe your findings.

List any services started or stopped (Event IDs 7035/7036) during the incident window. Are any of these services unusual or potentially related to unauthorized activity? Explain.

In 3–5 sentences, summarize what the Event Log evidence tells you about what happened during the incident window. What is your working hypothesis at this point?

Phase 2 — Prefetch File Analysis

Background

Windows Prefetch files (.pf) are stored in C:\Windows\Prefetch and are created when an application is executed. Each .pf file records the executable name, a run count, up to the last 8 execution timestamps (Windows 10), and the files/directories referenced during execution. Prefetch is one of the most reliable artifacts for proving program execution.

Tools

ToolPurpose
PECmd (Eric Zimmerman)Parse individual or batch Prefetch files; output to CSV or console
Timeline ExplorerView and filter PECmd CSV output

Step-by-Step Tasks

  1. Navigate to the Prefetch directory on the mounted forensic image: C:\Windows\Prefetch. Note the number of .pf files present.
  2. Run PECmd in batch mode to parse all Prefetch files: 
    PECmd.exe -d "E:\Windows\Prefetch" --csv "C:\Output" --csvf Prefetch_parsed.csv
    Open the resulting CSV in Timeline Explorer.
  3. Filter by the incident time window (Feb 28, 2026, 21:00–23:30 UTC). Identify all executables with a "last run" timestamp within this window.
  4. Identify suspicious executables. Look for command-line tools or utilities that are unusual on a clinical workstation, such as:
    • CMD.EXE, POWERSHELL.EXE — command shells
    • PSEXEC.EXE, WMIC.EXE — remote execution tools
    • NET.EXE, NET1.EXE — network/account commands
    • RAR.EXE, 7Z.EXE — archiving tools (potential data staging)
    • RCLONE.EXE, MEGASYNC.EXE — cloud sync tools (potential exfiltration)
    • Any executable with an unfamiliar or random-looking name
  5. For each suspicious executable, record:
    • Executable name and full path (from the .pf file name and referenced files)
    • Run count (how many times it has ever been executed)
    • Last run timestamp(s)
    • Notable referenced files or directories (e.g., paths to patient data folders)
  6. Correlate with Phase 1. Do the Prefetch timestamps for suspicious programs align with the logon events you identified in Phase 1? Note any correlations.
  • Low run count + incident-window timestamp: A run count of 1 with a last-run time during the incident strongly suggests the program was executed for the first time by the attacker.
  • Archiving tools: Execution of RAR.EXE or 7Z.EXE during the incident may indicate data was being packaged for exfiltration.
  • Referenced files: PECmd output includes files and directories the executable touched. Look for references to sensitive data paths (e.g., PatientRecords, D:\ for USB drives).
  • Sequence matters: If NET.EXE ran before RAR.EXE, which ran before a cloud sync tool, this suggests reconnaissance → staging → exfiltration.

Phase 2 — Response Fields

How many total Prefetch files were present in the directory? How many executables had a "last run" timestamp within the incident window (Feb 28, 21:00–23:30 UTC)?

List each suspicious executable you identified. For each, provide: (a) executable name, (b) run count, (c) last run timestamp(s), and (d) any notable referenced files or directories.

Compare your Prefetch findings with the Event Log evidence from Phase 1. Do the execution timestamps align with the suspicious logon session? Describe any correlations or discrepancies you observe.

In 3–5 sentences, summarize what the Prefetch evidence reveals about the attacker's actions. What programs were run, in what apparent sequence, and what does this suggest about the attacker's objectives?

Phase 3 — Volume Shadow Copy Examination

Background

The Volume Shadow Copy Service (VSS) creates point-in-time snapshots of NTFS volumes, triggered by events such as system restore points, Windows Update, or scheduled tasks. These snapshots allow forensic examiners to recover deleted or modified files from prior system states. In this phase you will access shadow copies from the forensic image to determine whether files were altered or deleted during or after the incident.

Tools

ToolPurpose
Arsenal Image MounterMount forensic image with VSS access (read-only)
vshadowmount (libvshadow)Linux alternative for mounting shadow copies
File comparison utilityCompare current vs. historical file versions (e.g., WinMerge, fc, diff)

Step-by-Step Tasks

  1. Mount the forensic image with VSS support. In Arsenal Image Mounter, select the forensic image and enable the option to expose Volume Shadow Copies. Mount as read-only.
    ⚠️ Important

    Always mount forensic images in read-only / write-protected mode to preserve evidence integrity. Never write to the original image.

  2. Enumerate available shadow copies. Note how many shadow copies exist and their creation dates. You can use the command: 
    vssadmin list shadows /for=E:
    or view them through Arsenal Image Mounter's interface. Record the shadow copy IDs and creation timestamps.
  3. Identify the most relevant shadow copy. Select the shadow copy with a creation date closest to but before the incident window (before Feb 28, 2026, 21:00 UTC). This represents the system state before the suspected intrusion.
  4. Browse the shadow copy. Navigate to directories of interest, such as:
    • User profile directories (e.g., Users\Dr.Chen\Documents)
    • Any paths identified in Phase 2 Prefetch referenced files
    • Temporary directories (Users\Dr.Chen\AppData\Local\Temp)
  5. Compare file versions. Select 2–3 files that exist in both the shadow copy (pre-incident) and the current image. Compare them:
    • Are any files present in the shadow copy but missing from the current image? (Deleted after incident)
    • Are any files present in the current image but absent from the shadow copy? (Created during/after incident)
    • Do any files have different content or sizes between the two versions? (Modified during incident)
  6. Extract relevant historical files. Copy any significant files from the shadow copy to your analysis output folder for documentation. Record the hash (MD5 or SHA-256) of each extracted file.
  7. Check for anti-forensic activity. Look for evidence that the attacker attempted to delete shadow copies. Check the System Event Log (from Phase 1) for events related to vssadmin delete shadows or the Volume Shadow Copy Service being stopped.
  • Deleted files recovered: If a file exists in the shadow copy but not on the live volume, the attacker (or someone) deleted it after the snapshot was taken. This is strong evidence of cover-up activity.
  • Log file tampering: Compare Event Log files between the shadow copy and the current image. If the current logs are smaller or missing entries, log clearing may have occurred.
  • New files post-incident: Files present on the current image but not in the pre-incident shadow copy were created during or after the intrusion — potential attacker tools or staged data.
  • Ransomware indicator: If shadow copies have been deleted (no shadows available, or VSS service stopped), this is a common ransomware or anti-forensic tactic.

Phase 3 — Response Fields

How many Volume Shadow Copies were available on the forensic image? List the creation date/time of each. Which shadow copy did you select as the pre-incident baseline, and why?

Describe the 2–3 files you compared between the shadow copy and the current image. For each file, state: (a) file name and path, (b) whether it was present/absent/modified in each version, (c) any size or content differences, and (d) the forensic significance of the difference.

List any files you extracted from the shadow copy. For each, provide the file name, path, hash value (MD5 or SHA-256), and explain why this file is relevant to the investigation.

Did you find any evidence that the attacker attempted to delete shadow copies or tamper with VSS? Describe what you found (or note the absence of such evidence).

In 3–5 sentences, summarize what the Volume Shadow Copy evidence adds to your understanding of the incident. How does it complement or expand upon the Event Log and Prefetch findings?
