🎯 Module Overview

📋 What You'll Learn

Cryptographic hash verification is the cornerstone of forensic evidence integrity. A hash value serves as a unique "digital fingerprint" that proves your forensic image is an exact, unmodified copy of the original evidence.

This module explains how hash functions work, which algorithms to use, and how to properly document hash verification for court admissibility.

✓ Learning Objectives

By the end of this module, you will be able to:

  • Explain the purpose of cryptographic hashing in forensic investigations
  • Compare MD5, SHA-1, and SHA-256 hash algorithms
  • Calculate hash values using command-line and GUI tools
  • Verify forensic image integrity through hash comparison
  • Document hash values according to forensic standards
  • Respond appropriately when hash values don't match
🎓 CSLO 3: Perform data collection using forensic technology methods in accordance with evidence handling procedures

💡 Why Hashing Matters

According to NIST Special Publication 800-86: "After a backup or imaging is performed, it is important to verify that the copied data is an exact duplicate of the original data. Computing the message digest of the copied data can be used to verify and ensure data integrity."

Without hash verification, you cannot prove your forensic image is an authentic copy of the original evidence.

🔐 Purpose of Hashing in Forensics

Cryptographic Hash Function

A mathematical algorithm that takes input data of any size and produces a fixed-size output (the hash value or message digest). The hash serves as a unique "fingerprint" of the data—any change to the input, no matter how small, produces a completely different hash.

Primary Purposes in Digital Forensics

Purposes of hashing in forensics
Purpose Description Example Use
Integrity Verification Prove the image hasn't changed since acquisition Compare hash before and after analysis
Authentication Confirm the image matches the original evidence Source hash = Image hash proves exact copy
Chain of Custody Document evidence state at each transfer Record hash when evidence changes hands
Deduplication Identify identical files across evidence sources Find same file on multiple devices
Known File Filtering Match against databases of known files NSRL database for OS files, known malware

The Hash Guarantee

Cryptographic hash functions provide four essential guarantees:

✓ Hash Function Properties
  1. Deterministic: Same input always produces the same hash output
  2. Unique (Collision Resistant): Different inputs produce different outputs with extremely high probability
  3. One-Way: Cannot reverse-engineer input from hash output
  4. Avalanche Effect: Small change in input produces completely different hash

⚠️ Understanding "Broken" Hash Algorithms

When we say MD5 is "broken," we mean researchers have found ways to create two different inputs that produce the same hash (a collision). This is a security concern for digital signatures and certificates.

For forensic integrity verification, MD5 is still useful because:

  • An attacker would need to modify evidence AND produce the same hash
  • Random accidental collisions are still astronomically unlikely
  • Using MD5 alongside SHA-256 provides defense in depth

However, always use SHA-256 as your primary hash and MD5/SHA-1 as supplementary verification.

ℹ️ NIST Guidance

According to NIST SP 800-86: "Because SHA-1 is a Federal Information Processing Standards (FIPS) approved algorithm and MD5 is not, Federal agencies should use SHA-1 instead of MD5 for message digests."

NIST has since recommended transitioning from SHA-1 to SHA-256 for stronger security.

💡 Best Practice: Use Multiple Hashes

Calculate both MD5 and SHA-256 during acquisition:

  • SHA-256: Primary hash for integrity verification
  • MD5: Secondary hash for compatibility with older systems and databases (like NSRL)

Most forensic tools (FTK Imager, Guymager, dcfldd) can calculate multiple hashes simultaneously with minimal performance impact.

🔧 Calculating Hash Values

Hash values can be calculated using command-line tools, GUI applications, or built into the acquisition process itself.

Command-Line Hash Calculation

Terminal - Hash Calculation Commands
# Linux/macOS - Calculate MD5 hash of a file examiner@forensic-ws:~$ md5sum /evidence/case001/evidence.dd a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6 /evidence/case001/evidence.dd # Linux/macOS - Calculate SHA-1 hash examiner@forensic-ws:~$ sha1sum /evidence/case001/evidence.dd da39a3ee5e6b4b0d3255bfef95601890afd80709 /evidence/case001/evidence.dd # Linux/macOS - Calculate SHA-256 hash examiner@forensic-ws:~$ sha256sum /evidence/case001/evidence.dd e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855 /evidence/case001/evidence.dd # Hash a raw device (evidence drive through write blocker) examiner@forensic-ws:~$ sudo md5sum /dev/sdb a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6 /dev/sdb # Calculate multiple hashes at once (using parallel commands) examiner@forensic-ws:~$ md5sum /evidence/evidence.dd & sha256sum /evidence/evidence.dd a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6 /evidence/evidence.dd e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855 /evidence/evidence.dd

GUI Tool Hash Calculation

FTK Imager calculates hashes automatically during acquisition:

  1. During image creation, select hash algorithms (MD5, SHA-1, SHA-256)
  2. Enable "Verify images after they are created"
  3. Hashes are calculated during acquisition and stored in the image metadata (E01) or separate .txt file (Raw)
  4. Verification compares computed hash against stored hash

To hash an existing file:

  1. File → Add Evidence Item → Image File
  2. Right-click on the image in Evidence Tree
  3. Select "Verify Drive/Image"

Guymager calculates hashes during acquisition:

  1. In the acquisition dialog, check the hash algorithms to calculate
  2. Check "Verify image after acquisition"
  3. Hashes are recorded in the .info file alongside the image

The .info file contains:

MD5 hash computed:    a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6
MD5 hash verified:    a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6
SHA256 hash computed: e3b0c44298fc1c149afbf4c8996fb924...
SHA256 hash verified: e3b0c44298fc1c149afbf4c8996fb924...

These enhanced dd tools calculate hashes during imaging:

# dcfldd with multiple hashes
sudo dcfldd if=/dev/sdb of=/evidence/image.dd \
    hash=md5,sha256 \
    hashlog=/evidence/hashes.txt

# dc3dd with hash logging
sudo dc3dd if=/dev/sdb hof=/evidence/image.dd \
    hash=md5 hash=sha256 \
    hlog=/evidence/hashes.log

On-the-fly hashing is more efficient than calculating hashes separately because the data is read only once.

✅ The Verification Process

Hash verification occurs at multiple points during a forensic investigation to ensure evidence integrity is maintained throughout.

When to Verify Hashes

1

Before Acquisition

Calculate the hash of the source evidence (through write blocker) before creating the image. This establishes the baseline hash that all subsequent verifications will compare against.

sudo md5sum /dev/sdb > /evidence/source_hash.txt
sudo sha256sum /dev/sdb >> /evidence/source_hash.txt
2

After Acquisition

Calculate the hash of the forensic image immediately after creation. Compare to the source hash—they must match exactly.

md5sum /evidence/image.dd
# Compare to source hash - must be identical

If hashes match: Image is verified as an exact copy. Proceed with analysis.

If hashes don't match: STOP. Investigate the discrepancy before proceeding.

3

Before Analysis Sessions

Re-verify the image hash before each analysis session to confirm the working copy hasn't been modified since acquisition.

4

After Analysis Sessions

Verify the hash again after completing analysis to document that the evidence remained unchanged during your examination.

5

At Evidence Transfer

When transferring evidence to another examiner, agency, or for court, verify and document hashes at the point of transfer.

What To Do When Hashes Don't Match

⚠️ Hash Mismatch Protocol

If source and image hashes don't match, follow these steps:

  1. STOP - Do not proceed with analysis
  2. Investigate the cause:
    • Were there read errors during acquisition?
    • Did the acquisition complete successfully?
    • Is the storage media failing?
    • Was the source accidentally modified?
    • Is the destination file corrupted?
  3. Re-acquire if possible - Create a new image from the source
  4. Document thoroughly - Record the mismatch and investigation
  5. Consult supervisor/legal counsel - Determine how to proceed

Example: Investigating a Hash Mismatch

Scenario: After imaging a 500GB drive, the image hash doesn't match the source.

Investigation:

  1. Check acquisition log for errors → Found 3 read errors at sectors 1048576-1048578
  2. Verify conv=noerror,sync was used → Yes, errors were handled
  3. Compare file sizes → Image is 500GB, source is 500GB ✓
  4. Re-hash source drive → Same hash as before acquisition ✓
  5. Root cause: Read errors during acquisition resulted in zeros being written for bad sectors

Resolution: Document the read errors and their locations. The hash difference is explained by the bad sectors. Note this in the forensic report and proceed with analysis, acknowledging that data in those sectors may be incomplete.

💡 Working Copy Best Practice

Always work from a verified copy, not the original image:

  1. Create forensic image from source → verify hash
  2. Create working copy from forensic image → verify hash
  3. Perform analysis on working copy
  4. Keep original image untouched as "master copy"

This protects the original image from any accidental modification during analysis.

📝 Hash Documentation

Proper documentation of hash values is essential for court admissibility and chain of custody. Your documentation should clearly show that evidence integrity was maintained throughout the investigation.

Required Documentation Elements

  • Case identification: Case number, evidence item number
  • Source device details: Make, model, serial number, capacity
  • Hash algorithms used: MD5, SHA-1, SHA-256
  • Source hash values: Hashes calculated from original evidence
  • Image hash values: Hashes calculated from forensic image
  • Verification result: Match/mismatch status
  • Date and time: When hashes were calculated
  • Examiner information: Name, signature
  • Tool information: Software/hardware used for hashing

ℹ️ Court Testimony Tip

When testifying about hash verification, be prepared to explain:

  • What a hash function is (in layperson terms: "digital fingerprint")
  • Why matching hashes prove the image is an exact copy
  • The specific algorithms used and why they were chosen
  • When and how you calculated and verified the hashes
  • That the process is a standard forensic practice

📋 Module Summary

✅ Key Takeaways

Purpose of Hashing

  • Hash values serve as "digital fingerprints" for evidence integrity
  • Primary purposes: integrity verification, authentication, chain of custody
  • Same input always produces same hash; any change produces different hash

Hash Algorithms

  • SHA-256: Recommended primary hash (64 hex characters, FIPS approved)
  • SHA-1: Acceptable but being phased out (40 hex characters)
  • MD5: Legacy, use alongside stronger hash (32 hex characters)
  • Best practice: Calculate both MD5 and SHA-256

Verification Process

  • Calculate source hash before acquisition
  • Verify image hash matches source after acquisition
  • Re-verify before and after analysis sessions
  • If hashes don't match: STOP, investigate, document, re-acquire

Documentation

  • Record all hash values with timestamps
  • Document verification results (pass/fail)
  • Include examiner certification
  • Maintain hash log as part of case file

Remember

Matching hash values are your proof that the forensic image is an exact, unmodified copy of the original evidence. Without proper hash verification and documentation, the authenticity of your evidence can be challenged in court.
